## 理论讲解

### 1. 为什么需要 Patroni

原生 PostgreSQL 流复制提供了数据冗余，但缺少**自动故障转移**（Automatic Failover）能力。当主库宕机时，需要人工介入将从库提升为新主库，这意味着停机时间取决于 DBA 的响应速度。

Patroni 是一个开源的 PostgreSQL 高可用解决方案，它可以：
- 自动检测主库故障并触发 Failover
- 自动将最佳从库提升为新主库
- 自动重新配置其他从库跟随新主库
- 提供 REST API 方便集成负载均衡器

### 2. Patroni 架构

```
┌───────────────────────────────────────────────────┐
│                  DCS (etcd / ZK / Consul)          │
│              存储集群状态、Leader锁                  │
└───────────┬──────────────┬──────────────┬─────────┘
            │              │              │
     ┌──────▼──────┐ ┌────▼────────┐ ┌──▼──────────┐
     │  Patroni     │ │  Patroni     │ │  Patroni     │
     │  Agent       │ │  Agent       │ │  Agent       │
     │  ┌────────┐  │ │  ┌────────┐  │ │  ┌────────┐  │
     │  │ PG 主库 │  │ │  │ PG 从库 │  │ │  │ PG 从库 │  │
     │  └────────┘  │ │  └────────┘  │ │  └────────┘  │
     │  Node 1      │ │  Node 2      │ │  Node 3      │
     └─────────────┘ └─────────────┘ └─────────────┘
            │              │              │
     ┌──────▼──────────────▼──────────────▼─────────┐
     │           HAProxy / 负载均衡                   │
     │        读写分离 → 主库:5432 / 从库:5433        │
     └──────────────────────────────────────────────┘
```

### 3. 核心组件

- **Patroni Agent**：运行在每个 PG 节点上的守护进程，管理本地 PostgreSQL 实例
- **DCS（Distributed Configuration Store）**：分布式配置存储，用于 Leader 选举和集群状态管理
- **REST API**：每个 Patroni 节点暴露 HTTP API（默认端口 8008），用于健康检查
- **patronictl**：命令行管理工具

### 4. DCS 选择对比

| 特性 | etcd | ZooKeeper | Consul |
|------|------|-----------|--------|
| 语言 | Go | Java | Go |
| 协议 | Raft | ZAB | Raft |
| 部署复杂度 | 低 | 中 | 中 |
| 资源占用 | 低 | 高（JVM） | 低 |
| 社区活跃度 | 高 | 高 | 高 |
| 推荐场景 | 首选方案 | 已有 ZK 集群 | 已有 Consul 环境 |

**推荐**：如果没有特殊偏好，推荐使用 etcd，它是 Patroni 社区最常用的 DCS。

### 5. Failover 决策流程

1. Patroni Agent 定期向 DCS 更新 Leader 锁的 TTL
2. 如果 Leader 未能在 TTL 内续期，DCS 释放 Leader 锁
3. 所有健康的从库开始竞争 Leader 锁
4. 获得锁的节点检查自身是否有资格成为新 Leader（数据最新、配置允许等）
5. 新 Leader 执行 `pg_promote()` 提升为主库
6. 其他从库自动重新配置 `primary_conninfo` 指向新主库
7. 旧主库恢复后自动以从库身份加入集群

### 6. 关键概念

- **Leader**：当前的主库节点，持有 DCS 中的 Leader 锁
- **Replica**：从库节点，跟随 Leader 复制数据
- **TTL**：Leader 锁的存活时间，超时未续期则触发选举
- **loop_wait**：Patroni 主循环的间隔时间
- **retry_timeout**：DCS 操作的重试超时
- **maximum_lag_on_failover**：Failover 时允许的最大复制延迟

## 代码示例

### 示例1：安装 Patroni 和 etcd

```bash
# ===== 在所有节点上安装 Patroni =====

# 方法1：使用 pip 安装
pip3 install patroni[etcd]

# 方法2：使用系统包管理器（以 Ubuntu/Debian 为例）
sudo apt-get install patroni

# 验证安装
patroni --version
patronictl --version

# ===== 在 3 个节点上部署 etcd 集群 =====

# 安装 etcd
sudo apt-get install etcd

# 或下载二进制文件
ETCD_VER=v3.5.12
curl -L https://github.com/etcd-io/etcd/releases/download/${ETCD_VER}/etcd-${ETCD_VER}-linux-amd64.tar.gz \
  | tar xz --strip-components=1 -C /usr/local/bin/ etcd-${ETCD_VER}-linux-amd64/etcd etcd-${ETCD_VER}-linux-amd64/etcdctl
```

### 示例2：etcd 集群配置

```ini
# /etc/etcd/etcd.conf — 节点1 配置示例

# 节点名称
ETCD_NAME=etcd1

# 数据目录
ETCD_DATA_DIR=/var/lib/etcd/default

# 客户端通信地址
ETCD_LISTEN_CLIENT_URLS=http://0.0.0.0:2379
ETCD_ADVERTISE_CLIENT_URLS=http://192.168.1.101:2379

# 集群成员通信地址
ETCD_LISTEN_PEER_URLS=http://0.0.0.0:2380
ETCD_INITIAL_ADVERTISE_PEER_URLS=http://192.168.1.101:2380

# 集群初始成员
ETCD_INITIAL_CLUSTER=etcd1=http://192.168.1.101:2380,etcd2=http://192.168.1.102:2380,etcd3=http://192.168.1.103:2380
ETCD_INITIAL_CLUSTER_TOKEN=pg-ha-cluster
ETCD_INITIAL_CLUSTER_STATE=new
```

```bash
# 启动 etcd 并检查集群健康
sudo systemctl start etcd
etcdctl member list
etcdctl endpoint health
```

### 示例3：Patroni 完整配置文件

```yaml
# /etc/patroni/patroni.yml — 节点1 配置

scope: pg-cluster          # 集群名称
namespace: /service/       # DCS 中的命名空间
name: pg-node1             # 本节点名称

restapi:
  listen: 0.0.0.0:8008
  connect_address: 192.168.1.101:8008
  # authentication:
  #   username: patroni
  #   password: SecretPatroni

etcd3:
  hosts:
    - 192.168.1.101:2379
    - 192.168.1.102:2379
    - 192.168.1.103:2379

bootstrap:
  dcs:
    ttl: 30                    # Leader 锁 TTL（秒）
    loop_wait: 10              # 主循环间隔（秒）
    retry_timeout: 10          # DCS 重试超时
    maximum_lag_on_failover: 1048576  # Failover 最大允许延迟（1MB）
    master_start_timeout: 300  # 等待旧主库恢复的超时
    synchronous_mode: false    # 是否启用同步复制
    postgresql:
      use_pg_rewind: true      # 使用 pg_rewind 恢复旧主库
      use_slots: true           # 使用复制槽
      parameters:
        wal_level: replica
        hot_standby: 'on'
        max_connections: 200
        max_wal_senders: 10
        max_replication_slots: 10
        wal_keep_size: '1GB'
        max_worker_processes: 8
        max_parallel_workers_per_gather: 2
        shared_preload_libraries: 'pg_stat_statements'

  initdb:
    - encoding: UTF8
    - data-checksums
    - locale: en_US.UTF-8

  pg_hba:
    - host replication replicator 192.168.1.0/24 scram-sha-256
    - host all all 192.168.1.0/24 scram-sha-256
    - host all all 0.0.0.0/0 scram-sha-256

  users:
    admin:
      password: AdminP@ss2024
      options:
        - createrole
        - createdb
    replicator:
      password: ReplP@ss2024
      options:
        - replication

postgresql:
  listen: 0.0.0.0:5432
  connect_address: 192.168.1.101:5432
  data_dir: /var/lib/postgresql/16/main
  bin_dir: /usr/lib/postgresql/16/bin
  pgpass: /tmp/pgpass0
  authentication:
    superuser:
      username: postgres
      password: PostgresP@ss
    replication:
      username: replicator
      password: ReplP@ss2024
    rewind:
      username: rewind_user
      password: RewindP@ss
  parameters:
    unix_socket_directories: '/var/run/postgresql'
  pg_hba:
    - local all all trust
    - host replication replicator 127.0.0.1/32 scram-sha-256
    - host replication replicator 192.168.1.0/24 scram-sha-256
    - host all all 0.0.0.0/0 scram-sha-256

tags:
  nofailover: false       # 是否禁止此节点成为 Leader
  noloadbalance: false    # 是否从负载均衡中排除
  clonefrom: false        # 是否允许从此节点克隆
  nosync: false           # 是否禁止此节点成为同步从库
```

### 示例5：Patroni 运维操作

```bash
# ---- 手动 Switchover（计划内主从切换）----
patronictl -c /etc/patroni/patroni.yml switchover
# 交互式选择目标节点和执行时间

# 非交互式 Switchover
patronictl -c /etc/patroni/patroni.yml switchover \
  --master pg-node1 --candidate pg-node2 --force

# ---- 手动 Failover（强制切换）----
patronictl -c /etc/patroni/patroni.yml failover
patronictl -c /etc/patroni/patroni.yml failover --candidate pg-node2 --force

# ---- 重新初始化节点 ----
patronictl -c /etc/patroni/patroni.yml reinit pg-cluster pg-node3

# ---- 重启 PostgreSQL ----
patronictl -c /etc/patroni/patroni.yml restart pg-cluster pg-node1

# ---- 重新加载配置 ----
patronictl -c /etc/patroni/patroni.yml reload pg-cluster pg-node1

# ---- 修改动态配置 ----
patronictl -c /etc/patroni/patroni.yml edit-config

# ---- 暂停/恢复自动 Failover ----
patronictl -c /etc/patroni/patroni.yml pause   # 暂停（维护时使用）
patronictl -c /etc/patroni/patroni.yml resume   # 恢复
```

### 示例6：使用 REST API 进行健康检查

```bash
# 检查节点是否为 Leader（返回 200 = Leader）
curl -s http://192.168.1.101:8008/primary
curl -s http://192.168.1.101:8008/leader

# 检查节点是否为 Replica（返回 200 = Replica）
curl -s http://192.168.1.101:8008/replica

# 获取节点详细信息
curl -s http://192.168.1.101:8008/patroni | python3 -m json.tool

# 获取集群信息
curl -s http://192.168.1.101:8008/cluster | python3 -m json.tool

# HAProxy 可利用这些端点做健康检查
# /primary → 200 表示是主库，用于写流量
# /replica → 200 表示是从库，用于读流量
```

### 示例7：HAProxy 配合 Patroni 配置

```ini
# /etc/haproxy/haproxy.cfg

global
    maxconn 1000

defaults
    mode tcp
    timeout connect 5s
    timeout client 30m
    timeout server 30m

# 写入流量 — 只路由到 Primary
frontend pg_write
    bind *:5000
    default_backend pg_primary

backend pg_primary
    option httpchk GET /primary
    http-check expect status 200
    default-server inter 3s fall 3 rise 2 on-marked-down shutdown-sessions
    server pg-node1 192.168.1.101:5432 maxconn 200 check port 8008
    server pg-node2 192.168.1.102:5432 maxconn 200 check port 8008
    server pg-node3 192.168.1.103:5432 maxconn 200 check port 8008

# 读取流量 — 路由到所有 Replica
frontend pg_read
    bind *:5001
    default_backend pg_replicas

backend pg_replicas
    balance roundrobin
    option httpchk GET /replica
    http-check expect status 200
    default-server inter 3s fall 3 rise 2 on-marked-down shutdown-sessions
    server pg-node1 192.168.1.101:5432 maxconn 200 check port 8008
    server pg-node2 192.168.1.102:5432 maxconn 200 check port 8008
    server pg-node3 192.168.1.103:5432 maxconn 200 check port 8008

# 管理界面
listen stats
    bind *:7000
    mode http
    stats enable
    stats uri /
```

## 常见问题

### Q1: Patroni 和 PostgreSQL 的关系是什么？

**A:** Patroni 不是 PostgreSQL 的替代品，而是一个"管理层"。它运行在 PostgreSQL 之上，负责管理 PostgreSQL 实例的生命周期（启动、停止、提升、重建等）。PostgreSQL 本身的流复制机制不变，Patroni 只是在此基础上增加了自动 Failover 和集群管理能力。启用 Patroni 后应通过 `patronictl` 而非直接使用 `pg_ctl` 来管理 PostgreSQL。

### Q2: etcd 集群需要几个节点？

**A:** 生产环境推荐至少 3 个 etcd 节点（允许1个节点故障），或 5 个节点（允许2个节点故障）。etcd 使用 Raft 协议，需要多数节点存活才能正常工作。不要使用偶数个节点，因为这不会比少一个节点的奇数集群有更好的容错性。etcd 可以与 Patroni 运行在同一台机器上，也可以独立部署。

### Q3: Patroni switchover 和 failover 有什么区别？

**A:** **Switchover** 是计划内的主从切换，旧主库会被优雅地降级为从库，整个过程有序进行。**Failover** 是在主库不可用时的紧急切换，可能涉及数据一致性检查。日常维护（如主库打补丁、升级硬件）应使用 switchover；只有在主库真正故障时才需要 failover。
