## 理论讲解

### 1. CROSS JOIN（交叉连接）

`CROSS JOIN` 返回两个表的**笛卡尔积**——左表每一行与右表每一行组合。

```sql
SELECT a.列, b.列
FROM 表A a
CROSS JOIN 表B b;
```

如果表A有 M 行，表B有 N 行，结果集有 M × N 行。

**典型用途**：
- 生成所有组合（如颜色 × 尺码）
- 配合日期序列生成报表框架
- 需要对比所有可能配对时

### 2. 自连接（Self Join）

自连接是表与自身进行 JOIN，常用于处理**层级关系**或**同表比较**。必须使用不同的别名来区分同一张表的两个实例。

```sql
SELECT a.列, b.列
FROM 表 a
JOIN 表 b ON a.关联列 = b.关联列;
```

**典型用途**：
- 员工-经理层级关系
- 查找同一表中满足条件的记录对
- 树形结构的父子关系查询

### 3. NATURAL JOIN

`NATURAL JOIN` 自动基于两个表中**同名列**进行连接，无需手动指定 `ON` 条件：

```sql
SELECT * FROM 表A NATURAL JOIN 表B;
```

**风险**：如果表结构发生变化（新增同名列），查询语义可能悄悄改变。**不建议在生产代码中使用**，因为隐式行为会导致维护困难。

### 4. 组合使用不同类型的 JOIN

一条查询可以混合使用多种 JOIN 类型：

```sql
SELECT ...
FROM 表A a
INNER JOIN 表B b ON a.id = b.a_id
LEFT JOIN 表C c ON b.id = c.b_id
CROSS JOIN 表D d;
```

### 5. 实际应用场景总结

| 场景 | 适用 JOIN 类型 |
|------|---------------|
| 查询订单及客户信息 | INNER JOIN |
| 查询所有客户（含无订单） | LEFT JOIN |
| 生成日期×产品报表框架 | CROSS JOIN |
| 查询员工及其经理 | Self JOIN |
| 对比同类别产品价格 | Self JOIN |
| 合并两表所有信息 | FULL OUTER JOIN |

## 代码示例

### 准备测试数据

```sql
-- 颜色表
CREATE TABLE colors (
    id   SERIAL PRIMARY KEY,
    name VARCHAR(20) NOT NULL
);

-- 尺码表
CREATE TABLE sizes (
    id   SERIAL PRIMARY KEY,
    name VARCHAR(10) NOT NULL
);

-- 员工表（含上级关系）
CREATE TABLE employees (
    id          SERIAL PRIMARY KEY,
    name        VARCHAR(50) NOT NULL,
    department  VARCHAR(50),
    salary      NUMERIC(10,2),
    manager_id  INT REFERENCES employees(id)
);

-- 插入颜色和尺码
INSERT INTO colors (name) VALUES ('红色'), ('蓝色'), ('白色');
INSERT INTO sizes (name) VALUES ('S'), ('M'), ('L'), ('XL');

-- 插入员工数据（层级结构）
INSERT INTO employees (name, department, salary, manager_id) VALUES
('王总', '管理层', 50000.00, NULL),        -- id=1, 无上级
('张经理', '研发部', 30000.00, 1),          -- id=2, 上级是王总
('李经理', '市场部', 28000.00, 1),          -- id=3, 上级是王总
('赵工', '研发部', 18000.00, 2),            -- id=4, 上级是张经理
('钱工', '研发部', 16000.00, 2),            -- id=5, 上级是张经理
('孙销售', '市场部', 15000.00, 3),          -- id=6, 上级是李经理
('周销售', '市场部', 14000.00, 3);          -- id=7, 上级是李经理
```

### 示例 1：CROSS JOIN 生成所有组合

```sql
-- 生成所有颜色和尺码的组合（SKU 基础数据）
SELECT
    c.name AS color,
    s.name AS size,
    c.name || '-' || s.name AS sku_code
FROM colors c
CROSS JOIN sizes s
ORDER BY c.name, s.id;

-- 结果：3 种颜色 × 4 种尺码 = 12 行
```

### 示例 2：CROSS JOIN 生成日期报表框架

```sql
-- 生成最近 7 天 × 各部门的报表框架
SELECT
    d.date_val,
    dept.department
FROM generate_series(
    CURRENT_DATE - INTERVAL '6 days',
    CURRENT_DATE,
    INTERVAL '1 day'
) AS d(date_val)
CROSS JOIN (
    SELECT DISTINCT department FROM employees
) AS dept
ORDER BY d.date_val, dept.department;
```

### 示例 3：自连接——员工与经理

```sql
-- 查询每个员工及其直属经理
SELECT
    e.name AS employee,
    e.department,
    e.salary,
    m.name AS manager
FROM employees e
LEFT JOIN employees m ON e.manager_id = m.id
ORDER BY e.id;
-- 使用 LEFT JOIN 确保王总（无上级）也能显示
```

### 示例 4：自连接——查找同部门薪资更高的人

```sql
-- 查找同部门中比自己薪资高的同事
SELECT
    e1.name AS employee,
    e1.salary AS my_salary,
    e2.name AS higher_paid_colleague,
    e2.salary AS colleague_salary,
    e1.department
FROM employees e1
JOIN employees e2
    ON e1.department = e2.department
    AND e2.salary > e1.salary
ORDER BY e1.department, e1.name;
```

### 示例 5：自连接——查找管理链

```sql
-- 查询两级管理链：员工 → 经理 → 总经理
SELECT
    e.name AS employee,
    m.name AS manager,
    gm.name AS general_manager
FROM employees e
LEFT JOIN employees m ON e.manager_id = m.id
LEFT JOIN employees gm ON m.manager_id = gm.id
ORDER BY e.id;
```

### 示例 6：NATURAL JOIN

```sql
-- 创建两个有同名列的表
CREATE TABLE student (student_id INT, name VARCHAR(50), class_id INT);
CREATE TABLE class (class_id INT, class_name VARCHAR(50));

INSERT INTO student VALUES (1, '小明', 101), (2, '小红', 102), (3, '小刚', 101);
INSERT INTO class VALUES (101, '一班'), (102, '二班'), (103, '三班');

-- NATURAL JOIN 自动基于 class_id 连接
SELECT * FROM student NATURAL JOIN class;

-- 等价于：
SELECT s.student_id, s.name, s.class_id, c.class_name
FROM student s
JOIN class c ON s.class_id = c.class_id;
```

### 示例 7：混合多种 JOIN 类型

```sql
-- 综合查询：所有客户的订单信息及商品详情
-- 使用 Day 014 创建的表
SELECT
    c.name AS customer_name,
    c.city,
    o.order_date,
    p.name AS product_name,
    oi.quantity,
    oi.unit_price * oi.quantity AS subtotal
FROM customers c
LEFT JOIN orders o ON c.id = o.customer_id           -- LEFT: 包含无订单客户
INNER JOIN order_items oi ON o.id = oi.order_id       -- INNER: 只要有明细的订单
INNER JOIN products p ON oi.product_id = p.id         -- INNER: 关联商品信息
ORDER BY c.name, o.order_date;
```

### 示例 8：实际业务场景——销售对比

```sql
-- 查找同一部门中薪资差距最大的两个员工
SELECT
    e1.department,
    e1.name AS high_earner,
    e1.salary AS high_salary,
    e2.name AS low_earner,
    e2.salary AS low_salary,
    e1.salary - e2.salary AS gap
FROM employees e1
JOIN employees e2
    ON e1.department = e2.department
    AND e1.salary > e2.salary
ORDER BY gap DESC
LIMIT 1;
```

## 常见问题

### Q1: CROSS JOIN 和不带 ON 条件的 JOIN 一样吗？

**A**：是的，`FROM a CROSS JOIN b` 等价于 `FROM a, b`。两者都产生笛卡尔积。区别只是写法上的：CROSS JOIN 明确表达了意图，逗号分隔容易让人误以为忘了写 WHERE 条件。建议需要笛卡尔积时显式使用 `CROSS JOIN`。

### Q2: NATURAL JOIN 为什么不推荐使用？

**A**：因为 `NATURAL JOIN` 隐式地基于所有同名列连接，当表结构变更（如新增了一个与另一表同名的列）时，连接行为会悄悄改变，导致难以排查的 bug。在生产代码中应该始终显式指定 `ON` 或 `USING` 条件。

### Q3: 自连接的性能如何？

**A**：自连接本质上与普通 JOIN 相同，PostgreSQL 优化器会按照正常的 JOIN 策略来处理。确保关联列上有索引即可。对于深层级的树形结构查询（如组织架构），自连接写法会很繁琐，推荐使用 PostgreSQL 的**递归 CTE**（`WITH RECURSIVE`）来处理。
