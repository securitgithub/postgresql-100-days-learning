## 理论讲解

### 1. 什么是 PostGIS

PostGIS 是 PostgreSQL 的空间数据库扩展，为 PostgreSQL 增加了对地理空间对象的支持。它是目前最强大的开源空间数据库解决方案，实现了 OGC（Open Geospatial Consortium）的 Simple Features 标准。

PostGIS 提供：
- 空间数据类型（点、线、面、多边形等）
- 空间索引（基于 GiST 和 SP-GiST）
- 数百个空间分析函数
- 坐标参考系统（CRS）支持
- 栅格数据处理
- 拓扑支持
- 3D 和 4D 几何支持

### 2. geometry vs geography

PostGIS 提供两种核心空间数据类型：

| 特性 | geometry | geography |
|------|----------|-----------|
| 坐标系 | 平面笛卡尔坐标 | 球面经纬度坐标 |
| 计算方式 | 欧几里得平面几何 | 球面/椭球体计算 |
| 精确度 | 小范围精确 | 全球范围精确 |
| 性能 | 更快 | 较慢（三角函数） |
| 距离单位 | 取决于 SRID | 默认米（meters） |
| 函数支持 | 全部（数百个） | 部分（核心函数） |
| 适用场景 | 城市级/区域级 | 全球级应用 |

**选择建议**：
- 如果数据范围在一个城市或区域内，使用 `geometry`（配合合适的投影坐标系）
- 如果数据跨越大范围（跨国、全球），使用 `geography`
- 如果不确定，使用 `geography` 更安全（结果更准确）

### 3. 常用空间对象类型

| 类型 | WKT 表示 | 说明 |
|------|---------|------|
| POINT | `POINT(经度 纬度)` | 点（如 POI、用户位置） |
| LINESTRING | `LINESTRING(x1 y1, x2 y2, ...)` | 线（如道路、河流） |
| POLYGON | `POLYGON((x1 y1, x2 y2, ..., x1 y1))` | 面（如行政区、围栏） |
| MULTIPOINT | `MULTIPOINT(...)` | 多点 |
| MULTILINESTRING | `MULTILINESTRING(...)` | 多线 |
| MULTIPOLYGON | `MULTIPOLYGON(...)` | 多面 |
| GEOMETRYCOLLECTION | `GEOMETRYCOLLECTION(...)` | 几何集合 |

### 4. SRID（Spatial Reference System Identifier）

SRID 定义了坐标参考系，决定了坐标值的含义：

| SRID | 名称 | 说明 |
|------|------|------|
| 4326 | WGS 84 | GPS 使用的经纬度坐标系（最常用） |
| 3857 | Web Mercator | Web 地图使用的投影（Google Maps 等） |
| 4490 | CGCS2000 | 中国国家大地坐标系 |
| 32650 | UTM Zone 50N | UTM 投影（中国东部适用） |

### 5. GiST 空间索引

GiST（Generalized Search Tree）是 PostgreSQL 支持的通用搜索树索引，PostGIS 利用它实现空间索引。空间索引使用 R-Tree 算法，通过包围盒（Bounding Box）快速过滤不相关的数据。

```
查询：找出距离某点 1km 内的所有 POI

无索引：逐行计算距离 → O(n)
有索引：
  1. 用包围盒快速排除大部分数据 → 极快
  2. 对候选结果精确计算距离 → 少量计算
```

### 6. 核心空间函数分类

| 类别 | 函数示例 | 说明 |
|------|---------|------|
| 关系判断 | ST_Contains, ST_Within, ST_Intersects | 判断空间关系 |
| 距离计算 | ST_Distance, ST_DWithin | 计算距离 |
| 几何操作 | ST_Buffer, ST_Union, ST_Intersection | 几何运算 |
| 构造函数 | ST_MakePoint, ST_GeomFromText | 创建几何对象 |
| 访问函数 | ST_X, ST_Y, ST_Area, ST_Length | 获取属性 |
| 转换函数 | ST_Transform, ST_AsGeoJSON | 坐标/格式转换 |

### 示例2：创建空间数据表

```sql
-- 创建城市 POI（兴趣点）表
CREATE TABLE pois (
    id SERIAL PRIMARY KEY,
    name VARCHAR(100) NOT NULL,
    category VARCHAR(50),
    -- 使用 geography 类型存储经纬度坐标（SRID=4326 WGS84）
    location GEOGRAPHY(POINT, 4326),
    address TEXT,
    created_at TIMESTAMP DEFAULT now()
);

-- 创建行政区域表
CREATE TABLE districts (
    id SERIAL PRIMARY KEY,
    name VARCHAR(100) NOT NULL,
    level VARCHAR(20),  -- province, city, district
    -- 使用 geometry 类型存储多边形
    boundary GEOMETRY(MULTIPOLYGON, 4326),
    area_km2 NUMERIC(10,2)
);

-- 创建道路表
CREATE TABLE roads (
    id SERIAL PRIMARY KEY,
    name VARCHAR(100),
    road_type VARCHAR(20),  -- highway, primary, secondary
    -- 线型几何
    geom GEOMETRY(LINESTRING, 4326),
    length_m NUMERIC(10,2)
);

-- 创建用户位置表（实时位置追踪）
CREATE TABLE user_locations (
    id BIGSERIAL PRIMARY KEY,
    user_id INT NOT NULL,
    location GEOGRAPHY(POINT, 4326),
    accuracy FLOAT,  -- 精度（米）
    recorded_at TIMESTAMP DEFAULT now()
);

-- 创建 GiST 空间索引
CREATE INDEX idx_pois_location ON pois USING GIST (location);
CREATE INDEX idx_districts_boundary ON districts USING GIST (boundary);
CREATE INDEX idx_roads_geom ON roads USING GIST (geom);
CREATE INDEX idx_user_locations ON user_locations USING GIST (location);
```

### 示例4：空间距离查询

```sql
-- 计算两点之间的距离
-- ST_Distance 对 geography 类型返回值单位为米
SELECT ST_Distance(
    ST_SetSRID(ST_MakePoint(116.3912, 39.9075), 4326)::geography,  -- 天安门
    ST_SetSRID(ST_MakePoint(116.3972, 39.9163), 4326)::geography   -- 故宫
) AS distance_meters;
-- 约 1063 米

-- 查找距离天安门 3 公里内的所有 POI
SELECT
    name,
    category,
    ROUND(ST_Distance(
        location,
        ST_SetSRID(ST_MakePoint(116.3912, 39.9075), 4326)::geography
    )::numeric, 0) AS distance_m
FROM pois
WHERE ST_DWithin(
    location,
    ST_SetSRID(ST_MakePoint(116.3912, 39.9075), 4326)::geography,
    3000  -- 3000米 = 3公里
)
ORDER BY distance_m;

-- ST_DWithin 比 ST_Distance < X 更高效，因为它能利用空间索引
-- 推荐用 ST_DWithin 代替 WHERE ST_Distance(...) < 3000

-- 查找最近的 5 个 POI（KNN 查询）
-- 使用 <-> 操作符进行 KNN 排序（利用空间索引）
SELECT
    name,
    category,
    ROUND(ST_Distance(
        location,
        ST_SetSRID(ST_MakePoint(116.3912, 39.9075), 4326)::geography
    )::numeric, 0) AS distance_m
FROM pois
ORDER BY location <-> ST_SetSRID(ST_MakePoint(116.3912, 39.9075), 4326)::geography
LIMIT 5;
```

### 示例5：空间关系查询

```sql
-- 创建一个测试用的区域多边形（简化的海淀区范围）
INSERT INTO districts (name, level, boundary) VALUES
    ('海淀区', 'district',
     ST_GeomFromText('MULTIPOLYGON(((
       116.25 39.95,
       116.25 40.02,
       116.35 40.02,
       116.35 39.95,
       116.25 39.95
     )))', 4326));

INSERT INTO districts (name, level, boundary) VALUES
    ('东城区', 'district',
     ST_GeomFromText('MULTIPOLYGON(((
       116.37 39.87,
       116.37 39.93,
       116.43 39.93,
       116.43 39.87,
       116.37 39.87
     )))', 4326));

-- ST_Contains：判断区域是否包含某个点
-- 查找海淀区内的所有 POI
SELECT p.name, p.category
FROM pois p
JOIN districts d ON ST_Contains(d.boundary, p.location::geometry)
WHERE d.name = '海淀区';

-- ST_Within：判断点是否在某个区域内（与 ST_Contains 相反）
SELECT d.name AS district_name
FROM districts d
WHERE ST_Within(
    ST_SetSRID(ST_MakePoint(116.3067, 39.9929), 4326),  -- 北京大学的位置
    d.boundary
);

-- ST_Intersects：判断两个几何对象是否相交
-- 查找与给定矩形区域相交的所有区域
SELECT name
FROM districts
WHERE ST_Intersects(
    boundary,
    ST_MakeEnvelope(116.30, 39.90, 116.40, 40.00, 4326)
);

-- 组合查询：每个区域内各类别 POI 的数量
SELECT
    d.name AS district_name,
    p.category,
    COUNT(*) AS poi_count
FROM districts d
JOIN pois p ON ST_Contains(d.boundary, p.location::geometry)
GROUP BY d.name, p.category
ORDER BY d.name, poi_count DESC;
```

### 示例6：GiST 索引效果验证

```sql
-- 生成大量测试数据（10万个随机 POI）
INSERT INTO pois (name, category, location)
SELECT
    'POI_' || i,
    (ARRAY['餐饮','购物','酒店','景点','医院'])[1 + (random()*4)::int],
    ST_SetSRID(
        ST_MakePoint(
            116.0 + random() * 0.8,   -- 经度范围 116.0-116.8
            39.7 + random() * 0.5     -- 纬度范围 39.7-40.2
        ),
        4326
    )::geography
FROM generate_series(1, 100000) AS i;

-- 分析表以更新统计信息
ANALYZE pois;

-- 测试1：无索引的范围查询性能
DROP INDEX IF EXISTS idx_pois_location;

EXPLAIN ANALYZE
SELECT count(*)
FROM pois
WHERE ST_DWithin(
    location,
    ST_SetSRID(ST_MakePoint(116.4, 39.95), 4326)::geography,
    2000
);
-- 注意执行时间和扫描方式（Seq Scan）

-- 测试2：创建 GiST 索引后的性能
CREATE INDEX idx_pois_location ON pois USING GIST (location);

EXPLAIN ANALYZE
SELECT count(*)
FROM pois
WHERE ST_DWithin(
    location,
    ST_SetSRID(ST_MakePoint(116.4, 39.95), 4326)::geography,
    2000
);
-- 注意执行时间和扫描方式（Index Scan）
-- 通常能看到 10x-100x 的性能提升
```

### 示例7：实用查询集锦

```sql
-- 1. 输出 GeoJSON 格式（便于前端地图展示）
SELECT
    json_build_object(
        'type', 'Feature',
        'geometry', ST_AsGeoJSON(location::geometry)::json,
        'properties', json_build_object(
            'name', name,
            'category', category
        )
    ) AS geojson_feature
FROM pois
WHERE category = '景点'
LIMIT 5;

-- 2. 计算多边形面积（平方公里）
SELECT
    name,
    ROUND(ST_Area(boundary::geography) / 1000000, 2) AS area_km2
FROM districts;

-- 3. 缓冲区分析：找出距离道路 100 米内的 POI
-- （假设 roads 表中有数据）
-- SELECT p.name
-- FROM pois p
-- JOIN roads r ON ST_DWithin(p.location, r.geom::geography, 100)
-- WHERE r.name = '长安街';

-- 4. 聚合空间范围：计算所有景点的最小包围矩形
SELECT ST_AsText(ST_Envelope(ST_Collect(location::geometry)))
FROM pois
WHERE category = '景点';

-- 5. 坐标转换：WGS84 转 Web Mercator
SELECT
    name,
    ST_AsText(ST_Transform(location::geometry, 3857)) AS web_mercator_point
FROM pois
LIMIT 3;

-- 6. 地理编码结果验证：检查坐标是否在中国境内
-- （使用简化的中国边界）
SELECT name, location
FROM pois
WHERE NOT ST_DWithin(
    location,
    ST_SetSRID(ST_MakePoint(104.0, 35.0), 4326)::geography,  -- 中国大致中心
    3000000  -- 3000公里半径
);
```

## 常见问题

### Q1: geometry 和 geography 类型如何选择？

**A:** 关键看数据范围和精度需求。如果数据集中在一个城市范围内（如外卖、打车应用），使用 `geometry` 配合适当的投影坐标系（如 UTM），计算速度更快。如果数据跨越大范围（如国际物流、航班追踪），必须使用 `geography` 以确保距离计算准确。一个实用的折中方案是：存储使用 `geography`（存 WGS84 经纬度），查询时根据需要转换为本地 `geometry` 投影进行计算。

### Q2: 为什么空间查询很慢？

**A:** 最常见的原因是缺少空间索引。确认以下几点：(1) 是否创建了 GiST 索引？(2) 是否执行了 `ANALYZE` 更新统计信息？(3) 是否使用了索引友好的函数（如 `ST_DWithin` 而不是 `ST_Distance < X`）？(4) 使用 `EXPLAIN ANALYZE` 确认查询是否真正使用了索引。另外，`geography` 类型的计算天然比 `geometry` 慢，对于高频查询可以考虑转为 `geometry` 使用投影坐标系。

### Q3: 如何处理中国常用的 GCJ-02（火星坐标）和 BD-09（百度坐标）？

**A:** PostGIS 原生支持 WGS-84（SRID 4326），不直接支持 GCJ-02 和 BD-09。解决方案：(1) 在应用层进行坐标纠偏后再存入数据库（推荐）；(2) 在 PostgreSQL 中编写 PL/pgSQL 转换函数；(3) 使用第三方扩展。最佳实践是统一将所有坐标转换为 WGS-84 后存储，在前端展示时根据地图服务商要求做反向转换。
