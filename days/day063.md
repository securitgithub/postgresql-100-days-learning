# Day 063: 索引深入（一）B-tree 与 Hash

## 学习目标

- 深入理解 B-tree 索引的内部结构与适用场景
- 掌握多列索引的列顺序选择原则
- 理解部分索引（Partial Index）的概念与应用场景
- 了解 Hash 索引的原理、优缺点与使用限制
- 能够根据查询模式选择合适的索引类型

## 理论讲解

### 一、B-tree 索引

B-tree（平衡多路搜索树）是 PostgreSQL 的**默认索引类型**，也是使用最广泛的索引。当你执行 `CREATE INDEX` 而不指定类型时，创建的就是 B-tree 索引。

#### 1.1 B-tree 的内部结构

B-tree 索引是一种多层树状结构：

- **根节点（Root）**：树的顶层，只有一个
- **内部节点（Internal Nodes）**：存储键值和指向下层节点的指针
- **叶子节点（Leaf Nodes）**：存储键值和指向堆表行的 TID（Tuple ID）

叶子节点之间通过双向链表连接，支持高效的范围扫描。

关键特性：
- 树是**平衡的**，从根到每个叶子的路径长度相同
- 查找时间复杂度为 O(log N)
- 每个节点通常对应一个磁盘页（8KB）
- 节点分裂和合并自动维护平衡

#### 1.2 B-tree 支持的操作符

B-tree 索引支持以下比较操作符：

| 操作符 | 说明 |
|--------|------|
| `<` | 小于 |
| `<=` | 小于等于 |
| `=` | 等于 |
| `>=` | 大于等于 |
| `>` | 大于 |
| `BETWEEN` | 范围 |
| `IN` | 集合成员 |
| `IS NULL` | 空值判断 |
| `IS NOT NULL` | 非空判断 |

此外，B-tree 还支持 `ORDER BY` 排序优化、`MIN()`/`MAX()` 聚合优化。

#### 1.3 B-tree 的适用场景

- 等值查询：`WHERE id = 100`
- 范围查询：`WHERE created_at > '2024-01-01'`
- 排序：`ORDER BY name`
- 前缀匹配：`WHERE name LIKE 'John%'`（注意不支持 `LIKE '%John'`）
- `GROUP BY` 和 `DISTINCT`

### 二、多列索引（复合索引）

#### 2.1 列顺序的重要性

多列 B-tree 索引中，列的顺序至关重要。索引按照列的定义顺序进行排序，类似于电话簿（先按姓排序，姓相同时再按名排序）。

**最左前缀原则**：多列索引只有在查询条件包含索引的最左列（第一列）时才能被使用。

```sql
-- 索引定义
CREATE INDEX idx_orders_status_date ON orders(status, order_date);

-- ✅ 可以使用索引（包含第一列）
SELECT * FROM orders WHERE status = 'active';
SELECT * FROM orders WHERE status = 'active' AND order_date > '2024-01-01';

-- ❌ 无法使用索引（缺少第一列）
SELECT * FROM orders WHERE order_date > '2024-01-01';
```

#### 2.2 列顺序选择原则

1. **等值条件的列放前面**：等值条件能精确定位到索引树的特定分支
2. **选择性高的列放前面**：选择性 = 不同值的数量 / 总行数，越接近 1 选择性越高
3. **范围条件的列放后面**：范围条件之后的列无法被有效使用
4. **考虑排序需求**：如果查询经常需要按某列排序，将其包含在索引中

```sql
-- 好的设计：等值在前，范围在后
CREATE INDEX idx_orders_composite ON orders(status, customer_id, order_date);

-- 这个查询能完整利用三列索引
SELECT * FROM orders
WHERE status = 'active'
  AND customer_id = 42
  AND order_date > '2024-01-01';
```

### 三、部分索引（Partial Index）

部分索引只对表中满足特定条件的行建立索引，通过 `WHERE` 子句定义。

#### 3.1 优势

- 索引更小，占用更少的磁盘空间
- 索引维护成本更低（INSERT/UPDATE 只影响满足条件的行）
- 查询时扫描的索引页更少，更快

#### 3.2 典型场景

```sql
-- 场景1：只有少量活跃订单
-- 90% 的订单是已完成的，只有 10% 是活跃的
CREATE INDEX idx_orders_active ON orders(customer_id)
WHERE status = 'active';

-- 查询活跃订单时，索引只有全表 10% 的大小
SELECT * FROM orders WHERE status = 'active' AND customer_id = 42;

-- 场景2：排除空值
CREATE INDEX idx_users_email ON users(email)
WHERE email IS NOT NULL;

-- 场景3：软删除场景
CREATE INDEX idx_posts_visible ON posts(created_at)
WHERE deleted_at IS NULL;
```

#### 3.3 使用条件

查询的 WHERE 条件必须**匹配或隐含**部分索引的条件，规划器才会使用它。

### 四、Hash 索引

#### 4.1 Hash 索引的原理

Hash 索引对键值计算哈希值，将其映射到固定数量的桶（bucket）中。查找时只需计算一次哈希即可定位到桶。

#### 4.2 特点与限制

| 特性 | B-tree | Hash |
|------|--------|------|
| 等值查询 | ✅ | ✅ |
| 范围查询 | ✅ | ❌ |
| 排序 | ✅ | ❌ |
| 多列索引 | ✅ | ❌ |
| WAL 支持 | ✅ | ✅（PostgreSQL 10+） |
| 索引大小 | 较大 | 通常更小 |

#### 4.3 何时考虑 Hash 索引

在 PostgreSQL 10 之前，Hash 索引不支持 WAL，崩溃后需要重建，基本不推荐使用。从 PostgreSQL 10 开始，Hash 索引已支持 WAL，变得安全可靠。

适用场景：
- 只有等值查询（`=`）
- 键值较长（如 UUID、长文本），Hash 索引可能比 B-tree 更小
- 不需要排序或范围查询

```sql
CREATE INDEX idx_sessions_token ON sessions USING hash (token);
```

实际上，在大多数场景下 B-tree 都足够好，Hash 索引的优势仅限于非常特殊的场景。

## 代码示例

### 示例1：B-tree 索引基本操作

```sql
-- 创建测试表
CREATE TABLE employees (
    id SERIAL PRIMARY KEY,
    name TEXT NOT NULL,
    department TEXT,
    salary NUMERIC,
    hire_date DATE,
    email TEXT
);

-- 插入测试数据
INSERT INTO employees (name, department, salary, hire_date, email)
SELECT
    'Employee_' || i,
    (ARRAY['Engineering','Sales','Marketing','HR','Finance'])[1 + (i % 5)],
    30000 + (random() * 70000)::int,
    '2015-01-01'::date + (random() * 3650)::int,
    'emp_' || i || '@example.com'
FROM generate_series(1, 500000) AS i;

ANALYZE employees;

-- 创建 B-tree 索引
CREATE INDEX idx_emp_department ON employees(department);
CREATE INDEX idx_emp_salary ON employees(salary);
CREATE INDEX idx_emp_hire_date ON employees(hire_date);

-- 等值查询
EXPLAIN ANALYZE SELECT * FROM employees WHERE department = 'Engineering';

-- 范围查询
EXPLAIN ANALYZE SELECT * FROM employees WHERE salary BETWEEN 50000 AND 60000;

-- 排序优化（利用索引避免额外排序）
EXPLAIN ANALYZE SELECT * FROM employees ORDER BY hire_date LIMIT 20;

-- 前缀匹配
CREATE INDEX idx_emp_name ON employees(name text_pattern_ops);
EXPLAIN ANALYZE SELECT * FROM employees WHERE name LIKE 'Employee_1%';
```

### 示例2：多列索引列顺序

```sql
-- 创建多列索引
CREATE INDEX idx_emp_dept_salary ON employees(department, salary);

-- ✅ 使用了索引的两列
EXPLAIN ANALYZE
SELECT * FROM employees
WHERE department = 'Sales' AND salary > 50000;

-- ✅ 只使用了第一列
EXPLAIN ANALYZE
SELECT * FROM employees WHERE department = 'Sales';

-- ❌ 无法使用索引（跳过了第一列）
EXPLAIN ANALYZE
SELECT * FROM employees WHERE salary > 50000;

-- 查看索引的选择性
SELECT
    COUNT(DISTINCT department) AS dept_distinct,
    COUNT(DISTINCT salary) AS salary_distinct,
    COUNT(*) AS total_rows
FROM employees;
```

### 示例3：部分索引

```sql
-- 添加状态列
ALTER TABLE employees ADD COLUMN status TEXT DEFAULT 'active';
UPDATE employees SET status = 'inactive' WHERE random() < 0.9;
ANALYZE employees;

-- 全索引
CREATE INDEX idx_emp_status_full ON employees(department) WHERE TRUE;

-- 部分索引：只索引活跃员工
CREATE INDEX idx_emp_status_partial ON employees(department)
WHERE status = 'active';

-- 对比索引大小
SELECT indexname, pg_size_pretty(pg_relation_size(indexrelid)) AS size
FROM pg_stat_user_indexes
WHERE tablename = 'employees'
  AND indexname IN ('idx_emp_status_full', 'idx_emp_status_partial');

-- 查询活跃员工时使用部分索引
EXPLAIN ANALYZE
SELECT * FROM employees
WHERE status = 'active' AND department = 'Engineering';

-- 查询非活跃员工时不会使用部分索引
EXPLAIN ANALYZE
SELECT * FROM employees
WHERE status = 'inactive' AND department = 'Engineering';
```

### 示例4：Hash 索引对比

```sql
-- 创建一个包含长键值的表
CREATE TABLE api_tokens (
    id SERIAL PRIMARY KEY,
    token TEXT NOT NULL,
    user_id INT,
    created_at TIMESTAMP DEFAULT now()
);

INSERT INTO api_tokens (token, user_id)
SELECT md5(random()::text) || md5(random()::text), (random() * 10000)::int
FROM generate_series(1, 200000);

ANALYZE api_tokens;

-- B-tree 索引
CREATE INDEX idx_token_btree ON api_tokens USING btree (token);

-- Hash 索引
CREATE INDEX idx_token_hash ON api_tokens USING hash (token);

-- 对比索引大小
SELECT indexname, pg_size_pretty(pg_relation_size(indexrelid)) AS size
FROM pg_stat_user_indexes
WHERE tablename = 'api_tokens'
  AND indexname IN ('idx_token_btree', 'idx_token_hash');

-- 等值查询两者都能使用
-- 先用 B-tree
DROP INDEX idx_token_hash;
EXPLAIN ANALYZE SELECT * FROM api_tokens WHERE token = (SELECT token FROM api_tokens LIMIT 1);

-- 再用 Hash
DROP INDEX idx_token_btree;
CREATE INDEX idx_token_hash ON api_tokens USING hash (token);
EXPLAIN ANALYZE SELECT * FROM api_tokens WHERE token = (SELECT token FROM api_tokens LIMIT 1);

-- Hash 索引无法支持范围查询
-- 下面的查询不会使用 Hash 索引
EXPLAIN ANALYZE SELECT * FROM api_tokens WHERE token > 'abc';
```

### 示例5：索引内部结构查看

```sql
-- 安装 pageinspect 扩展（需要超级用户权限）
CREATE EXTENSION IF NOT EXISTS pageinspect;

-- 查看 B-tree 索引的元信息
SELECT * FROM bt_metap('idx_emp_department');
-- 可以看到 level（层数）、root（根页面）等信息

-- 查看 B-tree 索引某一页的统计
SELECT * FROM bt_page_stats('idx_emp_department', 1);

-- 查看索引页中的条目
SELECT * FROM bt_page_items('idx_emp_department', 1) LIMIT 10;
```

## 常见问题

### Q1：B-tree 索引什么时候会失效？

**A**：以下情况 B-tree 索引可能无法使用：
1. 使用函数或表达式转换了索引列：`WHERE LOWER(name) = 'john'`（需要创建表达式索引）
2. 使用了 `LIKE '%abc'` 后缀匹配或 `LIKE '%abc%'` 中间匹配
3. 数据类型不匹配导致隐式转换：`WHERE varchar_col = 123`
4. 使用了 `NOT IN`、`<>` 等否定条件（低选择性时规划器会放弃索引）
5. 表的统计信息过时，规划器误判

### Q2：多列索引和多个单列索引哪个好？

**A**：取决于查询模式：
- **多列索引**适合查询经常同时过滤多个列的场景，效率更高
- **多个单列索引**更灵活，PostgreSQL 可以通过 Bitmap Scan 将多个索引结果合并
- 如果查询总是同时使用 A 和 B 两列，`(A, B)` 的多列索引更好
- 如果有时只用 A，有时只用 B，创建两个单列索引更合适

### Q3：Hash 索引在生产环境中推荐使用吗？

**A**：从 PostgreSQL 10 开始，Hash 索引已经支持 WAL 并且安全可靠。但在实际场景中，B-tree 在等值查询上的性能已经非常好，Hash 索引的优势微乎其微。只有在键值非常长（如 256 字节以上的字符串）且只需要等值查询时，Hash 索引才可能在索引大小上有明显优势。大部分生产环境中，坚持使用 B-tree 是更安全的选择。
