## 理论讲解

### 1. 为什么需要 JOIN？

关系型数据库的核心思想是将数据分散存储在多个表中，避免冗余。查询时需要将这些表**关联**起来获得完整信息。`JOIN` 就是实现表关联的方式。

### 2. INNER JOIN（内连接）

`INNER JOIN` 返回两个表中**都有匹配**的行。不匹配的行被丢弃。

```sql
SELECT a.列, b.列
FROM 表A a
INNER JOIN 表B b ON a.关联列 = b.关联列;
```

`INNER` 关键字可以省略，`JOIN` 默认就是内连接。

**韦恩图理解**：返回两个圆的交集部分。

### 3. LEFT JOIN（左外连接）

`LEFT JOIN` 返回左表的**所有行**，右表没有匹配的行用 NULL 填充：

```sql
SELECT a.列, b.列
FROM 表A a
LEFT JOIN 表B b ON a.关联列 = b.关联列;
```

**韦恩图理解**：返回左圆的全部，加上交集部分中右圆的数据。

### 4. RIGHT JOIN（右外连接）

`RIGHT JOIN` 与 `LEFT JOIN` 相反，返回右表的所有行：

```sql
SELECT a.列, b.列
FROM 表A a
RIGHT JOIN 表B b ON a.关联列 = b.关联列;
```

实践中 `RIGHT JOIN` 较少使用，通常可以通过交换表的顺序改写为 `LEFT JOIN`。

### 5. FULL OUTER JOIN（全外连接）

`FULL OUTER JOIN` 返回两个表的所有行。没有匹配的一侧用 NULL 填充：

```sql
SELECT a.列, b.列
FROM 表A a
FULL OUTER JOIN 表B b ON a.关联列 = b.关联列;
```

**韦恩图理解**：返回两个圆的并集。

### 6. ON 与 USING 的区别

- **ON**：指定任意连接条件，灵活度高
- **USING**：当两个表的关联列**同名**时使用，更简洁

```sql
-- ON 写法
SELECT * FROM orders o JOIN customers c ON o.customer_id = c.customer_id;

-- USING 写法（列名相同时）
SELECT * FROM orders JOIN customers USING (customer_id);
```

`USING` 的结果集中，连接列只出现一次；`ON` 的结果集中两个表的连接列都会出现。

### 7. 多表连接

可以在一条查询中连接多个表，按从左到右的顺序依次连接：

```sql
SELECT o.id, c.name, p.name, oi.quantity
FROM orders o
JOIN customers c ON o.customer_id = c.id
JOIN order_items oi ON o.id = oi.order_id
JOIN products p ON oi.product_id = p.id;
```

### 示例 1：INNER JOIN

```sql
-- 查询有订单的客户及其订单信息
SELECT
    c.name AS customer_name,
    o.id AS order_id,
    o.order_date,
    o.total_amount
FROM customers c
INNER JOIN orders o ON c.id = o.customer_id
ORDER BY o.order_date;
```

### 示例 2：LEFT JOIN

```sql
-- 查询所有客户及其订单（包括没有订单的客户）
SELECT
    c.name AS customer_name,
    o.id AS order_id,
    o.order_date,
    COALESCE(o.total_amount, 0) AS total_amount
FROM customers c
LEFT JOIN orders o ON c.id = o.customer_id
ORDER BY c.name;
-- 钱七的 order_id、order_date 和 total_amount 会显示为 NULL/0
```

### 示例 3：找出没有订单的客户

```sql
-- LEFT JOIN + WHERE IS NULL 模式：查找无匹配的行
SELECT c.name, c.email
FROM customers c
LEFT JOIN orders o ON c.id = o.customer_id
WHERE o.id IS NULL;
-- 结果：钱七
```

### 示例 4：RIGHT JOIN

```sql
-- 等价于将 LEFT JOIN 的表顺序反转
SELECT
    c.name AS customer_name,
    o.id AS order_id,
    o.order_date
FROM orders o
RIGHT JOIN customers c ON o.customer_id = c.id
ORDER BY c.name;
```

### 示例 5：FULL OUTER JOIN

```sql
-- 创建一个员工-部门的示例
CREATE TABLE departments (id INT PRIMARY KEY, name VARCHAR(50));
CREATE TABLE employees (id INT PRIMARY KEY, name VARCHAR(50), dept_id INT);

INSERT INTO departments VALUES (1, '研发部'), (2, '市场部'), (3, '财务部');
INSERT INTO employees VALUES (1, '小明', 1), (2, '小红', 2), (3, '小刚', NULL);

-- FULL OUTER JOIN：显示所有员工和所有部门
SELECT
    e.name AS employee,
    d.name AS department
FROM employees e
FULL OUTER JOIN departments d ON e.dept_id = d.id;
-- 小刚没有部门（department = NULL）
-- 财务部没有员工（employee = NULL）
```

### 示例 6：USING 关键字

```sql
-- 如果两个表的关联列同名，使用 USING 更简洁
-- 先创建同名列的表
CREATE TABLE authors (author_id INT PRIMARY KEY, name VARCHAR(50));
CREATE TABLE books (id INT PRIMARY KEY, title VARCHAR(100), author_id INT);

INSERT INTO authors VALUES (1, '鲁迅'), (2, '莫言');
INSERT INTO books VALUES (1, '呐喊', 1), (2, '红高粱', 2), (3, '朝花夕拾', 1);

SELECT b.title, a.name
FROM books b
JOIN authors a USING (author_id);
```

### 示例 7：多表连接

```sql
-- 查询完整的订单详情：客户名、订单日期、商品名、数量、小计
SELECT
    c.name AS customer_name,
    o.order_date,
    p.name AS product_name,
    oi.quantity,
    oi.unit_price,
    oi.quantity * oi.unit_price AS subtotal
FROM orders o
JOIN customers c ON o.customer_id = c.id
JOIN order_items oi ON o.id = oi.order_id
JOIN products p ON oi.product_id = p.id
ORDER BY o.order_date, c.name;
```

## 常见问题

### Q1: JOIN 时如果忘记写 ON 条件会怎样？

**A**：如果使用 `INNER JOIN` 或 `LEFT JOIN` 时省略 `ON` 子句，PostgreSQL 会报语法错误。但如果使用逗号分隔表名（隐式连接）且忘记 `WHERE` 条件，将产生笛卡尔积，结果行数 = 表A行数 x 表B行数，这通常不是你想要的结果。

### Q2: LEFT JOIN 和子查询 NOT IN 哪个性能更好？

**A**：在 PostgreSQL 中，`LEFT JOIN ... WHERE IS NULL`、`NOT IN` 和 `NOT EXISTS` 通常由优化器生成相似的执行计划。但当子查询结果可能包含 NULL 时，`NOT IN` 的语义会不同（任何值与 NULL 比较结果都是未知），可能返回空结果。推荐使用 `LEFT JOIN ... IS NULL` 或 `NOT EXISTS`，语义更明确。

### Q3: 多表 JOIN 的顺序重要吗？

**A**：从语义上说，`INNER JOIN` 的顺序不影响结果。PostgreSQL 的查询优化器会自动选择最优的连接顺序。但对于 `LEFT JOIN`，表的顺序会影响结果（左表和右表的角色不同）。在写多表查询时，按业务逻辑顺序书写（如从主表开始逐步关联）可以提高可读性。
