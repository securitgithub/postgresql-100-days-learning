# Day 075: 并发控制与锁 — 深入理解 PostgreSQL 锁机制

## 学习目标

- 理解 PostgreSQL 的行锁和表锁机制及其层次关系
- 掌握 `SELECT FOR UPDATE` 和 `SELECT FOR SHARE` 的使用场景
- 学会使用 `SKIP LOCKED` 和 `NOWAIT` 处理锁竞争
- 了解 Advisory Locks（咨询锁）的概念和应用场景
- 能够诊断和解决锁相关的性能问题

## 理论讲解

### 1. PostgreSQL 锁的层次

PostgreSQL 的锁机制分为多个层次：

**表级锁（Table-level Locks）**：共 8 种模式，从最弱到最强：
1. `ACCESS SHARE`：SELECT 语句获取，与几乎所有锁兼容
2. `ROW SHARE`：SELECT FOR UPDATE/SHARE 获取
3. `ROW EXCLUSIVE`：INSERT/UPDATE/DELETE 获取
4. `SHARE UPDATE EXCLUSIVE`：VACUUM、ANALYZE、CREATE INDEX CONCURRENTLY 获取
5. `SHARE`：CREATE INDEX（非 CONCURRENTLY）获取
6. `SHARE ROW EXCLUSIVE`：CREATE TRIGGER 等获取
7. `EXCLUSIVE`：REFRESH MATERIALIZED VIEW CONCURRENTLY 获取
8. `ACCESS EXCLUSIVE`：ALTER TABLE、DROP TABLE、VACUUM FULL 获取

**行级锁（Row-level Locks）**：4 种模式：
1. `FOR KEY SHARE`：最弱，不阻塞除 FOR UPDATE 外的其他锁
2. `FOR SHARE`：阻塞 UPDATE 和 DELETE
3. `FOR NO KEY UPDATE`：普通 UPDATE 使用（不修改主键/唯一键）
4. `FOR UPDATE`：最强，阻塞所有其他行锁

行锁不会阻塞读取（SELECT），这是 MVCC 的核心优势——读写不互相阻塞。

### 2. 显式行锁

`SELECT ... FOR UPDATE`：锁定选中的行，防止其他事务修改或以 FOR UPDATE 锁定这些行。常用于"先读后写"模式。

`SELECT ... FOR SHARE`：允许多个事务同时持有，但防止修改。适用于需要确保数据不被修改但不需要独占的场景。

### 3. NOWAIT 和 SKIP LOCKED

- `NOWAIT`：如果无法立即获取锁，立即报错而不是等待。适用于不能容忍延迟的场景。
- `SKIP LOCKED`：跳过已被锁定的行，只返回和锁定未被锁定的行。非常适合实现任务队列。

### 4. Advisory Locks（咨询锁）

Advisory Locks 是应用层面的锁，PostgreSQL 不强制执行其语义——完全由应用程序决定锁的含义。

特点：
- 基于一个 bigint 值或两个 int 值来标识
- 分为会话级（session-level）和事务级（transaction-level）
- 不会自动获取或释放，完全由应用控制
- 性能开销极低

## 代码示例

### 示例1：表锁的演示与查看

```sql
-- 手动获取表锁
BEGIN;
LOCK TABLE accounts IN SHARE MODE;
-- 此时其他事务不能修改 accounts，但可以读取
COMMIT;

-- 更强的锁
BEGIN;
LOCK TABLE accounts IN ACCESS EXCLUSIVE MODE;
-- 此时其他事务连 SELECT 都会被阻塞
COMMIT;

-- 查看当前系统中的锁
SELECT
    l.locktype,
    l.relation::regclass AS table_name,
    l.mode,
    l.granted,
    l.pid,
    a.usename,
    a.query,
    a.state
FROM pg_locks l
JOIN pg_stat_activity a ON l.pid = a.pid
WHERE l.relation IS NOT NULL
ORDER BY l.relation, l.mode;

-- 查看锁等待情况
SELECT
    blocked.pid AS blocked_pid,
    blocked.query AS blocked_query,
    blocking.pid AS blocking_pid,
    blocking.query AS blocking_query,
    blocked.wait_event_type,
    blocked.wait_event
FROM pg_stat_activity blocked
JOIN pg_locks bl ON bl.pid = blocked.pid AND NOT bl.granted
JOIN pg_locks kl ON kl.relation = bl.relation AND kl.granted
JOIN pg_stat_activity blocking ON kl.pid = blocking.pid
WHERE blocked.pid != blocking.pid;
```

### 示例2：SELECT FOR UPDATE / FOR SHARE

```sql
-- 场景：转账操作，需要先读余额再扣款
-- 错误做法（可能导致丢失更新）：
BEGIN;
SELECT balance FROM accounts WHERE id = 1;  -- 读到 10000
-- 另一个事务此时也读到 10000 并扣款
UPDATE accounts SET balance = balance - 1000 WHERE id = 1;
COMMIT;

-- 正确做法：使用 FOR UPDATE 锁定
BEGIN;
SELECT balance FROM accounts WHERE id = 1 FOR UPDATE;
-- 其他事务的 FOR UPDATE 或 UPDATE 会等待
-- 确认余额足够后扣款
UPDATE accounts SET balance = balance - 1000 WHERE id = 1;
COMMIT;

-- FOR SHARE 示例：验证外键引用
BEGIN;
-- 确保引用的用户在事务期间不被删除
SELECT id FROM users WHERE id = 100 FOR SHARE;
-- 现在安全地插入关联数据
INSERT INTO user_orders (user_id, amount) VALUES (100, 599);
COMMIT;

-- FOR UPDATE 的子查询用法
UPDATE accounts
SET balance = balance - 500
WHERE id = (
    SELECT id FROM accounts
    WHERE balance >= 500
    ORDER BY id
    LIMIT 1
    FOR UPDATE
);

-- FOR KEY SHARE —— 只锁定键，允许非键列更新
BEGIN;
SELECT * FROM accounts WHERE id = 1 FOR KEY SHARE;
-- 其他事务可以 UPDATE balance（非键列）
-- 但不能 DELETE 或 UPDATE id（键列）
COMMIT;
```

### 示例3：NOWAIT 避免长时间等待

```sql
-- 尝试锁定，如果被占用立即失败
BEGIN;
SELECT * FROM accounts WHERE id = 1 FOR UPDATE NOWAIT;
-- 如果行已被其他事务锁定：
-- ERROR: could not obtain lock on row in relation "accounts"
COMMIT;

-- 在应用中使用 NOWAIT 的模式
DO $$
BEGIN
    BEGIN
        -- 尝试获取锁
        PERFORM * FROM accounts WHERE id = 1 FOR UPDATE NOWAIT;
        -- 获取成功，执行操作
        UPDATE accounts SET balance = balance - 100 WHERE id = 1;
    EXCEPTION
        WHEN lock_not_available THEN
            RAISE NOTICE '账户正在被操作，请稍后重试';
    END;
END;
$$;
```

### 示例4：SKIP LOCKED 实现任务队列

```sql
-- 创建任务队列表
CREATE TABLE task_queue (
    id SERIAL PRIMARY KEY,
    task_type TEXT NOT NULL,
    payload JSONB,
    status TEXT DEFAULT 'pending'
        CHECK (status IN ('pending', 'processing', 'completed', 'failed')),
    created_at TIMESTAMP DEFAULT now(),
    started_at TIMESTAMP,
    completed_at TIMESTAMP,
    worker_id TEXT
);

-- 插入一些任务
INSERT INTO task_queue (task_type, payload) VALUES
('send_email', '{"to": "user@example.com", "subject": "Welcome"}'),
('process_image', '{"url": "/uploads/photo.jpg", "size": "thumb"}'),
('generate_report', '{"report_id": 42, "format": "pdf"}'),
('send_email', '{"to": "admin@example.com", "subject": "Alert"}'),
('sync_data', '{"source": "api", "target": "warehouse"}');

-- Worker 获取任务（使用 SKIP LOCKED）
-- 多个 Worker 可以并发执行此查询，不会获取同一任务
BEGIN;
WITH next_task AS (
    SELECT id
    FROM task_queue
    WHERE status = 'pending'
    ORDER BY created_at
    LIMIT 1
    FOR UPDATE SKIP LOCKED
)
UPDATE task_queue
SET status = 'processing',
    started_at = now(),
    worker_id = 'worker-1'
FROM next_task
WHERE task_queue.id = next_task.id
RETURNING task_queue.*;

-- Worker 处理完任务后更新状态
UPDATE task_queue
SET status = 'completed',
    completed_at = now()
WHERE id = 1 AND worker_id = 'worker-1';
COMMIT;

-- 批量获取任务
BEGIN;
WITH batch AS (
    SELECT id
    FROM task_queue
    WHERE status = 'pending'
    ORDER BY created_at
    LIMIT 5
    FOR UPDATE SKIP LOCKED
)
UPDATE task_queue
SET status = 'processing',
    started_at = now(),
    worker_id = 'worker-2'
FROM batch
WHERE task_queue.id = batch.id
RETURNING task_queue.*;
COMMIT;
```

### 示例5：Advisory Locks 咨询锁

```sql
-- 会话级咨询锁
-- 获取锁（阻塞直到获取成功）
SELECT pg_advisory_lock(12345);

-- 尝试获取锁（非阻塞，返回 boolean）
SELECT pg_try_advisory_lock(12345);
-- 返回 true 表示获取成功，false 表示已被其他会话持有

-- 释放锁
SELECT pg_advisory_unlock(12345);

-- 事务级咨询锁（事务结束自动释放）
BEGIN;
SELECT pg_advisory_xact_lock(12345);
-- ... 事务操作 ...
COMMIT;  -- 锁自动释放

-- 使用两个整数标识锁（适合按资源类型+资源ID锁定）
-- 第一个参数可以表示资源类型，第二个表示资源ID
SELECT pg_advisory_lock(1, 100);  -- 类型1（用户），ID 100
SELECT pg_advisory_lock(2, 200);  -- 类型2（订单），ID 200

-- 共享模式的咨询锁
SELECT pg_advisory_lock_shared(12345);  -- 多个会话可同时持有
SELECT pg_advisory_unlock_shared(12345);

-- 查看当前的咨询锁
SELECT
    l.locktype,
    l.classid,   -- 第一个 int（或 bigint 的高32位）
    l.objid,     -- 第二个 int（或 bigint 的低32位）
    l.mode,
    l.granted,
    a.pid,
    a.usename,
    a.query
FROM pg_locks l
JOIN pg_stat_activity a ON l.pid = a.pid
WHERE l.locktype = 'advisory';
```

### 示例6：Advisory Lock 实战 — 防止重复执行

```sql
-- 场景：确保某个定时任务只有一个实例在运行
CREATE OR REPLACE FUNCTION run_daily_aggregation()
RETURNS TEXT AS $$
DECLARE
    lock_acquired BOOLEAN;
BEGIN
    -- 尝试获取咨询锁（非阻塞）
    lock_acquired := pg_try_advisory_lock(hashtext('daily_aggregation'));

    IF NOT lock_acquired THEN
        RETURN '另一个实例正在运行，跳过';
    END IF;

    -- 执行聚合逻辑
    RAISE NOTICE '开始执行每日聚合...';

    -- 模拟耗时操作
    PERFORM pg_sleep(5);

    -- 释放锁
    PERFORM pg_advisory_unlock(hashtext('daily_aggregation'));

    RETURN '聚合完成';
END;
$$ LANGUAGE plpgsql;

-- 场景：基于行 ID 的细粒度锁
-- 避免对同一用户进行并发操作
CREATE OR REPLACE FUNCTION update_user_profile(
    p_user_id INTEGER,
    p_data JSONB
) RETURNS VOID AS $$
BEGIN
    -- 使用用户 ID 作为咨询锁
    PERFORM pg_advisory_xact_lock(hashtext('user_profile'), p_user_id);

    -- 执行更新操作（这里不需要 FOR UPDATE 了）
    UPDATE users SET profile = p_data WHERE id = p_user_id;
END;
$$ LANGUAGE plpgsql;
```

### 示例7：死锁诊断与预防

```sql
-- 设置死锁检测超时
SET deadlock_timeout = '1s';  -- 默认 1s

-- 模拟死锁场景
-- 会话 A:
BEGIN;
UPDATE accounts SET balance = balance - 100 WHERE id = 1;
-- 会话 B:
BEGIN;
UPDATE accounts SET balance = balance - 100 WHERE id = 2;
-- 会话 A:
UPDATE accounts SET balance = balance + 100 WHERE id = 2;  -- 等待 B
-- 会话 B:
UPDATE accounts SET balance = balance + 100 WHERE id = 1;  -- 死锁！

-- PostgreSQL 会检测到死锁，终止其中一个事务：
-- ERROR: deadlock detected

-- 预防死锁：总是按固定顺序更新
-- 例如按 id 升序锁定
BEGIN;
SELECT * FROM accounts WHERE id IN (1, 2) ORDER BY id FOR UPDATE;
-- 执行更新...
COMMIT;

-- 查看死锁日志
-- 在 postgresql.conf 中设置：
-- log_lock_waits = on
-- deadlock_timeout = 1s

-- 监控长时间持有的锁
SELECT
    pid,
    now() - pg_stat_activity.query_start AS duration,
    query,
    state
FROM pg_stat_activity
WHERE (now() - pg_stat_activity.query_start) > interval '5 minutes'
  AND state != 'idle'
ORDER BY duration DESC;
```

## 动手练习

1. **锁兼容性测试**：打开三个 psql 会话，测试不同锁模式的兼容性。例如，一个会话持有 `FOR UPDATE`，另一个尝试 `FOR SHARE`，观察阻塞行为。

2. **任务队列实现**：基于示例 4 的任务队列，用多个 psql 窗口模拟多个 Worker 并发获取任务，验证 `SKIP LOCKED` 确保每个任务只被一个 Worker 获取。

3. **Advisory Lock 实践**：实现一个分布式互斥锁，确保同一时间只有一个进程可以执行"每日报表生成"任务。

4. **死锁模拟与解决**：故意制造一个死锁场景，观察 PostgreSQL 的死锁检测和错误信息。然后修改代码使用固定的锁顺序来预防死锁。

5. **锁监控仪表盘**：编写一个查询，展示当前系统中所有的锁信息，包括持有者、等待者、锁类型、持续时间。

## 常见问题

### Q1: SELECT FOR UPDATE 和 Serializable 隔离级别应该怎么选？

**A:** 这两种方式解决的问题有所不同：
- `SELECT FOR UPDATE` 是悲观锁策略，显式锁定需要操作的行。适合明确知道要锁哪些行的场景，如单行更新、转账操作。性能开销主要在锁等待上。
- Serializable 是乐观锁策略，在提交时检查冲突。适合复杂的读写依赖场景，如写偏斜问题。性能开销主要在序列化失败重试上。
一般建议：简单的"读-改-写"场景用 `FOR UPDATE`；复杂的多行读写依赖用 Serializable。

### Q2: SKIP LOCKED 用于任务队列比消息队列（如 RabbitMQ）有什么优劣？

**A:** 优势：无需额外基础设施；任务与业务数据在同一事务中，保证一致性；适合中等规模的任务处理。劣势：不适合超高吞吐量场景（数万 TPS 以上）；缺乏消息队列的延迟投递、死信队列等高级功能；随着队列表增长需要维护（清理已完成任务）。一般来说，日任务量在百万级以下、对一致性要求高的场景，用 PostgreSQL 任务队列是很好的选择。

### Q3: 如何避免长事务导致的锁问题？

**A:** 长事务持有锁会阻塞其他事务，导致连锁等待。预防措施：
1. 设置 `idle_in_transaction_session_timeout` 自动终止空闲事务。
2. 设置 `lock_timeout` 限制等待锁的时间。
3. 设置 `statement_timeout` 限制单条语句执行时间。
4. 使用连接池（如 PgBouncer）自动管理连接。
5. 监控 `pg_stat_activity` 中长时间运行的事务，及时发现问题。
