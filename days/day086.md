## 理论讲解

### 1. 为什么需要读写分离

在典型的 Web 应用中，读操作远多于写操作（通常 80:20 甚至 95:5）。单一主库承载所有查询会成为瓶颈。读写分离通过将读请求分散到多个从库，有效降低主库压力并提升系统整体吞吐量。

```
                    ┌──────────┐
                    │  应用层   │
                    └────┬─────┘
                         │
              ┌──────────┴──────────┐
              │  读写分离路由层       │
              │  (中间件/应用逻辑)   │
              └──┬──────────────┬───┘
                 │              │
           写请求 │         读请求 │
                 ▼              ▼
           ┌─────────┐   ┌─────────┐
           │  主库     │   │ 从库集群 │
           │ (Primary) │   │ (多个    │
           │          │   │  Replica)│
           └─────────┘   └─────────┘
```

### 2. 读写分离的三种实现层次

#### 层次一：应用层读写分离

在应用代码中显式控制连接哪个数据库：
- 写操作 → 主库连接
- 读操作 → 从库连接

**优点**：最灵活、无额外组件、可精确控制
**缺点**：侵入业务代码、维护成本高

#### 层次二：中间件代理

部署中间件（PgPool-II 等）自动解析 SQL 并路由：
- SELECT → 从库
- INSERT/UPDATE/DELETE → 主库

**优点**：对应用透明、集中管理
**缺点**：中间件可能成为瓶颈和单点、SQL解析不完美

#### 层次三：驱动层/框架层

数据库驱动或 ORM 框架内建读写分离支持：
- JDBC 多主机 URL
- Django 数据库路由器
- Rails 多数据库支持

**优点**：对业务代码影响较小、无额外运维组件
**缺点**：受限于框架能力

### 3. 数据一致性问题

读写分离最大的挑战是**复制延迟**导致的数据不一致：

```
时间线 →
主库: INSERT user_id=100  ──────────────────────▶
从库:                     ... 延迟 ...  INSERT user_id=100
                               ↑
                          此时读从库找不到 user_id=100
```

解决策略：
- **强制读主**：关键查询（如刚写入后的读取）走主库
- **延迟感知路由**：监控从库延迟，超过阈值时将读请求回退到主库
- **会话粘滞**：同一会话在写操作后的一段时间内全部读主库
- **因果一致性**：记录写操作的 LSN，读请求等待从库追上该 LSN

### 4. 负载均衡策略

| 策略 | 说明 | 适用场景 |
|------|------|----------|
| 轮询（Round Robin） | 依次分配到各从库 | 从库性能一致 |
| 加权轮询 | 按权重比例分配 | 从库性能不一致 |
| 最少连接 | 分配到连接最少的从库 | 查询耗时差异大 |
| 延迟优先 | 优先选择延迟最低的从库 | 对一致性有要求 |
| 随机 | 随机选择从库 | 简单场景 |

### 示例4：PgPool-II 读写分离配置

```ini
# /etc/pgpool2/pgpool.conf — 读写分离核心配置

# 启用主从模式
master_slave_mode = on
master_slave_sub_mode = 'stream'

# 启用负载均衡
load_balance_mode = on

# 后端节点配置
backend_hostname0 = '192.168.1.100'
backend_port0 = 5432
backend_weight0 = 0            # 主库权重设为0，不参与读负载均衡
backend_flag0 = 'ALLOW_TO_FAILOVER'

backend_hostname1 = '192.168.1.101'
backend_port1 = 5432
backend_weight1 = 1            # 从库1 权重
backend_flag1 = 'ALLOW_TO_FAILOVER'

backend_hostname2 = '192.168.1.102'
backend_port2 = 5432
backend_weight2 = 1            # 从库2 权重
backend_flag2 = 'ALLOW_TO_FAILOVER'

# 写操作以及以下函数必须发送到主库
black_function_list = 'currval,lastval,nextval,setval'

# 在显式事务中，所有查询发送到主库
# （确保事务一致性）
allow_sql_comments = on

# 延迟检查：超过阈值的从库不参与负载均衡
delay_threshold = 10000000     # 10MB，超过则不路由到该从库

# 优先使用延迟最低的从库
prefer_lower_delay_standby = on

# 日志中记录路由决策（调试用）
log_per_node_statement = on
```

### 示例6：基于复制延迟的智能路由

```sql
-- 创建监控复制延迟的函数（在主库执行）
CREATE OR REPLACE FUNCTION get_replica_lag_info()
RETURNS TABLE (
    replica_name TEXT,
    replica_addr INET,
    lag_bytes BIGINT,
    lag_pretty TEXT,
    lag_time INTERVAL,
    is_healthy BOOLEAN
) AS $$
BEGIN
    RETURN QUERY
    SELECT
        r.application_name::TEXT,
        r.client_addr,
        pg_wal_lsn_diff(pg_current_wal_lsn(), r.replay_lsn)::BIGINT,
        pg_size_pretty(pg_wal_lsn_diff(pg_current_wal_lsn(), r.replay_lsn)),
        r.replay_lag,
        -- 延迟小于 5MB 且时间延迟小于 5 秒认为健康
        (pg_wal_lsn_diff(pg_current_wal_lsn(), r.replay_lsn) < 5242880
         AND COALESCE(r.replay_lag, '0'::interval) < '5 seconds'::interval)
    FROM pg_stat_replication r
    WHERE r.state = 'streaming';
END;
$$ LANGUAGE plpgsql;

-- 使用示例
SELECT * FROM get_replica_lag_info();

-- 结果示例：
--  replica_name | replica_addr  | lag_bytes | lag_pretty | lag_time | is_healthy
-- --------------+---------------+-----------+------------+----------+------------
--  standby1     | 192.168.1.101 |      1024 | 1024 bytes | 00:00:00 | t
--  standby2     | 192.168.1.102 |   8388608 | 8192 kB    | 00:00:03 | f
```

## 常见问题

### Q1: 写后读（Read-Your-Writes）一致性如何保证？

**A:** 几种实现方式：(1) 在写操作后的固定时间窗口（如2秒）内，将读请求路由到主库；(2) 写操作后返回当前 LSN，后续读请求携带 LSN 值，只路由到已追上该 LSN 的从库；(3) 使用 `synchronous_commit = remote_apply` 确保同步从库已回放完成。方案(2)是性能和一致性的最佳平衡。

### Q2: 从库挂了会影响读写分离吗？

**A:** 如果使用中间件（PgPool-II、HAProxy），它们会自动检测从库健康状态，将故障从库从负载均衡池中移除。如果所有从库都不可用，通常会将读请求回退到主库（fallback）。应用层实现需要自行处理连接失败和重试逻辑。建议始终保留至少2个从库以确保读服务的可用性。

### Q3: 读写分离能提升多少性能？

**A:** 取决于读写比例和从库数量。假设读写比为 80:20，2个从库各承担40%的读负载：主库负载降低约60%（只处理写和20%的读），每个从库处理40%的读。实际效果还受限于网络延迟、从库规格、查询复杂度等因素。对于读密集型应用（如内容网站），增加从库可以近乎线性地扩展读吞吐。
