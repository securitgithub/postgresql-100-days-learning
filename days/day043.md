## 理论讲解

### 1. 什么是行级安全（RLS）

行级安全是 PostgreSQL 9.5 引入的功能，允许数据库管理员定义策略（Policy），控制哪些行对哪些用户可见或可操作。

传统的 GRANT/REVOKE 只能控制**表级**或**列级**权限，而 RLS 可以控制到**行级**。例如：
- 员工只能看到自己部门的数据
- 租户只能看到自己的数据
- 不同角色看到不同范围的行

### 2. RLS 工作原理

启用 RLS 后，PostgreSQL 在每次查询时自动在 WHERE 子句中附加策略条件：

```
用户查询: SELECT * FROM orders;
实际执行: SELECT * FROM orders WHERE <策略条件>;
```

核心要点：
- RLS 默认是**关闭**的，需要对表显式启用
- 表的所有者默认**不受** RLS 限制（除非使用 FORCE）
- 超级用户和具有 `BYPASSRLS` 属性的角色不受 RLS 限制
- 如果启用了 RLS 但没有定义策略，默认拒绝所有行（deny-all）

### 3. 策略类型

PostgreSQL 支持两种策略类型：

- **PERMISSIVE（宽松型）**：默认类型。多个 PERMISSIVE 策略之间是 OR 关系，满足任一即可
- **RESTRICTIVE（限制型）**：多个 RESTRICTIVE 策略之间是 AND 关系，必须全部满足

最终的行可见性 = (任一 PERMISSIVE 满足) AND (所有 RESTRICTIVE 满足)

### 4. 策略命令类型

策略可以针对特定的 DML 操作：

| 命令 | 说明 |
|------|------|
| `ALL` | 适用于所有操作（默认） |
| `SELECT` | 仅适用于查询 |
| `INSERT` | 仅适用于插入（WITH CHECK） |
| `UPDATE` | 适用于更新（USING + WITH CHECK） |
| `DELETE` | 仅适用于删除 |

- `USING` 子句：控制哪些**现有行**可被访问（SELECT、UPDATE 的读取、DELETE）
- `WITH CHECK` 子句：控制哪些**新行**可被写入（INSERT、UPDATE 的写入）

### 5. 多租户隔离

RLS 是实现 SaaS 多租户数据隔离的理想方案：
- 所有租户数据存储在同一张表中
- 每行包含 `tenant_id` 字段
- 通过 RLS 策略确保每个租户只能访问自己的数据
- 应用层通过 `SET` 设置当前租户标识

## 代码示例

### 示例 1：RLS 基础启用与简单策略

```sql
-- 创建测试表
CREATE TABLE documents (
    id SERIAL PRIMARY KEY,
    title VARCHAR(200) NOT NULL,
    content TEXT,
    owner_name VARCHAR(100) NOT NULL,
    created_at TIMESTAMP DEFAULT NOW()
);

-- 插入测试数据
INSERT INTO documents (title, content, owner_name) VALUES
    ('项目计划书', '2026年度项目规划...', 'alice'),
    ('技术方案', 'PostgreSQL 架构设计...', 'alice'),
    ('市场报告', '第一季度市场分析...', 'bob'),
    ('财务报表', '年度财务概况...', 'carol');

-- 创建用户
CREATE ROLE alice LOGIN PASSWORD 'alice123';
CREATE ROLE bob LOGIN PASSWORD 'bob123';
CREATE ROLE carol LOGIN PASSWORD 'carol123';

-- 授予表权限
GRANT SELECT, INSERT, UPDATE, DELETE ON documents TO alice, bob, carol;
GRANT USAGE, SELECT ON SEQUENCE documents_id_seq TO alice, bob, carol;

-- 启用 RLS
ALTER TABLE documents ENABLE ROW LEVEL SECURITY;

-- 此时如果 alice 查询，会返回 0 行（deny-all 默认行为）

-- 创建策略：用户只能看到自己拥有的文档
CREATE POLICY doc_owner_policy ON documents
    FOR ALL
    USING (owner_name = current_user)
    WITH CHECK (owner_name = current_user);

-- 现在 alice 查询只能看到 owner_name = 'alice' 的行
-- bob 查询只能看到 owner_name = 'bob' 的行
```

### 示例 2：针对不同操作的策略

```sql
-- 删除之前的策略
DROP POLICY doc_owner_policy ON documents;

-- 创建更精细的策略

-- SELECT 策略：用户可以查看自己的文档
CREATE POLICY doc_select_policy ON documents
    FOR SELECT
    USING (owner_name = current_user);

-- INSERT 策略：用户只能插入 owner_name 为自己的文档
CREATE POLICY doc_insert_policy ON documents
    FOR INSERT
    WITH CHECK (owner_name = current_user);

-- UPDATE 策略：用户只能更新自己的文档，且不能更改所有者
CREATE POLICY doc_update_policy ON documents
    FOR UPDATE
    USING (owner_name = current_user)
    WITH CHECK (owner_name = current_user);

-- DELETE 策略：用户只能删除自己的文档
CREATE POLICY doc_delete_policy ON documents
    FOR DELETE
    USING (owner_name = current_user);

-- 测试：alice 尝试插入一条 bob 的文档（会被拒绝）
-- SET ROLE alice;
-- INSERT INTO documents (title, owner_name) VALUES ('Test', 'bob');
-- ERROR: new row violates row-level security policy
```

### 示例 3：PERMISSIVE 与 RESTRICTIVE 策略组合

```sql
-- 创建带部门和密级的表
CREATE TABLE classified_docs (
    id SERIAL PRIMARY KEY,
    title VARCHAR(200),
    department VARCHAR(50),
    security_level INT DEFAULT 1,  -- 1=公开, 2=内部, 3=机密
    content TEXT,
    author VARCHAR(100)
);

ALTER TABLE classified_docs ENABLE ROW LEVEL SECURITY;

-- 创建角色
CREATE ROLE tech_user LOGIN PASSWORD 'tech123';
GRANT ALL ON classified_docs TO tech_user;
GRANT USAGE, SELECT ON SEQUENCE classified_docs_id_seq TO tech_user;

-- PERMISSIVE 策略：技术部可以查看技术部的文档 OR 公开文档
CREATE POLICY dept_policy ON classified_docs
    AS PERMISSIVE
    FOR SELECT
    TO tech_user
    USING (department = '技术部' OR security_level = 1);

-- RESTRICTIVE 策略：额外限制，密级不能超过 2
CREATE POLICY level_policy ON classified_docs
    AS RESTRICTIVE
    FOR SELECT
    TO tech_user
    USING (security_level <= 2);

-- 最终效果：tech_user 能看到的行 =
--   (department = '技术部' OR security_level = 1) AND (security_level <= 2)
-- 即：技术部的公开和内部文档 + 其他部门的公开文档
-- 机密文档（security_level = 3）永远不可见
```

### 示例 4：多租户隔离方案

```sql
-- 创建多租户订单表
CREATE TABLE tenant_orders (
    id SERIAL PRIMARY KEY,
    tenant_id INT NOT NULL,
    customer_name VARCHAR(100),
    amount NUMERIC(12,2),
    order_date DATE DEFAULT CURRENT_DATE
);

-- 创建应用角色（所有租户共用同一个数据库角色）
CREATE ROLE app_user LOGIN PASSWORD 'app123';
GRANT ALL ON tenant_orders TO app_user;
GRANT USAGE, SELECT ON SEQUENCE tenant_orders_id_seq TO app_user;

-- 启用 RLS
ALTER TABLE tenant_orders ENABLE ROW LEVEL SECURITY;

-- 使用自定义 GUC 变量存储当前租户 ID
-- 应用在每次连接时设置：SET app.current_tenant = '1';

-- 创建策略：基于会话变量过滤
CREATE POLICY tenant_isolation ON tenant_orders
    FOR ALL
    USING (tenant_id = current_setting('app.current_tenant')::INT)
    WITH CHECK (tenant_id = current_setting('app.current_tenant')::INT);

-- 插入测试数据
SET app.current_tenant = '0';  -- 临时绕过（作为表所有者）
INSERT INTO tenant_orders (tenant_id, customer_name, amount) VALUES
    (1, '客户A', 1000.00),
    (1, '客户B', 2500.00),
    (2, '客户C', 3000.00),
    (2, '客户D', 1500.00),
    (3, '客户E', 4000.00);

-- 模拟租户 1 的访问
-- SET ROLE app_user;
-- SET app.current_tenant = '1';
-- SELECT * FROM tenant_orders;
-- 只能看到 tenant_id = 1 的 2 条记录

-- 模拟租户 2 的访问
-- SET app.current_tenant = '2';
-- SELECT * FROM tenant_orders;
-- 只能看到 tenant_id = 2 的 2 条记录
```

### 示例 5：FORCE ROW LEVEL SECURITY

```sql
-- 默认情况下，表所有者不受 RLS 限制
-- 使用 FORCE 可以强制所有者也遵守 RLS

-- 对表所有者也强制执行 RLS
ALTER TABLE tenant_orders FORCE ROW LEVEL SECURITY;

-- 现在即使表的所有者查询，也需要满足策略条件

-- 取消强制
ALTER TABLE tenant_orders NO FORCE ROW LEVEL SECURITY;

-- 查看哪些表启用了 RLS
SELECT relname, relrowsecurity, relforcerowsecurity
FROM pg_class
WHERE relkind = 'r'
  AND relnamespace = 'public'::regnamespace;
```

### 示例 6：管理和查看策略

```sql
-- 查看所有策略
SELECT schemaname, tablename, policyname,
       permissive, roles, cmd, qual, with_check
FROM pg_policies
ORDER BY tablename;

-- 修改策略
ALTER POLICY doc_select_policy ON documents
    USING (owner_name = current_user OR owner_name = 'public_user');

-- 重命名策略
ALTER POLICY doc_select_policy ON documents
    RENAME TO doc_read_policy;

-- 删除策略
DROP POLICY doc_read_policy ON documents;

-- 禁用 RLS（保留策略定义但不生效）
ALTER TABLE documents DISABLE ROW LEVEL SECURITY;
```

## 常见问题

### Q1：RLS 对性能有影响吗？

**A：** RLS 会在查询中添加额外的过滤条件，对性能的影响取决于：

1. **策略条件的复杂度**：简单的等值比较（如 `tenant_id = 1`）几乎无影响；复杂的子查询会显著影响性能
2. **索引支持**：确保策略中使用的列有适当的索引。例如多租户方案中 `tenant_id` 列应建立索引
3. **策略数量**：过多的 PERMISSIVE 策略会导致 OR 条件过长

最佳实践：
```sql
-- 为 RLS 过滤列创建索引
CREATE INDEX idx_tenant_orders_tenant_id ON tenant_orders(tenant_id);
-- 使用 EXPLAIN 验证策略条件是否使用了索引
EXPLAIN SELECT * FROM tenant_orders;
```

### Q2：RLS 能否被绕过？存在安全风险吗？

**A：** 以下情况可以绕过 RLS：
1. **超级用户**始终绕过 RLS
2. 具有 `BYPASSRLS` 属性的角色绕过 RLS
3. **表所有者**默认不受 RLS 限制（除非使用 `FORCE ROW LEVEL SECURITY`）

安全建议：
- 应用程序不应使用超级用户或表所有者角色连接
- 使用 `FORCE ROW LEVEL SECURITY` 确保一致性
- RLS 策略中避免使用可被用户控制的函数（防止泄漏通道攻击）
- 注意 `pg_dump` 导出时 RLS 策略会包含在内

### Q3：如何调试 RLS 策略不生效的问题？

**A：** 排查步骤：
1. 确认 RLS 已启用：`SELECT relrowsecurity FROM pg_class WHERE relname = 'table_name';`
2. 确认当前用户不是表所有者或超级用户
3. 查看策略定义：`SELECT * FROM pg_policies WHERE tablename = 'table_name';`
4. 确认策略的 `roles` 包含当前用户（`{}`表示适用所有用户）
5. 手动测试策略条件：将 USING 子句中的条件作为 WHERE 条件执行
6. 使用 `EXPLAIN` 查看查询计划中是否包含策略过滤条件
