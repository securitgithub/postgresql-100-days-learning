## 理论讲解

### 1. PostgreSQL 日志系统概述

PostgreSQL 的日志系统由内建的 logging collector（日志收集器）管理。日志记录对于故障诊断、性能调优、安全审计都至关重要。

日志系统的核心组件：

| 配置项 | 说明 |
|--------|------|
| `logging_collector` | 是否启用日志收集器，启用后日志写入文件 |
| `log_destination` | 日志输出目标格式 |
| `log_directory` | 日志文件存放目录 |
| `log_filename` | 日志文件命名模式 |
| `log_rotation_age` | 日志轮转时间间隔 |
| `log_rotation_size` | 日志轮转大小阈值 |
| `log_truncate_on_rotation` | 轮转时是否截断旧文件 |

### 2. log_destination 详解

PostgreSQL 支持多种日志输出格式：

**stderr（默认）**
- 最简单的纯文本格式
- 适合开发环境和小型部署
- 需要配合 `log_line_prefix` 自定义格式

**csvlog**
- 结构化的 CSV 格式输出
- 易于导入数据库分析
- 包含固定字段：时间戳、用户、数据库、PID、SQL 状态码等

**syslog**
- 输出到系统日志服务
- 适合集中式日志管理
- 通过 `syslog_facility` 和 `syslog_ident` 配置

**jsonlog（PostgreSQL 15+）**
- JSON 结构化格式
- 易于与 ELK、Loki 等现代日志系统集成

### 3. 日志内容控制

控制记录哪些信息：

| 参数 | 说明 | 推荐值 |
|------|------|--------|
| `log_min_messages` | 最低日志级别 | `warning` |
| `log_min_error_statement` | 记录错误 SQL 的最低级别 | `error` |
| `log_min_duration_statement` | 记录执行超过指定毫秒的 SQL | `1000`（1秒） |
| `log_statement` | 记录哪类 SQL：none/ddl/mod/all | `ddl` |
| `log_checkpoints` | 记录检查点信息 | `on` |
| `log_connections` | 记录连接事件 | `on` |
| `log_disconnections` | 记录断开事件 | `on` |
| `log_lock_waits` | 记录锁等待超时事件 | `on` |
| `log_temp_files` | 记录临时文件使用 | `0`（全部记录） |
| `log_autovacuum_min_duration` | 记录 autovacuum 活动 | `0` |

### 4. log_line_prefix 格式化

`log_line_prefix` 定义每行日志的前缀格式，常用转义符：

| 转义符 | 含义 |
|--------|------|
| `%t` | 时间戳（无毫秒） |
| `%m` | 时间戳（含毫秒） |
| `%u` | 用户名 |
| `%d` | 数据库名 |
| `%p` | 进程 PID |
| `%r` | 客户端地址 |
| `%a` | 应用名称 |
| `%l` | 会话行号 |
| `%q` | 非会话进程时停止输出 |

### 5. pgBadger 日志分析

pgBadger 是一个高性能的 PostgreSQL 日志分析器，能生成丰富的 HTML 报告，包括：
- 查询统计（最耗时、最频繁、最慢）
- 连接/断开统计
- 锁等待分析
- 临时文件使用
- 错误分布
- 检查点统计

### 6. 慢查询识别策略

识别慢查询的多种方法：
1. **日志方式**：`log_min_duration_statement` 记录超时 SQL
2. **扩展方式**：`pg_stat_statements` 统计所有 SQL 性能
3. **实时方式**：查询 `pg_stat_activity` 查看当前运行的长查询
4. **分析方式**：使用 `EXPLAIN ANALYZE` 分析执行计划

## 代码示例

### 示例1：生产环境推荐日志配置

```sql
-- 查看当前日志配置
SHOW logging_collector;
SHOW log_destination;
SHOW log_directory;
SHOW log_filename;
SHOW log_line_prefix;

-- 修改日志配置（需要超级用户权限）
-- 注意：部分参数需要重启，部分可以 reload

-- 以下参数可通过 ALTER SYSTEM 修改后 reload
ALTER SYSTEM SET logging_collector = 'on';           -- 需重启
ALTER SYSTEM SET log_destination = 'csvlog';          -- 需重启
ALTER SYSTEM SET log_directory = 'log';
ALTER SYSTEM SET log_filename = 'postgresql-%Y-%m-%d.log';
ALTER SYSTEM SET log_rotation_age = '1d';
ALTER SYSTEM SET log_rotation_size = '100MB';
ALTER SYSTEM SET log_truncate_on_rotation = 'on';
```

### 示例2：配置 log_line_prefix（stderr 模式）

```sql
-- pgBadger 推荐的 log_line_prefix 格式
ALTER SYSTEM SET log_line_prefix = '%t [%p]: user=%u,db=%d,app=%a,client=%h ';

-- 更详细的格式
ALTER SYSTEM SET log_line_prefix = '%m [%p] %q%u@%d [%a] [%r] ';

-- 应用配置
SELECT pg_reload_conf();
```

### 示例3：慢查询日志配置

```sql
-- 记录超过 500ms 的查询
ALTER SYSTEM SET log_min_duration_statement = 500;

-- 记录所有 DDL 语句
ALTER SYSTEM SET log_statement = 'ddl';

-- 记录检查点信息
ALTER SYSTEM SET log_checkpoints = 'on';

-- 记录锁等待
ALTER SYSTEM SET log_lock_waits = 'on';
ALTER SYSTEM SET deadlock_timeout = '1s';

-- 记录临时文件（任何大小）
ALTER SYSTEM SET log_temp_files = 0;

-- 记录 autovacuum 活动
ALTER SYSTEM SET log_autovacuum_min_duration = 0;

-- 应用配置
SELECT pg_reload_conf();
```

### 示例4：将 CSV 日志导入数据库分析

```sql
-- 创建日志分析表（匹配 csvlog 格式）
CREATE TABLE postgres_log (
    log_time               timestamp(3) with time zone,
    user_name              text,
    database_name          text,
    process_id             integer,
    connection_from        text,
    session_id             text,
    session_line_num       bigint,
    command_tag            text,
    session_start_time     timestamp with time zone,
    virtual_transaction_id text,
    transaction_id         bigint,
    error_severity         text,
    sql_state_code         text,
    message                text,
    detail                 text,
    hint                   text,
    internal_query         text,
    internal_query_pos     integer,
    context                text,
    query                  text,
    query_pos              integer,
    location               text,
    application_name       text,
    backend_type           text,
    leader_pid             integer,
    query_id               bigint
);

-- 导入 CSV 日志文件
COPY postgres_log FROM '/var/lib/postgresql/16/main/log/postgresql-2026-04-01.csv'
WITH (FORMAT csv);

-- 查询最慢的 SQL
SELECT log_time, user_name, database_name,
       substring(message from 'duration: ([0-9.]+) ms') AS duration_ms,
       query
FROM postgres_log
WHERE message LIKE 'duration:%'
ORDER BY substring(message from 'duration: ([0-9.]+) ms')::numeric DESC
LIMIT 20;

-- 统计错误分布
SELECT error_severity, sql_state_code, count(*)
FROM postgres_log
WHERE error_severity IN ('ERROR', 'FATAL', 'PANIC')
GROUP BY error_severity, sql_state_code
ORDER BY count(*) DESC;
```

### 示例5：使用 pgBadger 分析日志

```bash
# 安装 pgBadger
sudo apt-get install pgbadger
# 或
# sudo yum install pgbadger

# 分析 stderr 格式日志
pgbadger /var/log/postgresql/postgresql-2026-04-01.log \
    -o /tmp/pgbadger_report.html \
    -p '%t [%p]: user=%u,db=%d,app=%a,client=%h '

# 分析 CSV 格式日志
pgbadger /var/lib/postgresql/16/main/log/postgresql-2026-04-01.csv \
    -o /tmp/pgbadger_report.html \
    --format csv

# 分析多个日志文件
pgbadger /var/log/postgresql/postgresql-2026-04-*.log \
    -o /tmp/weekly_report.html

# 增量分析（适合定期执行）
pgbadger --last-parsed /tmp/.pgbadger_last \
    --incremental \
    /var/log/postgresql/postgresql-*.log \
    -O /var/www/html/pgbadger/

# 仅分析特定数据库
pgbadger --dbname myapp_production \
    /var/log/postgresql/postgresql-2026-04-01.log \
    -o /tmp/myapp_report.html
```

### 示例6：实时查看慢查询

```sql
-- 查看当前正在执行的慢查询（超过 5 秒）
SELECT pid,
       now() - pg_stat_activity.query_start AS duration,
       usename,
       datname,
       state,
       left(query, 100) AS query_preview
FROM pg_stat_activity
WHERE (now() - pg_stat_activity.query_start) > interval '5 seconds'
  AND state != 'idle'
  AND pid != pg_backend_pid()
ORDER BY duration DESC;

-- 查看等待事件
SELECT pid, wait_event_type, wait_event,
       state, left(query, 80) AS query
FROM pg_stat_activity
WHERE wait_event IS NOT NULL
  AND pid != pg_backend_pid();
```

### 示例7：日志轮转管理

```sql
-- 查看当前日志文件
SELECT pg_current_logfile();      -- stderr 日志
SELECT pg_current_logfile('csvlog'); -- CSV 日志

-- 手动触发日志轮转
SELECT pg_rotate_logfile();
```

```bash
# 定期清理旧日志（crontab 任务）
# 每天凌晨删除 30 天前的日志
0 0 * * * find /var/lib/postgresql/16/main/log -name "*.csv" -mtime +30 -delete
0 0 * * * find /var/lib/postgresql/16/main/log -name "*.log" -mtime +30 -delete
```

## 常见问题

### Q1：logging_collector 设为 on 后日志文件在哪里？

日志文件位于 `log_directory` 指定的路径下，默认是数据目录下的 `log` 子目录（如 `/var/lib/postgresql/16/main/log/`）。可通过 `SHOW log_directory;` 查看。如果 `log_directory` 是相对路径，则相对于数据目录。

### Q2：log_min_duration_statement 设为 0 会有什么影响？

设为 0 会记录所有 SQL 语句及其执行时间，这在生产环境中会产生大量日志，严重影响性能和磁盘空间。建议生产环境设置为 500 或 1000（毫秒），仅记录真正的慢查询。如需全量分析，应使用 `pg_stat_statements` 扩展。

### Q3：如何在不重启的情况下更改日志配置？

大部分日志参数都支持 `reload` 生效，使用 `ALTER SYSTEM SET ... ;` 修改后执行 `SELECT pg_reload_conf();` 即可。但 `logging_collector`、`log_destination` 等少数参数需要重启。可通过 `pg_settings` 视图的 `context` 列确认参数变更方式。
