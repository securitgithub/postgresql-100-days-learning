# Day 034: 存储过程 — CREATE PROCEDURE

## 学习目标

- 理解存储过程与函数的核心区别
- 掌握 CREATE PROCEDURE 的语法与使用方式
- 理解存储过程内部的事务控制（COMMIT/ROLLBACK）
- 熟练使用 CALL 语句调用存储过程
- 能够根据场景选择使用函数还是存储过程

## 理论讲解

### 存储过程 vs 函数

PostgreSQL 从 11 版本开始引入存储过程（`CREATE PROCEDURE`）。在此之前，所有可编程逻辑都通过函数（`CREATE FUNCTION`）实现。两者的核心区别如下：

| 特性 | 函数（FUNCTION） | 存储过程（PROCEDURE） |
|------|------------------|----------------------|
| 返回值 | 必须有返回值（RETURNS） | 无返回值（可用 INOUT 参数传出） |
| 调用方式 | `SELECT func()` 或在表达式中使用 | `CALL proc()` |
| 事务控制 | 不能 COMMIT/ROLLBACK | 可以内部 COMMIT/ROLLBACK |
| 在 SQL 中使用 | 可在 SELECT/WHERE 等中使用 | 不能在 SQL 表达式中使用 |
| 触发器 | 可作为触发器函数 | 不能用于触发器 |

### 为什么需要存储过程？

最核心的原因是**事务控制**。函数整体在一个事务中执行，无法在中间提交或回滚。而存储过程可以在执行过程中提交当前事务并开启新事务，这对以下场景非常重要：

- **批量数据处理**：每处理一批数据就提交一次，避免长事务
- **ETL 流程**：多步骤数据转换，每步独立提交
- **维护任务**：清理历史数据时分批提交，减少锁持有时间

### CREATE PROCEDURE 语法

```sql
CREATE [ OR REPLACE ] PROCEDURE procedure_name (
    [ [ argmode ] [ argname ] argtype [ DEFAULT default_expr ] [, ...] ]
)
LANGUAGE lang_name
AS $$
    -- 过程体
$$;
```

注意：
- 没有 `RETURNS` 子句
- 参数模式支持 IN、INOUT（不支持 OUT 和 VARIADIC）
- 不能声明 IMMUTABLE/STABLE/VOLATILE
- 使用 `CALL` 调用而非 `SELECT`

### 事务控制

存储过程中可以使用 `COMMIT` 和 `ROLLBACK` 控制事务：

```sql
CREATE PROCEDURE my_proc()
LANGUAGE plpgsql
AS $$
BEGIN
    -- 操作1
    INSERT INTO t1 VALUES (1);
    COMMIT;  -- 提交操作1

    -- 操作2
    INSERT INTO t2 VALUES (2);
    COMMIT;  -- 提交操作2

    -- 操作3（如果失败，只影响操作3）
    INSERT INTO t3 VALUES (3);
END;
$$;
```

**限制条件：**
- 如果存储过程是被另一个函数调用的，则不能使用事务控制
- 在显式事务块（`BEGIN...COMMIT`）中调用的过程不能使用事务控制
- 如果过程内部有 EXCEPTION 块，不能在同一层使用 COMMIT/ROLLBACK

## 代码示例

### 示例 1：基本存储过程

```sql
-- 创建一个员工调薪过程
CREATE OR REPLACE PROCEDURE adjust_salary(
    p_emp_id    INTEGER,
    p_new_salary NUMERIC
)
LANGUAGE plpgsql
AS $$
DECLARE
    v_old_salary NUMERIC;
    v_emp_name   TEXT;
BEGIN
    -- 获取当前信息
    SELECT salary, first_name || ' ' || last_name
    INTO v_old_salary, v_emp_name
    FROM employees
    WHERE employee_id = p_emp_id;

    IF NOT FOUND THEN
        RAISE EXCEPTION '员工 ID % 不存在', p_emp_id;
    END IF;

    -- 验证新薪资
    IF p_new_salary <= 0 THEN
        RAISE EXCEPTION '薪资必须大于零：%', p_new_salary;
    END IF;

    -- 更新薪资
    UPDATE employees
    SET salary = p_new_salary, updated_at = NOW()
    WHERE employee_id = p_emp_id;

    -- 记录调薪日志
    INSERT INTO salary_audit_log (employee_id, old_salary, new_salary, changed_at)
    VALUES (p_emp_id, v_old_salary, p_new_salary, NOW());

    RAISE NOTICE '员工 % 薪资已从 % 调整为 %', v_emp_name, v_old_salary, p_new_salary;
END;
$$;

-- 调用存储过程
CALL adjust_salary(101, 8500.00);
-- 使用命名参数
CALL adjust_salary(p_emp_id => 102, p_new_salary => 9000.00);
```

### 示例 2：使用 INOUT 参数返回结果

```sql
-- 存储过程没有 RETURNS，但可以通过 INOUT 参数传出结果
CREATE OR REPLACE PROCEDURE transfer_funds(
    p_from_account INTEGER,
    p_to_account   INTEGER,
    p_amount       NUMERIC,
    INOUT p_status TEXT DEFAULT NULL  -- 用 INOUT 传出状态
)
LANGUAGE plpgsql
AS $$
DECLARE
    v_balance NUMERIC;
BEGIN
    -- 检查源账户余额
    SELECT balance INTO v_balance
    FROM accounts
    WHERE account_id = p_from_account
    FOR UPDATE;  -- 加行锁

    IF NOT FOUND THEN
        p_status := '失败：源账户不存在';
        RETURN;
    END IF;

    IF v_balance < p_amount THEN
        p_status := format('失败：余额不足（当前余额：%s）', v_balance);
        RETURN;
    END IF;

    -- 检查目标账户
    IF NOT EXISTS (SELECT 1 FROM accounts WHERE account_id = p_to_account) THEN
        p_status := '失败：目标账户不存在';
        RETURN;
    END IF;

    -- 执行转账
    UPDATE accounts SET balance = balance - p_amount WHERE account_id = p_from_account;
    UPDATE accounts SET balance = balance + p_amount WHERE account_id = p_to_account;

    -- 记录交易
    INSERT INTO transactions (from_account, to_account, amount, tx_time)
    VALUES (p_from_account, p_to_account, p_amount, NOW());

    p_status := format('成功：从账户 %s 转 %s 到账户 %s',
                       p_from_account, p_amount, p_to_account);
END;
$$;

-- 调用并获取状态
CALL transfer_funds(1001, 1002, 500.00, NULL);
```

### 示例 3：事务控制 — 批量数据迁移

```sql
-- 批量迁移数据，每批提交一次
CREATE OR REPLACE PROCEDURE migrate_orders_archive(
    p_batch_size INTEGER DEFAULT 1000
)
LANGUAGE plpgsql
AS $$
DECLARE
    v_moved   INTEGER;
    v_total   INTEGER := 0;
    v_cutoff  DATE := CURRENT_DATE - INTERVAL '1 year';
BEGIN
    RAISE NOTICE '开始迁移 % 之前的订单...', v_cutoff;

    LOOP
        -- 每次移动一批数据
        WITH moved AS (
            DELETE FROM orders
            WHERE order_date < v_cutoff
            AND order_id IN (
                SELECT order_id FROM orders
                WHERE order_date < v_cutoff
                LIMIT p_batch_size
            )
            RETURNING *
        )
        INSERT INTO orders_archive
        SELECT * FROM moved;

        GET DIAGNOSTICS v_moved = ROW_COUNT;
        v_total := v_total + v_moved;

        -- 如果没有更多数据，退出
        EXIT WHEN v_moved = 0;

        -- 每批提交一次事务（关键！）
        COMMIT;
        RAISE NOTICE '已迁移 % 条记录（累计 %）', v_moved, v_total;
    END LOOP;

    RAISE NOTICE '迁移完成，共迁移 % 条记录', v_total;
END;
$$;

-- 调用
CALL migrate_orders_archive(500);
```

### 示例 4：事务控制 — 带回滚的多步骤任务

```sql
-- 多步骤用户注册流程
CREATE OR REPLACE PROCEDURE register_user_full(
    p_email    TEXT,
    p_name     TEXT,
    p_role     TEXT DEFAULT 'user'
)
LANGUAGE plpgsql
AS $$
DECLARE
    v_user_id INTEGER;
BEGIN
    -- 步骤1：创建用户
    INSERT INTO users (email, name, created_at)
    VALUES (p_email, p_name, NOW())
    RETURNING id INTO v_user_id;

    COMMIT;  -- 用户创建成功，先提交
    RAISE NOTICE '步骤1完成：用户 % 已创建（ID=%）', p_email, v_user_id;

    -- 步骤2：分配角色
    INSERT INTO user_roles (user_id, role_name, assigned_at)
    VALUES (v_user_id, p_role, NOW());

    COMMIT;  -- 角色分配成功，提交
    RAISE NOTICE '步骤2完成：角色 % 已分配', p_role;

    -- 步骤3：初始化用户配置
    INSERT INTO user_settings (user_id, setting_key, setting_value)
    VALUES
        (v_user_id, 'language', 'zh-CN'),
        (v_user_id, 'timezone', 'Asia/Shanghai'),
        (v_user_id, 'notifications', 'true');

    COMMIT;
    RAISE NOTICE '步骤3完成：用户配置已初始化';

    RAISE NOTICE '用户 % 注册完成', p_email;
END;
$$;

CALL register_user_full('newuser@example.com', '新用户', 'editor');
```

### 示例 5：清理过期数据

```sql
-- 定期清理过期数据的维护过程
CREATE OR REPLACE PROCEDURE maintenance_cleanup(
    p_days_to_keep INTEGER DEFAULT 90,
    p_batch_size   INTEGER DEFAULT 5000,
    INOUT p_summary TEXT DEFAULT NULL
)
LANGUAGE plpgsql
AS $$
DECLARE
    v_cutoff    TIMESTAMP := NOW() - (p_days_to_keep || ' days')::INTERVAL;
    v_deleted   INTEGER;
    v_total_log INTEGER := 0;
    v_total_ses INTEGER := 0;
BEGIN
    -- 清理日志表
    LOOP
        DELETE FROM system_logs
        WHERE created_at < v_cutoff
        AND id IN (
            SELECT id FROM system_logs
            WHERE created_at < v_cutoff
            LIMIT p_batch_size
        );
        GET DIAGNOSTICS v_deleted = ROW_COUNT;
        v_total_log := v_total_log + v_deleted;
        COMMIT;
        EXIT WHEN v_deleted < p_batch_size;
    END LOOP;

    -- 清理会话表
    LOOP
        DELETE FROM user_sessions
        WHERE expire_at < v_cutoff
        AND id IN (
            SELECT id FROM user_sessions
            WHERE expire_at < v_cutoff
            LIMIT p_batch_size
        );
        GET DIAGNOSTICS v_deleted = ROW_COUNT;
        v_total_ses := v_total_ses + v_deleted;
        COMMIT;
        EXIT WHEN v_deleted < p_batch_size;
    END LOOP;

    p_summary := format('清理完成：日志 %s 条，会话 %s 条', v_total_log, v_total_ses);
    RAISE NOTICE '%', p_summary;
END;
$$;

-- 调用
CALL maintenance_cleanup(60, 2000, NULL);
```

## 常见问题

### Q1：什么时候该用存储过程而不是函数？

简单规则：
- **需要事务控制** → 存储过程
- **需要返回值或在 SQL 中调用** → 函数
- **执行操作（副作用为主）** → 存储过程
- **计算并返回结果** → 函数

实际项目中，大多数场景用函数就够了。存储过程主要用于：批量数据处理、ETL 流程、定期维护任务等需要在中间提交事务的场景。

### Q2：存储过程中 COMMIT 后变量值还保留吗？

是的，COMMIT 只影响数据库事务状态，PL/pgSQL 的局部变量保持不变。这使得你可以在提交后继续使用之前计算的变量值。

```sql
CREATE OR REPLACE PROCEDURE demo_commit_vars()
LANGUAGE plpgsql
AS $$
DECLARE
    v_counter INTEGER := 0;
BEGIN
    v_counter := v_counter + 1;
    COMMIT;
    -- v_counter 仍然是 1
    RAISE NOTICE 'counter = %', v_counter;  -- 输出: counter = 1
END;
$$;
```

### Q3：为什么在 EXCEPTION 块内不能使用 COMMIT/ROLLBACK？

因为 EXCEPTION 块内部使用了隐式的 SAVEPOINT 机制。当异常发生时，PostgreSQL 会自动回滚到 SAVEPOINT，这与显式的 COMMIT/ROLLBACK 在语义上冲突。如果你需要同时使用事务控制和异常处理，需要将它们放在不同的嵌套层级中。
