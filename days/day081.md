# Day 081: 流复制基础 — 主从复制原理与配置

## 学习目标

- 理解 PostgreSQL 流复制（Streaming Replication）的基本原理与架构
- 掌握 WAL（Write-Ahead Log）日志发送与接收的工作机制
- 学会在主库配置 `wal_level`、`max_wal_senders` 等关键参数
- 学会在从库配置 `primary_conninfo` 实现数据同步
- 了解流复制与日志传送（Log Shipping）的区别

## 理论讲解

### 1. 什么是流复制

流复制是 PostgreSQL 9.0 引入的一种内建高可用方案。主库（Primary）将 WAL 日志流实时发送给一个或多个从库（Standby），从库接收并回放这些日志，从而保持与主库的数据一致。

与基于文件的日志传送（Log Shipping）相比，流复制的最大优势在于**实时性**——从库几乎可以在主库生成 WAL 记录的同一时刻就收到数据，复制延迟通常在毫秒级别。

### 2. 流复制架构

```
┌─────────────┐         WAL Stream         ┌─────────────┐
│   主库       │ ──────────────────────────▶ │   从库       │
│  (Primary)   │   walsender ─▶ walreceiver │  (Standby)   │
│              │                            │              │
│  wal_level   │                            │ primary_     │
│  = replica   │                            │ conninfo     │
└─────────────┘                            └─────────────┘
```

核心组件：
- **walsender 进程**：运行在主库上，负责将 WAL 数据流式发送给从库
- **walreceiver 进程**：运行在从库上，负责接收 WAL 数据并写入本地
- **startup 进程**：从库上的恢复进程，负责回放收到的 WAL 日志

### 3. WAL 发送与接收流程

1. 客户端在主库执行写操作（INSERT/UPDATE/DELETE）
2. 主库将变更写入 WAL 缓冲区，然后刷写到 WAL 文件
3. walsender 进程检测到新的 WAL 数据，通过 TCP 连接发送给从库
4. 从库的 walreceiver 进程接收 WAL 数据，写入本地 WAL 文件
5. startup 进程读取本地 WAL 文件并回放，更新数据页面

### 4. 关键参数详解

#### 主库参数

| 参数 | 说明 | 推荐值 |
|------|------|--------|
| `wal_level` | WAL 日志级别 | `replica` 或 `logical` |
| `max_wal_senders` | 最大 WAL 发送进程数 | 从库数量 + 2（留余量） |
| `wal_keep_size` | 保留的 WAL 文件大小 | `1GB`（根据需要调整） |
| `max_replication_slots` | 最大复制槽数量 | 从库数量 + 2 |
| `synchronous_standby_names` | 同步从库名称 | 按需设置 |

#### 从库参数

| 参数 | 说明 | 示例 |
|------|------|------|
| `primary_conninfo` | 连接主库的信息 | `'host=主库IP port=5432 user=replicator password=xxx'` |
| `hot_standby` | 是否允许从库查询 | `on` |
| `primary_slot_name` | 使用的复制槽名称 | `'standby1_slot'` |

### 5. wal_level 三种级别

- **minimal**：仅记录崩溃恢复所需的最少信息，不支持任何复制
- **replica**：记录足够物理复制（流复制）的信息，这是默认值
- **logical**：在 replica 基础上增加逻辑解码所需信息，支持逻辑复制

```
minimal < replica < logical
```

设置为更高级别会增加少量 WAL 写入开销，但通常可忽略不计。

### 6. 流复制 vs 日志传送

| 特性 | 流复制 | 日志传送 |
|------|--------|----------|
| 延迟 | 毫秒级 | 分钟级（取决于归档频率） |
| 传输单位 | WAL 记录 | WAL 文件（通常16MB） |
| 实时性 | 高 | 低 |
| 配置复杂度 | 中 | 低 |
| 带宽使用 | 持续、小量 | 间歇、大块 |

## 代码示例

### 示例1：主库配置 postgresql.conf

```ini
# ---- 流复制主库配置 ----

# WAL 级别设置为 replica（支持流复制的最低要求）
wal_level = replica

# 允许最多 5 个 WAL 发送进程（支持5个从库同时连接）
max_wal_senders = 5

# 保留 1GB 的 WAL 文件，防止从库断开后丢失未同步的 WAL
wal_keep_size = '1GB'

# 允许最多 5 个复制槽
max_replication_slots = 5

# 监听所有网络接口
listen_addresses = '*'

# 启用归档（推荐，作为流复制的补充）
archive_mode = on
archive_command = 'cp %p /var/lib/pgsql/archive/%f'
```

### 示例2：主库 pg_hba.conf 添加复制权限

```ini
# 允许从库通过复制协议连接主库
# TYPE  DATABASE        USER            ADDRESS                 METHOD
host    replication     replicator      192.168.1.0/24          scram-sha-256
host    replication     replicator      10.0.0.0/8              scram-sha-256
```

### 示例3：在主库创建复制用户

```sql
-- 创建专用复制用户
CREATE ROLE replicator WITH REPLICATION LOGIN PASSWORD 'StrongP@ss2024';

-- 验证用户权限
SELECT rolname, rolreplication
FROM pg_roles
WHERE rolname = 'replicator';
```

### 示例4：使用 pg_basebackup 创建从库基础备份

```bash
# 在从库机器上执行，从主库拉取完整基础备份
pg_basebackup \
  -h 192.168.1.100 \
  -p 5432 \
  -U replicator \
  -D /var/lib/postgresql/16/main \
  -Fp \          # 以 plain 格式输出
  -Xs \          # 使用流模式传输 WAL
  -P \           # 显示进度
  -R             # 自动生成 standby.signal 和 primary_conninfo

# -R 参数会自动创建以下文件：
#   standby.signal          -- 标识此实例为从库
#   postgresql.auto.conf    -- 包含 primary_conninfo 设置
```

### 示例5：从库配置 postgresql.conf

```ini
# ---- 流复制从库配置 ----

# 允许从库接受只读查询（Hot Standby）
hot_standby = on

# 连接主库的信息（如果没用 pg_basebackup -R 则需手动配置）
primary_conninfo = 'host=192.168.1.100 port=5432 user=replicator password=StrongP@ss2024 application_name=standby1'

# 指定使用的复制槽（可选但推荐）
primary_slot_name = 'standby1_slot'
```

### 示例6：在主库创建复制槽

```sql
-- 创建物理复制槽（确保主库为从库保留所需的 WAL）
SELECT pg_create_physical_replication_slot('standby1_slot');

-- 查看所有复制槽
SELECT slot_name, slot_type, active, restart_lsn
FROM pg_replication_slots;
```

### 示例7：检查复制状态

```sql
-- 在主库查看复制连接状态
SELECT
    pid,
    usesysid,
    usename,
    application_name,
    client_addr,
    state,
    sent_lsn,
    write_lsn,
    flush_lsn,
    replay_lsn,
    sync_state
FROM pg_stat_replication;

-- 在从库查看自身复制状态
SELECT
    pid,
    status,
    receive_start_lsn,
    received_lsn,
    last_msg_send_time,
    last_msg_receipt_time,
    sender_host,
    sender_port
FROM pg_stat_wal_receiver;

-- 在从库查看恢复状态
SELECT pg_is_in_recovery();           -- 是否处于恢复模式（从库返回 true）
SELECT pg_last_wal_receive_lsn();     -- 最后接收到的 WAL 位置
SELECT pg_last_wal_replay_lsn();      -- 最后回放的 WAL 位置
SELECT pg_last_xact_replay_timestamp(); -- 最后回放事务的时间戳
```

## 常见问题

### Q1: 修改 wal_level 需要重启数据库吗？

**A:** 是的，`wal_level` 是一个需要重启生效的参数（context = postmaster）。修改后必须重启 PostgreSQL 服务。同样，`max_wal_senders` 和 `max_replication_slots` 也需要重启才能生效。建议在业务低谷期进行变更。

### Q2: 从库能否执行写操作？

**A:** 不能。流复制的从库处于只读模式（Hot Standby），任何写操作（INSERT/UPDATE/DELETE/DDL）都会报错：`ERROR: cannot execute XXX in a read-only transaction`。如果需要在从库写入，可以考虑逻辑复制方案。

### Q3: max_wal_senders 设置过小会怎样？

**A:** 如果所有 walsender 进程都被占用，新的从库将无法连接到主库进行复制。同时，`pg_basebackup` 也需要占用一个 walsender 进程。建议设置为实际从库数量加 2~3 的余量。出现 `FATAL: number of requested standby connections exceeds max_wal_senders` 时说明需要增大此值。
