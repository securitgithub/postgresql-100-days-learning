## 理论讲解

### 1. 物理备份概述

物理备份是对 PostgreSQL 数据目录（包括所有数据文件、WAL 文件等）的完整复制。

**物理备份的优势：**
- 备份和恢复速度快（直接复制文件）
- 支持 PITR（基于时间点恢复）
- 备份了整个集群，包括所有数据库
- 适合大型数据库

**物理备份的限制：**
- 只能恢复到相同主版本的 PostgreSQL
- 只能恢复整个集群，不能选择性恢复单个表
- 需要相同或兼容的操作系统和硬件架构

### 2. 文件系统备份

最简单的物理备份方式是直接复制数据目录，但有严格要求：

- **必须停止 PostgreSQL** 才能确保一致性
- 或者使用文件系统快照（LVM 快照、ZFS 快照等）配合 WAL
- 需要备份整个数据目录 `$PGDATA`
- 如果使用了表空间，还需要备份表空间目录

### 3. pg_basebackup

`pg_basebackup` 是 PostgreSQL 内置的在线物理备份工具：

- 通过复制协议从运行中的服务器获取备份
- 自动处理备份开始和结束的检查点
- 支持压缩
- 支持将备份传输到远程服务器
- 可以包含 WAL 文件

### 4. WAL 归档

WAL 归档是实现 PITR 的前提条件。其工作原理：

```
事务操作 → 写入 WAL → WAL 段文件填满（16MB）→ 触发归档
                                                    ↓
                                            执行 archive_command
                                                    ↓
                                            将 WAL 文件复制到归档目录
```

**关键参数：**
- `wal_level`：必须设为 `replica` 或 `logical`
- `archive_mode`：设为 `on` 启用归档
- `archive_command`：定义归档时执行的 shell 命令

### 5. PITR（Point-In-Time Recovery）

PITR 允许将数据库恢复到过去任意一个时间点。实现条件：

1. 一个基础备份（pg_basebackup）
2. 从基础备份时间到目标时间点之间的所有 WAL 归档文件

**恢复流程：**
```
还原基础备份 → 配置恢复参数 → 启动 PostgreSQL
                                    ↓
                          自动重放 WAL 归档文件
                                    ↓
                          到达目标时间点后停止
                                    ↓
                          数据库恢复完成
```

## 代码示例

### 示例 1：文件系统备份

```bash
# === 冷备份（停止服务） ===

# 1. 停止 PostgreSQL
pg_ctl stop -D /var/lib/postgresql/data

# 2. 复制整个数据目录
cp -a /var/lib/postgresql/data /backup/pg_data_$(date +%Y%m%d)

# 3. 如果有表空间，也需要备份
# cp -a /path/to/tablespace /backup/tablespace_$(date +%Y%m%d)

# 4. 重新启动 PostgreSQL
pg_ctl start -D /var/lib/postgresql/data

# === 使用 LVM 快照（热备份） ===

# 1. 创建 LVM 快照
# lvcreate -L 10G -s -n pg_snap /dev/vg_data/lv_pgdata

# 2. 挂载快照
# mount /dev/vg_data/pg_snap /mnt/pg_snapshot

# 3. 复制数据
# cp -a /mnt/pg_snapshot/pgdata/* /backup/pg_snapshot_$(date +%Y%m%d)/

# 4. 卸载并删除快照
# umount /mnt/pg_snapshot
# lvremove /dev/vg_data/pg_snap

# === 使用 rsync 增量复制（配合 WAL 归档） ===
# 注意：单独使用 rsync 不保证一致性，必须配合 WAL
# rsync -av --delete /var/lib/postgresql/data/ /backup/pg_rsync/
```

# archive_mode = on

# archive_command = 'cp %p /data/wal_archive/%f'

# %f = WAL 文件名

# archive_timeout = 300   # 5 分钟无 WAL 写入也强制归档

```sql
-- 验证归档配置
SHOW wal_level;
SHOW archive_mode;
SHOW archive_command;
SHOW archive_timeout;

-- 查看归档状态
SELECT * FROM pg_stat_archiver;
-- archived_count: 已归档的 WAL 文件数
-- last_archived_wal: 最后归档的 WAL 文件
-- last_archived_time: 最后归档时间
-- failed_count: 归档失败次数
-- last_failed_wal: 最后失败的 WAL 文件
-- last_failed_time: 最后失败时间

-- 手动触发 WAL 切换（测试归档是否工作）
SELECT pg_switch_wal();

-- 检查当前 WAL 位置
SELECT pg_current_wal_lsn();
SELECT pg_walfile_name(pg_current_wal_lsn());
```

### 示例 3：pg_basebackup 基础用法

```bash
# === 基本备份 ===
# 备份到指定目录
pg_basebackup -h localhost -U repl_user -D /backup/base_$(date +%Y%m%d) \
    -Fp -Xs -P -v

# 参数说明：
# -D : 备份目标目录
# -Fp: plain 格式（直接文件）
# -Xs: 通过流复制传输 WAL（推荐）
# -P : 显示进度
# -v : 详细输出

# === 压缩备份（tar 格式） ===
pg_basebackup -h localhost -U repl_user -D /backup/base_$(date +%Y%m%d) \
    -Ft -z -Xs -P -v

# -Ft: tar 格式
# -z : gzip 压缩
# 生成文件：base.tar.gz 和 pg_wal.tar.gz

# === 指定压缩级别 ===
pg_basebackup -h localhost -U repl_user -D /backup/base_$(date +%Y%m%d) \
    -Ft -Z 9 -Xs -P

# === 设置最大传输速率（避免影响生产） ===
pg_basebackup -h localhost -U repl_user -D /backup/base_$(date +%Y%m%d) \
    -Fp -Xs -P --max-rate=100M

# === 创建复制槽以确保 WAL 不被回收 ===
pg_basebackup -h localhost -U repl_user -D /backup/base_$(date +%Y%m%d) \
    -Fp -Xs -P -S backup_slot --create-slot

# === 备份到标准输出（管道传输） ===
pg_basebackup -h localhost -U repl_user -Ft -Xs \
    | ssh backup@remote "cat > /backup/base_$(date +%Y%m%d).tar"
```

# 完整的 pg_basebackup 示例
pg_basebackup \
    -h localhost \
    -p 5432 \
    -U repl_user \
    -D /backup/basebackup_$(date +%Y%m%d_%H%M%S) \
    -Fp \

    -P \
    -v \

    --label="daily_backup_$(date +%Y%m%d)" \
    2>&1 | tee /backup/logs/basebackup_$(date +%Y%m%d).log

### 示例 5：PITR 恢复流程

```bash
# === PITR 恢复完整步骤 ===

# 假设场景：在 2026-04-02 15:30 误删了重要数据
# 需要恢复到 2026-04-02 15:29（误操作前一分钟）

# 1. 停止 PostgreSQL
pg_ctl stop -D /var/lib/postgresql/data

# 2. 保存当前数据目录（以防万一）
mv /var/lib/postgresql/data /var/lib/postgresql/data_broken

# 3. 还原基础备份
cp -a /backup/basebackup_20260402_020000 /var/lib/postgresql/data

# 或者使用 tar 格式的备份：
# mkdir /var/lib/postgresql/data
# tar xzf /backup/base.tar.gz -C /var/lib/postgresql/data/
# tar xzf /backup/pg_wal.tar.gz -C /var/lib/postgresql/data/pg_wal/

# 4. 清理旧的 WAL（如果有的话）
# rm -f /var/lib/postgresql/data/pg_wal/*

# 5. 创建恢复配置文件
cat > /var/lib/postgresql/data/postgresql.auto.conf << 'CONF'
# PITR 恢复配置
restore_command = 'cp /data/wal_archive/%f %p'
recovery_target_time = '2026-04-02 15:29:00+08'
recovery_target_action = 'pause'
CONF

# 6. 创建恢复信号文件
touch /var/lib/postgresql/data/recovery.signal

# 7. 确保文件权限正确
chown -R postgres:postgres /var/lib/postgresql/data
chmod 700 /var/lib/postgresql/data

# 8. 启动 PostgreSQL（会自动进入恢复模式）
pg_ctl start -D /var/lib/postgresql/data

# 9. 检查恢复状态
# psql -c "SELECT pg_is_in_recovery();"
# 应返回 true（正在恢复中）

# 10. 验证数据是否正确
# psql -c "SELECT count(*) FROM important_table;"

# 11. 确认恢复完成后，提升为正常模式
# psql -c "SELECT pg_wal_replay_resume();"  -- 如果 pause 了
# 或者创建新的时间线：
# psql -c "SELECT pg_promote();"   -- PostgreSQL 12+
```

### 示例 7：物理备份自动化脚本

```bash
#!/bin/bash
# === PostgreSQL 物理备份 + WAL 归档管理脚本 ===

BACKUP_DIR="/data/backups/physical"
WAL_ARCHIVE="/data/wal_archive"
PG_HOST="localhost"
PG_PORT="5432"
PG_USER="repl_user"
RETENTION_DAYS=7
DATE=$(date +%Y%m%d_%H%M%S)
LOG="${BACKUP_DIR}/logs/backup_${DATE}.log"

mkdir -p "${BACKUP_DIR}/base" "${BACKUP_DIR}/logs"

log() {
    echo "[$(date '+%Y-%m-%d %H:%M:%S')] $1" | tee -a "${LOG}"
}

log "=== 开始物理备份 ==="

# 1. 执行基础备份
BACKUP_PATH="${BACKUP_DIR}/base/backup_${DATE}"
log "执行 pg_basebackup 到 ${BACKUP_PATH}..."

pg_basebackup \
    -h ${PG_HOST} \
    -p ${PG_PORT} \
    -U ${PG_USER} \
    -D "${BACKUP_PATH}" \
    -Ft -z \
    -Xs \
    -P \
    --checkpoint=fast \
    --label="auto_backup_${DATE}" \
    2>> "${LOG}"

if [ $? -eq 0 ]; then
    SIZE=$(du -sh "${BACKUP_PATH}" | cut -f1)
    log "基础备份完成，大小: ${SIZE}"
else
    log "错误：基础备份失败！"
    exit 1
fi

# 2. 清理旧的基础备份
log "清理 ${RETENTION_DAYS} 天前的备份..."
find "${BACKUP_DIR}/base" -maxdepth 1 -type d -name "backup_*" \
    -mtime +${RETENTION_DAYS} -exec rm -rf {} +

# 3. 清理旧的 WAL 归档（保留比最老备份更新的）
OLDEST_BACKUP=$(ls -1d ${BACKUP_DIR}/base/backup_* 2>/dev/null | head -1)
if [ -n "${OLDEST_BACKUP}" ]; then
    log "保留 ${OLDEST_BACKUP} 之后的 WAL 归档"
    # 使用 pg_archivecleanup 清理不需要的 WAL 文件
    if [ -f "${OLDEST_BACKUP}/backup_label" ]; then
        START_WAL=$(grep "START WAL LOCATION" "${OLDEST_BACKUP}/backup_label" \
            | awk '{print $6}' | tr -d ')')
        if [ -n "${START_WAL}" ]; then
            pg_archivecleanup "${WAL_ARCHIVE}" "${START_WAL}"
            log "WAL 归档清理完成"
        fi
    fi
fi

# 4. 统计归档状态
WAL_COUNT=$(ls -1 ${WAL_ARCHIVE}/ 2>/dev/null | wc -l)
WAL_SIZE=$(du -sh ${WAL_ARCHIVE}/ 2>/dev/null | cut -f1)
log "当前 WAL 归档：${WAL_COUNT} 个文件，总大小 ${WAL_SIZE}"

log "=== 备份完成 ==="
```

## 常见问题

### Q1：pg_basebackup 和 pg_dump 应该如何选择？

**A：** 根据场景选择：

| 场景 | 推荐工具 |
|------|----------|
| 需要 PITR 能力 | pg_basebackup + WAL 归档 |
| 跨版本迁移 | pg_dump |
| 只备份特定表或 Schema | pg_dump |
| 大数据量快速备份 | pg_basebackup |
| 需要最大灵活性 | 两者结合 |

最佳实践是同时使用两种方式：
- pg_basebackup + WAL 归档作为主要备份方案（支持 PITR）
- pg_dump 作为补充（跨版本兼容、选择性恢复）

### Q2：WAL 归档失败会影响数据库运行吗？

**A：** 会的。如果 `archive_command` 执行失败（返回非零退出码），PostgreSQL 会：

1. 保留未归档的 WAL 文件，不会删除
2. 周期性地重试归档命令
3. 如果 WAL 文件持续累积，可能耗尽 `pg_wal` 目录的磁盘空间
4. 磁盘满时 PostgreSQL 会停止接受新事务

监控建议：
```sql
-- 监控归档失败
SELECT archived_count, failed_count,
       last_failed_wal, last_failed_time
FROM pg_stat_archiver;
```

确保 `archive_command` 可靠：使用 `test ! -f` 避免覆盖，使用充足的归档存储空间。

### Q3：PITR 恢复后可以继续正常使用吗？需要注意什么？

**A：** 可以正常使用，但需要注意：

1. **新时间线**：PITR 恢复后会创建新的时间线（Timeline），WAL 文件名包含时间线 ID。后续的 WAL 归档使用新时间线，不会与旧的冲突。

2. **流复制重建**：如果有备库，恢复后需要重建所有备库（因为时间线变了）。

3. **逻辑复制**：逻辑复制槽在 PITR 后可能无效，需要重新创建。

4. **恢复后建议操作**：
```sql
-- 确认恢复完成
SELECT pg_is_in_recovery();  -- 应返回 false

-- 检查时间线
SELECT timeline_id FROM pg_control_checkpoint();

-- 立即做一次完整备份
-- pg_basebackup ...

-- 确保 WAL 归档正常
SELECT * FROM pg_stat_archiver;
```
