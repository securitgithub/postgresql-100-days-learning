- 了解 postgresql.conf 的整体结构与参数分类
- 深入理解 shared_buffers 的作用与调优方法
- 掌握 work_mem 对查询性能的影响及合理设置

## 理论讲解

### 1. postgresql.conf 概览

`postgresql.conf` 是 PostgreSQL 的主配置文件，包含数百个参数，主要分为以下类别：

| 类别 | 说明 | 示例参数 |
|------|------|----------|
| 连接与认证 | 监听地址、最大连接数 | listen_addresses, max_connections |
| 内存 | 共享缓冲区、工作内存 | shared_buffers, work_mem |
| WAL | 预写日志配置 | wal_level, max_wal_size |
| 查询优化 | 规划器参数 | effective_cache_size, random_page_cost |
| 日志 | 日志级别与格式 | log_statement, log_min_duration_statement |
| 自动清理 | autovacuum 配置 | autovacuum_vacuum_threshold |
| 复制 | 流复制参数 | max_wal_senders, hot_standby |

参数的生效方式：
- **无需重启**（reload 即可）：大部分参数
- **需要重启**：shared_buffers、max_connections、wal_level 等
- **会话级可修改**：work_mem、statement_timeout 等

### 2. shared_buffers 深入

`shared_buffers` 是 PostgreSQL 最重要的内存参数，定义了数据库用于缓存数据页的共享内存区域大小。

**工作原理：**
- PostgreSQL 读取数据时，先检查 shared_buffers
- 如果数据在缓冲区中（缓存命中），直接返回
- 如果不在（缓存未命中），从磁盘读取并放入缓冲区
- 修改的数据页先在缓冲区中修改（脏页），后续由后台进程写入磁盘

**调优建议：**
- 推荐设置为系统总内存的 **25%**
- 不建议超过系统总内存的 **40%**（操作系统也需要内存用于文件系统缓存）
- 对于小于 1GB 内存的系统，建议设为 128MB ~ 256MB
- 需要重启 PostgreSQL 才能生效

### 3. work_mem 深入

`work_mem` 定义了**每个排序或哈希操作**可使用的内存量。

**关键理解：**
- 这不是每个连接的总内存，而是每个**操作**的内存
- 一个复杂查询可能同时有多个排序/哈希操作，每个都分配 work_mem
- 总内存使用 ≈ work_mem × 并发连接数 × 每个查询的操作数

**影响的操作：**
- ORDER BY 排序
- DISTINCT
- Hash Join
- Hash Aggregate
- 位图扫描中的位图创建

**调优建议：**
- 默认值 4MB 通常偏小
- 总内存 / max_connections / 4 是一个参考公式
- 对于分析型查询，可以在会话级别临时增大
- 设置过大可能导致 OOM（内存耗尽）

### 4. maintenance_work_mem

`maintenance_work_mem` 定义了维护操作可使用的内存量。

**影响的操作：**
- VACUUM
- CREATE INDEX
- ALTER TABLE ADD FOREIGN KEY
- CLUSTER
- REINDEX

**调优建议：**
- 默认 64MB，通常偏小
- 建议设为系统总内存的 5%~10%，或 512MB ~ 2GB
- 维护操作通常不会大量并发，因此可以设置得较大
- 对 VACUUM 而言，还受 `autovacuum_work_mem` 单独控制

### 5. effective_cache_size

`effective_cache_size` 是一个**估算值**，告诉查询规划器操作系统和 PostgreSQL 可用于缓存的总内存量。

**关键理解：**
- 这个参数**不分配任何内存**，只是给规划器的提示
- 影响规划器对索引扫描 vs 顺序扫描的决策
- 值越大，规划器越倾向于使用索引扫描（因为假设数据在缓存中）

**调优建议：**
- 设置为系统总内存的 **50%~75%**
- 计算方式：shared_buffers + 操作系统可用于文件缓存的内存

### 6. wal_buffers

`wal_buffers` 定义了 WAL（预写日志）数据在写入磁盘前的内存缓冲区大小。

**调优建议：**
- 默认值 `-1` 表示自动计算（shared_buffers 的 1/32，最小 64KB，最大 16MB）
- 对大多数场景，默认的自动计算已足够
- 对于大量并发写入的场景，可以手动设为 16MB ~ 64MB
- 需要重启才能生效

## 代码示例

### 示例 1：查看和修改参数

```sql
-- 查看当前参数值
SHOW shared_buffers;
SHOW work_mem;
SHOW maintenance_work_mem;
SHOW effective_cache_size;
SHOW wal_buffers;

-- 查看所有参数及其来源
SELECT name, setting, unit, source, context
FROM pg_settings
WHERE name IN (
    'shared_buffers', 'work_mem', 'maintenance_work_mem',
    'effective_cache_size', 'wal_buffers'
);

-- context 字段说明：
-- postmaster: 需要重启
-- sighup: reload 即可
-- user: 会话级可修改
-- superuser: 超级用户可在会话级修改

-- 查看参数的详细信息
SELECT name, setting, unit, min_val, max_val,
       boot_val, reset_val, context, short_desc
FROM pg_settings
WHERE name = 'shared_buffers';
```

### 示例 2：内存参数推荐配置

```sql
-- 假设服务器配置：32GB 内存，专用数据库服务器

-- postgresql.conf 配置示例：

-- 共享缓冲区：25% 的系统内存
-- shared_buffers = '8GB'

-- 工作内存：根据连接数和查询复杂度
-- 假设 max_connections = 200
-- work_mem = '32MB'  -- 32GB / 200 / 4 ≈ 40MB，保守取 32MB

-- 维护工作内存
-- maintenance_work_mem = '2GB'

-- 有效缓存大小：75% 的系统内存
-- effective_cache_size = '24GB'

-- WAL 缓冲区
-- wal_buffers = '64MB'

-- 在 SQL 中检查是否需要重启
SELECT name, setting, pending_restart
FROM pg_settings
WHERE pending_restart = true;
```

### 示例 3：会话级 work_mem 调整

```sql
-- 查看当前 work_mem
SHOW work_mem;  -- 假设全局设置为 32MB

-- 对于需要大量排序的分析查询，临时增大 work_mem
SET work_mem = '256MB';

-- 执行复杂查询
EXPLAIN (ANALYZE, BUFFERS)
SELECT department, COUNT(*), AVG(salary)
FROM employees
GROUP BY department
ORDER BY AVG(salary) DESC;

-- 查看查询是否使用了内存排序而非磁盘排序
-- Sort Method: quicksort  Memory: xxxkB  -- 内存排序（好）
-- Sort Method: external merge  Disk: xxxkB  -- 磁盘排序（需要更大 work_mem）

-- 恢复默认值
RESET work_mem;

-- 也可以为特定角色设置
ALTER ROLE analyst SET work_mem = '128MB';
ALTER ROLE web_app SET work_mem = '16MB';
```

### 示例 4：shared_buffers 命中率分析

```sql
-- 查看缓冲区命中率
SELECT
    sum(blks_hit) AS buffer_hits,
    sum(blks_read) AS disk_reads,
    CASE
        WHEN sum(blks_hit) + sum(blks_read) = 0 THEN 0
        ELSE round(sum(blks_hit)::numeric /
             (sum(blks_hit) + sum(blks_read)) * 100, 2)
    END AS hit_ratio_pct
FROM pg_stat_database;

-- 理想的命中率应该在 99% 以上
-- 如果低于 95%，考虑增大 shared_buffers

-- 查看每个数据库的命中率
SELECT datname,
    blks_hit,
    blks_read,
    CASE WHEN blks_hit + blks_read = 0 THEN 0
         ELSE round(blks_hit::numeric / (blks_hit + blks_read) * 100, 2)
    END AS hit_ratio_pct
FROM pg_stat_database
WHERE datname NOT LIKE 'template%'
ORDER BY hit_ratio_pct;

-- 查看特定表的缓冲区使用情况
SELECT relname,
    heap_blks_read,
    heap_blks_hit,
    CASE WHEN heap_blks_hit + heap_blks_read = 0 THEN 0
         ELSE round(heap_blks_hit::numeric /
              (heap_blks_hit + heap_blks_read) * 100, 2)
    END AS hit_ratio_pct
FROM pg_statio_user_tables
ORDER BY heap_blks_read DESC
LIMIT 20;
```

### 示例 5：work_mem 使用监控

```sql
-- 查看临时文件使用情况（work_mem 不足时会写临时文件）
SELECT datname, temp_files, temp_bytes,
       pg_size_pretty(temp_bytes) AS temp_size
FROM pg_stat_database
WHERE temp_files > 0;

-- 启用临时文件日志记录
-- 在 postgresql.conf 中设置：
-- log_temp_files = 0  -- 记录所有临时文件（0 表示记录所有大小）

-- 查看当前所有连接的内存使用
SELECT pid, usename, application_name,
       state, query
FROM pg_stat_activity
WHERE state != 'idle'
ORDER BY pid;

-- 使用 pg_stat_statements 找出需要大量排序的查询
-- 需要先安装 pg_stat_statements 扩展
-- CREATE EXTENSION pg_stat_statements;
SELECT query, calls, total_exec_time,
       rows, temp_blks_read, temp_blks_written
FROM pg_stat_statements
WHERE temp_blks_written > 0
ORDER BY temp_blks_written DESC
LIMIT 10;
```

### 示例 6：不同场景的配置模板

```sql
-- === Web 应用场景（16GB 内存，高并发短查询） ===
-- shared_buffers = '4GB'
-- work_mem = '8MB'
-- maintenance_work_mem = '1GB'
-- effective_cache_size = '12GB'
-- wal_buffers = '16MB'
-- max_connections = 300

-- === 数据分析场景（64GB 内存，低并发复杂查询） ===
-- shared_buffers = '16GB'
-- work_mem = '512MB'
-- maintenance_work_mem = '4GB'
-- effective_cache_size = '48GB'
-- wal_buffers = '64MB'
-- max_connections = 30

-- === 混合场景（32GB 内存） ===
-- shared_buffers = '8GB'
-- work_mem = '32MB'          -- 会话级按需调大
-- maintenance_work_mem = '2GB'
-- effective_cache_size = '24GB'
-- wal_buffers = '32MB'
-- max_connections = 200

-- 根据系统内存动态计算推荐值
SELECT
    pg_size_pretty(os_mem * 0.25) AS recommended_shared_buffers,
    pg_size_pretty(os_mem * 0.05) AS recommended_maintenance_work_mem,
    pg_size_pretty(os_mem * 0.75) AS recommended_effective_cache_size
FROM (
    -- 这里需要替换为实际的系统内存大小（字节）
    SELECT 32::bigint * 1024 * 1024 * 1024 AS os_mem
) t;
```

## 常见问题

### Q1：shared_buffers 设得越大越好吗？

**A：** 不是。设置过大会带来以下问题：

1. **挤占操作系统文件缓存**：PostgreSQL 依赖操作系统的页缓存来加速磁盘 I/O。shared_buffers 过大会减少操作系统可用的缓存内存
2. **双重缓存**：数据可能同时存在于 shared_buffers 和操作系统缓存中，造成浪费
3. **检查点压力增大**：更大的 shared_buffers 意味着更多的脏页需要在检查点时写入

经验法则：
- 专用服务器：系统内存的 25%
- 共享服务器：系统内存的 15%
- 超过 40% 通常没有更好的效果，反而可能性能下降

### Q2：work_mem 导致 OOM 怎么排查和解决？

**A：** work_mem 导致内存问题的排查步骤：

1. **计算最大内存消耗**：`max_connections × work_mem × 每个查询的操作数`
2. **检查临时文件日志**：设置 `log_temp_files = 0` 找出需要大内存的查询
3. **使用分层策略**：
```sql
-- 全局保守设置
-- work_mem = '8MB'

-- 对需要的角色单独设置
ALTER ROLE analyst SET work_mem = '256MB';

-- 对需要的查询临时设置
SET work_mem = '512MB';
-- 执行查询
RESET work_mem;
```
4. **优化查询**：减少排序和哈希操作比增大 work_mem 更有效

### Q3：如何判断 effective_cache_size 是否设置正确？

**A：** `effective_cache_size` 的设置可以通过以下方式验证：

1. 查看操作系统的可用缓存：
```bash
# Linux 下查看可用内存（包含文件缓存）
free -h
# 关注 "available" 或 "buff/cache" 列
```

2. 公式：`effective_cache_size ≈ shared_buffers + OS 可用文件缓存`

3. 验证方法：对比相同查询在不同 `effective_cache_size` 下的执行计划。如果设置合理，规划器应该在数据量大时选择索引扫描而非顺序扫描。
