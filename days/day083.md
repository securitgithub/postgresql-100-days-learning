# Day 083: 逻辑复制 — 灵活的数据同步方案

## 学习目标

- 理解逻辑复制与物理复制的核心区别
- 掌握 `CREATE PUBLICATION` 和 `CREATE SUBSCRIPTION` 的使用方法
- 了解逻辑复制的限制与注意事项
- 学会利用逻辑复制实现跨版本升级和部分表同步
- 掌握逻辑复制的监控与故障排查

## 理论讲解

### 1. 逻辑复制 vs 物理复制

| 特性 | 物理复制（流复制） | 逻辑复制 |
|------|-------------------|----------|
| 复制单位 | WAL 字节流（整个集群） | 逻辑变更（行级别） |
| 粒度 | 整个数据库集群 | 可选择特定表 |
| 从库写入 | 不支持（只读） | 支持（可读写） |
| 跨版本 | 不支持 | 支持 |
| DDL 复制 | 自动（WAL包含DDL） | 不支持（需手动同步） |
| 序列复制 | 自动 | 不支持 |
| 性能开销 | 较低 | 较高（需解码WAL） |
| 适用场景 | 高可用/灾备 | 数据分发/迁移/升级 |

### 2. 逻辑复制的工作原理

逻辑复制基于**发布-订阅**（Publish-Subscribe）模型：

```
发布端 (Publisher)                    订阅端 (Subscriber)
┌──────────────────┐                ┌──────────────────┐
│  WAL 日志        │                │                  │
│    ↓              │                │                  │
│  逻辑解码        │   逻辑变更流    │  逻辑复制 Worker │
│  (wal_level =    │ ─────────────▶ │    ↓              │
│   logical)       │                │  应用变更到目标表 │
│    ↓              │                │                  │
│  Publication      │                │  Subscription    │
└──────────────────┘                └──────────────────┘
```

工作流程：
1. 发布端将 WAL 记录通过逻辑解码插件（pgoutput）转换为逻辑变更事件
2. 逻辑变更以行级别的 INSERT/UPDATE/DELETE 操作形式发送
3. 订阅端接收并应用这些变更到对应的目标表

### 3. 逻辑复制的限制

- **不复制 DDL**：表结构变更（ALTER TABLE等）不会自动同步
- **不复制序列值**：序列（SEQUENCE）的值不会同步
- **不复制大对象**：Large Object 不支持
- **需要主键或 REPLICA IDENTITY**：UPDATE/DELETE 操作需要唯一标识行
- **初始快照**：订阅创建时会做全量数据拷贝（可配置跳过）
- **冲突处理**：如果订阅端有冲突数据，复制会中断

### 4. REPLICA IDENTITY

逻辑复制需要能唯一标识每一行才能正确执行 UPDATE 和 DELETE：

- **DEFAULT**：使用主键（如果存在）
- **USING INDEX**：使用指定的唯一索引
- **FULL**：使用整行数据（性能较差但最灵活）
- **NOTHING**：不记录旧值（UPDATE/DELETE 无法复制）

### 5. 典型应用场景

- **跨版本升级**：PG 14 → PG 16 的在线滚动升级
- **部分表同步**：只同步部分表到报表库
- **多主写入**：多个数据库各自写入不同的表
- **数据分发**：将数据从中心库分发到多个下游系统
- **数据聚合**：将多个库的数据汇聚到一个分析库

## 代码示例

### 示例1：基础逻辑复制配置

```sql
-- ========== 发布端（Publisher）配置 ==========

-- 1. 确保 wal_level 设置为 logical
ALTER SYSTEM SET wal_level = 'logical';
-- 需要重启生效

-- 2. 创建示例表
CREATE TABLE products (
    id SERIAL PRIMARY KEY,
    name VARCHAR(100) NOT NULL,
    price NUMERIC(10,2),
    category VARCHAR(50),
    updated_at TIMESTAMP DEFAULT now()
);

CREATE TABLE orders (
    id SERIAL PRIMARY KEY,
    product_id INT REFERENCES products(id),
    quantity INT,
    total NUMERIC(10,2),
    order_date TIMESTAMP DEFAULT now()
);

-- 3. 插入测试数据
INSERT INTO products (name, price, category) VALUES
    ('笔记本电脑', 6999.00, '电子产品'),
    ('机械键盘', 599.00, '外设'),
    ('显示器', 2499.00, '外设');

-- 4. 创建发布（发布所有表）
CREATE PUBLICATION my_pub FOR ALL TABLES;

-- 或者只发布指定表
CREATE PUBLICATION products_pub FOR TABLE products;

-- 或者发布特定表并带过滤条件（PG 15+）
CREATE PUBLICATION expensive_pub
    FOR TABLE products WHERE (price > 1000);
```

### 示例2：订阅端配置

```sql
-- ========== 订阅端（Subscriber）配置 ==========

-- 1. 在订阅端创建相同结构的表（DDL 不会自动同步）
CREATE TABLE products (
    id SERIAL PRIMARY KEY,
    name VARCHAR(100) NOT NULL,
    price NUMERIC(10,2),
    category VARCHAR(50),
    updated_at TIMESTAMP DEFAULT now()
);

CREATE TABLE orders (
    id SERIAL PRIMARY KEY,
    product_id INT REFERENCES products(id),
    quantity INT,
    total NUMERIC(10,2),
    order_date TIMESTAMP DEFAULT now()
);

-- 2. 创建订阅
CREATE SUBSCRIPTION my_sub
    CONNECTION 'host=192.168.1.100 port=5432 dbname=mydb user=replicator password=SecureP@ss'
    PUBLICATION my_pub;

-- 订阅创建后会自动：
--   a) 做初始数据快照（全量同步现有数据）
--   b) 开始实时同步增量变更

-- 3. 检查订阅状态
SELECT * FROM pg_stat_subscription;
```

### 示例3：发布管理操作

```sql
-- 查看所有发布
SELECT * FROM pg_publication;

-- 查看发布包含的表
SELECT * FROM pg_publication_tables WHERE pubname = 'my_pub';

-- 向发布添加表
ALTER PUBLICATION my_pub ADD TABLE new_table;

-- 从发布移除表
ALTER PUBLICATION my_pub DROP TABLE old_table;

-- 修改发布选项（控制哪些操作被发布）
ALTER PUBLICATION my_pub
    SET (publish = 'insert, update, delete');
-- 可选值: insert, update, delete, truncate

-- 删除发布
DROP PUBLICATION my_pub;
```

### 示例4：订阅管理操作

```sql
-- 查看所有订阅
SELECT * FROM pg_subscription;

-- 查看订阅的详细状态
SELECT
    subname,
    pid,
    relid::regclass AS table_name,
    received_lsn,
    last_msg_send_time,
    last_msg_receipt_time,
    latest_end_lsn
FROM pg_stat_subscription;

-- 查看每个表的同步状态
SELECT
    srsubid,
    srrelid::regclass AS table_name,
    srsublsn,
    srsubstate  -- i=初始化, d=数据拷贝, s=已同步, r=正常复制
FROM pg_subscription_rel;

-- 临时禁用订阅（不删除）
ALTER SUBSCRIPTION my_sub DISABLE;

-- 重新启用订阅
ALTER SUBSCRIPTION my_sub ENABLE;

-- 刷新发布端的表列表（发布端添加新表后）
ALTER SUBSCRIPTION my_sub REFRESH PUBLICATION;

-- 删除订阅
DROP SUBSCRIPTION my_sub;
```

### 示例5：REPLICA IDENTITY 配置

```sql
-- 查看当前表的 REPLICA IDENTITY
SELECT relname, relreplident
FROM pg_class
WHERE relname IN ('products', 'orders');
-- d = default (主键), n = nothing, f = full, i = index

-- 对没有主键的表设置 REPLICA IDENTITY
CREATE TABLE logs (
    ts TIMESTAMP,
    level VARCHAR(10),
    message TEXT
);

-- 方法1：使用 FULL（所有列作为标识，性能较差）
ALTER TABLE logs REPLICA IDENTITY FULL;

-- 方法2：创建唯一索引后指定
CREATE UNIQUE INDEX logs_ts_idx ON logs (ts);
ALTER TABLE logs REPLICA IDENTITY USING INDEX logs_ts_idx;
```

### 示例6：跨版本升级场景

```sql
-- ===== 使用逻辑复制实现 PG 14 → PG 16 滚动升级 =====

-- 步骤1: 在 PG 14 (源库) 创建发布
-- [PG 14]
CREATE PUBLICATION upgrade_pub FOR ALL TABLES;

-- 步骤2: 在 PG 16 (目标库) 创建表结构
-- [PG 16]
-- 使用 pg_dump --schema-only 导出结构并在新库执行
-- pg_dump -h old_host -p 5432 -d mydb --schema-only | psql -h new_host -p 5432 -d mydb

-- 步骤3: 在 PG 16 创建订阅
-- [PG 16]
CREATE SUBSCRIPTION upgrade_sub
    CONNECTION 'host=old_host port=5432 dbname=mydb user=replicator password=xxx'
    PUBLICATION upgrade_pub;

-- 步骤4: 等待初始同步完成，监控状态
-- [PG 16]
SELECT * FROM pg_stat_subscription;
SELECT * FROM pg_subscription_rel;  -- 等所有表状态变为 'r'

-- 步骤5: 确认数据一致后，切换应用连接到新库
-- 步骤6: 在新库删除订阅
-- [PG 16]
DROP SUBSCRIPTION upgrade_sub;

-- 步骤7: 修复序列值（逻辑复制不同步序列）
-- [PG 16]
SELECT setval('products_id_seq', (SELECT MAX(id) FROM products));
SELECT setval('orders_id_seq', (SELECT MAX(id) FROM orders));
```

### 示例7：冲突处理

```sql
-- 当订阅端出现冲突时（如主键重复），复制会中断
-- 查看复制错误
SELECT * FROM pg_stat_subscription;

-- 查看日志中的错误信息
-- LOG: logical replication apply worker for subscription "my_sub" has started
-- ERROR: duplicate key value violates unique constraint "products_pkey"

-- 解决方法1: 删除冲突数据后让复制自动恢复
DELETE FROM products WHERE id = 999;  -- 删除冲突行

-- 解决方法2: 跳过有问题的事务（谨慎使用）
-- 先找到当前订阅的 LSN
SELECT * FROM pg_stat_subscription;

-- 通过 ALTER SUBSCRIPTION 跳过
ALTER SUBSCRIPTION my_sub
    SKIP (lsn = '0/12345678');  -- PG 15+

-- 解决方法3: 禁用订阅，手动修复后重新启用
ALTER SUBSCRIPTION my_sub DISABLE;
-- ... 修复数据 ...
ALTER SUBSCRIPTION my_sub ENABLE;
```

## 动手练习

### 练习1：创建发布和订阅

在同一台机器上用两个不同的数据库模拟发布-订阅：

```sql
-- 创建发布端数据库
CREATE DATABASE pub_db;

-- 创建订阅端数据库
CREATE DATABASE sub_db;

-- 在 pub_db 中创建表和发布
\c pub_db
CREATE TABLE users (id serial PRIMARY KEY, name text, email text);
INSERT INTO users (name, email) VALUES ('张三', 'zhangsan@example.com');
CREATE PUBLICATION user_pub FOR TABLE users;

-- 在 sub_db 中创建表和订阅
\c sub_db
CREATE TABLE users (id serial PRIMARY KEY, name text, email text);
CREATE SUBSCRIPTION user_sub
    CONNECTION 'host=localhost port=5432 dbname=pub_db'
    PUBLICATION user_pub;

-- 验证数据同步
SELECT * FROM users;
```

### 练习2：测试行过滤发布（PG 15+）

```sql
-- 在发布端创建带过滤条件的发布
CREATE PUBLICATION vip_pub
    FOR TABLE users WHERE (name LIKE 'VIP%');

-- 在订阅端创建对应订阅并观察哪些行被同步
```

### 练习3：模拟 DDL 变更同步

```sql
-- 在发布端添加列
ALTER TABLE users ADD COLUMN phone VARCHAR(20);

-- 观察订阅端是否自动同步了这个变更
-- （应该没有，需要手动在订阅端执行相同的 DDL）

```

### 练习4：监控逻辑复制状态

```sql
-- 在发布端查看复制槽
SELECT * FROM pg_replication_slots WHERE slot_type = 'logical';

-- 在订阅端查看订阅状态
SELECT
    subname,
    received_lsn,
    latest_end_lsn,
    last_msg_send_time,
    last_msg_receipt_time
FROM pg_stat_subscription;
```

## 常见问题

### Q1: 逻辑复制能复制 DDL 吗？

**A:** 不能。逻辑复制只复制 DML 操作（INSERT/UPDATE/DELETE/TRUNCATE）。所有的表结构变更（CREATE TABLE、ALTER TABLE、CREATE INDEX 等）都需要手动在订阅端执行。在做 DDL 变更时，建议先在订阅端执行（因为新增列有默认值），然后再在发布端执行。一些第三方工具（如 pglogical）提供了 DDL 复制的扩展功能。

### Q2: 逻辑复制初始同步很慢怎么办？

**A:** 初始快照（initial data copy）对大表可能很耗时。优化方法：(1) 创建订阅时指定 `copy_data = false` 跳过初始拷贝（需自行确保数据一致）；(2) 先用 `pg_dump/pg_restore` 做全量同步，再创建订阅从特定 LSN 开始；(3) 增大 `wal_sender_timeout` 和 `wal_receiver_timeout` 避免超时。

### Q3: 逻辑复制和流复制可以同时使用吗？

**A:** 可以。一个 PostgreSQL 实例可以同时作为流复制的主库和逻辑复制的发布端。只需要将 `wal_level` 设置为 `logical`（这是最高级别，兼容 `replica` 级别的所有功能）。常见的架构是：使用流复制做高可用（主从切换），同时使用逻辑复制向分析库同步数据。
