## 理论讲解

### 1. 为什么在 Kubernetes 上运行 PostgreSQL

随着云原生架构的普及，越来越多的组织希望将数据库也纳入 Kubernetes 统一管理：

- **统一运维**：应用和数据库使用相同的部署、监控、日志体系
- **自动化**：Operator 自动处理备份、Failover、扩缩容
- **声明式管理**：通过 YAML 文件定义期望状态，Operator 负责实现
- **弹性调度**：利用 K8s 的调度能力优化资源使用

### 2. Kubernetes Operator 模式

Operator 是 Kubernetes 的扩展模式，将运维知识编码为软件：

```
用户定义 CR (Custom Resource)
    │
    ▼
┌─────────────┐     Watch      ┌─────────────────┐
│  Kubernetes  │ ─────────────▶ │   Operator       │
│  API Server  │                │  (控制器逻辑)     │
└─────────────┘                │                   │
                               │  - 创建 Pod       │
    Reconcile                  │  - 配置复制        │
    ◀──────────────────────────│  - 执行备份        │
                               │  - 处理 Failover  │
                               └─────────────────┘
```

Operator 持续监听 CR 的变更，通过 Reconcile 循环将实际状态调和为期望状态。

| 特性 | CloudNativePG | Zalando Operator | CrunchyData PGO |

### 4. CloudNativePG（CNPG）架构

CloudNativePG 是目前最受推荐的 Kubernetes PostgreSQL Operator（CNCF Sandbox 项目）：

```
┌──────────────────────────────────────────┐
│            Kubernetes Cluster             │
│                                          │
│  ┌──────────────────┐                    │
│  │ CNPG Operator    │                    │
│  │ (Deployment)     │                    │
│  └────────┬─────────┘                    │
│           │ 管理                          │
│  ┌────────▼─────────────────────────┐    │
│  │       Cluster CR                  │    │
│  │  ┌─────────┐ ┌────────┐ ┌──────┐│    │
│  │  │ Primary  │ │Replica1│ │Repl.2││    │
│  │  │ (Pod)    │ │ (Pod)  │ │(Pod) ││    │
│  │  │  PG 16   │ │ PG 16  │ │PG 16 ││    │
│  │  │  PVC     │ │ PVC    │ │ PVC  ││    │
│  │  └─────────┘ └────────┘ └──────┘│    │
│  └──────────────────────────────────┘    │
│                                          │
│  ┌──────────┐  ┌───────────┐             │
│  │ Service  │  │ Service   │             │
│  │ (RW)     │  │ (RO)      │             │
│  │ → Primary│  │ → Replicas│             │
│  └──────────┘  └───────────┘             │
└──────────────────────────────────────────┘
```

核心特点：
- 每个 Pod 运行一个 PostgreSQL 实例（非 sidecar 模式）
- 使用原生流复制，不依赖 Patroni
- 自动 Failover（通过 fencing 和 promotion）
- 支持 Barman Cloud 进行持续归档和备份
- 支持滚动更新和在线扩缩容

### 5. 有状态服务注意事项

在 Kubernetes 上运行数据库面临特殊挑战：

- **存储**：必须使用 PersistentVolume（PV），推荐使用高性能存储类（如 SSD）
- **网络**：Pod IP 会变化，需要稳定的 Service 端点
- **调度**：需要反亲和性（anti-affinity）确保副本分布在不同节点
- **资源**：CPU/内存必须设置 requests 和 limits
- **备份**：不能依赖节点本地存储，需要对象存储（S3、GCS 等）
- **升级**：需要滚动升级策略，避免同时重启所有实例

## 代码示例

### 示例1：安装 CloudNativePG Operator

```bash
# 方法1：使用 kubectl 直接安装
kubectl apply --server-side -f \
  https://raw.githubusercontent.com/cloudnative-pg/cloudnative-pg/release-1.22/releases/cnpg-1.22.1.yaml

# 方法2：使用 Helm 安装（推荐）
helm repo add cnpg https://cloudnative-pg.github.io/charts
helm repo update
helm install cnpg cnpg/cloudnative-pg \
  --namespace cnpg-system \
  --create-namespace

# 验证 Operator 运行状态
kubectl get pods -n cnpg-system
kubectl get deploy -n cnpg-system

# 检查 CRD 是否已安装
kubectl get crd | grep cnpg
# 应该看到 clusters.postgresql.cnpg.io
```

### 示例2：CNPG 创建 PostgreSQL 集群

```yaml
# pg-cluster.yaml — CloudNativePG 集群定义
apiVersion: postgresql.cnpg.io/v1
kind: Cluster
metadata:
  name: pg-production
  namespace: database
spec:
  # 实例数量（1主 + 2从 = 3个实例）
  instances: 3

  # PostgreSQL 版本
  imageName: ghcr.io/cloudnative-pg/postgresql:16.2

  # PostgreSQL 配置
  postgresql:
    parameters:
      shared_buffers: "512MB"
      effective_cache_size: "1536MB"
      work_mem: "8MB"
      maintenance_work_mem: "128MB"
      max_connections: "200"
      log_statement: "ddl"
      log_min_duration_statement: "1000"
      shared_preload_libraries: "pg_stat_statements"
      pg_stat_statements.track: "all"
    pg_hba:
      - host all all 10.0.0.0/8 scram-sha-256
      - host all all 172.16.0.0/12 scram-sha-256

  # 存储配置
  storage:
    size: 50Gi
    storageClass: fast-ssd    # 使用高性能存储类

  # 资源限制
  resources:
    requests:
      memory: "1Gi"
      cpu: "1"
    limits:
      memory: "2Gi"
      cpu: "2"

  # 反亲和性：确保 Pod 分布在不同节点
  affinity:
    topologyKey: kubernetes.io/hostname

  # 超级用户密码
  superuserSecret:
    name: pg-superuser-secret

  # 备份配置（持续归档到 S3）
  backup:
    barmanObjectStore:
      destinationPath: s3://my-pg-backups/production/
      s3Credentials:
        accessKeyId:
          name: aws-s3-creds
          key: ACCESS_KEY_ID
        secretAccessKey:
          name: aws-s3-creds
          key: SECRET_ACCESS_KEY
      wal:
        compression: gzip
      data:
        compression: gzip
    retentionPolicy: "30d"

  # 启用监控
  monitoring:
    enablePodMonitor: true
```

```bash
# 创建命名空间和密钥
kubectl create namespace database

kubectl create secret generic pg-superuser-secret \
  --namespace database \
  --from-literal=username=postgres \
  --from-literal=password='SuperSecureP@ss2024'

kubectl create secret generic aws-s3-creds \
  --namespace database \
  --from-literal=ACCESS_KEY_ID=AKIAXXXXXXX \
  --from-literal=SECRET_ACCESS_KEY=XXXXXXX

# 部署集群
kubectl apply -f pg-cluster.yaml

# 查看集群状态
kubectl get cluster -n database
kubectl describe cluster pg-production -n database

# 查看 Pod 状态
kubectl get pods -n database -l cnpg.io/cluster=pg-production

# 输出示例：
# NAME                READY   STATUS    ROLE      AGE
# pg-production-1     1/1     Running   primary   5m
# pg-production-2     1/1     Running   replica   4m
# pg-production-3     1/1     Running   replica   3m
```

### 示例4：CNPG 备份与恢复

```yaml
# scheduled-backup.yaml — 定时备份
apiVersion: postgresql.cnpg.io/v1
kind: ScheduledBackup
metadata:
  name: pg-daily-backup
  namespace: database
spec:
  schedule: "0 2 * * *"          # 每天凌晨2点
  cluster:
    name: pg-production
  backupOwnerReference: self
  method: barmanObjectStore
```

```yaml
# restore-cluster.yaml — 从备份恢复新集群
apiVersion: postgresql.cnpg.io/v1
kind: Cluster
metadata:
  name: pg-restored
  namespace: database
spec:
  instances: 3
  imageName: ghcr.io/cloudnative-pg/postgresql:16.2

  storage:
    size: 50Gi
    storageClass: fast-ssd

  # 从备份恢复
  bootstrap:
    recovery:
      source: pg-production
      # 可选：恢复到指定时间点（PITR）
      recoveryTarget:
        targetTime: "2024-06-15T10:30:00Z"

  externalClusters:
    - name: pg-production
      barmanObjectStore:
        destinationPath: s3://my-pg-backups/production/
        s3Credentials:
          accessKeyId:
            name: aws-s3-creds
            key: ACCESS_KEY_ID
          secretAccessKey:
            name: aws-s3-creds
            key: SECRET_ACCESS_KEY
```

```bash
# 创建定时备份
kubectl apply -f scheduled-backup.yaml

# 手动触发一次备份
kubectl cnpg backup pg-production -n database

# 查看备份列表
kubectl get backup -n database

# 从备份恢复
kubectl apply -f restore-cluster.yaml
```

### 示例5：Zalando Postgres Operator 部署

```bash
# 安装 Zalando Postgres Operator
helm repo add postgres-operator-charts https://opensource.zalando.com/postgres-operator/charts/postgres-operator
helm install postgres-operator postgres-operator-charts/postgres-operator \

  --create-namespace

# 安装 Operator UI（可选）
helm install postgres-operator-ui postgres-operator-charts/postgres-operator-ui \

```

```yaml
# zalando-cluster.yaml — Zalando Operator 集群定义
apiVersion: "acid.zalan.do/v1"
kind: postgresql
metadata:
  name: pg-team-myapp
  namespace: database
spec:
  teamId: "myteam"
  volume:
    size: 50Gi
    storageClass: fast-ssd
  numberOfInstances: 3

  postgresql:
    version: "16"
    parameters:
      shared_buffers: "512MB"
      max_connections: "200"
      log_statement: "ddl"

  users:
    myapp_owner:
      - superuser
      - createdb
    myapp_user: []
    myapp_readonly:
      - login

  databases:
    myapp: myapp_owner

  resources:
    requests:
      cpu: "1"
      memory: "1Gi"
    limits:
      cpu: "2"
      memory: "2Gi"

  patroni:
    ttl: 30
    loop_wait: 10
    retry_timeout: 10
    maximum_lag_on_failover: 33554432   # 32MB
    synchronous_mode: false
```

```bash
# 部署 Zalando 集群
kubectl apply -f zalando-cluster.yaml

# 查看集群状态
kubectl get postgresql -n database

# 获取自动生成的密码
kubectl get secret myapp-owner.pg-team-myapp.credentials.postgresql.acid.zalan.do \
  -n database -o jsonpath='{.data.password}' | base64 -d
```

### 示例6：应用连接 Kubernetes 中的 PostgreSQL

```yaml
# app-deployment.yaml — 连接 CNPG 集群的应用示例
apiVersion: apps/v1
kind: Deployment
metadata:
  name: myapp
  namespace: database
spec:
  replicas: 3
  selector:
    matchLabels:
      app: myapp
  template:
    metadata:
      labels:
        app: myapp
    spec:
      containers:
        - name: myapp
          image: myapp:latest
          env:
            # 读写连接（主库）
            - name: DATABASE_URL
              value: "postgresql://app_user:$(DB_PASSWORD)@pg-production-rw:5432/myapp"
            # 只读连接（从库）
            - name: DATABASE_READONLY_URL
              value: "postgresql://app_user:$(DB_PASSWORD)@pg-production-ro:5432/myapp"
            - name: DB_PASSWORD
              valueFrom:
                secretKeyRef:
                  name: pg-app-credentials
                  key: password
```

### 示例7：集群运维操作

```bash
# ===== CloudNativePG 运维命令 =====

# 查看集群详细状态
kubectl cnpg status pg-production -n database

# 手动 Switchover（主从切换）
kubectl cnpg promote pg-production pg-production-2 -n database

# 扩展副本数量（从3扩到5）
kubectl patch cluster pg-production -n database \
  --type merge -p '{"spec":{"instances":5}}'

# 缩减副本数量
kubectl patch cluster pg-production -n database \
  --type merge -p '{"spec":{"instances":3}}'

# 滚动重启（不影响服务）
kubectl cnpg restart pg-production -n database

# 查看 PostgreSQL 日志
kubectl logs pg-production-1 -n database

# 进入 psql 会话
kubectl cnpg psql pg-production -n database -- -c "SELECT version();"

# 修改 PostgreSQL 参数（自动滚动重启）
kubectl patch cluster pg-production -n database --type merge -p '
{
  "spec": {
    "postgresql": {
      "parameters": {
        "max_connections": "300"
      }
    }
  }
}'
```

## 常见问题

### Q1: Kubernetes 上运行数据库真的可靠吗？

**A:** 是的，前提是正确配置。关键要点：(1) 使用可靠的 PersistentVolume（推荐云厂商提供的 SSD 存储类）；(2) 配置 Pod 反亲和性确保副本分布在不同节点；(3) 设置合理的资源 requests/limits 防止资源争抢；(4) 使用 Operator 自动管理而非手工维护。许多大型公司（如 Zalando、EDB、GitLab）已在生产环境大规模使用 K8s 上的 PostgreSQL。

### Q2: CloudNativePG 和 Zalando Operator 选哪个？

**A:** CloudNativePG（CNPG）是 CNCF Sandbox 项目，设计更现代，不依赖 Patroni（使用原生的 fencing 和 promotion 机制），配置更简洁。Zalando Operator 基于 Patroni/Spilo，更成熟，社区经验更丰富。如果是新项目，推荐 CNPG；如果已有 Patroni 经验或需要特定的 Spilo 功能，可选择 Zalando Operator。

### Q3: PV 数据丢失怎么办？

**A:** 这就是为什么备份至关重要。K8s 的 PV 依赖底层存储系统，如果存储故障导致 PV 数据丢失，需要从备份恢复。最佳实践：(1) 配置持续 WAL 归档到对象存储（S3/GCS/MinIO）；(2) 定期执行全量备份；(3) 测试恢复流程确保备份可用；(4) 使用支持快照的存储类作为额外保险。
