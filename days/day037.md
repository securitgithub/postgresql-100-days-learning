## 理论讲解

### 数据类型选择原则

选择正确的数据类型直接影响存储效率、查询性能和数据完整性。核心原则：

1. **选择能满足需求的最小类型**：节省存储空间，提高缓存命中率
2. **优先使用内置类型**：利用 PostgreSQL 的优化和索引支持
3. **考虑未来扩展**：预留一定的增长空间
4. **利用约束保护数据**：类型本身就是一种约束

#### 常用数据类型对比

| 需求 | 推荐类型 | 说明 |
|------|----------|------|
| 小整数（0-255） | `SMALLINT` | 2字节 |
| 常规整数 | `INTEGER` | 4字节，最常用 |
| 大整数/自增主键 | `BIGINT` / `BIGSERIAL` | 8字节 |
| 精确小数（金额） | `NUMERIC(p,s)` | 精确计算，无浮点误差 |
| 近似小数 | `REAL` / `DOUBLE PRECISION` | 有精度损失，计算快 |
| 短字符串 | `VARCHAR(n)` | 有长度限制 |
| 任意长度文本 | `TEXT` | 无长度限制，性能与 VARCHAR 相同 |
| 布尔值 | `BOOLEAN` | true/false/null |
| 日期 | `DATE` | 仅日期，4字节 |
| 日期时间 | `TIMESTAMP` | 日期+时间，8字节 |
| 带时区日期时间 | `TIMESTAMPTZ` | 推荐用于存储时间点 |
| JSON文档 | `JSONB` | 二进制JSON，支持索引和操作符 |
| UUID | `UUID` | 16字节，全局唯一 |
| IP地址 | `INET` / `CIDR` | 内置网络地址类型 |
| 金额 | `NUMERIC(12,2)` | 避免使用 `MONEY` 类型 |

#### 特别注意

```sql
-- 金额：永远不要用浮点数！
-- 错误
CREATE TABLE bad_prices (price DOUBLE PRECISION);
INSERT INTO bad_prices VALUES (0.1 + 0.2);  -- 结果不是 0.3！

-- 正确
CREATE TABLE good_prices (price NUMERIC(12,2));
INSERT INTO good_prices VALUES (0.1 + 0.2);  -- 精确得到 0.30

-- 时间戳：推荐使用 TIMESTAMPTZ
-- 存储时自动转为UTC，查询时按会话时区显示
CREATE TABLE events (
    event_time TIMESTAMPTZ DEFAULT NOW()  -- 推荐
    -- event_time TIMESTAMP                -- 不推荐，无时区信息
);

-- TEXT vs VARCHAR：在 PostgreSQL 中性能没有差异
-- VARCHAR(n) 的优势在于它提供了长度校验
CREATE TABLE example (
    code    VARCHAR(10),    -- 有明确长度限制的用 VARCHAR
    content TEXT            -- 无明确限制的用 TEXT
);
```

### 主键策略

#### 自增 ID（SERIAL / BIGSERIAL / IDENTITY）

```sql
-- 传统方式（SERIAL）
CREATE TABLE t1 (id SERIAL PRIMARY KEY);

-- 推荐方式（GENERATED AS IDENTITY，SQL 标准）
CREATE TABLE t2 (
    id INTEGER GENERATED ALWAYS AS IDENTITY PRIMARY KEY
);

-- 允许手动指定值的 IDENTITY
CREATE TABLE t3 (
    id INTEGER GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY
);
```

**优势：**
- 自增有序，B-tree 索引效率高
- 空间小（4 或 8 字节）
- 可读性好，便于调试
- JOIN 性能优越

**劣势：**
- 可预测，有安全隐患（可枚举资源）
- 分布式环境下难以保证唯一
- 数据迁移时可能冲突

#### UUID

```sql
-- 需要启用扩展（PostgreSQL 13+ 内置 gen_random_uuid）
CREATE EXTENSION IF NOT EXISTS pgcrypto;

CREATE TABLE t4 (
    id UUID PRIMARY KEY DEFAULT gen_random_uuid()
);

-- 或使用 uuid-ossp 扩展
CREATE EXTENSION IF NOT EXISTS "uuid-ossp";
CREATE TABLE t5 (
    id UUID PRIMARY KEY DEFAULT uuid_generate_v4()
);
```

**优势：**
- 全局唯一，适合分布式系统
- 不可预测，安全性好
- 可以在应用层生成，无需数据库参与
- 数据合并无冲突

**劣势：**
- 16 字节，比 INTEGER（4字节）大 4 倍
- 随机性导致 B-tree 索引频繁分裂，写入性能下降
- 不可读，调试困难
- JOIN 性能比整数差

#### 选择建议

| 场景 | 推荐 |
|------|------|
| 内部系统、单机部署 | BIGINT + IDENTITY |
| 需要暴露给外部的 ID | UUID（或 BIGINT + 混淆） |
| 分布式系统 | UUID 或 UUIDv7（有序UUID） |
| 高写入量的表 | BIGINT（避免 UUID 的索引分裂） |

### 索引策略规划

#### 索引类型选择

| 索引类型 | 适用场景 | 示例 |
|----------|----------|------|
| B-tree（默认） | 等值查询、范围查询、排序 | `WHERE id = 1`、`ORDER BY date` |
| Hash | 仅等值查询 | `WHERE code = 'ABC'`（较少使用） |
| GIN | 全文搜索、JSONB、数组 | `WHERE tags @> '{sql}'` |
| GiST | 几何数据、范围类型、全文搜索 | `WHERE point <@ box` |
| BRIN | 物理排序与值排序相关的大表 | 时间序列数据 |

#### 索引规划原则

```sql
-- 1. 为外键列创建索引（PostgreSQL 不自动创建）
CREATE INDEX idx_employees_dept_id ON employees(dept_id);

-- 2. 为常用查询条件创建索引
CREATE INDEX idx_orders_status ON orders(status);
CREATE INDEX idx_orders_customer_date ON orders(customer_id, order_date);

-- 3. 为排序列创建索引
CREATE INDEX idx_orders_date_desc ON orders(order_date DESC);

-- 4. 使用部分索引减少索引大小
CREATE INDEX idx_orders_pending ON orders(order_date)
    WHERE status = 'pending';

-- 5. 使用覆盖索引避免回表
CREATE INDEX idx_products_category ON products(category)
    INCLUDE (name, price);

-- 6. 为 JSONB 列创建 GIN 索引
CREATE INDEX idx_articles_metadata ON articles USING GIN(metadata);

-- 7. 表达式索引
CREATE INDEX idx_users_email_lower ON users(LOWER(email));
```

### 命名规范

一致的命名规范提高代码可读性和团队协作效率。

| 对象类型 | 命名规则 | 示例 |
|----------|----------|------|
| 表 | 小写、复数、下划线分隔 | `users`, `order_items` |
| 列 | 小写、下划线分隔 | `first_name`, `created_at` |
| 主键 | `表名单数_id` 或 `id` | `user_id` 或 `id` |
| 外键 | `引用表单数_id` | `department_id` |
| 索引 | `idx_表名_列名` | `idx_users_email` |
| 唯一索引 | `uk_表名_列名` | `uk_users_email` |
| 约束 | `ck_表名_描述` | `ck_employees_salary_positive` |
| 函数 | `动词_名词` | `get_user_by_id`, `calculate_total` |
| 触发器 | `trg_表名_动作` | `trg_orders_update_timestamp` |
| 视图 | `v_描述` | `v_active_users` |

## 代码示例

### 示例 1：完整的用户系统表设计

```sql
-- 用户表：使用 IDENTITY 主键
CREATE TABLE users (
    id          BIGINT GENERATED ALWAYS AS IDENTITY PRIMARY KEY,
    username    VARCHAR(50)  NOT NULL,
    email       VARCHAR(255) NOT NULL,
    phone       VARCHAR(20),
    status      VARCHAR(20)  NOT NULL DEFAULT 'active'
                CHECK (status IN ('active', 'inactive', 'banned')),
    created_at  TIMESTAMPTZ  NOT NULL DEFAULT NOW(),
    updated_at  TIMESTAMPTZ  NOT NULL DEFAULT NOW(),

    CONSTRAINT uk_users_username UNIQUE (username),
    CONSTRAINT uk_users_email    UNIQUE (email)
);

-- 添加注释
COMMENT ON TABLE users IS '用户基本信息表';
COMMENT ON COLUMN users.id IS '用户ID，自增主键';
COMMENT ON COLUMN users.status IS '用户状态：active/inactive/banned';

-- 索引规划
CREATE INDEX idx_users_status ON users(status);
CREATE INDEX idx_users_created_at ON users(created_at);
-- 表达式索引：支持不区分大小写的邮箱查询
CREATE INDEX idx_users_email_lower ON users(LOWER(email));
```

### 示例 2：商品表 — 类型选择示例

```sql
CREATE TABLE products (
    id           BIGINT GENERATED ALWAYS AS IDENTITY PRIMARY KEY,
    sku          VARCHAR(50) NOT NULL UNIQUE,         -- 固定格式编码
    name         VARCHAR(200) NOT NULL,                -- 有长度限制
    description  TEXT,                                  -- 无长度限制
    price        NUMERIC(12,2) NOT NULL CHECK (price >= 0),  -- 精确金额
    weight_kg    NUMERIC(8,3),                         -- 重量，精确到克
    is_active    BOOLEAN NOT NULL DEFAULT TRUE,        -- 布尔状态
    stock_count  INTEGER NOT NULL DEFAULT 0
                 CHECK (stock_count >= 0),             -- 库存不能为负
    category_id  INTEGER REFERENCES categories(id),
    tags         TEXT[],                                -- 数组类型存标签
    metadata     JSONB DEFAULT '{}',                   -- JSONB存储灵活属性
    created_at   TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    updated_at   TIMESTAMPTZ NOT NULL DEFAULT NOW()
);

-- 各类索引
CREATE INDEX idx_products_category ON products(category_id);
CREATE INDEX idx_products_active ON products(is_active) WHERE is_active = TRUE;
CREATE INDEX idx_products_tags ON products USING GIN(tags);
CREATE INDEX idx_products_metadata ON products USING GIN(metadata);
CREATE INDEX idx_products_price ON products(price);
```

### 示例 3：订单系统 — 完整设计

```sql
-- 订单表
CREATE TABLE orders (
    id            BIGINT GENERATED ALWAYS AS IDENTITY PRIMARY KEY,
    order_no      VARCHAR(30) NOT NULL UNIQUE,  -- 业务订单号
    customer_id   BIGINT NOT NULL REFERENCES users(id),
    status        VARCHAR(20) NOT NULL DEFAULT 'pending'
                  CHECK (status IN ('pending','paid','shipped','delivered','cancelled')),
    total_amount  NUMERIC(14,2) NOT NULL DEFAULT 0,
    shipping_addr JSONB,           -- 地址快照（反范式化，下单时固定）
    notes         TEXT,
    ordered_at    TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    paid_at       TIMESTAMPTZ,
    shipped_at    TIMESTAMPTZ,
    delivered_at  TIMESTAMPTZ
);

-- 订单明细表
CREATE TABLE order_items (
    id           BIGINT GENERATED ALWAYS AS IDENTITY PRIMARY KEY,
    order_id     BIGINT NOT NULL REFERENCES orders(id) ON DELETE CASCADE,
    product_id   BIGINT NOT NULL REFERENCES products(id),
    product_name VARCHAR(200) NOT NULL,  -- 商品名快照（反范式化）
    quantity     INTEGER NOT NULL CHECK (quantity > 0),
    unit_price   NUMERIC(12,2) NOT NULL CHECK (unit_price >= 0),

    CONSTRAINT uk_order_items_order_product UNIQUE (order_id, product_id)
);

-- 索引规划
CREATE INDEX idx_orders_customer ON orders(customer_id);
CREATE INDEX idx_orders_status ON orders(status);
CREATE INDEX idx_orders_ordered_at ON orders(ordered_at);
-- 部分索引：只索引未完成的订单
CREATE INDEX idx_orders_pending ON orders(ordered_at)
    WHERE status IN ('pending', 'paid', 'shipped');

CREATE INDEX idx_order_items_order ON order_items(order_id);
CREATE INDEX idx_order_items_product ON order_items(product_id);
```

### 示例 4：UUID 主键 + 公开 ID 混合策略

```sql
-- 内部用自增，对外暴露 UUID
CREATE TABLE api_resources (
    id          BIGINT GENERATED ALWAYS AS IDENTITY PRIMARY KEY,
    public_id   UUID NOT NULL DEFAULT gen_random_uuid() UNIQUE,
    name        VARCHAR(200) NOT NULL,
    data        JSONB,
    created_at  TIMESTAMPTZ NOT NULL DEFAULT NOW()
);

-- 内部查询用 id（性能好）
SELECT * FROM api_resources WHERE id = 12345;

-- API 对外暴露 public_id（安全）
SELECT * FROM api_resources
WHERE public_id = 'a0eebc99-9c0b-4ef8-bb6d-6bb9bd380a11';

-- 索引已通过 UNIQUE 自动创建
```

### 示例 5：查看表结构和索引

```sql
-- 查看表的列信息
SELECT column_name, data_type, character_maximum_length,
       is_nullable, column_default
FROM information_schema.columns
WHERE table_name = 'orders'
ORDER BY ordinal_position;

-- 查看表的索引
SELECT indexname, indexdef
FROM pg_indexes
WHERE tablename = 'orders';

-- 查看表和索引的大小
SELECT pg_size_pretty(pg_total_relation_size('orders')) AS total_size,
       pg_size_pretty(pg_relation_size('orders')) AS table_size,
       pg_size_pretty(pg_indexes_size('orders')) AS index_size;

-- 查看所有表的大小排名
SELECT relname AS table_name,
       pg_size_pretty(pg_total_relation_size(relid)) AS total_size
FROM pg_catalog.pg_statio_user_tables
ORDER BY pg_total_relation_size(relid) DESC
LIMIT 10;
```

## 常见问题

### Q1：VARCHAR(n) 和 TEXT 在 PostgreSQL 中有性能差异吗？

没有。PostgreSQL 内部对 VARCHAR、VARCHAR(n)、TEXT 使用相同的存储机制。VARCHAR(n) 只是增加了一个长度检查约束。选择建议：
- 有明确业务长度限制的字段用 VARCHAR(n)（如手机号、邮编）
- 没有明确限制的用 TEXT（如评论、描述）

### Q2：SERIAL 和 IDENTITY 有什么区别？该用哪个？

`SERIAL` 是 PostgreSQL 的传统语法糖，实际创建了一个序列并设置默认值。`IDENTITY` 是 SQL 标准语法（PostgreSQL 10+ 支持）。

推荐使用 `IDENTITY`，因为：
- 符合 SQL 标准
- 权限管理更清晰（序列与列绑定）
- `GENERATED ALWAYS` 可以防止意外手动指定值
- `DROP COLUMN` 时自动清理序列

### Q3：索引是不是越多越好？

不是。索引的代价包括：
- **写入开销**：每次 INSERT/UPDATE/DELETE 都要更新所有相关索引
- **存储空间**：索引占用额外磁盘空间
- **维护成本**：需要定期 REINDEX 或 VACUUM 维护

**建议**：只为实际使用的查询模式创建索引。可以通过 `pg_stat_user_indexes` 监控索引使用情况，移除从未被使用的索引。
