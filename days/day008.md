## 理论讲解

### 1. 主键（PRIMARY KEY）

主键是表中唯一标识每一行的列或列组合，具备两个特性：
- **唯一性**（UNIQUE）：同一张表中不能有两行具有相同的主键值
- **非空**（NOT NULL）：主键列不允许 NULL 值

**底层实现**：PostgreSQL 自动为主键创建一个 B-tree 唯一索引。

**主键选择策略：**

| 策略 | 示例 | 优点 | 缺点 |
|------|------|------|------|
| 自增整数 | SERIAL/IDENTITY | 简单高效、索引紧凑 | 可预测、分布式不友好 |
| UUID | gen_random_uuid() | 全局唯一、分布式友好 | 较大(16字节)、索引碎片 |
| 自然键 | 身份证号、邮箱 | 业务含义清晰 | 可能变更、多列组合复杂 |
| 复合主键 | (user_id, course_id) | 直接表达业务关系 | JOIN 复杂、子表外键冗长 |

### 2. 外键（FOREIGN KEY）

外键是建立表与表之间关联关系的约束，确保引用完整性（Referential Integrity）：
- 子表（引用表）的外键列值必须在父表（被引用表）的主键/唯一键中存在
- 或者外键列值为 NULL（除非设置了 NOT NULL）

### 3. 外键操作（Referential Actions）

当父表中被引用的行被更新或删除时，可以指定不同的行为：

| 操作 | ON DELETE 行为 | ON UPDATE 行为 |
|------|--------------|--------------|
| NO ACTION | 默认。如有子行则报错 | 默认。如有子行则报错 |
| RESTRICT | 立即报错（与 NO ACTION 类似） | 立即报错 |
| CASCADE | 删除对应的子行 | 更新对应的子行外键值 |
| SET NULL | 将子行外键设为 NULL | 将子行外键设为 NULL |
| SET DEFAULT | 将子行外键设为默认值 | 将子行外键设为默认值 |

**NO ACTION vs RESTRICT 的细微区别**：
- RESTRICT 是立即检查，不可延迟
- NO ACTION 可以通过 DEFERRABLE 延迟到事务提交时检查

### 4. 约束命名规范

良好的约束命名让数据库维护更清晰。推荐命名规范：

| 约束类型 | 命名格式 | 示例 |
|---------|---------|------|
| 主键 | `{表名}_pkey` | `orders_pkey` |
| 外键 | `{表名}_{列名}_fkey` | `orders_customer_id_fkey` |
| 唯一 | `{表名}_{列名}_key` | `users_email_key` |
| 检查 | `{表名}_{列名}_check` | `products_price_check` |

## 代码示例

### 主键定义方式

```sql
-- 方式1：列级主键（单列）
CREATE TABLE categories (
    id integer GENERATED ALWAYS AS IDENTITY PRIMARY KEY,
    name text NOT NULL UNIQUE
);

-- 方式2：表级主键（支持多列）
CREATE TABLE order_items (
    order_id integer,
    product_id integer,
    quantity integer NOT NULL DEFAULT 1,
    unit_price numeric(10,2) NOT NULL,
    PRIMARY KEY (order_id, product_id)  -- 复合主键
);

-- 方式3：命名主键约束
CREATE TABLE departments (
    id integer GENERATED ALWAYS AS IDENTITY,
    name text NOT NULL,
    CONSTRAINT departments_pkey PRIMARY KEY (id)
);
```

### 自增主键的三种方式

```sql
-- 方式1：SERIAL（传统方式）
CREATE TABLE t_serial (
    id serial PRIMARY KEY,
    name text
);

-- 等价于：
-- CREATE SEQUENCE t_serial_id_seq;
-- CREATE TABLE t_serial (
--     id integer NOT NULL DEFAULT nextval('t_serial_id_seq') PRIMARY KEY,
--     name text
-- );
-- ALTER SEQUENCE t_serial_id_seq OWNED BY t_serial.id;

-- 方式2：BIGSERIAL（大范围自增）
CREATE TABLE t_bigserial (
    id bigserial PRIMARY KEY,
    name text
);

-- 方式3：GENERATED AS IDENTITY（推荐，SQL 标准）
CREATE TABLE t_identity_always (
    id integer GENERATED ALWAYS AS IDENTITY PRIMARY KEY,
    name text
);

CREATE TABLE t_identity_default (
    id integer GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
    name text
);

-- 对比行为差异
INSERT INTO t_serial (name) VALUES ('a');          -- id = 1
INSERT INTO t_serial (id, name) VALUES (100, 'b'); -- 允许手动指定

INSERT INTO t_identity_always (name) VALUES ('a');            -- id = 1
INSERT INTO t_identity_always (id, name) VALUES (100, 'b');   -- ERROR！
-- 必须使用 OVERRIDING SYSTEM VALUE
INSERT INTO t_identity_always (id, name)
    OVERRIDING SYSTEM VALUE VALUES (100, 'b');                 -- OK

INSERT INTO t_identity_default (name) VALUES ('a');           -- id = 1
INSERT INTO t_identity_default (id, name) VALUES (100, 'b'); -- OK，允许手动指定

-- 查看序列当前值
SELECT currval('t_serial_id_seq');
SELECT lastval();  -- 最近在当前会话中使用的序列值
```

### 外键定义

```sql
-- 先创建父表
CREATE TABLE customers (
    id integer GENERATED ALWAYS AS IDENTITY PRIMARY KEY,
    name text NOT NULL,
    email text UNIQUE
);

CREATE TABLE products (
    id integer GENERATED ALWAYS AS IDENTITY PRIMARY KEY,
    name text NOT NULL,
    price numeric(10,2) NOT NULL CHECK (price >= 0)
);

-- 创建带外键的子表
CREATE TABLE orders (
    id integer GENERATED ALWAYS AS IDENTITY PRIMARY KEY,
    customer_id integer NOT NULL,
    order_date date NOT NULL DEFAULT current_date,
    total numeric(12,2) NOT NULL DEFAULT 0,
    -- 列级外键（简写）
    CONSTRAINT orders_customer_id_fkey
        FOREIGN KEY (customer_id) REFERENCES customers(id)
        ON DELETE RESTRICT
        ON UPDATE CASCADE
);

CREATE TABLE order_details (
    id integer GENERATED ALWAYS AS IDENTITY PRIMARY KEY,
    order_id integer NOT NULL,
    product_id integer NOT NULL,
    quantity integer NOT NULL CHECK (quantity > 0),
    unit_price numeric(10,2) NOT NULL,
    -- 表级外键
    CONSTRAINT order_details_order_id_fkey
        FOREIGN KEY (order_id) REFERENCES orders(id)
        ON DELETE CASCADE,
    CONSTRAINT order_details_product_id_fkey
        FOREIGN KEY (product_id) REFERENCES products(id)
        ON DELETE RESTRICT
);

-- 插入测试数据
INSERT INTO customers (name, email) VALUES
    ('张三', 'zhangsan@example.com'),
    ('李四', 'lisi@example.com');

INSERT INTO products (name, price) VALUES
    ('笔记本电脑', 5999.00),
    ('无线鼠标', 129.00),
    ('机械键盘', 599.00);

INSERT INTO orders (customer_id, total) VALUES (1, 6128.00);
INSERT INTO order_details (order_id, product_id, quantity, unit_price) VALUES
    (1, 1, 1, 5999.00),
    (1, 2, 1, 129.00);
```

### 外键引用完整性测试

```sql
-- 测试1：插入不存在的外键值
INSERT INTO orders (customer_id, total) VALUES (999, 0);
-- ERROR: insert or update on table "orders" violates foreign key constraint
-- DETAIL: Key (customer_id)=(999) is not present in table "customers".

-- 测试2：删除被引用的父行（RESTRICT）
DELETE FROM customers WHERE id = 1;
-- ERROR: update or delete on table "customers" violates foreign key constraint
-- DETAIL: Key (id)=(1) is still referenced from table "orders".

-- 测试3：删除有级联删除的父行（CASCADE）
DELETE FROM orders WHERE id = 1;
-- 成功！同时 order_details 中 order_id = 1 的行也被删除了
SELECT * FROM order_details;  -- 已为空
```

### 级联操作详解

```sql
-- CASCADE 示例
CREATE TABLE parent_cascade (
    id integer PRIMARY KEY,
    name text
);
CREATE TABLE child_cascade (
    id integer PRIMARY KEY,
    parent_id integer REFERENCES parent_cascade(id)
        ON DELETE CASCADE
        ON UPDATE CASCADE
);

INSERT INTO parent_cascade VALUES (1, '父级1'), (2, '父级2');
INSERT INTO child_cascade VALUES (10, 1), (20, 1), (30, 2);

-- CASCADE DELETE：删除父行，子行自动删除
DELETE FROM parent_cascade WHERE id = 1;
SELECT * FROM child_cascade;  -- 只剩 (30, 2)

-- CASCADE UPDATE：更新父键，子行外键自动更新
UPDATE parent_cascade SET id = 200 WHERE id = 2;
SELECT * FROM child_cascade;  -- (30, 200)

-- SET NULL 示例
DROP TABLE child_cascade;
DROP TABLE parent_cascade;

CREATE TABLE parent_setnull (
    id integer PRIMARY KEY,
    name text
);
CREATE TABLE child_setnull (
    id integer PRIMARY KEY,
    parent_id integer REFERENCES parent_setnull(id)
        ON DELETE SET NULL
);

INSERT INTO parent_setnull VALUES (1, '父级1'), (2, '父级2');
INSERT INTO child_setnull VALUES (10, 1), (20, 1), (30, 2);

DELETE FROM parent_setnull WHERE id = 1;
SELECT * FROM child_setnull;
-- (10, NULL), (20, NULL), (30, 2) -- 外键变为 NULL

-- SET DEFAULT 示例
DROP TABLE child_setnull;
DROP TABLE parent_setnull;

CREATE TABLE parent_setdefault (
    id integer PRIMARY KEY,
    name text
);
CREATE TABLE child_setdefault (
    id integer PRIMARY KEY,
    parent_id integer DEFAULT 0 REFERENCES parent_setdefault(id)
        ON DELETE SET DEFAULT
);

INSERT INTO parent_setdefault VALUES (0, '默认'), (1, '父级1');
INSERT INTO child_setdefault VALUES (10, 1);

DELETE FROM parent_setdefault WHERE id = 1;
SELECT * FROM child_setdefault;  -- (10, 0) -- 外键变为默认值 0
```

### 自引用外键（Self-Referencing）

```sql
-- 员工表：manager_id 引用自身的 id
CREATE TABLE staff (
    id integer GENERATED ALWAYS AS IDENTITY PRIMARY KEY,
    name text NOT NULL,
    title text NOT NULL,
    manager_id integer,
    CONSTRAINT staff_manager_fkey
        FOREIGN KEY (manager_id) REFERENCES staff(id)
        ON DELETE SET NULL
);

INSERT INTO staff (name, title) VALUES ('CEO 张总', 'CEO');  -- id=1, manager_id=NULL
INSERT INTO staff (name, title, manager_id) VALUES
    ('CTO 李总', 'CTO', 1),   -- id=2
    ('VP 王总', 'VP', 1),      -- id=3
    ('开发经理 赵工', 'Manager', 2),  -- id=4
    ('高级开发 钱工', 'Senior Dev', 4);  -- id=5

-- 查询组织结构
SELECT
    s.name AS employee,
    s.title,
    m.name AS manager
FROM staff s
LEFT JOIN staff m ON s.manager_id = m.id
ORDER BY s.id;
```

### 延迟约束检查（DEFERRABLE）

```sql
-- 创建可延迟的外键约束
CREATE TABLE nodes (
    id integer PRIMARY KEY,
    next_id integer,
    CONSTRAINT nodes_next_fkey
        FOREIGN KEY (next_id) REFERENCES nodes(id)
        DEFERRABLE INITIALLY DEFERRED  -- 延迟到事务结束时检查
);

-- 在一个事务中插入循环引用
BEGIN;
    INSERT INTO nodes VALUES (1, 2);  -- next_id=2 还不存在，但延迟检查不报错
    INSERT INTO nodes VALUES (2, 3);
    INSERT INTO nodes VALUES (3, 1);  -- 形成循环: 1->2->3->1
COMMIT;  -- 此时检查所有外键约束，全部满足

-- 如果约束不是 DEFERRABLE，上面的第一条 INSERT 就会报错
```

### 查看约束信息

```sql
-- 查看表的所有约束
SELECT
    conname AS constraint_name,
    contype AS type,  -- p=主键, f=外键, u=唯一, c=检查
    pg_get_constraintdef(oid) AS definition
FROM pg_constraint
WHERE conrelid = 'orders'::regclass;

-- 查看外键详细信息
SELECT
    tc.constraint_name,
    tc.table_name AS child_table,
    kcu.column_name AS child_column,
    ccu.table_name AS parent_table,
    ccu.column_name AS parent_column,
    rc.update_rule,
    rc.delete_rule
FROM information_schema.table_constraints tc
JOIN information_schema.key_column_usage kcu
    ON tc.constraint_name = kcu.constraint_name
JOIN information_schema.constraint_column_usage ccu
    ON tc.constraint_name = ccu.constraint_name
JOIN information_schema.referential_constraints rc
    ON tc.constraint_name = rc.constraint_name
WHERE tc.constraint_type = 'FOREIGN KEY'
  AND tc.table_schema = 'public';

-- 使用 psql 快捷命令
\d orders      -- 显示表结构和约束
\d+ orders     -- 显示更详细信息
```

## 常见问题

### Q1: 主键应该用自增整数还是 UUID？

**A:** 取决于应用场景：
- **自增整数**适合：单体应用、不需要对外暴露 ID、追求高性能
- **UUID**适合：分布式系统、微服务架构、需要在客户端预生成 ID、安全性要求高（ID 不可预测）
- 折中方案：对外使用 UUID，内部主键用自增整数，同时保留 UUID 列用于 API

### Q2: 外键会影响性能吗？

**A:** 会有一定影响，但通常是值得的：
- **INSERT/UPDATE/DELETE 时**：需要检查引用完整性，增加少量开销
- **优化建议**：确保外键列有索引。PostgreSQL 自动为主键和唯一键创建索引，但不会自动为外键列创建索引，需要手动添加
- 在数据仓库等批量导入场景中，有时会临时禁用外键以加速导入

### Q3: 可以创建指向非主键列的外键吗？

**A:** 可以，外键可以引用有 UNIQUE 约束的列，不一定是主键。例如：
```sql
-- users 表的 email 列有 UNIQUE 约束
ALTER TABLE orders ADD CONSTRAINT orders_email_fkey
    FOREIGN KEY (customer_email) REFERENCES users(email);
```
但通常推荐引用主键，因为主键更稳定且一定有索引。
