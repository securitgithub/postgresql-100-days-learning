# Day 076: 应用集成（一）连接池 — PgBouncer 配置与优化

## 学习目标

- 理解数据库连接池的必要性和工作原理
- 掌握 PgBouncer 的安装、配置和管理
- 理解事务池（Transaction Pooling）、会话池（Session Pooling）、语句池（Statement Pooling）的区别
- 学会优化连接参数以获得最佳性能
- 理解长连接和短连接的适用场景及权衡

## 理论讲解

### 1. 为什么需要连接池

PostgreSQL 为每个客户端连接创建一个独立的后端进程（backend process），这带来了几个问题：

- **进程创建开销大**：每次建立连接需要 fork 进程、分配内存（通常 5-10MB）、进行身份认证。
- **内存消耗高**：1000 个连接就需要 5-10GB 内存，即使大部分连接处于空闲状态。
- **上下文切换**：过多的进程导致操作系统频繁的上下文切换，降低 CPU 效率。
- **max_connections 限制**：默认 100，即使增大也不宜超过几百。

连接池通过复用少量数据库连接服务大量客户端连接来解决这些问题。

### 2. PgBouncer 概述

PgBouncer 是最流行的 PostgreSQL 连接池中间件，具有以下特点：
- 轻量级：单进程、事件驱动，内存占用极低（每个连接约 2KB）
- 支持三种池模式
- 支持多种认证方式
- 支持在线重启和配置重载
- 可以管理多个数据库连接

### 3. 三种池模式

**Session Pooling（会话池）**：
- 客户端连接时分配一个后端连接，断开时归还
- 支持所有 PostgreSQL 特性（包括 PREPARE、SET、LISTEN 等）
- 复用程度最低，但兼容性最好

**Transaction Pooling（事务池）**：
- 事务开始时分配后端连接，事务结束时归还
- 同一客户端的不同事务可能使用不同的后端连接
- 不支持跨事务的会话状态（如 PREPARE、SET、LISTEN/NOTIFY、临时表）
- 复用程度高，是最常用的模式

**Statement Pooling（语句池）**：
- 每条 SQL 语句执行完立即归还连接
- 不允许多语句事务（自动提交模式）
- 复用程度最高，但限制最大
- 适合简单的只读查询场景

### 4. 连接参数优化

关键参数：
- `max_connections`（PostgreSQL）：数据库最大连接数
- `pool_size`（PgBouncer）：每个数据库的连接池大小
- `max_client_conn`（PgBouncer）：最大客户端连接数
- `reserve_pool_size`：保留连接池大小
- `server_idle_timeout`：空闲后端连接超时

经验法则：PostgreSQL 的 `max_connections` 应设置为 CPU 核心数的 2-4 倍（考虑 I/O 等待）。

### 5. 长连接 vs 短连接

**长连接**：连接建立后保持，反复使用。减少连接建立开销，但需要管理连接健康状态。

**短连接**：每次操作新建连接，操作完关闭。编程简单，但频繁建连开销大。在有连接池的情况下，短连接在应用层的开销被连接池吸收。

## 代码示例

### 示例1：PgBouncer 安装与基本配置

```sql
-- PgBouncer 安装（Ubuntu/Debian）
-- sudo apt-get install pgbouncer

-- PgBouncer 安装（CentOS/RHEL）
-- sudo yum install pgbouncer
```

```ini
;; /etc/pgbouncer/pgbouncer.ini —— 主配置文件

[databases]
;; 数据库连接配置
;; 格式: dbname = connection_string
mydb = host=127.0.0.1 port=5432 dbname=mydb
;; 可以映射不同名称
production = host=db-master.internal port=5432 dbname=app_production
readonly = host=db-replica.internal port=5432 dbname=app_production

;; 通配符：将所有未明确列出的数据库转发到同一主机
* = host=127.0.0.1 port=5432

[pgbouncer]
;; 监听地址和端口
listen_addr = 0.0.0.0
listen_port = 6432

;; 认证相关
auth_type = md5
auth_file = /etc/pgbouncer/userlist.txt

;; 池模式（session, transaction, statement）
pool_mode = transaction

;; 连接池大小
default_pool_size = 20
min_pool_size = 5
reserve_pool_size = 5
reserve_pool_timeout = 3

;; 客户端连接限制
max_client_conn = 1000
max_db_connections = 50

;; 超时设置
server_idle_timeout = 600
server_connect_timeout = 15
server_login_retry = 15
client_idle_timeout = 0
client_login_timeout = 60
query_timeout = 0
query_wait_timeout = 120

;; 日志
log_connections = 1
log_disconnections = 1
log_pooler_errors = 1
stats_period = 60

;; 管理界面
admin_users = pgbouncer_admin
stats_users = pgbouncer_monitor
```

### 示例2：用户认证配置

```ini
;; /etc/pgbouncer/userlist.txt
;; 格式: "username" "password"
;; 密码可以是明文、md5 或 scram-sha-256

;; md5 格式
"myuser" "md5d41d8cd98f00b204e9800998ecf8427e"

;; 明文格式
"app_user" "secret_password"

;; 使用 auth_query 自动从数据库获取密码
;; 在 pgbouncer.ini 中配置:
;; auth_type = md5
;; auth_query = SELECT usename, passwd FROM pg_shadow WHERE usename=$1
```

```sql
-- 在 PostgreSQL 中创建 PgBouncer 认证查询用户
CREATE ROLE pgbouncer_auth LOGIN PASSWORD 'auth_password';
GRANT SELECT ON pg_shadow TO pgbouncer_auth;

-- 创建认证函数（更安全的方式）
CREATE OR REPLACE FUNCTION pgbouncer.get_auth(p_usename TEXT)
RETURNS TABLE(username TEXT, password TEXT) AS $$
BEGIN
    RETURN QUERY
    SELECT usename::TEXT, passwd::TEXT
    FROM pg_catalog.pg_shadow
    WHERE usename = p_usename;
END;
$$ LANGUAGE plpgsql SECURITY DEFINER;
```

-- 每个数据库的统计：
-- total_xact_count: 总事务数

-- 查看配置

### 示例4：不同池模式的行为差异

```sql
-- === Session Pooling 模式下 ===
-- 所有特性正常工作
SET work_mem = '256MB';         -- OK：整个会话有效
PREPARE my_stmt AS SELECT $1;   -- OK：跨事务有效
LISTEN my_channel;               -- OK：持续监听
CREATE TEMP TABLE tmp (id INT);  -- OK：整个会话可用

-- === Transaction Pooling 模式下 ===
-- 以下操作会有问题：

-- SET 只在当前事务有效
BEGIN;
SET LOCAL work_mem = '256MB';  -- OK：用 SET LOCAL
-- ... 查询 ...
COMMIT;
-- work_mem 恢复默认（连接已归还池中）

-- PREPARE 不可跨事务使用
BEGIN;
PREPARE my_stmt AS SELECT $1;
EXECUTE my_stmt(42);  -- OK
COMMIT;
-- 连接归还，下一事务可能是不同的后端连接
-- EXECUTE my_stmt(42);  -- 错误！语句不存在

-- LISTEN/NOTIFY 不可用
-- LISTEN my_channel;  -- 无意义，连接随时归还

-- 临时表不可跨事务
BEGIN;
CREATE TEMP TABLE tmp (id INT) ON COMMIT DROP;  -- 使用 ON COMMIT DROP
INSERT INTO tmp VALUES (1);
SELECT * FROM tmp;  -- OK
COMMIT;
-- 表已删除

-- === Statement Pooling 模式下 ===
-- 不允许显式事务
-- BEGIN;  -- 会报错
-- 所有查询自动提交
SELECT * FROM accounts;  -- OK，自动提交
UPDATE accounts SET balance = 100 WHERE id = 1;  -- OK，自动提交
```

### 示例5：连接参数优化实践

```sql
-- 在 PostgreSQL 中查看当前连接信息
SELECT
    count(*) AS total_connections,
    count(*) FILTER (WHERE state = 'active') AS active,
    count(*) FILTER (WHERE state = 'idle') AS idle,
    count(*) FILTER (WHERE state = 'idle in transaction') AS idle_in_tx,
    count(*) FILTER (WHERE wait_event_type = 'Client') AS waiting_client
FROM pg_stat_activity
WHERE backend_type = 'client backend';

-- 查看连接占用的内存（估算）
SELECT
    pg_size_pretty(
        count(*) * 5 * 1024 * 1024  -- 每连接约 5MB
    ) AS estimated_connection_memory,
    count(*) AS connection_count
FROM pg_stat_activity
WHERE backend_type = 'client backend';

-- 推荐的 max_connections 计算
-- max_connections = CPU核心数 * 2 + 有效磁盘数
-- 例如：8核CPU，1个SSD → 8*2+1 = 17，可设为 20-30

-- 查看当前 max_connections
SHOW max_connections;

-- PostgreSQL 端设置合理的超时
ALTER SYSTEM SET idle_in_transaction_session_timeout = '30s';
ALTER SYSTEM SET statement_timeout = '60s';
SELECT pg_reload_conf();
```

1. **PgBouncer 安装与配置**：安装 PgBouncer，配置 Transaction Pooling 模式，使用 `psql` 通过 PgBouncer 连接数据库，验证基本查询正常工作。

## 常见问题

### Q1: Transaction Pooling 模式下如何使用 PREPARE？

**A:** 在 Transaction Pooling 模式下，Prepared Statement 不能跨事务使用，因为不同事务可能分配到不同的后端连接。解决方案：
1. 在每个事务内部 PREPARE 并使用，事务结束时 DEALLOCATE。
2. 使用 PgBouncer 的 `server_reset_query` 配置项（默认 `DISCARD ALL`）清理连接状态。
3. 大多数连接池驱动（如 JDBC 的 `prepareThreshold`）会自动处理这个问题，将 PREPARE 推迟到多次执行后才发送。
4. 考虑使用 PgBouncer 3.0+ 的 `prepared_statement` 支持特性。

### Q2: 如何确定最佳的连接池大小？

**A:** 连接池大小并不是越大越好。经验公式：`pool_size = CPU核心数 * 2 + 有效磁盘数`。对于纯 SSD 系统，可以简化为 `CPU核心数 * 2` 到 `CPU核心数 * 4`。过大的连接池会导致过多进程竞争 CPU 和 I/O，反而降低吞吐量。建议通过压测确定最佳值：逐步增加 pool_size，观察 TPS 变化，当 TPS 不再增长甚至下降时，取之前的值。

### Q3: PgBouncer 和 PostgreSQL 内置的连接池（如 pg_pool）有什么区别？

**A:** PostgreSQL 自身没有内置连接池（截至 PG 16），但社区有相关提案。PgBouncer 是独立的中间件，优势在于：
1. 可以部署在应用服务器侧，减少网络延迟。
2. 支持多种池模式灵活选择。
3. 极低的资源占用。
4. 可以管理多个 PostgreSQL 实例。
另一个选择是 Pgpool-II，它除了连接池还提供负载均衡、主从切换等功能，但比 PgBouncer 更重。如果只需要连接池，推荐 PgBouncer。
