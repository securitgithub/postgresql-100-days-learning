# Day 064: 索引深入（二）GiST、GIN、BRIN

## 学习目标

- 理解 GiST 索引的原理及其在几何数据、范围类型上的应用
- 掌握 GIN 索引在全文搜索、数组和 JSONB 上的使用方法
- 了解 BRIN 索引的适用场景及其极小的存储开销
- 认识 SP-GiST 索引的特点与应用
- 能够根据数据类型和查询需求选择恰当的索引类型

## 理论讲解

### 一、PostgreSQL 索引类型总览

PostgreSQL 提供了丰富的索引类型，除了昨天学习的 B-tree 和 Hash，还有以下专用索引：

| 索引类型 | 适用场景 | 典型数据类型 |
|----------|----------|-------------|
| GiST | 几何数据、范围类型、全文搜索 | geometry, tsquery, range |
| GIN | 多值元素搜索 | array, jsonb, tsvector |
| BRIN | 物理排序与逻辑排序一致的大表 | timestamp, serial |
| SP-GiST | 非平衡分区树结构 | inet, text, geometry |

### 二、GiST 索引（Generalized Search Tree）

GiST 是一种通用搜索树框架，不同数据类型可以在此框架上实现自己的索引策略。

#### 2.1 GiST 的核心思想

GiST 使用**边界区域**（bounding region）来组织数据。每个内部节点存储一个"边界"，包含其所有子节点的数据范围。查找时通过检查边界是否与查询条件重叠来决定是否深入子树。

#### 2.2 适用场景

**几何数据**：
- 点、线、多边形的包含、重叠、相邻查询
- PostGIS 的空间索引基于 GiST

**范围类型**：
- `int4range`、`tsrange`、`daterange` 等
- 支持 `@>`（包含）、`<@`（被包含）、`&&`（重叠）等操作符

**全文搜索**：
- GiST 可以索引 `tsvector`，但通常 GIN 更适合（GiST 有损，GIN 无损）

**排除约束**：
- GiST 索引支持排除约束（EXCLUDE），用于防止数据冲突（如时间段不重叠）

#### 2.3 GiST 的特点

- 支持最近邻搜索（KNN，`ORDER BY <-> distance`）
- 索引可能是"有损的"（lossy）：索引告诉你"可能匹配"，需要回表确认
- 索引大小通常介于 B-tree 和 GIN 之间
- 写入性能比 GIN 好

### 三、GIN 索引（Generalized Inverted Index）

GIN 是一种倒排索引，非常适合包含多个元素的值（如数组、全文搜索向量、JSONB）。

#### 3.1 GIN 的核心思想

倒排索引将每个元素映射到包含该元素的所有行。例如对于全文搜索，每个词项（term）指向包含该词的所有文档 ID。

```
"数据库" → [doc_1, doc_5, doc_12, ...]
"索引"   → [doc_1, doc_3, doc_7, ...]
"优化"   → [doc_2, doc_5, doc_9, ...]
```

#### 3.2 适用场景

**全文搜索（tsvector）**：
- 查找包含特定词项的文档
- 支持 `@@` 匹配操作符

**数组（array）**：
- `@>` 包含、`<@` 被包含、`&&` 重叠
- 查找数组中包含某个元素的行

**JSONB**：
- `@>` 包含、`?` 键存在、`?|` 任一键存在、`?&` 所有键存在

**三元组（pg_trgm）**：
- 模糊匹配 `LIKE '%keyword%'`
- 相似度搜索

#### 3.3 GIN 的特点

- 精确索引（不是有损的）
- 索引体积较大（存储每个元素到行的映射）
- 写入开销大（每次插入可能需要更新多个索引项）
- `fastupdate` 机制：新数据先写入待处理列表，批量合并到主索引
- 读取性能非常好

#### 3.4 GIN 的 Pending List

为了缓解写入性能问题，GIN 使用了 pending list（待处理列表）。新插入的数据先暂存到 pending list 中，待积累到一定程度后批量合并到主索引。

```sql
-- 查看 GIN 待处理列表大小
SELECT * FROM gin_metapage_info(get_raw_page('idx_name', 0));

-- 手动触发 pending list 合并
VACUUM table_name;
```

### 四、BRIN 索引（Block Range Index）

BRIN 是 PostgreSQL 9.5 引入的一种极轻量的索引类型。

#### 4.1 BRIN 的核心思想

BRIN 不为每一行创建索引条目，而是为**连续的物理页面块**（称为 block range）存储摘要信息（最小值和最大值）。

例如，对一个按时间顺序插入的日志表，BRIN 索引可能记录：
- 页面 1-128：时间范围 2024-01-01 ~ 2024-01-15
- 页面 129-256：时间范围 2024-01-15 ~ 2024-01-30
- ...

#### 4.2 适用场景

BRIN 的有效前提是**数据的物理存储顺序与索引列的逻辑顺序高度相关**。

最佳场景：
- 时间序列数据（按时间顺序插入的日志表）
- 自增 ID 列
- 大表（百万行以上），索引大小差异才明显

不适用场景：
- 数据被频繁随机更新
- 物理存储顺序与列值无关

#### 4.3 BRIN 的优势

- **极小的存储空间**：几百 MB 表的 BRIN 索引可能只有几十 KB
- **维护成本极低**
- 适合只追加（append-only）的场景

#### 4.4 pages_per_range 参数

BRIN 的精度由 `pages_per_range` 参数控制（默认 128）。值越小，精度越高但索引越大。

### 五、SP-GiST 索引（Space-Partitioned GiST）

SP-GiST 支持分区搜索树结构，如四叉树、k-d 树、基数树等。

适用场景：
- IP 地址范围查询（`inet` 类型）
- 电话号码前缀匹配
- 几何点数据的空间分区查询

```sql
-- IP 地址范围查询
CREATE INDEX idx_ip ON access_log USING spgist (client_ip);
```

## 代码示例

### 示例1：GiST 索引与几何数据

```sql
-- 创建包含位置信息的表
CREATE TABLE shops (
    id SERIAL PRIMARY KEY,
    name TEXT,
    location POINT
);

-- 插入测试数据（模拟城市中的店铺坐标）
INSERT INTO shops (name, location)
SELECT
    'Shop_' || i,
    point(
        116.0 + random() * 1.0,  -- 经度范围
        39.5 + random() * 1.0    -- 纬度范围
    )
FROM generate_series(1, 100000) AS i;

ANALYZE shops;

-- 创建 GiST 索引
CREATE INDEX idx_shops_location ON shops USING gist (location);

-- 查找指定区域内的店铺（矩形范围）
EXPLAIN ANALYZE
SELECT * FROM shops
WHERE location <@ box '((116.3,39.8),(116.5,40.0))';

-- 最近邻搜索（KNN）：找距离目标点最近的 10 家店铺
EXPLAIN ANALYZE
SELECT name, location,
       location <-> point '(116.4, 39.9)' AS distance
FROM shops
ORDER BY location <-> point '(116.4, 39.9)'
LIMIT 10;
```

### 示例2：GiST 索引与范围类型

```sql
-- 会议室预约表（时间范围不能重叠）
CREATE TABLE room_bookings (
    id SERIAL PRIMARY KEY,
    room_id INT NOT NULL,
    during TSRANGE NOT NULL,
    booked_by TEXT
);

-- 创建 GiST 索引
CREATE INDEX idx_bookings_during ON room_bookings USING gist (during);

-- 使用排除约束防止同一房间的预约时间重叠
ALTER TABLE room_bookings
ADD CONSTRAINT no_overlap
EXCLUDE USING gist (room_id WITH =, during WITH &&);

-- 插入预约
INSERT INTO room_bookings (room_id, during, booked_by)
VALUES (1, '[2024-06-01 09:00, 2024-06-01 10:00)', '张三');

INSERT INTO room_bookings (room_id, during, booked_by)
VALUES (1, '[2024-06-01 10:00, 2024-06-01 11:30)', '李四');

-- 这条会失败（与张三的预约重叠）
-- INSERT INTO room_bookings (room_id, during, booked_by)
-- VALUES (1, '[2024-06-01 09:30, 2024-06-01 10:30)', '王五');

-- 查询某时间段有哪些预约
EXPLAIN ANALYZE
SELECT * FROM room_bookings
WHERE during && '[2024-06-01 09:00, 2024-06-01 12:00)'::tsrange;
```

### 示例3：GIN 索引与数组

```sql
-- 文章标签表
CREATE TABLE articles (
    id SERIAL PRIMARY KEY,
    title TEXT,
    tags TEXT[]
);

-- 插入数据
INSERT INTO articles (title, tags)
SELECT
    'Article_' || i,
    ARRAY(
        SELECT (ARRAY['PostgreSQL','MySQL','Redis','MongoDB','Kafka',
                       'Docker','K8s','Linux','Python','Go'])[1 + (random()*9)::int]
        FROM generate_series(1, 3)
    )
FROM generate_series(1, 200000) AS i;

ANALYZE articles;

-- 创建 GIN 索引
CREATE INDEX idx_articles_tags ON articles USING gin (tags);

-- 查找包含特定标签的文章
EXPLAIN ANALYZE
SELECT * FROM articles WHERE tags @> ARRAY['PostgreSQL'];

-- 查找包含多个标签的文章
EXPLAIN ANALYZE
SELECT * FROM articles WHERE tags @> ARRAY['PostgreSQL', 'Docker'];

-- 查找包含任一标签的文章
EXPLAIN ANALYZE
SELECT * FROM articles WHERE tags && ARRAY['Redis', 'MongoDB'];
```

### 示例4：GIN 索引与 JSONB

```sql
-- 事件日志表（JSONB 格式存储）
CREATE TABLE events (
    id SERIAL PRIMARY KEY,
    data JSONB NOT NULL
);

-- 插入数据
INSERT INTO events (data)
SELECT jsonb_build_object(
    'type', (ARRAY['click','view','purchase','signup'])[1 + (random()*3)::int],
    'user_id', (random() * 10000)::int,
    'page', '/page/' || (random() * 100)::int,
    'metadata', jsonb_build_object('browser', 'Chrome', 'os', 'Linux')
)
FROM generate_series(1, 300000);

ANALYZE events;

-- 创建 GIN 索引（默认 jsonb_ops）
CREATE INDEX idx_events_data ON events USING gin (data);

-- 查询包含特定键值对的事件
EXPLAIN ANALYZE
SELECT * FROM events WHERE data @> '{"type": "purchase"}';

-- 查询嵌套键值
EXPLAIN ANALYZE
SELECT * FROM events WHERE data @> '{"metadata": {"browser": "Chrome"}}';

-- jsonb_path_ops 操作符类（更小的索引，只支持 @>）
CREATE INDEX idx_events_data_path ON events USING gin (data jsonb_path_ops);

-- 对比两种操作符类的索引大小
SELECT indexname, pg_size_pretty(pg_relation_size(indexrelid)) AS size
FROM pg_stat_user_indexes
WHERE tablename = 'events'
  AND indexname LIKE 'idx_events_data%';
```

### 示例5：BRIN 索引

```sql
-- 模拟时间序列日志表
CREATE TABLE sensor_data (
    id BIGSERIAL PRIMARY KEY,
    sensor_id INT,
    reading NUMERIC,
    recorded_at TIMESTAMP NOT NULL DEFAULT now()
);

-- 按时间顺序插入数据（BRIN 的理想场景）
INSERT INTO sensor_data (sensor_id, reading, recorded_at)
SELECT
    (random() * 100)::int,
    random() * 1000,
    '2023-01-01'::timestamp + (i || ' seconds')::interval
FROM generate_series(1, 2000000) AS i;

ANALYZE sensor_data;

-- 创建 BRIN 索引
CREATE INDEX idx_sensor_brin ON sensor_data USING brin (recorded_at);

-- 创建 B-tree 索引做对比
CREATE INDEX idx_sensor_btree ON sensor_data USING btree (recorded_at);

-- 对比索引大小（差距巨大）
SELECT indexname, pg_size_pretty(pg_relation_size(indexrelid)) AS size
FROM pg_stat_user_indexes
WHERE tablename = 'sensor_data'
  AND indexname IN ('idx_sensor_brin', 'idx_sensor_btree');

-- 时间范围查询
EXPLAIN ANALYZE
SELECT * FROM sensor_data
WHERE recorded_at BETWEEN '2023-06-01' AND '2023-06-02';

-- 调整 pages_per_range 获得更高精度
CREATE INDEX idx_sensor_brin_16 ON sensor_data
USING brin (recorded_at) WITH (pages_per_range = 16);
```

### 示例6：SP-GiST 索引

```sql
-- IP 地址查询
CREATE TABLE access_log (
    id SERIAL PRIMARY KEY,
    client_ip INET,
    path TEXT,
    accessed_at TIMESTAMP
);

INSERT INTO access_log (client_ip, path, accessed_at)
SELECT
    ('192.168.' || (random()*255)::int || '.' || (random()*255)::int)::inet,
    '/api/v1/resource/' || (random()*100)::int,
    now() - (random() * 30 || ' days')::interval
FROM generate_series(1, 200000);

ANALYZE access_log;

-- SP-GiST 索引
CREATE INDEX idx_log_ip ON access_log USING spgist (client_ip);

-- 子网范围查询
EXPLAIN ANALYZE
SELECT * FROM access_log
WHERE client_ip << '192.168.1.0/24'::inet;
```

## 常见问题

### Q1：GIN 索引写入太慢怎么办？

**A**：GIN 索引因为需要维护倒排列表，写入开销确实较大。几种优化方法：
1. 开启 `fastupdate`（默认已开启）：新数据先进入 pending list，批量合并
2. 增大 `gin_pending_list_limit`（默认 4MB）：允许 pending list 更大，减少合并频率
3. 在批量导入时先删除索引，导入完毕再重建
4. 考虑是否可以用 GiST 代替（GiST 写入更快，但查询可能较慢且有损）

### Q2：BRIN 索引什么时候不生效？

**A**：BRIN 的有效性完全取决于数据的物理存储顺序与查询列值的相关性。以下情况 BRIN 效果差：
1. 数据被大量随机 UPDATE，导致行在物理上被移动到各处
2. 表经历过大量 DELETE + INSERT，打乱了物理顺序
3. 使用了 `CLUSTER` 命令按其他列重排了表
解决方法是定期 `CLUSTER` 表按目标列排序，或使用 `pg_repack` 重组。

### Q3：GiST 和 GIN 都支持全文搜索，该选哪个？

**A**：两者各有特点：
- **GIN**：精确索引，查询更快，适合读多写少场景，索引体积更大
- **GiST**：有损索引（需要回表验证），查询较慢，但写入更快，索引更小，支持 KNN 排序

一般建议：如果主要是精确的全文搜索匹配，选 GIN；如果需要相关度排序或写入频繁，考虑 GiST。
