## 理论讲解

### 1. PostgreSQL 统计系统概述

PostgreSQL 内置了一套强大的统计收集系统（Statistics Collector），它跟踪数据库活动并将统计信息暴露在一组系统视图中。这些视图分为：

- **动态活动视图**：`pg_stat_activity`、`pg_stat_replication`
- **累计统计视图**：`pg_stat_user_tables`、`pg_stat_user_indexes`
- **后台进程视图**：`pg_stat_bgwriter`、`pg_stat_wal`（PG 14+）
- **IO 统计视图**：`pg_stat_io`（PG 16+）

统计数据通过共享内存传递，开销很低。使用 `track_activities`、`track_counts`、`track_io_timing` 等参数控制采集范围。

### 2. pg_stat_activity 详解

`pg_stat_activity` 是最重要的动态监控视图，显示每个服务器进程的当前状态。

关键字段：

| 字段 | 说明 |
|------|------|
| `pid` | 后端进程 ID |
| `datname` | 连接的数据库 |
| `usename` | 连接用户 |
| `application_name` | 应用名称 |
| `client_addr` | 客户端 IP |
| `state` | 连接状态：active/idle/idle in transaction 等 |
| `query` | 当前或最近执行的查询 |
| `query_start` | 查询开始时间 |
| `state_change` | 状态最后变更时间 |
| `wait_event_type` | 等待事件类型 |
| `wait_event` | 等待事件名称 |
| `backend_type` | 后端类型 |

连接状态含义：
- `active`：正在执行查询
- `idle`：等待新命令
- `idle in transaction`：事务中但无活动查询（危险状态）
- `idle in transaction (aborted)`：事务中且有错误
- `fastpath function call`：执行快速路径函数
- `disabled`：tracking 被禁用

### 3. pg_stat_user_tables 详解

记录用户表的访问统计：

| 字段 | 说明 |
|------|------|
| `seq_scan` | 顺序扫描次数 |
| `seq_tup_read` | 顺序扫描读取的行数 |
| `idx_scan` | 索引扫描次数 |
| `idx_tup_fetch` | 索引扫描获取的行数 |
| `n_tup_ins` | 插入行数 |
| `n_tup_upd` | 更新行数 |
| `n_tup_del` | 删除行数 |
| `n_live_tup` | 存活行数估计 |
| `n_dead_tup` | 死元组数 |
| `last_vacuum` | 最后 vacuum 时间 |
| `last_autovacuum` | 最后 autovacuum 时间 |
| `last_analyze` | 最后 analyze 时间 |

### 4. pg_stat_user_indexes 详解

记录索引的使用情况：

| 字段 | 说明 |
|------|------|
| `idx_scan` | 索引被扫描的次数 |
| `idx_tup_read` | 索引扫描返回的条目数 |
| `idx_tup_fetch` | 通过索引获取的实际行数 |

通过这个视图可以识别未使用的索引，减少不必要的写入开销。

### 5. pg_stat_bgwriter 详解

监控后台写入器和检查点的性能：

| 字段 | 说明 |
|------|------|
| `checkpoints_timed` | 定时触发的检查点次数 |
| `checkpoints_req` | 请求触发的检查点次数 |
| `checkpoint_write_time` | 检查点写入耗时（ms） |
| `checkpoint_sync_time` | 检查点同步耗时（ms） |
| `buffers_checkpoint` | 检查点写出的缓冲区数 |
| `buffers_clean` | 后台写入器写出的缓冲区数 |
| `buffers_backend` | 后端进程直接写出的缓冲区数 |
| `maxwritten_clean` | 后台写入器因达到上限而停止的次数 |
| `buffers_alloc` | 分配的缓冲区数 |

如果 `buffers_backend` 远大于 `buffers_clean`，说明后台写入器不够积极，后端进程被迫自己写脏页，影响性能。

## 代码示例

### 示例1：pg_stat_activity 常用查询

```sql
-- 查看所有活跃连接
SELECT pid, usename, datname, state,
       application_name, client_addr,
       now() - query_start AS query_duration,
       left(query, 80) AS query_preview
FROM pg_stat_activity
WHERE pid != pg_backend_pid()
ORDER BY query_start ASC;

-- 按状态统计连接数
SELECT state, count(*)
FROM pg_stat_activity
WHERE pid != pg_backend_pid()
GROUP BY state
ORDER BY count(*) DESC;

-- 查找长时间运行的查询（超过 30 秒）
SELECT pid, usename, datname,
       now() - query_start AS duration,
       wait_event_type, wait_event,
       state, left(query, 120) AS query
FROM pg_stat_activity
WHERE state = 'active'
  AND now() - query_start > interval '30 seconds'
  AND pid != pg_backend_pid()
ORDER BY duration DESC;

-- 查找 idle in transaction 超时的连接
SELECT pid, usename, datname,
       now() - state_change AS idle_duration,
       left(query, 100) AS last_query
FROM pg_stat_activity
WHERE state = 'idle in transaction'
  AND now() - state_change > interval '5 minutes';

-- 按数据库和用户统计连接
SELECT datname, usename, state, count(*)
FROM pg_stat_activity
GROUP BY datname, usename, state
ORDER BY datname, count(*) DESC;
```

### 示例2：查看等待事件

```sql
-- 查看正在等待的进程
SELECT pid, usename, datname,
       wait_event_type, wait_event,
       state, left(query, 80) AS query
FROM pg_stat_activity
WHERE wait_event IS NOT NULL
  AND state = 'active'
  AND pid != pg_backend_pid()
ORDER BY wait_event_type;

-- 按等待事件统计
SELECT wait_event_type, wait_event, count(*)
FROM pg_stat_activity
WHERE wait_event IS NOT NULL
GROUP BY wait_event_type, wait_event
ORDER BY count(*) DESC;
```

### 示例3：表统计分析

```sql
-- 查找频繁顺序扫描的大表（可能缺少索引）
SELECT schemaname, relname,
       seq_scan,
       seq_tup_read,
       idx_scan,
       CASE WHEN seq_scan + idx_scan > 0
            THEN round(100.0 * idx_scan / (seq_scan + idx_scan), 2)
            ELSE 0 END AS idx_scan_pct,
       n_live_tup
FROM pg_stat_user_tables
WHERE seq_scan > 100
  AND n_live_tup > 10000
ORDER BY seq_tup_read DESC
LIMIT 20;

-- 查看死元组较多的表（需要 VACUUM）
SELECT schemaname, relname,
       n_live_tup,
       n_dead_tup,
       CASE WHEN n_live_tup > 0
            THEN round(100.0 * n_dead_tup / n_live_tup, 2)
            ELSE 0 END AS dead_pct,
       last_vacuum,
       last_autovacuum,
       last_analyze
FROM pg_stat_user_tables
WHERE n_dead_tup > 1000
ORDER BY n_dead_tup DESC
LIMIT 20;

-- 表的读写活跃度排名
SELECT schemaname, relname,
       n_tup_ins AS inserts,
       n_tup_upd AS updates,
       n_tup_del AS deletes,
       n_tup_ins + n_tup_upd + n_tup_del AS total_writes,
       seq_scan + COALESCE(idx_scan, 0) AS total_reads
FROM pg_stat_user_tables
ORDER BY (n_tup_ins + n_tup_upd + n_tup_del) DESC
LIMIT 20;
```

### 示例4：索引使用分析

```sql
-- 查找未使用的索引（idx_scan = 0）
SELECT s.schemaname, s.relname AS table_name,
       s.indexrelname AS index_name,
       s.idx_scan,
       pg_size_pretty(pg_relation_size(s.indexrelid)) AS index_size
FROM pg_stat_user_indexes s
JOIN pg_index i ON s.indexrelid = i.indexrelid
WHERE s.idx_scan = 0
  AND NOT i.indisunique         -- 排除唯一索引
  AND NOT i.indisprimary        -- 排除主键
ORDER BY pg_relation_size(s.indexrelid) DESC;

-- 索引效率分析
SELECT schemaname, relname, indexrelname,
       idx_scan,
       idx_tup_read,
       idx_tup_fetch,
       CASE WHEN idx_tup_read > 0
            THEN round(100.0 * idx_tup_fetch / idx_tup_read, 2)
            ELSE 0 END AS fetch_pct
FROM pg_stat_user_indexes
WHERE idx_scan > 0
ORDER BY idx_scan DESC
LIMIT 20;

-- 表的索引概览
SELECT t.schemaname, t.relname,
       pg_size_pretty(pg_table_size(t.relid)) AS table_size,
       count(i.indexrelname) AS index_count,
       pg_size_pretty(sum(pg_relation_size(i.indexrelid))) AS total_index_size,
       sum(i.idx_scan) AS total_idx_scans
FROM pg_stat_user_tables t
LEFT JOIN pg_stat_user_indexes i ON t.relid = i.relid
GROUP BY t.schemaname, t.relname, t.relid
ORDER BY pg_table_size(t.relid) DESC
LIMIT 20;
```

### 示例5：后台写入器与检查点监控

```sql
-- 查看 bgwriter 统计
SELECT checkpoints_timed,
       checkpoints_req,
       checkpoint_write_time / 1000 AS write_time_seconds,
       checkpoint_sync_time / 1000 AS sync_time_seconds,
       buffers_checkpoint,
       buffers_clean,
       buffers_backend,
       maxwritten_clean,
       buffers_alloc,
       stats_reset
FROM pg_stat_bgwriter;

-- 计算缓冲区写入来源比例
SELECT round(100.0 * buffers_checkpoint / NULLIF(buffers_checkpoint + buffers_clean + buffers_backend, 0), 1) AS pct_checkpoint,
       round(100.0 * buffers_clean / NULLIF(buffers_checkpoint + buffers_clean + buffers_backend, 0), 1) AS pct_bgwriter,
       round(100.0 * buffers_backend / NULLIF(buffers_checkpoint + buffers_clean + buffers_backend, 0), 1) AS pct_backend
FROM pg_stat_bgwriter;

-- 重置统计信息（用于重新开始计量）
SELECT pg_stat_reset_shared('bgwriter');
```

### 示例6：综合监控仪表盘查询

```sql
-- 数据库概览
SELECT datname,
       numbackends AS connections,
       xact_commit AS commits,
       xact_rollback AS rollbacks,
       CASE WHEN xact_commit + xact_rollback > 0
            THEN round(100.0 * xact_commit / (xact_commit + xact_rollback), 2)
            ELSE 100 END AS commit_ratio,
       blks_read,
       blks_hit,
       CASE WHEN blks_read + blks_hit > 0
            THEN round(100.0 * blks_hit / (blks_read + blks_hit), 2)
            ELSE 100 END AS cache_hit_ratio,
       tup_returned, tup_fetched,
       tup_inserted, tup_updated, tup_deleted
FROM pg_stat_database
WHERE datname NOT LIKE 'template%'
ORDER BY numbackends DESC;

-- 缓存命中率（应 > 99%）
SELECT sum(blks_hit) * 100.0 / NULLIF(sum(blks_hit) + sum(blks_read), 0) AS cache_hit_ratio
FROM pg_stat_database;
```

## 常见问题

### Q1：pg_stat_user_tables 中的数据何时更新？

统计数据由 stats collector 进程（PG 15 之前）或通过共享内存（PG 15+）更新。数据存在轻微延迟（通常不超过 500ms）。统计计数器从上次 `pg_stat_reset()` 之后开始累计。如果需要准确的增量数据，应定期采集并计算差值。

### Q2：如何终止一个长时间运行的查询？

使用 `pg_cancel_backend(pid)` 发送取消信号（温和方式），如果无效则使用 `pg_terminate_backend(pid)` 强制终止连接。注意区分：cancel 只取消当前查询，terminate 会断开整个连接。

```sql
-- 取消查询
SELECT pg_cancel_backend(12345);
-- 终止连接
SELECT pg_terminate_backend(12345);
```

### Q3：统计信息可以重置吗？

可以。`SELECT pg_stat_reset();` 重置当前数据库的统计。`SELECT pg_stat_reset_shared('bgwriter');` 重置共享统计。建议在调优前重置，以便获得干净的基线数据。注意重置是全局操作，在多人使用的环境中需协调。
