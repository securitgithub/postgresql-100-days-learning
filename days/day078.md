# Day 078: 应用集成（三）FDW — 外部数据包装器

## 学习目标

- 理解外部数据包装器（Foreign Data Wrapper）的概念和架构
- 掌握 `postgres_fdw` 实现跨 PostgreSQL 实例查询
- 学会使用 `file_fdw` 查询外部 CSV/日志文件
- 了解其他常见 FDW（mysql_fdw、redis_fdw 等）的使用
- 能够在实际场景中利用 FDW 进行异构数据源集成

## 理论讲解

### 1. FDW 概述

Foreign Data Wrapper（外部数据包装器）是 SQL/MED（Management of External Data）标准的 PostgreSQL 实现。它允许你像查询本地表一样查询外部数据源。

FDW 的核心组件：
- **Foreign Data Wrapper**：连接外部数据源的"驱动程序"
- **Foreign Server**：外部数据源的连接信息
- **User Mapping**：本地用户到外部数据源用户的映射
- **Foreign Table**：本地定义的外部表结构

查询流程：
1. 用户查询外部表
2. PostgreSQL 将查询发送给 FDW
3. FDW 连接外部数据源获取数据
4. 数据返回给 PostgreSQL 做进一步处理

### 2. postgres_fdw

`postgres_fdw` 是 PostgreSQL 自带的 FDW，用于连接其他 PostgreSQL 实例。它支持：
- 远程查询下推（WHERE 条件、JOIN、排序、聚合等）
- 远程事务管理
- 读写操作（可以 INSERT/UPDATE/DELETE 远程表）
- 连接缓存和复用

### 3. file_fdw

`file_fdw` 也是 PostgreSQL 自带的 FDW，可以将服务器上的文件作为表来查询。支持：
- CSV 文件
- 固定宽度的文本文件
- COPY 格式兼容的任何文件
- 程序输出（通过 `PROGRAM` 选项）

### 4. 其他常见 FDW

社区提供了丰富的 FDW 扩展：
- `mysql_fdw`：连接 MySQL/MariaDB
- `redis_fdw`：连接 Redis
- `mongo_fdw`：连接 MongoDB
- `oracle_fdw`：连接 Oracle Database
- `tds_fdw`：连接 SQL Server/Sybase
- `multicorn`：Python 编写的 FDW 框架

### 5. 性能考虑

FDW 查询的性能关键在于"下推"（Pushdown）：
- **条件下推**：WHERE 条件在远程执行，减少数据传输
- **JOIN 下推**：两个远程表的 JOIN 在远程执行
- **聚合下推**：SUM、COUNT 等在远程执行
- **排序下推**：ORDER BY 在远程执行
- **LIMIT 下推**：LIMIT 在远程执行

不能下推的操作会导致全量数据拉取到本地再处理，性能很差。

## 代码示例

### 示例1：postgres_fdw 基本配置

```sql
-- 在本地数据库（local_db）上操作

-- 第 1 步：安装扩展
CREATE EXTENSION postgres_fdw;

-- 第 2 步：创建外部服务器
CREATE SERVER remote_server
    FOREIGN DATA WRAPPER postgres_fdw
    OPTIONS (
        host '192.168.1.100',
        port '5432',
        dbname 'remote_db',
        -- 连接优化选项
        fetch_size '10000',          -- 每次获取的行数（默认 100）
        extensions 'hstore,pg_trgm'  -- 远程安装的扩展
    );

-- 第 3 步：创建用户映射
CREATE USER MAPPING FOR current_user
    SERVER remote_server
    OPTIONS (
        user 'remote_user',
        password 'remote_password'
    );

-- 也可以为特定用户创建映射
CREATE USER MAPPING FOR app_user
    SERVER remote_server
    OPTIONS (user 'remote_readonly', password 'readonly_pass');

-- 公共映射（所有用户使用同一远程账号）
CREATE USER MAPPING FOR PUBLIC
    SERVER remote_server
    OPTIONS (user 'remote_readonly', password 'readonly_pass');

-- 第 4 步：创建外部表
CREATE FOREIGN TABLE remote_orders (
    id INTEGER,
    customer_name TEXT,
    product TEXT,
    amount NUMERIC(10,2),
    order_date DATE,
    status TEXT
)
SERVER remote_server
OPTIONS (
    schema_name 'public',
    table_name 'orders'
);

-- 查询外部表（像本地表一样使用）
SELECT * FROM remote_orders WHERE order_date >= '2024-01-01' LIMIT 10;
```

### 示例2：批量导入外部表结构

```sql
-- 使用 IMPORT FOREIGN SCHEMA 批量导入
-- 导入远程 public 模式下的所有表
IMPORT FOREIGN SCHEMA public
    FROM SERVER remote_server
    INTO local_foreign_schema;

-- 只导入指定的表
IMPORT FOREIGN SCHEMA public
    LIMIT TO (orders, customers, products)
    FROM SERVER remote_server
    INTO foreign_tables;

-- 排除某些表
IMPORT FOREIGN SCHEMA public
    EXCEPT (audit_log, temp_data)
    FROM SERVER remote_server
    INTO foreign_tables;

-- 查看已创建的外部表
SELECT
    foreign_table_schema,
    foreign_table_name,
    foreign_server_name
FROM information_schema.foreign_tables;

-- 查看外部表的列信息
SELECT
    table_name,
    column_name,
    data_type
FROM information_schema.columns
WHERE table_schema = 'foreign_tables'
ORDER BY table_name, ordinal_position;
```

### 示例3：跨实例 JOIN 和查询下推

```sql
-- 跨实例 JOIN：本地表与远程表关联
CREATE TABLE local_customers (
    id SERIAL PRIMARY KEY,
    name TEXT,
    vip_level INTEGER
);

INSERT INTO local_customers VALUES
(1, '张三', 3),
(2, '李四', 1),
(3, '王五', 2);

-- 本地客户 JOIN 远程订单
SELECT
    c.name,
    c.vip_level,
    count(o.id) AS order_count,
    sum(o.amount) AS total_amount
FROM local_customers c
LEFT JOIN remote_orders o ON o.customer_name = c.name
WHERE c.vip_level >= 2
GROUP BY c.name, c.vip_level
ORDER BY total_amount DESC;

-- 查看查询是否下推
EXPLAIN VERBOSE
SELECT * FROM remote_orders
WHERE order_date >= '2024-01-01'
  AND amount > 1000
ORDER BY order_date DESC
LIMIT 20;
-- 查看 Remote SQL 行，确认条件被下推

-- 两个远程表之间的 JOIN（同一服务器的会被下推）
CREATE FOREIGN TABLE remote_customers (
    id INTEGER,
    name TEXT,
    email TEXT
)
SERVER remote_server
OPTIONS (schema_name 'public', table_name 'customers');

EXPLAIN VERBOSE
SELECT c.name, o.product, o.amount
FROM remote_customers c
JOIN remote_orders o ON o.customer_name = c.name
WHERE c.id < 100;
-- 如果两个表在同一远程服务器，JOIN 会被下推
```

### 示例4：远程写操作

```sql
-- postgres_fdw 支持写操作
-- 插入远程数据
INSERT INTO remote_orders (customer_name, product, amount, order_date, status)
VALUES ('新客户', '测试商品', 199.00, CURRENT_DATE, 'pending');

-- 更新远程数据
UPDATE remote_orders
SET status = 'shipped'
WHERE id = 1001 AND status = 'pending';

-- 删除远程数据
DELETE FROM remote_orders
WHERE status = 'cancelled' AND order_date < '2023-01-01';

-- 事务性：远程操作包含在本地事务中
BEGIN;
-- 本地操作
INSERT INTO local_customers (name, vip_level) VALUES ('赵六', 1);
-- 远程操作
INSERT INTO remote_orders (customer_name, product, amount, order_date, status)
VALUES ('赵六', '首单商品', 99.00, CURRENT_DATE, 'pending');
COMMIT;
-- 两个操作要么都成功，要么都回滚（两阶段提交）

-- 如果不需要写操作，可以将外部表设为只读
ALTER FOREIGN TABLE remote_orders OPTIONS (ADD updatable 'false');
```

### 示例5：file_fdw 查询外部文件

```sql
-- 安装 file_fdw 扩展
CREATE EXTENSION file_fdw;

-- 创建文件服务器
CREATE SERVER file_server FOREIGN DATA WRAPPER file_fdw;

-- === 查询 CSV 文件 ===
-- 假设有文件 /tmp/sales_data.csv 内容如下：
-- date,product,quantity,price
-- 2024-01-01,Widget A,100,9.99
-- 2024-01-01,Widget B,50,19.99
-- 2024-01-02,Widget A,120,9.99

CREATE FOREIGN TABLE csv_sales (
    sale_date DATE,
    product TEXT,
    quantity INTEGER,
    price NUMERIC(10,2)
)
SERVER file_server
OPTIONS (
    filename '/tmp/sales_data.csv',
    format 'csv',
    header 'true',
    delimiter ','
);

-- 查询 CSV 文件
SELECT * FROM csv_sales WHERE quantity > 80;

-- 聚合分析
SELECT
    product,
    sum(quantity) AS total_qty,
    sum(quantity * price) AS total_revenue
FROM csv_sales
GROUP BY product;

-- === 查询日志文件 ===
CREATE FOREIGN TABLE pg_log (
    log_time TIMESTAMP WITH TIME ZONE,
    user_name TEXT,
    database_name TEXT,
    process_id INTEGER,
    connection_from TEXT,
    session_id TEXT,
    session_line_num BIGINT,
    command_tag TEXT,
    session_start_time TIMESTAMP WITH TIME ZONE,
    virtual_transaction_id TEXT,
    transaction_id BIGINT,
    error_severity TEXT,
    sql_state_code TEXT,
    message TEXT,
    detail TEXT,
    hint TEXT,
    internal_query TEXT,
    internal_query_pos INTEGER,
    context TEXT,
    query TEXT,
    query_pos INTEGER,
    location TEXT,
    application_name TEXT
)
SERVER file_server
OPTIONS (
    filename '/var/log/postgresql/postgresql-16-main.csv',
    format 'csv'
);

-- 查询错误日志
SELECT log_time, error_severity, message
FROM pg_log
WHERE error_severity IN ('ERROR', 'FATAL')
ORDER BY log_time DESC
LIMIT 20;

-- === 查询程序输出 ===
CREATE FOREIGN TABLE disk_usage (
    filesystem TEXT,
    size TEXT,
    used TEXT,
    avail TEXT,
    use_pct TEXT,
    mounted_on TEXT
)
SERVER file_server
OPTIONS (
    program 'df -h',
    format 'csv',
    delimiter ' '
);

SELECT * FROM disk_usage;
```

### 示例7：FDW 性能优化

```sql
-- 1. 增大 fetch_size 减少网络往返
ALTER SERVER remote_server OPTIONS (SET fetch_size '50000');

-- 2. 查看查询下推情况
EXPLAIN (VERBOSE, COSTS OFF)
SELECT customer_name, sum(amount)
FROM remote_orders
WHERE order_date >= '2024-01-01'
GROUP BY customer_name
HAVING sum(amount) > 10000;
-- 检查 "Remote SQL" 确认下推

-- 3. 使用物化视图缓存远程数据
CREATE MATERIALIZED VIEW mv_remote_summary AS
SELECT
    customer_name,
    count(*) AS order_count,
    sum(amount) AS total_amount,
    max(order_date) AS last_order_date
FROM remote_orders
GROUP BY customer_name;

CREATE INDEX idx_mv_remote_customer ON mv_remote_summary(customer_name);

-- 定期刷新
REFRESH MATERIALIZED VIEW CONCURRENTLY mv_remote_summary;

-- 4. 创建定时刷新任务（使用 pg_cron）
-- SELECT cron.schedule('refresh_remote_summary', '0 */2 * * *',
--     'REFRESH MATERIALIZED VIEW CONCURRENTLY mv_remote_summary');

-- 5. 为外部表添加本地统计信息
ANALYZE remote_orders;
-- 让优化器更准确地估算成本

-- 6. 连接优化
ALTER SERVER remote_server OPTIONS (
    SET fetch_size '50000',
    SET keepalives '1',
    SET keepalives_idle '30',
    SET keepalives_interval '10',
    SET keepalives_count '3'
);

-- 7. 使用分区表 + FDW 实现数据分层
-- 近期数据在本地，历史数据在远程
CREATE TABLE all_orders (
    id INTEGER,
    customer_name TEXT,
    amount NUMERIC,
    order_date DATE
) PARTITION BY RANGE (order_date);

-- 本地分区（近期数据）
CREATE TABLE orders_current PARTITION OF all_orders
    FOR VALUES FROM ('2024-01-01') TO ('2025-01-01');

-- 远程分区（历史数据）
CREATE FOREIGN TABLE orders_archive PARTITION OF all_orders
    FOR VALUES FROM ('2020-01-01') TO ('2024-01-01')
    SERVER remote_server
    OPTIONS (schema_name 'public', table_name 'orders_archive');

-- 查询会自动路由：近期走本地，历史走远程
SELECT * FROM all_orders WHERE order_date >= '2023-06-01';
```

## 动手练习

1. **postgres_fdw 实战**：在同一台机器上创建两个数据库，使用 `postgres_fdw` 将数据库 A 的表映射到数据库 B 中。测试跨库查询、JOIN 和写操作。

2. **file_fdw 日志分析**：将 PostgreSQL 的 CSV 格式日志通过 `file_fdw` 挂载为外部表，查询最近的错误信息、最慢的查询、最频繁的连接来源。

3. **查询下推分析**：使用 `EXPLAIN VERBOSE` 分析不同查询模式（简单 WHERE、JOIN、聚合、子查询）的下推情况。找出哪些操作不能下推，以及如何改写查询以促进下推。

4. **物化视图缓存**：为频繁查询的远程数据创建物化视图。对比直接查询外部表和查询物化视图的性能差异。

5. **数据分层存储**：实现一个分区表，近 3 个月的数据存储在本地分区，更早的数据通过 FDW 存储在远程实例。验证查询能够透明地访问所有数据。

## 常见问题

### Q1: FDW 查询的事务一致性如何保证？

**A:** `postgres_fdw` 使用两阶段提交（2PC）来保证本地事务和远程事务的一致性。当本地事务包含对远程表的修改时，COMMIT 阶段会先在远程执行 PREPARE TRANSACTION，然后在本地提交，最后在远程执行 COMMIT PREPARED。但这不是完美的分布式事务——如果最后一步失败（网络断开），需要手动处理远程的 prepared transaction。对于只读查询，远程使用 REPEATABLE READ 隔离级别确保快照一致性。

### Q2: 为什么 FDW 查询很慢？

**A:** 常见原因：
1. **条件未下推**：使用了远程不支持的函数或操作符，导致全表拉取。用 `EXPLAIN VERBOSE` 检查 Remote SQL。
2. **fetch_size 太小**：默认 100，大表查询会产生大量网络往返。增大到 10000-50000。
3. **网络延迟**：远程服务器距离远。考虑使用物化视图缓存。
4. **缺少远程索引**：确保远程表有适当的索引。
5. **统计信息不准确**：对外部表执行 `ANALYZE` 更新本地统计信息。

### Q3: FDW 能替代 ETL 吗？

**A:** FDW 适合实时查询少量数据的场景，不适合替代 ETL：
- **适合**：跨系统查询、数据验证、小表关联、元数据查询。
- **不适合**：大数据量定期同步、复杂数据转换、对延迟敏感的分析查询。
最佳实践是将 FDW 和物化视图结合：FDW 提供实时查询能力，物化视图定期刷新作为"轻量级 ETL"。对于大规模数据集成，仍然建议使用专业的 ETL 工具。
