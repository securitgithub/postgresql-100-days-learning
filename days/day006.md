## 理论讲解

### 1. 日期时间类型

| 类型 | 存储 | 范围 | 精度 |
|------|------|------|------|
| date | 4 字节 | 4713 BC - 5874897 AD | 天 |
| time | 8 字节 | 00:00:00 - 24:00:00 | 微秒 |
| time with time zone | 12 字节 | 同上 + 时区 | 微秒 |
| timestamp | 8 字节 | 4713 BC - 294276 AD | 微秒 |
| timestamp with time zone (timestamptz) | 8 字节 | 同上 | 微秒 |
| interval | 16 字节 | -178000000年 - 178000000年 | 微秒 |

**timestamp vs timestamptz 的关键区别：**

- `timestamp`（without time zone）：存储字面时间值，不做任何时区转换
- `timestamptz`（with time zone）：存储 UTC 时间，显示时根据会话时区自动转换

**最佳实践**：除非有特殊理由，始终使用 `timestamptz`。这样无论用户在哪个时区，数据都能正确转换。

### 2. interval 类型

interval 表示时间间隔，是 PostgreSQL 独有的强大类型。它支持年、月、日、时、分、秒的组合，并且可以参与日期运算。

### 3. boolean 类型

PostgreSQL 的 boolean 类型接受多种输入格式：

| 真值输入 | 假值输入 |
|---------|---------|
| TRUE, 't', 'true', 'y', 'yes', 'on', '1' | FALSE, 'f', 'false', 'n', 'no', 'off', '0' |

存储大小：1 字节。还有一个特殊值 `NULL`（未知）。

### 4. UUID 类型

UUID（通用唯一标识符）是一个 128 位的标识符，显示为 32 个十六进制字符（加 4 个连字符）。格式如：`a0eebc99-9c0b-4ef8-bb6d-6bb9bd380a11`

优点：全局唯一，无需中心化生成，适合分布式系统。
缺点：占用 16 字节（比 integer 的 4 字节大），随机性导致索引碎片。

### 5. ENUM 类型

枚举类型定义一组固定的允许值，类似于编程语言中的 enum。值的顺序由定义时的顺序决定。

### 6. Serial 与 IDENTITY

| 类型 | 底层实现 | 范围 | 推荐度 |
|------|---------|------|--------|
| serial | integer + sequence | 1 - 2,147,483,647 | 传统方式 |
| bigserial | bigint + sequence | 1 - 9.2 x 10^18 | 传统方式 |
| smallserial | smallint + sequence | 1 - 32,767 | 传统方式 |
| GENERATED AS IDENTITY | SQL 标准 | 取决于列类型 | **推荐** |

PostgreSQL 10 开始推荐使用 `GENERATED AS IDENTITY`，这是 SQL 标准的写法。

## 代码示例

### 日期时间操作

```sql
-- 获取当前时间
SELECT
    current_date,                    -- 当前日期
    current_time,                    -- 当前时间（带时区）
    current_timestamp,               -- 当前时间戳（带时区）
    now(),                           -- 同 current_timestamp
    localtime,                       -- 当前本地时间
    localtimestamp,                  -- 当前本地时间戳
    clock_timestamp();               -- 精确当前时间（每次调用不同）

-- 日期时间输入格式
SELECT
    '2025-06-15'::date,                         -- ISO 格式
    '15-Jun-2025'::date,                        -- 英文月份
    '20250615'::date,                           -- 紧凑格式
    '2025-06-15 14:30:00'::timestamp,           -- 时间戳
    '2025-06-15 14:30:00+08'::timestamptz,      -- 带时区
    '14:30:00'::time;                            -- 时间

-- 时区转换
SET timezone = 'Asia/Shanghai';
SELECT
    '2025-06-15 12:00:00+00'::timestamptz,           -- 显示为上海时间（+8）
    '2025-06-15 12:00:00+00'::timestamptz AT TIME ZONE 'US/Eastern',  -- 转为纽约时间
    '2025-06-15 12:00:00+00'::timestamptz AT TIME ZONE 'Asia/Tokyo';  -- 转为东京时间

-- 日期部分提取
SELECT
    extract(year FROM timestamp '2025-06-15 14:30:45'),    -- 2025
    extract(month FROM timestamp '2025-06-15 14:30:45'),   -- 6
    extract(day FROM timestamp '2025-06-15 14:30:45'),     -- 15
    extract(hour FROM timestamp '2025-06-15 14:30:45'),    -- 14
    extract(dow FROM timestamp '2025-06-15 14:30:45'),     -- 0=周日
    extract(epoch FROM timestamp '2025-06-15 14:30:45');   -- Unix 时间戳

-- date_part 函数（等价于 extract）
SELECT
    date_part('year', now()),
    date_part('quarter', now()),
    date_part('week', now());

-- 日期截断
SELECT
    date_trunc('year', timestamp '2025-06-15 14:30:45'),    -- 2025-01-01 00:00:00
    date_trunc('month', timestamp '2025-06-15 14:30:45'),   -- 2025-06-01 00:00:00
    date_trunc('day', timestamp '2025-06-15 14:30:45'),     -- 2025-06-15 00:00:00
    date_trunc('hour', timestamp '2025-06-15 14:30:45');    -- 2025-06-15 14:00:00
```

### interval 运算

```sql
-- interval 的多种表示
SELECT
    interval '1 year 2 months 3 days',
    interval '1-2',               -- 1 year 2 months (ISO)
    interval '3 4:05:06',         -- 3 days 4 hours 5 minutes 6 seconds
    interval '1.5 hours',         -- 1 hour 30 minutes
    '30 days'::interval;

-- 日期加减运算
SELECT
    current_date + interval '7 days'   AS next_week,
    current_date - interval '1 month'  AS last_month,
    current_date + 30                  AS plus_30_days,  -- date + integer = date
    now() + interval '2 hours 30 minutes' AS later;

-- 两个日期之间的间隔
SELECT
    '2025-12-31'::date - '2025-01-01'::date AS days_diff,  -- 结果是 integer（天数）
    age('2025-12-31'::date, '2025-01-01'::date),            -- 结果是 interval
    age(timestamp '2000-06-15');                              -- 从生日到现在的年龄

-- 实际应用：计算到期日
CREATE TABLE subscriptions (
    id serial PRIMARY KEY,
    user_name text,
    start_date date NOT NULL DEFAULT current_date,
    duration interval NOT NULL,
    expire_date date GENERATED ALWAYS AS (start_date + duration) STORED
);

-- 注意：上面的 GENERATED 列不支持 interval 加法，改为手动计算
DROP TABLE IF EXISTS subscriptions;
CREATE TABLE subscriptions (
    id serial PRIMARY KEY,
    user_name text,
    start_date date NOT NULL DEFAULT current_date,
    duration_days integer NOT NULL
);

INSERT INTO subscriptions (user_name, duration_days) VALUES
    ('张三', 30),
    ('李四', 365),
    ('王五', 90);

SELECT
    user_name,
    start_date,
    duration_days,
    start_date + duration_days AS expire_date,
    start_date + duration_days - current_date AS remaining_days
FROM subscriptions;
```

### 布尔类型

```sql
-- 创建包含布尔列的表
CREATE TABLE features (
    id serial PRIMARY KEY,
    name text NOT NULL,
    is_enabled boolean NOT NULL DEFAULT false,
    is_premium boolean DEFAULT NULL  -- NULL 表示"未设置"
);

INSERT INTO features (name, is_enabled, is_premium) VALUES
    ('暗色模式', true, false),
    ('自动备份', 'yes', true),        -- 'yes' 也是合法输入
    ('数据导出', 't', NULL),           -- 't' 也是合法输入
    ('API访问', '1', 'f'),             -- '1' 和 'f' 也是合法输入
    ('消息推送', false, 'no');

SELECT * FROM features;

-- 布尔值在 WHERE 中的使用
SELECT name FROM features WHERE is_enabled;           -- 等价于 is_enabled = true
SELECT name FROM features WHERE NOT is_enabled;       -- 等价于 is_enabled = false
SELECT name FROM features WHERE is_premium IS NULL;   -- 查找未设置的

-- 布尔聚合
SELECT
    count(*) FILTER (WHERE is_enabled) AS enabled_count,
    count(*) FILTER (WHERE NOT is_enabled) AS disabled_count,
    count(*) FILTER (WHERE is_premium IS NULL) AS unknown_count
FROM features;

-- 三值逻辑（true, false, NULL）
SELECT
    true AND NULL,    -- NULL
    false AND NULL,   -- false
    true OR NULL,     -- true
    false OR NULL,    -- NULL
    NOT NULL;         -- NULL
```

### UUID 类型

```sql
-- 启用 uuid 生成扩展（PostgreSQL 13+ 内置 gen_random_uuid）
-- CREATE EXTENSION IF NOT EXISTS "uuid-ossp";  -- 旧方式

-- 生成 UUID
SELECT gen_random_uuid();  -- PostgreSQL 13+ 内置函数

-- 使用 UUID 作为主键
CREATE TABLE sessions (
    id uuid PRIMARY KEY DEFAULT gen_random_uuid(),
    user_id integer NOT NULL,
    token text,
    created_at timestamptz DEFAULT now()
);

INSERT INTO sessions (user_id, token) VALUES
    (1, 'abc123'),
    (2, 'def456');

SELECT * FROM sessions;
-- id 列会自动生成类似 a0eebc99-9c0b-4ef8-bb6d-6bb9bd380a11 的值

-- UUID 比较（不区分大小写）
SELECT 'A0EEBC99-9C0B-4EF8-BB6D-6BB9BD380A11'::uuid =
       'a0eebc99-9c0b-4ef8-bb6d-6bb9bd380a11'::uuid;  -- true
```

### ENUM 类型

```sql
-- 创建枚举类型
CREATE TYPE mood AS ENUM ('sad', 'ok', 'happy');
CREATE TYPE order_status AS ENUM ('pending', 'paid', 'shipped', 'delivered', 'cancelled');

-- 使用枚举
CREATE TABLE daily_log (
    id serial PRIMARY KEY,
    log_date date NOT NULL DEFAULT current_date,
    feeling mood NOT NULL,
    note text
);

INSERT INTO daily_log (feeling, note) VALUES
    ('happy', '今天学了 PostgreSQL！'),
    ('ok', '进度正常'),
    ('sad', '遇到了 bug');

-- 枚举有顺序，可以比较
SELECT * FROM daily_log WHERE feeling > 'ok';  -- 只返回 'happy'
SELECT * FROM daily_log ORDER BY feeling;       -- sad, ok, happy

-- 插入不合法的值会报错
INSERT INTO daily_log (feeling) VALUES ('angry');
-- ERROR: invalid input value for enum mood: "angry"

-- 修改枚举（添加新值）
ALTER TYPE mood ADD VALUE 'excited' AFTER 'happy';
ALTER TYPE mood ADD VALUE 'anxious' BEFORE 'sad';

-- 查看枚举的所有值
SELECT enum_range(NULL::mood);
SELECT unnest(enum_range(NULL::order_status));
```

### Serial 与 IDENTITY

```sql
-- 传统方式：serial
CREATE TABLE old_style (
    id serial PRIMARY KEY,  -- 自动创建 sequence
    name text
);

-- 推荐方式：GENERATED AS IDENTITY（SQL 标准）
CREATE TABLE new_style (
    id integer GENERATED ALWAYS AS IDENTITY PRIMARY KEY,
    name text
);

-- GENERATED ALWAYS vs GENERATED BY DEFAULT
CREATE TABLE identity_demo (
    id1 integer GENERATED ALWAYS AS IDENTITY,     -- 不允许手动插入
    id2 integer GENERATED BY DEFAULT AS IDENTITY,  -- 允许手动插入
    name text
);

INSERT INTO identity_demo (name) VALUES ('自动生成');

-- 尝试手动指定 ALWAYS 列的值
INSERT INTO identity_demo (id1, id2, name) VALUES (100, 200, '手动');
-- ERROR: cannot insert a non-DEFAULT value into column "id1"

-- 使用 OVERRIDING SYSTEM VALUE 强制覆盖
INSERT INTO identity_demo (id1, id2, name)
    OVERRIDING SYSTEM VALUE
    VALUES (100, 200, '强制覆盖');

-- 自定义 IDENTITY 的起始值和步长
CREATE TABLE custom_seq (
    id integer GENERATED ALWAYS AS IDENTITY (START WITH 1000 INCREMENT BY 10),
    name text
);

INSERT INTO custom_seq (name) VALUES ('a'), ('b'), ('c');
SELECT * FROM custom_seq;  -- id: 1000, 1010, 1020
```

### 自定义域（DOMAIN）

```sql
-- 创建自定义域：带约束的类型别名
CREATE DOMAIN positive_int AS integer CHECK (VALUE > 0);
CREATE DOMAIN email AS text CHECK (VALUE ~* '^[A-Za-z0-9._%+-]+@[A-Za-z0-9.-]+\.[A-Za-z]{2,}$');
CREATE DOMAIN phone_cn AS varchar(11) CHECK (VALUE ~ '^1[3-9][0-9]{9}$');

-- 使用自定义域
CREATE TABLE contacts (
    id integer GENERATED ALWAYS AS IDENTITY PRIMARY KEY,
    name text NOT NULL,
    age positive_int,
    email_addr email,
    phone phone_cn
);

-- 合法插入
INSERT INTO contacts (name, age, email_addr, phone) VALUES
    ('张三', 28, 'zhangsan@example.com', '13800138000');

-- 非法插入测试
INSERT INTO contacts (name, age) VALUES ('李四', -5);
-- ERROR: value for domain positive_int violates check constraint

INSERT INTO contacts (name, email_addr) VALUES ('王五', 'not-an-email');
-- ERROR: value for domain email violates check constraint

-- 查看已定义的域
SELECT domain_name, data_type, domain_default
FROM information_schema.domains
WHERE domain_schema = 'public';
```

## 常见问题

### Q1: timestamp 和 timestamptz 到底该用哪个？

**A:** 几乎所有场景都应该使用 `timestamptz`（timestamp with time zone）。原因：
- 它会自动处理时区转换，避免时间歧义
- 存储的是 UTC 时间，显示时根据客户端时区自动转换
- 只有在你确定不需要时区信息（如生日只需日期）时才用 `timestamp`
- PostgreSQL 官方文档也推荐默认使用 `timestamptz`

### Q2: serial 和 IDENTITY 有什么区别？应该用哪个？

**A:**
- `serial` 是 PostgreSQL 传统的自增列实现，底层创建了一个独立的 sequence，有一些权限管理的小问题
- `GENERATED AS IDENTITY` 是 SQL:2003 标准，行为更明确，权限管理更干净
- 新项目推荐使用 `GENERATED AS IDENTITY`
- 旧项目中的 `serial` 也没有迁移的必要，继续使用即可

### Q3: enum 类型可以删除某个值吗？

**A:** PostgreSQL 目前不支持直接从 enum 中删除值（只能添加）。如果需要"禁用"某个值，有以下替代方案：
1. 使用 CHECK 约束代替 enum
2. 创建新的 enum 类型，迁移数据后删除旧类型
3. 在应用层做校验，数据库存储为 text

这也是为什么有些人建议使用 `text + CHECK` 而非 `enum` 的原因——灵活性更好。
