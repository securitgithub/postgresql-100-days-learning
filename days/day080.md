## 理论讲解

### 1. 慢数据库诊断方法论

诊断慢数据库需要系统化的方法，从全局到局部逐步缩小范围：

**第一步：全局概览**
- 检查系统资源（CPU、内存、I/O、网络）
- 查看 `pg_stat_database` 数据库级统计
- 检查连接数和等待情况

**第二步：识别慢查询**
- 使用 `pg_stat_statements` 找出消耗资源最多的 SQL
- 按总时间、平均时间、调用次数等维度排序
- 识别高频低效查询和低频高耗查询

**第三步：深入分析**
- 对慢查询使用 `EXPLAIN (ANALYZE, BUFFERS, TIMING)` 分析
- 识别全表扫描、嵌套循环、排序溢出等问题
- 检查统计信息是否过时

**第四步：实施优化**
- 创建或调整索引
- 重写查询
- 调整配置参数
- 考虑分区改造

**第五步：验证效果**
- 重置统计信息
- 监控优化后的性能指标
- 对比优化前后的执行计划

### 2. pg_stat_statements 详解

`pg_stat_statements` 是 PostgreSQL 最重要的性能分析扩展，它记录所有 SQL 语句的执行统计信息。

关键指标：
- `calls`：执行次数
- `total_exec_time`：总执行时间（毫秒）
- `mean_exec_time`：平均执行时间
- `rows`：返回/影响的总行数
- `shared_blks_hit`：共享缓冲区命中次数
- `shared_blks_read`：从磁盘读取的块数
- `temp_blks_written`：临时文件写入的块数

### 3. EXPLAIN ANALYZE 深度解读

EXPLAIN ANALYZE 的关键输出：
- **actual time**：实际执行时间（启动时间..总时间）
- **rows**：实际行数 vs 估计行数（差距大说明统计信息不准）
- **loops**：循环次数（嵌套循环中需要乘以 loops）
- **Buffers**：缓冲区访问情况（hit=缓存 / read=磁盘）
- **Sort Method**：排序方法（quicksort=内存 / external merge=磁盘溢出）

### 4. 索引优化策略

根据 EXPLAIN ANALYZE 结果选择索引类型：
- **Seq Scan on large table**：考虑 B-tree 索引
- **Filter with low selectivity**：考虑部分索引
- **LIKE '%xxx%'**：考虑 pg_trgm GIN 索引
- **JSONB 查询**：考虑 GIN 索引
- **范围查询**：考虑 BRIN 索引（有序数据）
- **全文搜索**：GIN 索引
- **地理数据/范围类型**：GiST 索引

### 5. 分区改造评估

什么时候考虑分区：
- 单表数据量超过数千万行
- 查询经常带有分区键条件（如日期范围）
- 需要高效的数据归档和清理
- 索引维护成本过高

### 示例2：Top 10 慢查询分析

```sql
-- === Top 10：按总执行时间排序（最消耗资源的查询）===
SELECT
    round(total_exec_time::numeric, 2) AS total_time_ms,
    calls,
    round(mean_exec_time::numeric, 2) AS avg_time_ms,
    round((100 * total_exec_time / sum(total_exec_time) OVER ())::numeric, 2) AS pct,
    rows,
    query
FROM pg_stat_statements
ORDER BY total_exec_time DESC
LIMIT 10;

-- === Top 10：按平均执行时间排序（最慢的单次查询）===
SELECT
    round(mean_exec_time::numeric, 2) AS avg_time_ms,
    round(max_exec_time::numeric, 2) AS max_time_ms,
    calls,
    rows / NULLIF(calls, 0) AS avg_rows,
    query
FROM pg_stat_statements
WHERE calls >= 10  -- 至少执行过 10 次
ORDER BY mean_exec_time DESC
LIMIT 10;

-- === Top 10：按调用次数排序（最频繁的查询）===
SELECT
    calls,
    round(total_exec_time::numeric, 2) AS total_time_ms,
    round(mean_exec_time::numeric, 2) AS avg_time_ms,
    rows,
    query
FROM pg_stat_statements
ORDER BY calls DESC
LIMIT 10;

-- === Top 10：按 I/O 消耗排序（最消耗磁盘的查询）===
SELECT
    shared_blks_read + shared_blks_dirtied AS total_io_blocks,
    shared_blks_hit,
    shared_blks_read,
    round(
        shared_blks_hit::numeric * 100 /
        NULLIF(shared_blks_hit + shared_blks_read, 0), 2
    ) AS cache_hit_pct,
    calls,
    query
FROM pg_stat_statements
WHERE shared_blks_read > 0
ORDER BY shared_blks_read DESC
LIMIT 10;

-- === Top 10：使用临时文件的查询（内存不足导致溢出）===
SELECT
    temp_blks_written,
    temp_blks_read,
    calls,
    round(mean_exec_time::numeric, 2) AS avg_time_ms,
    query
FROM pg_stat_statements
WHERE temp_blks_written > 0
ORDER BY temp_blks_written DESC
LIMIT 10;

-- === 综合诊断看板 ===
WITH totals AS (
    SELECT sum(total_exec_time) AS total_time FROM pg_stat_statements
)
SELECT
    queryid,
    calls,
    round(total_exec_time::numeric, 2) AS total_ms,
    round(mean_exec_time::numeric, 2) AS avg_ms,
    round(stddev_exec_time::numeric, 2) AS stddev_ms,
    round((total_exec_time / t.total_time * 100)::numeric, 2) AS pct_total,
    rows / NULLIF(calls, 0) AS avg_rows,
    round(
        shared_blks_hit::numeric * 100 /
        NULLIF(shared_blks_hit + shared_blks_read, 0), 2
    ) AS cache_hit_pct,
    left(query, 80) AS query_preview
FROM pg_stat_statements, totals t
ORDER BY total_exec_time DESC
LIMIT 20;
```

### 示例4：索引优化实战

```sql
-- === 问题1：高频查询缺少索引 ===
-- pg_stat_statements 发现此查询调用 50000 次/天，平均 200ms
-- SELECT * FROM orders_large WHERE customer_id = $1 AND status = $2

-- 创建复合索引
CREATE INDEX idx_orders_customer_status ON orders_large(customer_id, status);

-- 验证
EXPLAIN (ANALYZE, BUFFERS)
SELECT * FROM orders_large WHERE customer_id = 5000 AND status = 'pending';
-- 应该看到 Index Scan，时间降至几毫秒

-- === 问题2：日期范围查询慢 ===
-- SELECT count(*) FROM orders_large WHERE order_date BETWEEN $1 AND $2

-- 方案A：B-tree 索引
CREATE INDEX idx_orders_date ON orders_large(order_date);

-- 方案B：BRIN 索引（如果数据按时间顺序插入）
CREATE INDEX idx_orders_date_brin ON orders_large USING brin(order_date);
-- BRIN 索引极小，但要求数据物理有序

-- 对比两种索引的大小
SELECT
    indexname,
    pg_size_pretty(pg_relation_size(indexname::regclass)) AS index_size
FROM pg_indexes
WHERE tablename = 'orders_large'
ORDER BY pg_relation_size(indexname::regclass) DESC;

-- === 问题3：模糊搜索慢 ===
-- SELECT * FROM orders_large WHERE region LIKE '%华%'
CREATE EXTENSION IF NOT EXISTS pg_trgm;
CREATE INDEX idx_orders_region_trgm ON orders_large USING gin(region gin_trgm_ops);

EXPLAIN (ANALYZE)
SELECT * FROM orders_large WHERE region LIKE '%华%';

-- === 问题4：覆盖索引优化 ===
-- 高频查询只需要 id、amount、status
CREATE INDEX idx_orders_covering
    ON orders_large(customer_id, status)
    INCLUDE (amount, order_date);

-- Index Only Scan（无需回表）
EXPLAIN (ANALYZE, BUFFERS)
SELECT customer_id, status, amount, order_date
FROM orders_large
WHERE customer_id = 5000 AND status = 'shipped';

-- === 问题5：部分索引 ===
-- 95% 的查询只关心未完成的订单
CREATE INDEX idx_orders_active
    ON orders_large(customer_id, order_date)
    WHERE status NOT IN ('delivered', 'cancelled');

-- 索引只包含活跃订单，更小更快
EXPLAIN (ANALYZE)
SELECT * FROM orders_large
WHERE customer_id = 5000
  AND status NOT IN ('delivered', 'cancelled')
  AND order_date >= '2023-06-01';

-- === 检查索引使用情况 ===
SELECT
    schemaname,
    relname AS table_name,
    indexrelname AS index_name,
    idx_scan AS times_used,
    idx_tup_read,
    idx_tup_fetch,
    pg_size_pretty(pg_relation_size(indexrelid)) AS index_size
FROM pg_stat_user_indexes
WHERE relname = 'orders_large'
ORDER BY idx_scan DESC;

-- 找出从未使用的索引
SELECT
    indexrelname,
    idx_scan,
    pg_size_pretty(pg_relation_size(indexrelid)) AS size
FROM pg_stat_user_indexes
WHERE idx_scan = 0
  AND schemaname = 'public'
ORDER BY pg_relation_size(indexrelid) DESC;
```

### 示例5：分区改造实战

```sql
-- === 评估是否需要分区 ===
-- 检查表大小
SELECT
    pg_size_pretty(pg_total_relation_size('orders_large')) AS total_size,
    pg_size_pretty(pg_relation_size('orders_large')) AS table_size,
    pg_size_pretty(pg_indexes_size('orders_large')) AS indexes_size,
    (SELECT count(*) FROM orders_large) AS row_count;

-- === 创建分区表 ===
CREATE TABLE orders_partitioned (
    id SERIAL,
    customer_id INTEGER,
    product_id INTEGER,
    amount NUMERIC(10,2),
    status TEXT,
    order_date TIMESTAMP NOT NULL,
    region TEXT,
    metadata JSONB
) PARTITION BY RANGE (order_date);

-- 创建月度分区
CREATE TABLE orders_p2023_01 PARTITION OF orders_partitioned
    FOR VALUES FROM ('2023-01-01') TO ('2023-02-01');
CREATE TABLE orders_p2023_02 PARTITION OF orders_partitioned
    FOR VALUES FROM ('2023-02-01') TO ('2023-03-01');
CREATE TABLE orders_p2023_03 PARTITION OF orders_partitioned
    FOR VALUES FROM ('2023-03-01') TO ('2023-04-01');
CREATE TABLE orders_p2023_04 PARTITION OF orders_partitioned
    FOR VALUES FROM ('2023-04-01') TO ('2023-05-01');
CREATE TABLE orders_p2023_05 PARTITION OF orders_partitioned
    FOR VALUES FROM ('2023-05-01') TO ('2023-06-01');
CREATE TABLE orders_p2023_06 PARTITION OF orders_partitioned
    FOR VALUES FROM ('2023-06-01') TO ('2023-07-01');
-- ... 继续创建其他月份

-- 创建默认分区（接收不匹配任何分区的数据）
CREATE TABLE orders_p_default PARTITION OF orders_partitioned DEFAULT;

-- 批量创建分区的函数
CREATE OR REPLACE FUNCTION create_monthly_partitions(
    p_table TEXT,
    p_start_date DATE,
    p_end_date DATE
) RETURNS VOID AS $$
DECLARE
    v_date DATE := p_start_date;
    v_partition_name TEXT;
BEGIN
    WHILE v_date < p_end_date LOOP
        v_partition_name := p_table || '_p' || to_char(v_date, 'YYYY_MM');
        EXECUTE format(
            'CREATE TABLE IF NOT EXISTS %I PARTITION OF %I
             FOR VALUES FROM (%L) TO (%L)',
            v_partition_name, p_table,
            v_date, v_date + interval '1 month'
        );
        v_date := v_date + interval '1 month';
    END LOOP;
END;
$$ LANGUAGE plpgsql;

-- === 数据迁移 ===
-- 将旧表数据迁移到分区表
INSERT INTO orders_partitioned
    (customer_id, product_id, amount, status, order_date, region, metadata)
SELECT
    customer_id, product_id, amount, status, order_date, region, metadata
FROM orders_large
WHERE order_date >= '2023-01-01' AND order_date < '2024-01-01';

-- 在分区表上创建索引（会自动应用到所有分区）
CREATE INDEX idx_part_customer ON orders_partitioned(customer_id);
CREATE INDEX idx_part_date_status ON orders_partitioned(order_date, status);

-- === 验证分区裁剪 ===
EXPLAIN (ANALYZE)
SELECT count(*), sum(amount)
FROM orders_partitioned
WHERE order_date >= '2023-03-01' AND order_date < '2023-04-01';
-- 应该只扫描 orders_p2023_03 分区

EXPLAIN (ANALYZE)
SELECT *
FROM orders_partitioned
WHERE order_date >= '2023-06-15' AND order_date < '2023-07-15'
  AND customer_id = 5000;
-- 应该只扫描 2 个分区

-- === 分区维护 ===
-- 归档旧分区（分离后变成普通表）
ALTER TABLE orders_partitioned DETACH PARTITION orders_p2023_01;
-- 可以将 orders_p2023_01 移到归档库或删除

-- 清理旧数据（比 DELETE 快得多）
DROP TABLE orders_p2023_01;

-- 查看分区信息
SELECT
    parent.relname AS parent_table,
    child.relname AS partition_name,
    pg_size_pretty(pg_relation_size(child.oid)) AS size,
    pg_stat_get_live_tuples(child.oid) AS live_rows
FROM pg_inherits
JOIN pg_class parent ON pg_inherits.inhparent = parent.oid
JOIN pg_class child ON pg_inherits.inhrelid = child.oid
WHERE parent.relname = 'orders_partitioned'
ORDER BY child.relname;
```

### 示例6：综合诊断流程示例

```sql
-- === 第1步：数据库全局健康检查 ===
SELECT
    datname,
    numbackends AS connections,
    xact_commit AS commits,
    xact_rollback AS rollbacks,
    round(
        xact_commit::numeric * 100 /
        NULLIF(xact_commit + xact_rollback, 0), 2
    ) AS commit_ratio_pct,
    blks_hit,
    blks_read,
    round(
        blks_hit::numeric * 100 /
        NULLIF(blks_hit + blks_read, 0), 2
    ) AS cache_hit_pct,
    tup_returned,
    tup_fetched,
    tup_inserted,
    tup_updated,
    tup_deleted,
    deadlocks,
    temp_files,
    pg_size_pretty(temp_bytes) AS temp_bytes
FROM pg_stat_database
WHERE datname = current_database();

-- === 第2步：检查当前活跃查询 ===
SELECT
    pid,
    now() - query_start AS duration,
    state,
    wait_event_type,
    wait_event,
    left(query, 100) AS query
FROM pg_stat_activity
WHERE state != 'idle'
  AND pid != pg_backend_pid()
ORDER BY duration DESC;

-- === 第3步：检查锁等待 ===
SELECT
    blocked.pid AS blocked_pid,
    blocked.query AS blocked_query,
    now() - blocked.query_start AS waiting_duration,
    blocking.pid AS blocking_pid,
    blocking.query AS blocking_query
FROM pg_stat_activity blocked
JOIN pg_locks bl ON bl.pid = blocked.pid AND NOT bl.granted
JOIN pg_locks kl ON kl.relation = bl.relation AND kl.granted AND kl.pid != bl.pid
JOIN pg_stat_activity blocking ON kl.pid = blocking.pid
WHERE blocked.pid != blocking.pid;

-- === 第4步：表膨胀检查 ===
SELECT
    relname,
    n_live_tup,
    n_dead_tup,
    round(n_dead_tup::numeric * 100 / NULLIF(n_live_tup + n_dead_tup, 0), 2) AS dead_pct,
    last_vacuum,
    last_autovacuum,
    last_analyze,
    last_autoanalyze
FROM pg_stat_user_tables
ORDER BY n_dead_tup DESC
LIMIT 20;

-- === 第5步：综合优化建议 ===
-- 基于以上诊断结果，形成优化报告

-- 查看最需要优化的查询（高总时间 + 低缓存命中率）
WITH query_stats AS (
    SELECT
        queryid,
        query,
        calls,
        total_exec_time,
        mean_exec_time,
        shared_blks_hit,
        shared_blks_read,
        CASE WHEN shared_blks_hit + shared_blks_read > 0
            THEN round(shared_blks_hit::numeric * 100 / (shared_blks_hit + shared_blks_read), 2)
            ELSE 100
        END AS cache_hit_pct,
        temp_blks_written
    FROM pg_stat_statements
)
SELECT
    queryid,
    calls,
    round(total_exec_time::numeric, 0) AS total_ms,
    round(mean_exec_time::numeric, 2) AS avg_ms,
    cache_hit_pct,
    temp_blks_written,
    CASE
        WHEN cache_hit_pct < 90 THEN '需要索引优化'
        WHEN mean_exec_time > 1000 THEN '需要查询优化'
        WHEN temp_blks_written > 0 THEN '需要增加 work_mem'
        WHEN calls > 100000 AND mean_exec_time > 10 THEN '高频查询需优化'
        ELSE '暂无问题'
    END AS suggestion,
    left(query, 80) AS query
FROM query_stats
WHERE total_exec_time > 1000  -- 至少消耗 1 秒
ORDER BY total_exec_time DESC
LIMIT 20;
```

### 示例7：优化前后对比报告

```sql
-- 生成优化报告的查询
-- 在优化前保存基线数据
CREATE TABLE perf_baseline AS
SELECT
    queryid,
    query,
    calls,
    total_exec_time,
    mean_exec_time,
    shared_blks_hit,
    shared_blks_read,
    now() AS snapshot_time
FROM pg_stat_statements
WHERE total_exec_time > 100;

-- 重置统计
SELECT pg_stat_statements_reset();

-- ... 执行优化操作 ...
-- ... 等待一段时间收集新数据 ...

-- 对比优化前后
SELECT
    n.queryid,
    left(n.query, 60) AS query,
    b.calls AS old_calls,
    n.calls AS new_calls,
    round(b.mean_exec_time::numeric, 2) AS old_avg_ms,
    round(n.mean_exec_time::numeric, 2) AS new_avg_ms,
    round(
        ((b.mean_exec_time - n.mean_exec_time) / NULLIF(b.mean_exec_time, 0) * 100)::numeric, 1
    ) AS improvement_pct
FROM pg_stat_statements n
JOIN perf_baseline b USING (queryid)
WHERE b.mean_exec_time > 10  -- 只关注原来大于 10ms 的查询
ORDER BY (b.mean_exec_time - n.mean_exec_time) DESC
LIMIT 20;
```

## 动手练习

1. **Top 10 分析**：在你的测试数据库上安装 `pg_stat_statements`，执行各种查询后，使用示例 2 中的查询找出 Top 10 慢查询。分析这些查询为什么慢。

2. **EXPLAIN ANALYZE 实战**：选择 Top 10 中最慢的 3 条查询，使用 `EXPLAIN (ANALYZE, BUFFERS, TIMING)` 详细分析。画出执行计划树，标注每个节点的时间和行数。

3. **索引优化**：根据 EXPLAIN ANALYZE 的结果，为慢查询创建合适的索引。对比创建索引前后的执行计划和执行时间。目标：将每条查询的平均执行时间降低 80% 以上。

4. **分区改造**：将测试表改造为按月分区的表。验证分区裁剪是否生效，对比分区前后的查询性能。测试分区维护操作（添加新分区、归档旧分区）。

5. **综合诊断报告**：编写一个完整的数据库健康检查脚本，输出以下信息：数据库大小、连接数、缓存命中率、Top 10 慢查询、未使用的索引、表膨胀率、锁等待情况。

## 常见问题

### Q1: pg_stat_statements 的 query 字段参数被替换为 $1、$2 后如何找到原始 SQL？

**A:** `pg_stat_statements` 会将查询中的常量替换为参数占位符以合并统计。要找到原始 SQL：
1. 通过 `queryid` 在 `pg_stat_activity` 中查找正在执行的实例。
2. 开启 `log_min_duration_statement` 记录慢查询日志，日志中包含完整 SQL。
3. 使用 `auto_explain` 扩展自动记录慢查询的执行计划。
4. 在应用日志中通过查询模式匹配找到原始参数。

### Q2: 优化后执行计划变差了怎么办？

**A:** 可能的原因和解决方案：
1. **统计信息不准确**：执行 `ANALYZE` 更新表的统计信息。
2. **执行计划缓存**：如果使用 Prepared Statements，PostgreSQL 可能使用了过时的通用计划。执行 `DISCARD ALL` 清除缓存。
3. **新索引导致优化器选错计划**：检查是否需要调整 `random_page_cost`、`effective_cache_size` 等参数。
4. **可以使用 `pg_hint_plan` 扩展强制指定执行计划**（但这是最后手段）。

### Q3: 第四阶段（Day 071-080）的知识如何在这次诊断中综合运用？

**A:** 各知识点的应用场景：
- **Day 071-072 全文搜索/JSONB**：检查是否有不走索引的 JSONB 或全文搜索查询，优化索引类型。
- **Day 073 范围类型**：范围查询应使用 GiST 索引，排他约束需要 GiST 支持。
- **Day 074-075 事务与锁**：检查长事务和锁等待是否导致性能下降。
- **Day 076 连接池**：确认连接数是否合理，是否需要 PgBouncer。
- **Day 077 应用开发**：检查是否存在 N+1 查询、是否使用了批量操作。
- **Day 078 FDW**：外部表查询可能是性能瓶颈，检查下推情况。
- **Day 079 安全**：审计日志是否影响性能，权限检查是否增加开销。
