## 理论讲解

### 什么是数据建模？

数据建模是将现实世界的业务需求转化为数据库结构的过程。一个好的数据模型应该：
- 准确反映业务实体及其关系
- 减少数据冗余，保证数据一致性
- 支持高效的查询和更新操作
- 具备良好的扩展性

### ER 模型（实体-关系模型）

ER 模型由 Peter Chen 在 1976 年提出，是数据建模最经典的方法。核心概念：

**实体（Entity）**：现实世界中可以独立标识的事物。例如：用户、订单、商品。在数据库中对应一张表。

**属性（Attribute）**：实体的特征。例如：用户的姓名、邮箱。在数据库中对应列。

**关系（Relationship）**：实体之间的关联。例如：用户"下单"订单。

**主键（Primary Key）**：唯一标识实体实例的属性或属性组合。

**外键（Foreign Key）**：引用另一个实体主键的属性，用于建立关系。

### 数据库范式

范式（Normal Form）是一组规则，用于评估和改进数据库设计的质量。每个更高的范式建立在前一个范式的基础上。

#### 第一范式（1NF）

**规则：每个列都是原子的（不可再分），每行都是唯一的。**

违反 1NF 的例子：
```
| 学生ID | 姓名 | 电话                  |
|--------|------|-----------------------|
| 1      | 张三 | 138xxx, 139xxx        |  ← 一个字段存多个值，违反1NF
```

符合 1NF：
```
| 学生ID | 姓名 | 电话号码 |
|--------|------|----------|
| 1      | 张三 | 138xxx   |
| 1      | 张三 | 139xxx   |
```

或者更好的方式是拆成两张表。

#### 第二范式（2NF）

**规则：满足 1NF，且所有非主键列都完全依赖于整个主键（消除部分依赖）。**

违反 2NF 的例子（复合主键：学生ID + 课程ID）：
```
| 学生ID | 课程ID | 学生姓名 | 课程名称 | 成绩 |
|--------|--------|----------|----------|------|
```
- "学生姓名"只依赖于"学生ID"，不依赖"课程ID" → 部分依赖
- "课程名称"只依赖于"课程ID" → 部分依赖

符合 2NF：拆分为学生表、课程表、选课表。

#### 第三范式（3NF）

**规则：满足 2NF，且所有非主键列直接依赖于主键（消除传递依赖）。**

违反 3NF 的例子：
```
| 员工ID | 姓名 | 部门ID | 部门名称   | 部门位置 |
|--------|------|--------|-----------|---------|
```
- "部门名称"和"部门位置"依赖于"部门ID"，而"部门ID"依赖于"员工ID" → 传递依赖

符合 3NF：拆分为员工表和部门表。

### 反范式化

虽然范式化减少了冗余，但过度范式化可能导致过多的 JOIN 操作，影响查询性能。反范式化是有意引入冗余以提高查询效率的策略。

**常见反范式化手段：**
- **冗余列**：在订单表中存储商品名称（避免每次 JOIN 商品表）
- **汇总表**：预计算的统计数据（如每日销售汇总）
- **物化视图**：定期刷新的查询结果缓存

**何时考虑反范式化：**
- 查询远多于写入的场景
- JOIN 操作成为明显的性能瓶颈
- 数据变更频率低，冗余数据维护成本可控

### 关系设计模式

#### 一对一（1:1）

一个实体实例最多对应另一个实体的一个实例。

```
用户 ←→ 用户详情
```

实现方式：在一方表中放置另一方的主键作为外键（加 UNIQUE 约束），或共享主键。

#### 一对多（1:N）

一个实体实例对应多个另一实体的实例。

```
部门 ←→ 员工（一个部门有多个员工）
```

实现方式：在"多"方表中放置"一"方的主键作为外键。

#### 多对多（M:N）

两个实体之间互相可以有多个对应。

```
学生 ←→ 课程（一个学生选多门课，一门课有多个学生）
```

实现方式：创建中间表（关联表/连接表）。

## 代码示例

### 示例 1：一对一关系

```sql
-- 用户基本信息表
CREATE TABLE users (
    user_id    SERIAL PRIMARY KEY,
    username   VARCHAR(50) UNIQUE NOT NULL,
    email      VARCHAR(100) UNIQUE NOT NULL,
    created_at TIMESTAMP DEFAULT NOW()
);

-- 用户详情表（一对一：共享主键方式）
CREATE TABLE user_profiles (
    user_id    INTEGER PRIMARY KEY
               REFERENCES users(user_id) ON DELETE CASCADE,
    real_name  VARCHAR(100),
    phone      VARCHAR(20),
    address    TEXT,
    birthday   DATE,
    avatar_url TEXT
);

-- 插入数据
INSERT INTO users (username, email) VALUES ('alice', 'alice@example.com');
INSERT INTO user_profiles (user_id, real_name, phone)
VALUES (1, '爱丽丝', '13800001111');

-- 查询：JOIN 两表获取完整信息
SELECT u.username, u.email, p.real_name, p.phone
FROM users u
LEFT JOIN user_profiles p ON u.user_id = p.user_id;
```

### 示例 2：一对多关系

```sql
-- 部门表（"一"方）
CREATE TABLE departments (
    dept_id   SERIAL PRIMARY KEY,
    dept_name VARCHAR(100) NOT NULL UNIQUE,
    location  VARCHAR(200)
);

-- 员工表（"多"方，包含外键）
CREATE TABLE employees (
    emp_id     SERIAL PRIMARY KEY,
    first_name VARCHAR(50) NOT NULL,
    last_name  VARCHAR(50) NOT NULL,
    salary     NUMERIC(10,2) NOT NULL CHECK (salary > 0),
    dept_id    INTEGER REFERENCES departments(dept_id)
               ON DELETE SET NULL
               ON UPDATE CASCADE,
    hire_date  DATE DEFAULT CURRENT_DATE
);

-- 插入数据
INSERT INTO departments (dept_name, location) VALUES
    ('技术部', '3楼A区'),
    ('市场部', '5楼B区');

INSERT INTO employees (first_name, last_name, salary, dept_id) VALUES
    ('张', '三', 15000, 1),
    ('李', '四', 12000, 1),
    ('王', '五', 10000, 2);

-- 查询某部门的所有员工
SELECT d.dept_name, e.first_name || e.last_name AS name, e.salary
FROM departments d
JOIN employees e ON d.dept_id = e.dept_id
WHERE d.dept_name = '技术部';
```

### 示例 3：多对多关系

```sql
-- 学生表
CREATE TABLE students (
    student_id SERIAL PRIMARY KEY,
    name       VARCHAR(50) NOT NULL,
    grade      INTEGER NOT NULL
);

-- 课程表
CREATE TABLE courses (
    course_id   SERIAL PRIMARY KEY,
    course_name VARCHAR(100) NOT NULL,
    credit      INTEGER NOT NULL
);

-- 选课中间表（多对多关联）
CREATE TABLE student_courses (
    student_id INTEGER REFERENCES students(student_id) ON DELETE CASCADE,
    course_id  INTEGER REFERENCES courses(course_id)  ON DELETE CASCADE,
    enrolled_at DATE DEFAULT CURRENT_DATE,
    score       NUMERIC(5,2),
    PRIMARY KEY (student_id, course_id)  -- 复合主键防止重复选课
);

-- 插入数据
INSERT INTO students (name, grade) VALUES ('张三', 2023), ('李四', 2023);
INSERT INTO courses (course_name, credit) VALUES ('数据库原理', 4), ('操作系统', 3);

INSERT INTO student_courses (student_id, course_id) VALUES
    (1, 1), (1, 2),  -- 张三选了两门课
    (2, 1);           -- 李四选了一门课

-- 查询某学生的所有课程
SELECT s.name, c.course_name, c.credit
FROM students s
JOIN student_courses sc ON s.student_id = sc.student_id
JOIN courses c ON sc.course_id = c.course_id
WHERE s.name = '张三';

-- 查询某课程的所有学生
SELECT c.course_name, s.name, sc.score
FROM courses c
JOIN student_courses sc ON c.course_id = sc.course_id
JOIN students s ON sc.student_id = s.student_id
WHERE c.course_name = '数据库原理';
```

### 示例 4：范式化设计 — 从非规范到 3NF

```sql
-- 非规范化的订单表（违反多个范式）
-- | 订单ID | 客户名 | 客户电话 | 商品列表          | 总金额 |
-- | 1      | 张三   | 138xxx  | 手机,电脑,耳机     | 15000 |

-- 规范化后的 3NF 设计：

-- 客户表
CREATE TABLE customers (
    customer_id SERIAL PRIMARY KEY,
    name        VARCHAR(100) NOT NULL,
    phone       VARCHAR(20),
    email       VARCHAR(100)
);

-- 商品表
CREATE TABLE products (
    product_id  SERIAL PRIMARY KEY,
    name        VARCHAR(200) NOT NULL,
    price       NUMERIC(10,2) NOT NULL,
    category    VARCHAR(50)
);

-- 订单表
CREATE TABLE orders (
    order_id    SERIAL PRIMARY KEY,
    customer_id INTEGER NOT NULL REFERENCES customers(customer_id),
    order_date  TIMESTAMP DEFAULT NOW(),
    status      VARCHAR(20) DEFAULT 'pending'
);

-- 订单明细表（订单与商品的多对多关系）
CREATE TABLE order_items (
    item_id    SERIAL PRIMARY KEY,
    order_id   INTEGER NOT NULL REFERENCES orders(order_id) ON DELETE CASCADE,
    product_id INTEGER NOT NULL REFERENCES products(product_id),
    quantity   INTEGER NOT NULL CHECK (quantity > 0),
    unit_price NUMERIC(10,2) NOT NULL,  -- 下单时的价格快照（反范式化）
    UNIQUE (order_id, product_id)
);

-- 查询订单详情
SELECT o.order_id, c.name AS customer,
       p.name AS product, oi.quantity,
       oi.unit_price, (oi.quantity * oi.unit_price) AS subtotal
FROM orders o
JOIN customers c ON o.customer_id = c.customer_id
JOIN order_items oi ON o.order_id = oi.order_id
JOIN products p ON oi.product_id = p.product_id
WHERE o.order_id = 1;
```

### 示例 5：反范式化示例

```sql
-- 在订单表中添加冗余的客户名称和订单总额
ALTER TABLE orders ADD COLUMN customer_name VARCHAR(100);
ALTER TABLE orders ADD COLUMN total_amount NUMERIC(12,2) DEFAULT 0;

-- 创建触发器自动维护冗余数据
CREATE OR REPLACE FUNCTION update_order_total()
RETURNS TRIGGER
LANGUAGE plpgsql
AS $$
BEGIN
    UPDATE orders
    SET total_amount = (
        SELECT COALESCE(SUM(quantity * unit_price), 0)
        FROM order_items WHERE order_id = NEW.order_id
    )
    WHERE order_id = NEW.order_id;
    RETURN NEW;
END;
$$;

CREATE TRIGGER trg_update_order_total
    AFTER INSERT OR UPDATE OR DELETE ON order_items
    FOR EACH ROW
    EXECUTE FUNCTION update_order_total();

-- 这样查询订单列表时不需要 JOIN order_items 计算总额
SELECT order_id, customer_name, total_amount, order_date
FROM orders
ORDER BY order_date DESC;
```

## 常见问题

### Q1：范式化到什么程度比较合适？

实际项目中，3NF 通常是一个很好的平衡点。大多数 OLTP（在线事务处理）系统以 3NF 为目标。更高的范式（BCNF、4NF、5NF）在大多数场景下是"过度设计"。

对于 OLAP（在线分析处理）/数据仓库，通常使用星型模型或雪花模型，有意引入反范式化以提高查询性能。

### Q2：一对一关系什么时候应该拆成两张表？

以下情况建议拆分：
- **数据访问模式不同**：基本信息频繁查询，详情信息偶尔查询
- **安全隔离**：敏感数据（如身份证号）单独存放，方便权限控制
- **可选数据**：不是每个用户都有详情信息，避免大量 NULL
- **数据大小**：某些列特别大（如个人简介 TEXT），拆分可提高主表查询效率

### Q3：中间表（关联表）是否需要自己的主键？

两种做法都可以：
- **复合主键**：`PRIMARY KEY (student_id, course_id)`，更简洁，天然防止重复
- **独立主键**：`id SERIAL PRIMARY KEY` + `UNIQUE (student_id, course_id)`，方便 ORM 框架使用

如果中间表有自己的业务属性（如成绩、创建时间），推荐使用独立主键。
