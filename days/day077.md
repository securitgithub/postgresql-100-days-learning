- 理解 ORM 的 N+1 查询问题及其解决方案

- 学会批量操作（Bulk Insert/Update/Upsert）的最佳实践

## 理论讲解

### 1. ORM 与 N+1 问题

ORM（Object-Relational Mapping）将数据库表映射为编程语言中的对象，简化开发。但 ORM 常见的 N+1 查询问题严重影响性能：

**N+1 问题**：查询一个列表（1 次查询），然后对列表中的每个元素分别查询关联数据（N 次查询）。例如，查询 100 篇文章及其作者需要 1 + 100 = 101 次查询。

解决方案：
- **Eager Loading（预加载）**：使用 JOIN 一次查询所有数据
- **Batch Loading（批量加载）**：使用 WHERE IN (...) 批量获取关联数据
- **Subquery Loading**：使用子查询加载关联数据

### 2. Prepared Statements

Prepared Statements（预处理语句）将 SQL 的解析、重写、规划阶段与执行阶段分离。同一 SQL 模板只需准备一次，可以用不同参数多次执行。

优势：
- 减少重复的解析和规划开销
- 防止 SQL 注入（参数与 SQL 分离）
- 在某些情况下生成更优的执行计划（通用计划 vs 自定义计划）

PostgreSQL 在前 5 次执行使用自定义计划（Custom Plan），之后如果通用计划（Generic Plan）成本不明显更差，就切换到通用计划。

### 3. 批量操作

单条 INSERT 的网络往返和事务开销在大量数据时非常可观。批量操作技术：
- **多值 INSERT**：`INSERT INTO t VALUES (...), (...), (...)`
- **COPY**：PostgreSQL 的高速数据加载命令
- **Unnest + INSERT**：使用数组参数批量插入
- **批量 UPDATE**：使用 FROM + VALUES 或 CTE
- **UPSERT**：`INSERT ... ON CONFLICT DO UPDATE`

### 4. LISTEN/NOTIFY 机制

PostgreSQL 内置的发布/订阅消息系统：
- `NOTIFY channel [, payload]`：向频道发送消息
- `LISTEN channel`：订阅频道
- `UNLISTEN channel`：取消订阅

特点：
- 消息在发送事务提交后才会传递
- 消息是临时的，不持久化
- 同一事务中相同频道的相同消息会合并
- 负载（payload）最大 8000 字节

## 代码示例

### 示例1：N+1 问题演示与解决

```sql
-- 创建示例表
CREATE TABLE authors (
    id SERIAL PRIMARY KEY,
    name TEXT NOT NULL,
    email TEXT
);

CREATE TABLE posts (
    id SERIAL PRIMARY KEY,
    author_id INTEGER REFERENCES authors(id),
    title TEXT NOT NULL,
    content TEXT,
    created_at TIMESTAMP DEFAULT now()
);

CREATE TABLE comments (
    id SERIAL PRIMARY KEY,
    post_id INTEGER REFERENCES posts(id),
    commenter TEXT NOT NULL,
    body TEXT NOT NULL,
    created_at TIMESTAMP DEFAULT now()
);

-- 插入测试数据
INSERT INTO authors (name, email) VALUES
('张三', 'zhang@example.com'),
('李四', 'li@example.com'),
('王五', 'wang@example.com');

INSERT INTO posts (author_id, title, content)
SELECT
    (random() * 2 + 1)::int,
    'Article ' || i,
    'Content of article ' || i
FROM generate_series(1, 50) i;

INSERT INTO comments (post_id, commenter, body)
SELECT
    (random() * 49 + 1)::int,
    'User' || (random() * 10 + 1)::int,
    'Comment text ' || i
FROM generate_series(1, 200) i;

-- === N+1 问题（ORM 常见的查询模式）===
-- 第 1 步：查询所有文章（1 次查询）
SELECT id, title, author_id FROM posts LIMIT 10;

-- 第 2-11 步：对每篇文章查询作者（10 次查询！）
SELECT name FROM authors WHERE id = 1;
SELECT name FROM authors WHERE id = 2;
SELECT name FROM authors WHERE id = 3;
-- ... 重复 N 次

-- === 解决方案 1：JOIN 预加载 ===
SELECT
    p.id, p.title, p.content,
    a.name AS author_name, a.email AS author_email
FROM posts p
JOIN authors a ON p.author_id = a.id
LIMIT 10;
-- 只需 1 次查询

-- === 解决方案 2：IN 批量加载 ===
-- 第 1 步：查询文章
SELECT id, title, author_id FROM posts LIMIT 10;
-- 第 2 步：批量查询所有需要的作者（1 次查询）
SELECT id, name, email FROM authors WHERE id IN (1, 2, 3);
-- 总共 2 次查询（而非 11 次）

-- === 解决方案 3：子查询 + 横向连接 ===
SELECT
    p.id, p.title,
    a.name AS author_name,
    (SELECT count(*) FROM comments c WHERE c.post_id = p.id) AS comment_count
FROM posts p
JOIN authors a ON p.author_id = a.id
ORDER BY p.created_at DESC
LIMIT 10;

-- === 解决方案 4：使用聚合避免多次查询 ===
SELECT
    p.id, p.title,
    a.name AS author_name,
    coalesce(cc.cnt, 0) AS comment_count
FROM posts p
JOIN authors a ON p.author_id = a.id
LEFT JOIN (
    SELECT post_id, count(*) AS cnt
    FROM comments
    GROUP BY post_id
) cc ON cc.post_id = p.id
ORDER BY p.created_at DESC
LIMIT 10;
```

### 示例2：Prepared Statements

```sql
-- 创建 Prepared Statement
PREPARE get_post_by_id(integer) AS
    SELECT p.id, p.title, p.content, a.name AS author
    FROM posts p
    JOIN authors a ON p.author_id = a.id
    WHERE p.id = $1;

-- 执行 Prepared Statement
EXECUTE get_post_by_id(1);
EXECUTE get_post_by_id(5);
EXECUTE get_post_by_id(10);

-- 查看当前会话的 Prepared Statements
SELECT name, statement, parameter_types, result_types
FROM pg_prepared_statements;

-- 查看执行计划（Generic Plan vs Custom Plan）
EXPLAIN EXECUTE get_post_by_id(1);

-- 释放 Prepared Statement
DEALLOCATE get_post_by_id;

-- 释放所有
DEALLOCATE ALL;

-- 带多参数的 Prepared Statement
PREPARE search_posts(text, integer, integer) AS
    SELECT id, title, created_at
    FROM posts
    WHERE title ILIKE '%' || $1 || '%'
    ORDER BY created_at DESC
    LIMIT $2 OFFSET $3;

EXECUTE search_posts('Article', 10, 0);
EXECUTE search_posts('Article', 10, 10);

-- 自定义计划 vs 通用计划
-- PostgreSQL 前 5 次使用自定义计划（考虑参数值）
-- 之后可能切换到通用计划（不考虑参数值）
-- 可以通过 plan_cache_mode 控制行为
SET plan_cache_mode = 'force_generic_plan';   -- 强制通用计划
SET plan_cache_mode = 'force_custom_plan';    -- 强制自定义计划
SET plan_cache_mode = 'auto';                 -- 自动选择（默认）
```

### 示例4：批量 UPDATE 和 UPSERT

```sql
-- === 批量 UPDATE：使用 FROM + VALUES ===
-- 同时更新多行的不同值
UPDATE posts
SET title = v.new_title
FROM (VALUES
    (1, 'New Title for Post 1'),
    (2, 'New Title for Post 2'),
    (5, 'New Title for Post 5')
) AS v(id, new_title)
WHERE posts.id = v.id;

-- === 批量 UPDATE：使用 UNNEST ===
UPDATE posts
SET title = data.new_title
FROM unnest(
    ARRAY[1, 2, 3],
    ARRAY['Title A', 'Title B', 'Title C']
) AS data(id, new_title)
WHERE posts.id = data.id;

-- === UPSERT（INSERT ... ON CONFLICT）===
-- 创建带唯一约束的表
CREATE TABLE user_settings (
    user_id INTEGER,
    key TEXT,
    value TEXT,
    updated_at TIMESTAMP DEFAULT now(),
    PRIMARY KEY (user_id, key)
);

-- 单条 UPSERT
INSERT INTO user_settings (user_id, key, value) VALUES (1, 'theme', 'dark')
ON CONFLICT (user_id, key)
DO UPDATE SET
    value = EXCLUDED.value,
    updated_at = now();

-- 批量 UPSERT
INSERT INTO user_settings (user_id, key, value) VALUES
(1, 'theme', 'light'),
(1, 'language', 'zh'),
(2, 'theme', 'dark'),
(2, 'notifications', 'on')
ON CONFLICT (user_id, key)
DO UPDATE SET
    value = EXCLUDED.value,
    updated_at = now();

-- 批量 UPSERT + UNNEST
INSERT INTO user_settings (user_id, key, value)
SELECT * FROM unnest(
    ARRAY[1, 1, 2],
    ARRAY['font_size', 'theme', 'language'],
    ARRAY['14', 'dark', 'en']
)
ON CONFLICT (user_id, key)
DO UPDATE SET
    value = EXCLUDED.value,
    updated_at = now();

-- UPSERT + DO NOTHING（忽略冲突）
INSERT INTO user_settings (user_id, key, value) VALUES (1, 'theme', 'dark')
ON CONFLICT DO NOTHING;
```

### 示例5：LISTEN/NOTIFY 基本用法

```sql
-- === 会话 A（监听者）===
LISTEN order_events;
LISTEN user_events;

-- 等待通知（在 psql 中）
-- 收到通知后会显示：
-- Asynchronous notification "order_events" with payload "..." received from server process with PID 12345.

-- === 会话 B（发送者）===
-- 发送简单通知
NOTIFY order_events, 'new_order:1001';

-- 在事务中发送（提交后才会传递）
BEGIN;
INSERT INTO orders (customer_name, product, amount, order_date)
VALUES ('新客户', '商品X', 299, CURRENT_DATE);
NOTIFY order_events, '{"type":"new_order","order_id":1001,"amount":299}';
COMMIT;
-- 此时会话 A 收到通知

-- 事务回滚则通知不会发送
BEGIN;
NOTIFY order_events, 'this_will_not_be_sent';
ROLLBACK;
-- 会话 A 不会收到通知

-- 使用 pg_notify 函数（支持动态频道名）
SELECT pg_notify('order_events', '{"type":"status_change","order_id":1001,"status":"shipped"}');

-- === 回到会话 A ===
-- 取消监听
UNLISTEN order_events;
UNLISTEN *;  -- 取消所有监听
```

### 示例6：LISTEN/NOTIFY 触发器通知

```sql
-- 创建通知触发器：当订单状态变更时自动发送通知
CREATE OR REPLACE FUNCTION notify_order_change()
RETURNS TRIGGER AS $$
DECLARE
    payload JSON;
BEGIN
    payload := json_build_object(
        'operation', TG_OP,
        'table', TG_TABLE_NAME,
        'id', COALESCE(NEW.id, OLD.id),
        'timestamp', now()
    );

    IF TG_OP = 'INSERT' THEN
        payload := json_build_object(
            'operation', 'INSERT',
            'order_id', NEW.id,
            'customer', NEW.customer_name,
            'amount', NEW.amount
        );
    ELSIF TG_OP = 'UPDATE' THEN
        payload := json_build_object(
            'operation', 'UPDATE',
            'order_id', NEW.id,
            'old_product', OLD.product,
            'new_product', NEW.product
        );
    ELSIF TG_OP = 'DELETE' THEN
        payload := json_build_object(
            'operation', 'DELETE',
            'order_id', OLD.id
        );
    END IF;

    PERFORM pg_notify('order_changes', payload::text);
    RETURN COALESCE(NEW, OLD);
END;
$$ LANGUAGE plpgsql;

CREATE TRIGGER trg_order_changes
    AFTER INSERT OR UPDATE OR DELETE ON orders
    FOR EACH ROW
    EXECUTE FUNCTION notify_order_change();

-- 测试：监听者先执行 LISTEN order_changes;
-- 然后插入/更新/删除订单，监听者自动收到通知
INSERT INTO orders (customer_name, product, amount, order_date)
VALUES ('测试用户', '测试商品', 99, CURRENT_DATE);
```

### 示例7：使用 LISTEN/NOTIFY 实现缓存失效

```sql
-- 缓存失效通知系统
CREATE OR REPLACE FUNCTION notify_cache_invalidation()
RETURNS TRIGGER AS $$
BEGIN
    -- 通知应用层清除相关缓存
    PERFORM pg_notify(
        'cache_invalidation',
        json_build_object(
            'table', TG_TABLE_NAME,
            'schema', TG_TABLE_SCHEMA,
            'operation', TG_OP,
            'key', CASE
                WHEN TG_OP = 'DELETE' THEN OLD.id
                ELSE NEW.id
            END
        )::text
    );
    RETURN COALESCE(NEW, OLD);
END;
$$ LANGUAGE plpgsql;

-- 为需要缓存的表添加触发器
CREATE TRIGGER trg_authors_cache
    AFTER INSERT OR UPDATE OR DELETE ON authors
    FOR EACH ROW EXECUTE FUNCTION notify_cache_invalidation();

CREATE TRIGGER trg_posts_cache
    AFTER INSERT OR UPDATE OR DELETE ON posts
```

## 动手练习

1. **N+1 查询检测**：在你常用的 ORM 框架中，编写一个查询获取文章列表及其作者和评论数。开启 SQL 日志，观察生成的 SQL 数量。然后使用 Eager Loading 优化，对比优化前后的 SQL 数量和执行时间。

2. **批量插入性能测试**：分别使用单条 INSERT、多值 INSERT（每批 500 条）、COPY、UNNEST 四种方式插入 10 万条数据。记录每种方式的耗时，得出性能比较结论。

3. **UPSERT 实战**：设计一个商品库存同步场景。外部系统定时推送库存数据（可能是新增也可能是更新），使用 `ON CONFLICT DO UPDATE` 实现幂等的同步操作。

4. **LISTEN/NOTIFY 实时通知**：实现一个简单的实时通知系统。当用户表有新注册时，通过 NOTIFY 通知应用层发送欢迎邮件。用两个 psql 窗口模拟发送者和监听者。

5. **Prepared Statements 性能对比**：编写一个循环执行 1000 次相同模板查询的脚本，分别使用普通查询和 Prepared Statements，对比执行时间。

## 常见问题

### Q1: 如何在不修改 ORM 代码的情况下检测 N+1 查询？

**A:** 几种方法：
1. **开启 PostgreSQL 慢查询日志**：设置 `log_min_duration_statement = 0` 临时记录所有 SQL，分析重复模式。
2. **使用 pg_stat_statements**：查看执行次数（calls）异常高的查询模板。
3. **ORM 框架工具**：Django 有 `django-debug-toolbar`，Rails 有 `bullet` gem，都可以检测 N+1。
4. **应用层日志**：在数据库连接层添加查询计数中间件。
关键指标：如果一个页面请求产生了超过 20 条 SQL 查询，很可能存在 N+1 问题。

### Q2: LISTEN/NOTIFY 在连接池（PgBouncer Transaction Pooling）下能用吗？

**A:** 不能。在 Transaction Pooling 模式下，事务结束后连接归还池中，`LISTEN` 的注册会丢失。解决方案：
1. 使用 Session Pooling 模式专门处理 LISTEN 连接。
2. 维护一个不经过 PgBouncer 的直连连接专用于 LISTEN。
3. 使用外部消息队列（如 Redis Pub/Sub、RabbitMQ）替代 LISTEN/NOTIFY。
推荐方案 2：大多数应用只需要 1-2 个 LISTEN 连接，直连不会带来连接数问题。

### Q3: 批量 INSERT 时应该每批多少条？

**A:** 取决于多个因素：
- 每行数据大小：行越大，每批条数应越少
- 网络带宽：本地连接可以更大批次
- 事务日志压力：过大的单事务会产生大量 WAL
- 一般经验值：每批 500-5000 条，或每批数据总量不超过几 MB
- COPY 方式没有"批次"概念，可以一次性加载大量数据
建议通过实际压测确定最佳批次大小。
