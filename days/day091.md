## 理论讲解

### 1. 什么是时序数据

时序数据（Time-Series Data）是指按照时间顺序记录的数据点集合，常见于：

- **IoT 传感器数据**：温度、湿度、压力等采集值
- **服务器监控指标**：CPU、内存、磁盘IO
- **金融交易数据**：股票价格、交易量
- **日志数据**：应用日志、访问日志

时序数据的典型特征：

| 特征 | 说明 |
|------|------|
| 时间戳为主键 | 每条记录都有一个时间戳 |
| 写多读少 | 数据以追加方式写入，很少更新 |
| 数据量巨大 | 每秒可能产生成千上万条记录 |
| 查询按时间范围 | 常见按最近1小时/1天等查询 |
| 数据老化 | 旧数据可压缩或删除 |

### 2. TimescaleDB 简介

TimescaleDB 是基于 PostgreSQL 的开源时序数据库扩展，完全兼容 SQL 标准。它的核心优势：

- **完全兼容 PostgreSQL**：支持所有 SQL 语法、索引、JOIN、存储过程
- **透明分区**：自动按时间对数据进行分区（chunk），用户无感知
- **高性能写入**：针对时序数据写入场景做了大量优化
- **内置压缩**：原生列式压缩，通常可节省 90%+ 存储空间
- **连续聚合**：物化视图的增量刷新版本，实时聚合效率极高
- **数据保留策略**：自动删除过期数据

### 3. Hypertable 概念

Hypertable 是 TimescaleDB 的核心抽象。它在逻辑上是一张普通的 PostgreSQL 表，但底层会自动按时间维度分割成多个 chunk（块）。

```
┌──────────────────────────────────┐
│          Hypertable (逻辑表)       │
├──────────┬──────────┬────────────┤
│ Chunk 1  │ Chunk 2  │ Chunk 3    │
│ 1月1日-7日│ 1月8日-14日│ 1月15日-21日│
└──────────┴──────────┴────────────┘
```

每个 chunk 就是一个标准的 PostgreSQL 表，继承父表的索引和约束。TimescaleDB 会根据查询条件自动裁剪不需要的 chunk，极大提升查询速度。

### 4. 时序数据压缩

TimescaleDB 的原生压缩将行存格式转换为列存格式，并应用多种压缩算法：

- **Gorilla 压缩**：适用于浮点数（如传感器读数）
- **Delta-delta 编码**：适用于时间戳
- **字典编码**：适用于低基数字符串
- **LZ 压缩**：通用文本压缩

压缩率通常在 **10x - 20x**，即原来占用 100GB 的数据压缩后仅需 5-10GB。

### 5. Continuous Aggregates 连续聚合

Continuous Aggregates 是 TimescaleDB 对 PostgreSQL 物化视图的增强版本。它能够：

- 仅刷新发生变化的时间桶（增量刷新）
- 支持实时查询（结合物化数据和最新原始数据）
- 自动根据刷新策略定时更新

与传统物化视图的对比：

| 特性 | 物化视图 | Continuous Aggregates |
|------|---------|----------------------|
| 刷新方式 | 全量刷新 | 增量刷新 |
| 实时性 | 刷新后才更新 | 可实时合并最新数据 |
| 性能 | 全量扫描 | 仅处理新增数据 |
| 自动化 | 需手动/定时 | 内置自动刷新策略 |

## 代码示例

### 示例1：安装与配置 TimescaleDB

```bash
# Ubuntu/Debian 安装 TimescaleDB
sudo apt install timescaledb-2-postgresql-16

# 运行配置工具调优
sudo timescaledb-tune --pg-config=/usr/bin/pg_config

# 重启 PostgreSQL
sudo systemctl restart postgresql

# 在数据库中启用扩展
psql -d mydb -c "CREATE EXTENSION IF NOT EXISTS timescaledb;"
```

```sql
-- 验证安装
SELECT extname, extversion
FROM pg_extension
WHERE extname = 'timescaledb';

-- 查看 TimescaleDB 信息
SELECT * FROM timescaledb_information.hypertables;
```

### 示例2：创建 Hypertable

```sql
-- 第一步：创建普通表
CREATE TABLE sensor_data (
    time        TIMESTAMPTZ NOT NULL,
    sensor_id   INTEGER NOT NULL,
    location    TEXT NOT NULL,
    temperature DOUBLE PRECISION,
    humidity    DOUBLE PRECISION,
    pressure    DOUBLE PRECISION
);

-- 第二步：将普通表转换为 Hypertable
-- chunk_time_interval 指定每个 chunk 覆盖的时间范围
SELECT create_hypertable(
    'sensor_data',
    'time',
    chunk_time_interval => INTERVAL '1 day'
);

-- 创建带有空间分区的 Hypertable（适用于多租户场景）
SELECT create_hypertable(
    'sensor_data',
    'time',
    partitioning_column => 'sensor_id',
    number_partitions   => 4,
    chunk_time_interval => INTERVAL '1 day'
);

-- 查看 Hypertable 的 chunk 信息
SELECT show_chunks('sensor_data');

-- 查看 chunk 详细信息
SELECT * FROM timescaledb_information.chunks
WHERE hypertable_name = 'sensor_data'
ORDER BY range_start DESC;
```

### 示例3：批量写入时序数据

```sql
-- 单条插入
INSERT INTO sensor_data (time, sensor_id, location, temperature, humidity, pressure)
VALUES (NOW(), 1, '机房A-01', 23.5, 45.2, 1013.25);

-- 批量插入（推荐方式，效率远高于逐条插入）
INSERT INTO sensor_data (time, sensor_id, location, temperature, humidity, pressure)
VALUES
    ('2025-01-01 00:00:00+08', 1, '机房A-01', 22.1, 44.0, 1012.5),
    ('2025-01-01 00:00:00+08', 2, '机房A-02', 23.4, 46.1, 1013.0),
    ('2025-01-01 00:00:00+08', 3, '机房B-01', 21.8, 43.5, 1011.8),
    ('2025-01-01 00:01:00+08', 1, '机房A-01', 22.3, 44.2, 1012.6),
    ('2025-01-01 00:01:00+08', 2, '机房A-02', 23.5, 46.3, 1013.1);

-- 使用 generate_series 生成测试数据
INSERT INTO sensor_data (time, sensor_id, location, temperature, humidity, pressure)
SELECT
    ts,
    (random() * 10 + 1)::int AS sensor_id,
    '机房' || chr(65 + (random()*3)::int) || '-0' || (random()*5+1)::int AS location,
    20 + random() * 10 AS temperature,
    40 + random() * 20 AS humidity,
    1010 + random() * 10 AS pressure
FROM generate_series(
    '2025-01-01'::timestamptz,
    '2025-03-31'::timestamptz,
    INTERVAL '1 minute'
) AS ts;
```

### 示例4：时序查询与 TimescaleDB 函数

```sql
-- 按时间桶聚合（最常用的时序查询模式）
-- time_bucket 是 TimescaleDB 的核心函数
SELECT
    time_bucket('1 hour', time) AS hour,
    sensor_id,
    AVG(temperature) AS avg_temp,
    MAX(temperature) AS max_temp,
    MIN(temperature) AS min_temp,
    COUNT(*) AS readings
FROM sensor_data
WHERE time > NOW() - INTERVAL '7 days'
GROUP BY hour, sensor_id
ORDER BY hour DESC, sensor_id;

-- 使用 first() 和 last() 获取每个时间桶的首末值
SELECT
    time_bucket('1 day', time) AS day,
    sensor_id,
    first(temperature, time) AS open_temp,
    last(temperature, time)  AS close_temp,
    MAX(temperature) AS high_temp,
    MIN(temperature) AS low_temp
FROM sensor_data
WHERE time > NOW() - INTERVAL '30 days'
GROUP BY day, sensor_id
ORDER BY day DESC;

-- 按时间桶统计并填充空缺（使用 time_bucket_gapfill）
SELECT
    time_bucket_gapfill('1 hour', time) AS hour,
    sensor_id,
    AVG(temperature) AS avg_temp,
    locf(AVG(temperature)) AS avg_temp_filled  -- 用前一个值填充空缺
FROM sensor_data
WHERE time BETWEEN '2025-01-01' AND '2025-01-02'
  AND sensor_id = 1
GROUP BY hour, sensor_id
ORDER BY hour;
```

### 示例5：压缩配置

```sql
-- 启用压缩并指定分段和排序规则
ALTER TABLE sensor_data SET (
    timescaledb.compress,
    timescaledb.compress_segmentby = 'sensor_id',
    timescaledb.compress_orderby = 'time DESC'
);

-- 手动压缩指定 chunk
SELECT compress_chunk(c.chunk_name)
FROM timescaledb_information.chunks c
WHERE c.hypertable_name = 'sensor_data'
  AND c.range_end < NOW() - INTERVAL '7 days'
  AND NOT c.is_compressed;

-- 添加自动压缩策略：7天前的数据自动压缩
SELECT add_compression_policy('sensor_data', INTERVAL '7 days');

-- 查看压缩效果
SELECT
    pg_size_pretty(before_compression_total_bytes) AS before_size,
    pg_size_pretty(after_compression_total_bytes)  AS after_size,
    round(
        (1 - after_compression_total_bytes::numeric / before_compression_total_bytes) * 100, 1
    ) AS compression_ratio_pct
FROM hypertable_compression_stats('sensor_data');

-- 解压缩（如需修改历史数据）
SELECT decompress_chunk(c.chunk_name)
FROM timescaledb_information.chunks c
WHERE c.hypertable_name = 'sensor_data'
  AND c.range_start >= '2025-01-01'
  AND c.range_end <= '2025-01-08';
```

### 示例6：Continuous Aggregates

```sql
-- 创建连续聚合视图：每小时温度统计
CREATE MATERIALIZED VIEW sensor_hourly
WITH (timescaledb.continuous) AS
SELECT
    time_bucket('1 hour', time) AS bucket,
    sensor_id,
    AVG(temperature) AS avg_temp,
    MAX(temperature) AS max_temp,
    MIN(temperature) AS min_temp,
    AVG(humidity) AS avg_humidity,
    COUNT(*) AS sample_count
FROM sensor_data
GROUP BY bucket, sensor_id
WITH NO DATA;  -- 先不填充历史数据

-- 手动刷新指定时间范围的数据
CALL refresh_continuous_aggregate(
    'sensor_hourly',
    '2025-01-01',
    '2025-04-01'
);

-- 添加自动刷新策略
SELECT add_continuous_aggregate_policy('sensor_hourly',
    start_offset  => INTERVAL '3 days',   -- 刷新范围起点
    end_offset    => INTERVAL '1 hour',    -- 刷新范围终点
    schedule_interval => INTERVAL '1 hour' -- 执行间隔
);

-- 在连续聚合之上再建连续聚合（层级聚合）
CREATE MATERIALIZED VIEW sensor_daily
WITH (timescaledb.continuous) AS
SELECT
    time_bucket('1 day', bucket) AS bucket,
    sensor_id,
    AVG(avg_temp) AS avg_temp,
    MAX(max_temp) AS max_temp,
    MIN(min_temp) AS min_temp,
    SUM(sample_count) AS total_samples
FROM sensor_hourly
GROUP BY time_bucket('1 day', bucket), sensor_id
WITH NO DATA;

-- 查询连续聚合（与查询普通表完全一致）
SELECT * FROM sensor_hourly
WHERE bucket > NOW() - INTERVAL '24 hours'
  AND sensor_id = 1
ORDER BY bucket DESC;
```

### 示例7：数据保留策略

```sql
-- 自动删除90天前的原始数据
SELECT add_retention_policy('sensor_data', INTERVAL '90 days');

-- 手动删除指定时间前的 chunk
SELECT drop_chunks('sensor_data', older_than => INTERVAL '180 days');

-- 查看已配置的所有策略（压缩/保留/聚合刷新）
SELECT * FROM timescaledb_information.jobs
WHERE hypertable_name = 'sensor_data';

-- 移除保留策略
SELECT remove_retention_policy('sensor_data');
```

## 常见问题

### Q1: TimescaleDB 的 chunk_time_interval 应该设置多大？

**A:** 推荐让每个 chunk 的大小在 25% 左右的可用内存范围内。一般经验法则：
- 数据量小（< 1GB/天）：`INTERVAL '7 days'`
- 数据量中等（1-10GB/天）：`INTERVAL '1 day'`
- 数据量大（> 10GB/天）：`INTERVAL '12 hours'` 或更短

可以通过 `SELECT * FROM chunks_detailed_size('sensor_data')` 查看每个 chunk 的实际大小来调整。

### Q2: 压缩后的数据还能查询和修改吗？

**A:** 压缩后的数据**可以正常查询**，TimescaleDB 会自动在查询时解压缩。但**不能直接修改**压缩数据，需要先调用 `decompress_chunk()` 解压缩对应的 chunk，修改后再重新压缩。INSERT 到已压缩 chunk 的时间范围内的新数据会被暂存，直到下次压缩时合并。

### Q3: Continuous Aggregates 与 PostgreSQL 原生物化视图有什么区别？

**A:** 主要区别在于刷新机制。PostgreSQL 原生物化视图的 `REFRESH MATERIALIZED VIEW` 是全量刷新，数据量大时非常慢。TimescaleDB 的 Continuous Aggregates 是增量刷新，只处理上次刷新后新增或修改的数据，效率高出数个数量级。此外 Continuous Aggregates 还支持实时模式（`timescaledb.materialized_only = false`），查询时会自动合并物化结果和最新的未物化数据。
