## 理论讲解

### 1. 字符编码基础

字符编码决定了数据库如何存储和处理文本数据。PostgreSQL 支持多种编码：

| 编码 | 说明 | 适用场景 |
|------|------|---------|
| UTF8 | Unicode 变长编码（1-4 字节） | 推荐，支持所有语言 |
| LATIN1 | ISO 8859-1，西欧语言 | 旧系统兼容 |
| EUC_CN | GB2312 扩展 | 简体中文（旧系统） |
| EUC_JP | 日文编码 | 日文系统 |
| SQL_ASCII | 无编码检查 | 不推荐，无法保证一致性 |

**强烈建议**：新项目统一使用 UTF-8 编码，这是现代应用的标准选择。

编码在三个层级设置：
1. **集群级别**：`initdb` 时通过 `--encoding` 指定
2. **数据库级别**：`CREATE DATABASE` 时指定
3. **客户端级别**：`client_encoding` 参数

### 2. Locale 概念

Locale（区域设置）影响文本的排序、大小写转换和分类行为。PostgreSQL 中的关键 locale 参数：

| 参数 | 影响范围 |
|------|---------|
| `LC_COLLATE` | 字符串排序顺序（ORDER BY 的结果） |
| `LC_CTYPE` | 字符分类（大小写转换、正则中的 `\w` 等） |
| `LC_MESSAGES` | 系统消息语言 |
| `LC_MONETARY` | 货币格式 |
| `LC_NUMERIC` | 数字格式 |
| `LC_TIME` | 日期时间格式 |

`LC_COLLATE` 和 `LC_CTYPE` 在数据库创建后不可更改（因为它们影响索引排序），其他参数可以随时修改。

### 3. Collation 排序规则

Collation 定义了字符串比较和排序的规则。不同的 collation 会导致完全不同的排序结果。

PostgreSQL 支持两种 collation 提供者：
- **libc**：使用操作系统的 C 库提供的 locale
- **ICU**（PG 10+）：使用 International Components for Unicode 库

关键区别：

| 特性 | libc | ICU |
|------|------|-----|
| 来源 | 操作系统 | ICU 库 |
| 跨平台一致性 | 差（不同 OS 结果可能不同） | 好 |
| 版本稳定性 | OS 升级可能改变排序 | 版本化，可控 |
| 功能丰富度 | 基本 | 更多选项（强度级别等） |
| 非确定性 collation | PG 12+ | PG 12+ |

### 4. ICU Collation 深入

ICU collation 提供更精细的排序控制：

**排序强度级别**：
1. **Primary**：忽略重音和大小写（a = á = A）
2. **Secondary**：区分重音，忽略大小写（a ≠ á, a = A）
3. **Tertiary**：区分重音和大小写（默认）
4. **Quaternary**：区分标点符号

**非确定性 collation（PG 12+）**：
允许不同的字符串在比较时视为相等。例如，大小写不敏感的比较。

### 5. 编码转换

客户端编码与服务器编码不同时，PostgreSQL 会自动进行编码转换。转换规则：
- 服务器编码为 UTF8 时，可以与大部分客户端编码互转
- `SQL_ASCII` 不会进行任何转换（也不会验证）

## 代码示例

### 示例1：查看当前编码和 locale 设置

```sql
-- 查看服务器编码
SHOW server_encoding;

-- 查看客户端编码
SHOW client_encoding;

-- 查看当前数据库的 locale 设置
SELECT datname, encoding, datcollate, datctype,
       pg_encoding_to_char(encoding) AS encoding_name
FROM pg_database;

-- 查看所有 locale 相关参数
SHOW lc_collate;
SHOW lc_ctype;
SHOW lc_messages;
SHOW lc_monetary;
SHOW lc_numeric;
SHOW lc_time;

-- 查看集群默认 locale
SELECT name, setting
FROM pg_settings
WHERE name LIKE 'lc_%';
```

### 示例2：创建指定编码的数据库

```sql
-- 创建 UTF-8 编码的数据库，使用中文排序规则
CREATE DATABASE myapp_cn
    ENCODING 'UTF8'
    LC_COLLATE 'zh_CN.UTF-8'
    LC_CTYPE 'zh_CN.UTF-8'
    TEMPLATE template0;

-- 创建使用 C locale 的数据库（纯字节序排序，性能最好）
CREATE DATABASE myapp_c
    ENCODING 'UTF8'
    LC_COLLATE 'C'
    LC_CTYPE 'C'
    TEMPLATE template0;

-- 创建使用 ICU collation 的数据库（PG 15+）
CREATE DATABASE myapp_icu
    ENCODING 'UTF8'
    LOCALE_PROVIDER icu
    ICU_LOCALE 'zh-CN'
    LC_COLLATE 'C'
    LC_CTYPE 'C'
    TEMPLATE template0;
```

### 示例3：Collation 的使用

```sql
-- 查看系统中可用的 collation
SELECT collname, collprovider, collencoding,
       pg_encoding_to_char(collencoding) AS encoding,
       colliculocale, collcollate
FROM pg_collation
ORDER BY collname
LIMIT 30;

-- 查看 ICU 提供的 collation
SELECT collname, colliculocale
FROM pg_collation
WHERE collprovider = 'i'
ORDER BY collname;

-- 在查询中指定 collation
SELECT name
FROM products
ORDER BY name COLLATE "C";  -- 纯字节序

SELECT name
FROM products
ORDER BY name COLLATE "zh_CN";  -- 中文排序（如果可用）

-- 对比不同 collation 的排序结果
WITH test_data AS (
    SELECT unnest(ARRAY['apple', 'Banana', 'cherry', 'Ångström', 'über', 'café']) AS word
)
SELECT word,
       row_number() OVER (ORDER BY word COLLATE "C") AS c_order,
       row_number() OVER (ORDER BY word COLLATE "en_US") AS en_order
FROM test_data
ORDER BY word COLLATE "C";
```

### 示例4：在列上指定 Collation

```sql
-- 创建表时指定列的 collation
CREATE TABLE multilingual_data (
    id serial PRIMARY KEY,
    name_en text COLLATE "en_US.utf8",
    name_cn text COLLATE "zh_CN.utf8",
    name_de text COLLATE "de_DE.utf8"
);

-- 修改已有列的 collation
ALTER TABLE products ALTER COLUMN name SET DATA TYPE text COLLATE "C";

-- 在索引上使用特定 collation
CREATE INDEX idx_name_c ON products (name COLLATE "C");
-- 使用 C locale 的索引支持 LIKE 'prefix%' 查询优化
```

### 示例5：创建自定义 Collation

```sql
-- 基于已有 collation 创建新的
CREATE COLLATION my_collation (PROVIDER = libc, LOCALE = 'zh_CN.UTF-8');

-- 创建 ICU collation
CREATE COLLATION chinese_pinyin (PROVIDER = icu, LOCALE = 'zh-CN');

-- 创建大小写不敏感的 ICU collation（PG 12+）
CREATE COLLATION case_insensitive (
    PROVIDER = icu,
    LOCALE = 'und-u-ks-level2',
    DETERMINISTIC = false
);

-- 使用非确定性 collation
CREATE TABLE ci_test (
    id serial PRIMARY KEY,
    name text COLLATE case_insensitive
);

INSERT INTO ci_test (name) VALUES ('Hello'), ('World');

-- 大小写不敏感的查询
SELECT * FROM ci_test WHERE name = 'hello';  -- 匹配 'Hello'！
SELECT * FROM ci_test WHERE name = 'WORLD';  -- 匹配 'World'！

-- 注意：非确定性 collation 不支持某些操作
-- 不支持 pattern 操作符（LIKE）和 hash 索引/分区
```

### 示例6：ICU Collation 高级特性

```sql
-- ICU locale 自定义标签
-- 语言-地区-u-扩展键-扩展值

-- 不区分重音的排序
CREATE COLLATION ignore_accents (
    PROVIDER = icu,
    LOCALE = 'und-u-ks-level1',
    DETERMINISTIC = false
);

-- 测试
SELECT 'café' = 'cafe' COLLATE ignore_accents;  -- true

-- 数字感知排序（natural sort）
CREATE COLLATION natural_sort (
    PROVIDER = icu,
    LOCALE = 'en-u-kn-true'
);

-- 对比排序效果
WITH files AS (
    SELECT unnest(ARRAY['file1', 'file10', 'file2', 'file20', 'file3']) AS name
)
SELECT name,
       row_number() OVER (ORDER BY name COLLATE "C") AS byte_order,
       row_number() OVER (ORDER BY name COLLATE natural_sort) AS natural_order
FROM files;
-- natural_sort: file1, file2, file3, file10, file20（符合直觉）
-- C collation: file1, file10, file2, file20, file3（字典序）

-- 电话簿排序（德语特殊排序）
CREATE COLLATION german_phonebook (
    PROVIDER = icu,
    LOCALE = 'de-u-co-phonebk'
);
```

### 示例7：编码转换

```sql
-- 查看支持的编码转换
SELECT * FROM pg_conversion WHERE condefault ORDER BY conname;

-- 设置客户端编码
SET client_encoding = 'UTF8';
SET client_encoding = 'GBK';  -- 适用于某些旧客户端

-- 编码转换函数
SELECT convert('Hello', 'UTF8', 'LATIN1');
SELECT convert_from(E'\\x48656c6c6f', 'UTF8');
SELECT convert_to('Hello', 'UTF8');

-- 字符与编码点
SELECT chr(20013);          -- 返回: 中
SELECT ascii('中');          -- 返回: 20013（Unicode 码点）
SELECT length('你好世界');    -- 返回: 4（字符数）
SELECT octet_length('你好世界');  -- 返回: 12（UTF-8 字节数）
```

### 示例8：C locale 与 LIKE 优化

```sql
-- C locale 下，LIKE 前缀匹配可以使用 B-tree 索引
-- 非 C locale 需要特殊的操作符类

-- 方法1：使用 C collation 的索引
CREATE INDEX idx_name_c_like ON products (name COLLATE "C");
-- 查询时需要匹配 collation
SELECT * FROM products WHERE name COLLATE "C" LIKE 'iPhone%';

-- 方法2：使用 text_pattern_ops 操作符类
CREATE INDEX idx_name_pattern ON products (name text_pattern_ops);
-- 这个索引支持 LIKE 前缀匹配，不需要 C locale
SELECT * FROM products WHERE name LIKE 'iPhone%';  -- 使用索引

-- 方法3：使用 pg_trgm GIN 索引（支持任意位置的 LIKE）
CREATE INDEX idx_name_trgm ON products USING gin (name gin_trgm_ops);
SELECT * FROM products WHERE name LIKE '%iPhone%';  -- 使用索引
```

## 常见问题

### Q1：创建数据库时报错 "locale not available" 怎么办？

这说明操作系统中没有安装对应的 locale。在 Debian/Ubuntu 上，使用 `sudo locale-gen zh_CN.UTF-8 && sudo dpkg-reconfigure locales` 安装。在 RHEL 上，确保 `glibc-langpack-zh` 包已安装。安装后无需重启 PostgreSQL，新创建的数据库即可使用新 locale。

### Q2：数据库创建后能否更改 LC_COLLATE？

不能直接更改。`LC_COLLATE` 和 `LC_CTYPE` 在数据库创建时固定，因为它们影响索引的排序顺序。如需更改，必须用 `pg_dump` 导出数据，创建使用新 locale 的数据库，再导入数据。但可以在查询或列级别使用 `COLLATE` 子句指定不同的排序规则。

### Q3：为什么推荐使用 ICU 而非 libc collation？

libc collation 依赖操作系统的 C 库，不同操作系统和版本可能产生不同的排序结果。操作系统升级后 glibc 版本变化可能导致索引损坏（排序不一致）。ICU collation 有版本化机制，行为更可预测且跨平台一致。PostgreSQL 16 开始将 ICU 作为默认推荐的 collation provider。
