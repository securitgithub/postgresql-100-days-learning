## 理论讲解

### 1. 逻辑备份概述

逻辑备份是将数据库对象和数据导出为 SQL 语句或自定义格式文件的过程。PostgreSQL 提供两个核心工具：

| 工具 | 说明 |
|------|------|
| `pg_dump` | 备份单个数据库 |
| `pg_dumpall` | 备份整个集群（所有数据库 + 全局对象） |

**逻辑备份 vs 物理备份：**

| 特性 | 逻辑备份 | 物理备份 |
|------|----------|----------|
| 粒度 | 数据库/表/Schema | 整个集群 |
| 跨版本恢复 | 支持 | 不支持 |
| 备份速度 | 较慢 | 较快 |
| 恢复速度 | 较慢 | 较快 |
| 文件大小 | 较大（可压缩） | 取决于数据量 |
| PITR 支持 | 不支持 | 支持 |
| 选择性恢复 | 支持 | 有限支持 |

### 2. pg_dump 备份格式

| 格式 | 参数 | 文件类型 | 特点 |
|------|------|----------|------|
| 纯文本 | `-Fp`（默认） | .sql | 可直接用 psql 执行，可读性好 |
| 自定义 | `-Fc` | .dump | 压缩存储，支持选择性恢复，最常用 |
| 目录 | `-Fd` | 目录 | 支持并行备份/恢复，每个表一个文件 |
| tar | `-Ft` | .tar | 兼容 tar 工具，不压缩 |

**推荐：** 生产环境使用自定义格式（`-Fc`）或目录格式（`-Fd`），兼具压缩和灵活恢复的优势。

### 3. pg_dump 一致性保证

pg_dump 在执行时会获取一个**快照**，确保备份的数据一致性：
- 使用 `SERIALIZABLE` 或 `REPEATABLE READ` 隔离级别
- 备份过程中其他事务的修改不会影响备份数据
- 不会阻塞正常的读写操作（但会阻止 DDL 操作如 DROP TABLE）

### 4. pg_dumpall 特殊用途

`pg_dumpall` 用于备份集群级别的全局对象：
- 角色和角色属性
- 表空间定义
- 所有数据库

常见用法：用 `pg_dumpall --globals-only` 单独备份全局对象，配合 `pg_dump` 备份各个数据库。

### 5. pg_restore 恢复工具

`pg_restore` 用于恢复 pg_dump 创建的非纯文本格式备份：
- 支持选择性恢复（指定表、Schema）
- 支持并行恢复
- 支持恢复前生成 SQL 脚本预览

## 代码示例

### 示例 1：pg_dump 基础用法

```bash
# === 纯文本格式备份 ===
# 备份整个数据库为 SQL 文件
pg_dump -h localhost -U postgres -d mydb -Fp -f mydb_backup.sql

# 仅备份数据（不含 Schema 定义）
pg_dump -h localhost -U postgres -d mydb --data-only -f mydb_data.sql

# 仅备份 Schema 定义（不含数据）
pg_dump -h localhost -U postgres -d mydb --schema-only -f mydb_schema.sql

# 备份特定表
pg_dump -h localhost -U postgres -d mydb -t orders -t customers -f tables_backup.sql

# 备份特定 Schema
pg_dump -h localhost -U postgres -d mydb -n sales -n inventory -f schemas_backup.sql

# 排除特定表
pg_dump -h localhost -U postgres -d mydb -T audit_log -T temp_data -f partial_backup.sql

# 备份时包含 CREATE DATABASE 语句
pg_dump -h localhost -U postgres -d mydb -C -f mydb_with_create.sql

# 使用 gzip 压缩纯文本备份
pg_dump -h localhost -U postgres -d mydb | gzip > mydb_backup.sql.gz
```

### 示例 2：自定义格式和目录格式

```bash
# === 自定义格式备份（推荐） ===
# 自动压缩，支持 pg_restore 选择性恢复
pg_dump -h localhost -U postgres -d mydb -Fc -f mydb_backup.dump

# 指定压缩级别（0-9，默认为 6）
pg_dump -h localhost -U postgres -d mydb -Fc -Z 9 -f mydb_backup.dump

# === 目录格式备份（支持并行） ===
# 并行备份（使用 4 个进程）
pg_dump -h localhost -U postgres -d mydb -Fd -j 4 -f mydb_backup_dir/

# 目录结构：
# mydb_backup_dir/
#   toc.dat          -- 目录文件
#   1234.dat.gz      -- 表数据文件（自动压缩）
#   1235.dat.gz
#   ...

# === tar 格式备份 ===
pg_dump -h localhost -U postgres -d mydb -Ft -f mydb_backup.tar

# 查看备份内容（不实际恢复）
pg_restore -l mydb_backup.dump
```

### 示例 3：pg_dump 高级选项

```bash
# 使用 INSERT 语句代替 COPY（兼容性更好，但更慢）
pg_dump -h localhost -U postgres -d mydb --inserts -f mydb_inserts.sql

# 使用多行 INSERT（速度比单行 INSERT 快）
pg_dump -h localhost -U postgres -d mydb --rows-per-insert=1000 -f mydb_bulk.sql

# 排除大对象（Large Objects）
pg_dump -h localhost -U postgres -d mydb --no-blobs -Fc -f mydb_noblobs.dump

# 不备份权限信息
pg_dump -h localhost -U postgres -d mydb --no-privileges -Fc -f mydb_nopriv.dump

# 不备份所有者信息
pg_dump -h localhost -U postgres -d mydb --no-owner -Fc -f mydb_noowner.dump

# 设置备份时的 search_path 不含 public
pg_dump -h localhost -U postgres -d mydb --no-publications --no-subscriptions \
    -Fc -f mydb_clean.dump

# 包含 IF EXISTS 子句（恢复时更安全）
pg_dump -h localhost -U postgres -d mydb --if-exists --clean -Fc -f mydb_safe.dump

# 详细输出模式
pg_dump -h localhost -U postgres -d mydb -Fc -v -f mydb_backup.dump 2>&1 | tee dump.log
```

### 示例 4：pg_dumpall 用法

```bash
# 备份整个集群（所有数据库）
pg_dumpall -h localhost -U postgres -f cluster_backup.sql

# 仅备份全局对象（角色、表空间）
pg_dumpall -h localhost -U postgres --globals-only -f globals.sql

# 仅备份角色定义
pg_dumpall -h localhost -U postgres --roles-only -f roles.sql

# 仅备份表空间定义
pg_dumpall -h localhost -U postgres --tablespaces-only -f tablespaces.sql

# 最佳实践：组合使用
# 1. 备份全局对象
pg_dumpall -h localhost -U postgres --globals-only -f globals.sql

# 2. 分别备份每个数据库（自定义格式，支持并行）
pg_dump -h localhost -U postgres -d db1 -Fc -f db1.dump
pg_dump -h localhost -U postgres -d db2 -Fc -f db2.dump
pg_dump -h localhost -U postgres -d db3 -Fc -f db3.dump
```

### 示例 5：pg_restore 恢复操作

```bash
# === 恢复纯文本格式 ===
# 使用 psql 直接执行 SQL
psql -h localhost -U postgres -d target_db -f mydb_backup.sql

# 恢复 gzip 压缩的备份
gunzip -c mydb_backup.sql.gz | psql -h localhost -U postgres -d target_db

# === 恢复自定义格式 ===
# 恢复到已存在的数据库
pg_restore -h localhost -U postgres -d target_db mydb_backup.dump

# 创建数据库并恢复（如果备份时使用了 -C）
pg_restore -h localhost -U postgres -C -d postgres mydb_backup.dump

# 先清空再恢复
pg_restore -h localhost -U postgres -d target_db --clean --if-exists mydb_backup.dump

# === 选择性恢复 ===
# 只恢复特定表
pg_restore -h localhost -U postgres -d target_db -t orders mydb_backup.dump

# 只恢复特定 Schema
pg_restore -h localhost -U postgres -d target_db -n sales mydb_backup.dump

# 只恢复数据（不恢复 Schema）
pg_restore -h localhost -U postgres -d target_db --data-only mydb_backup.dump

# 只恢复 Schema（不恢复数据）
pg_restore -h localhost -U postgres -d target_db --schema-only mydb_backup.dump

# === 并行恢复（目录格式） ===
pg_restore -h localhost -U postgres -d target_db -j 4 mydb_backup_dir/

# === 生成 SQL 脚本（不实际执行） ===
pg_restore -f restore_script.sql mydb_backup.dump

# === 使用恢复列表控制恢复内容 ===
# 1. 生成目录列表
pg_restore -l mydb_backup.dump > toc.list

# 2. 编辑 toc.list，注释掉不需要恢复的项
# 3. 按编辑后的列表恢复
pg_restore -h localhost -U postgres -d target_db -L toc.list mydb_backup.dump

# === 错误处理 ===
# 遇到错误继续（默认行为）
pg_restore -h localhost -U postgres -d target_db mydb_backup.dump

# 遇到第一个错误即停止
pg_restore -h localhost -U postgres -d target_db --exit-on-error mydb_backup.dump

# 使用单事务（要么全部成功，要么全部回滚）
pg_restore -h localhost -U postgres -d target_db --single-transaction mydb_backup.dump
```

### 示例 6：自动备份脚本

```bash
#!/bin/bash
# === PostgreSQL 自动逻辑备份脚本 ===

# 配置变量
PG_HOST="localhost"
PG_PORT="5432"
PG_USER="backup_user"
BACKUP_DIR="/data/backups/postgresql"
RETENTION_DAYS=30
DATE=$(date +%Y%m%d_%H%M%S)
LOG_FILE="${BACKUP_DIR}/logs/backup_${DATE}.log"

# 需要备份的数据库列表
DATABASES="production staging analytics"

# 创建目录
mkdir -p "${BACKUP_DIR}/daily" "${BACKUP_DIR}/globals" "${BACKUP_DIR}/logs"

# 日志函数
log() {
    echo "[$(date '+%Y-%m-%d %H:%M:%S')] $1" | tee -a "${LOG_FILE}"
}

log "=== 开始备份 ==="

# 1. 备份全局对象
log "备份全局对象..."
pg_dumpall -h ${PG_HOST} -p ${PG_PORT} -U ${PG_USER} \
    --globals-only \
    -f "${BACKUP_DIR}/globals/globals_${DATE}.sql" 2>> "${LOG_FILE}"

if [ $? -eq 0 ]; then
    log "全局对象备份成功"
else
    log "错误：全局对象备份失败！"
fi

# 2. 备份各个数据库
for DB in ${DATABASES}; do
    log "开始备份数据库: ${DB}..."
    DUMP_FILE="${BACKUP_DIR}/daily/${DB}_${DATE}.dump"

    pg_dump -h ${PG_HOST} -p ${PG_PORT} -U ${PG_USER} \
        -d ${DB} \
        -Fc \
        -Z 6 \
        -v \
        -f "${DUMP_FILE}" 2>> "${LOG_FILE}"

    if [ $? -eq 0 ]; then
        SIZE=$(du -sh "${DUMP_FILE}" | cut -f1)
        log "数据库 ${DB} 备份成功，大小: ${SIZE}"
    else
        log "错误：数据库 ${DB} 备份失败！"
    fi
done

# 3. 验证备份完整性
log "验证备份完整性..."
for DB in ${DATABASES}; do
    DUMP_FILE="${BACKUP_DIR}/daily/${DB}_${DATE}.dump"
    if [ -f "${DUMP_FILE}" ]; then
        # 尝试列出备份内容
        pg_restore -l "${DUMP_FILE}" > /dev/null 2>&1
        if [ $? -eq 0 ]; then
            log "验证通过: ${DB}"
        else
            log "警告：${DB} 备份文件可能损坏！"
        fi
    fi
done

# 4. 清理过期备份
log "清理 ${RETENTION_DAYS} 天前的备份..."
find "${BACKUP_DIR}/daily" -name "*.dump" -mtime +${RETENTION_DAYS} -delete
find "${BACKUP_DIR}/globals" -name "*.sql" -mtime +${RETENTION_DAYS} -delete
find "${BACKUP_DIR}/logs" -name "*.log" -mtime +${RETENTION_DAYS} -delete

log "=== 备份完成 ==="

# 5. 发送通知（可选）
# mail -s "PostgreSQL 备份报告 ${DATE}" admin@example.com < "${LOG_FILE}"
```

```bash
# 设置 cron 定时任务
# 每天凌晨 2:00 执行备份
# crontab -e
# 0 2 * * * /path/to/pg_backup.sh

# 使用 .pgpass 文件避免交互式输入密码
# 创建 ~/.pgpass 文件
# hostname:port:database:username:password
# localhost:5432:*:backup_user:your_password
# chmod 600 ~/.pgpass
```

## 常见问题

### Q1：pg_dump 过程中数据库可以正常使用吗？

**A：** 可以。pg_dump 使用快照机制，不会阻塞正常的 DML 操作（SELECT/INSERT/UPDATE/DELETE）。但需要注意：

1. pg_dump 会获取 `ACCESS SHARE` 锁，会阻止 `DROP TABLE` 和 `ALTER TABLE` 等 DDL 操作
2. 大表的备份可能耗时较长，期间会占用一个连接
3. 备份期间的数据修改不会包含在备份中（因为使用快照）
4. 长时间的 pg_dump 会阻止 autovacuum 清理死元组

建议在低峰期执行备份。

### Q2：备份文件太大怎么办？

**A：** 减小备份文件大小的方法：

1. **使用压缩**：自定义格式（`-Fc`）自动压缩；纯文本可用管道压缩
```bash
pg_dump -Fc -Z 9 -f backup.dump mydb        # 最高压缩级别
pg_dump mydb | gzip -9 > backup.sql.gz       # gzip 压缩
pg_dump mydb | zstd -19 > backup.sql.zst     # zstd 压缩（更快更小）
```

2. **排除大表或无需备份的表**：`-T log_table -T temp_table`

3. **分离数据和 Schema**：Schema 定义很小，大数据表单独备份

4. **只备份增量变化的表**：配合时间戳列实现伪增量备份

### Q3：pg_dump 和 pg_dumpall 哪个更适合生产环境？

**A：** 推荐组合使用：

```bash
# 最佳实践
# 1. pg_dumpall 备份全局对象
pg_dumpall --globals-only > globals.sql

# 2. pg_dump 分别备份每个数据库（自定义格式）
for db in $(psql -At -c "SELECT datname FROM pg_database WHERE datistemplate = false AND datname != 'postgres'"); do
    pg_dump -Fc -f "${db}.dump" "$db"
done
```

原因：
- `pg_dumpall` 只支持纯文本格式，不支持并行和选择性恢复
- 分离全局对象和数据库备份，恢复更灵活
- 每个数据库使用自定义格式，支持并行和选择性恢复
