## 理论讲解

### 1. 同步复制 vs 异步复制

PostgreSQL 流复制默认是**异步**的。主库提交事务后不等待从库确认即返回成功。这意味着如果主库突然宕机，可能有极少量已提交但未同步到从库的数据丢失。

**同步复制**要求主库在提交事务时等待至少一个从库确认已收到（甚至已回放）WAL 数据后才返回成功。这保证了零数据丢失（RPO=0），但会增加事务提交延迟。

```
异步复制：主库 COMMIT → 立即返回 → 后台发送 WAL → 从库接收回放
同步复制：主库 COMMIT → 等待从库确认 → 返回成功
```

### 2. synchronous_commit 级别

| 级别 | 说明 | 性能 | 安全性 |
|------|------|------|--------|
| `off` | 不等待 WAL 写入磁盘 | 最快 | 最低 |
| `local` | 仅等待本地 WAL 写入磁盘 | 快 | 本地安全 |
| `remote_write` | 等待从库写入操作系统缓存 | 中等 | 较高 |
| `remote_apply` | 等待从库回放完成 | 最慢 | 最高 |
| `on`（默认） | 同步模式下等待从库 flush | 较慢 | 高 |

### 3. Replication Slots 复制槽

复制槽是 PostgreSQL 9.4 引入的机制，解决了一个关键问题：**确保主库不会在从库消费之前删除所需的 WAL 文件**。

没有复制槽时，如果从库断开时间较长，主库可能已回收了从库需要的 WAL 文件，导致从库无法继续同步，必须重新做基础备份。

复制槽分两种：
- **物理复制槽**：用于流复制，保留 WAL 文件
- **逻辑复制槽**：用于逻辑复制，保留 WAL 和逻辑解码信息

**注意**：复制槽的风险在于，如果从库长时间断开，主库会持续保留 WAL 文件，可能导致磁盘空间耗尽。

### 4. 复制延迟的来源

复制延迟可能由以下原因造成：
- **网络延迟**：主从之间网络带宽不足或延迟高
- **从库负载**：从库执行大量查询影响 WAL 回放速度
- **从库 I/O 瓶颈**：从库磁盘写入速度跟不上 WAL 接收速度
- **长事务**：从库上的长查询可能导致回放冲突暂停
- **大批量写入**：主库瞬间产生大量 WAL 数据

### 5. 级联复制

PostgreSQL 9.2 起支持级联复制，即从库也可以作为 WAL 发送者，向更下游的从库转发 WAL 数据：

```
主库 (Primary)
  └── 从库A (Standby) → 接收主库 WAL
        └── 从库B (Cascading Standby) → 接收从库A 转发的 WAL
```

级联复制可以减轻主库的网络负担，适用于跨地域部署场景。

## 代码示例

### 示例1：完整搭建一主一从（主库操作）

```bash
#!/bin/bash
# ==== 主库配置脚本 ====

# 1. 创建复制用户
sudo -u postgres psql -c "
CREATE ROLE replicator WITH REPLICATION LOGIN PASSWORD 'Rep1icaP@ss';
"

# 2. 修改 pg_hba.conf 允许从库连接
echo "host replication replicator 192.168.1.201/32 scram-sha-256" \
  | sudo tee -a /etc/postgresql/16/main/pg_hba.conf

# 3. 修改 postgresql.conf
sudo -u postgres psql -c "
ALTER SYSTEM SET wal_level = 'replica';
ALTER SYSTEM SET max_wal_senders = 5;
ALTER SYSTEM SET max_replication_slots = 5;
ALTER SYSTEM SET wal_keep_size = '1GB';
ALTER SYSTEM SET listen_addresses = '*';
"

# 4. 创建复制槽
sudo -u postgres psql -c "
SELECT pg_create_physical_replication_slot('standby1_slot');
"

# 5. 重启主库使配置生效
sudo systemctl restart postgresql
```

### 示例2：完整搭建一主一从（从库操作）

```bash
#!/bin/bash
# ==== 从库配置脚本 ====

# 1. 停止从库上的 PostgreSQL 服务
sudo systemctl stop postgresql

# 2. 清空从库数据目录
sudo rm -rf /var/lib/postgresql/16/main/*

# 3. 使用 pg_basebackup 从主库拉取基础备份
sudo -u postgres pg_basebackup \
  -h 192.168.1.200 \
  -p 5432 \
  -U replicator \
  -D /var/lib/postgresql/16/main \
  -Fp -Xs -P -R

# 4. 检查自动生成的配置
cat /var/lib/postgresql/16/main/standby.signal
cat /var/lib/postgresql/16/main/postgresql.auto.conf
# 应该能看到 primary_conninfo 已自动配置

# 5. 追加从库专用配置
sudo -u postgres bash -c "cat >> /var/lib/postgresql/16/main/postgresql.auto.conf << 'EOF'
primary_slot_name = 'standby1_slot'
hot_standby = on
EOF"

# 6. 启动从库
sudo systemctl start postgresql

# 7. 验证从库处于恢复模式
sudo -u postgres psql -c "SELECT pg_is_in_recovery();"
# 应返回 t (true)
```

### 示例3：配置同步复制

```ini
# 在主库 postgresql.conf 中配置同步复制

# 方式一：指定单个同步从库（application_name 需匹配）
synchronous_standby_names = 'standby1'

# 方式二：FIRST 模式 — 前 N 个按优先级排列的从库为同步
# 至少等待列表中前1个从库确认
synchronous_standby_names = 'FIRST 1 (standby1, standby2)'

# 方式三：ANY 模式 — 任意 N 个从库确认即可
# 任意2个从库确认即返回成功
synchronous_standby_names = 'ANY 2 (standby1, standby2, standby3)'

# 设置同步提交级别
synchronous_commit = 'remote_apply'   # 最严格：等待从库回放完成
```

```sql
-- 也可以在会话级别单独控制同步行为
-- 对于不重要的写操作，可临时关闭同步等待
SET synchronous_commit = 'local';
INSERT INTO logs (msg) VALUES ('不重要的日志');

-- 对于关键数据，确保同步
SET synchronous_commit = 'remote_apply';
UPDATE accounts SET balance = balance - 1000 WHERE id = 1;
```

### 示例4：复制槽管理

```sql
-- 创建物理复制槽
SELECT pg_create_physical_replication_slot('standby1_slot');
SELECT pg_create_physical_replication_slot('standby2_slot');

-- 查看所有复制槽详情
SELECT
    slot_name,
    slot_type,
    active,
    active_pid,
    restart_lsn,
    confirmed_flush_lsn,
    pg_size_pretty(
        pg_wal_lsn_diff(pg_current_wal_lsn(), restart_lsn)
    ) AS retained_wal_size
FROM pg_replication_slots;

-- 删除不再使用的复制槽（重要！避免 WAL 堆积）
SELECT pg_drop_replication_slot('standby2_slot');

-- 查看因复制槽保留的 WAL 大小
SELECT
    slot_name,
    pg_size_pretty(
        pg_wal_lsn_diff(pg_current_wal_lsn(), restart_lsn)
    ) AS lag_size
FROM pg_replication_slots
WHERE NOT active;
```

### 示例5：监控复制延迟

```sql
-- 方法1：在主库查看每个从库的延迟（推荐）
SELECT
    application_name,
    client_addr,
    state,
    sync_state,
    sent_lsn,
    write_lsn,
    flush_lsn,
    replay_lsn,
    -- 发送延迟（字节）
    pg_wal_lsn_diff(sent_lsn, replay_lsn) AS replay_lag_bytes,
    pg_size_pretty(pg_wal_lsn_diff(sent_lsn, replay_lsn)) AS replay_lag_pretty,
    -- 时间维度的延迟
    write_lag,
    flush_lag,
    replay_lag
FROM pg_stat_replication;

-- 方法2：在从库查看自身延迟
SELECT
    now() - pg_last_xact_replay_timestamp() AS replication_lag,
    pg_last_wal_receive_lsn() AS last_received,
    pg_last_wal_replay_lsn() AS last_replayed,
    pg_wal_lsn_diff(
        pg_last_wal_receive_lsn(),
        pg_last_wal_replay_lsn()
    ) AS receive_replay_diff_bytes;

-- 方法3：创建监控视图（便于反复查询）
CREATE VIEW v_replication_status AS
SELECT
    application_name AS standby_name,
    client_addr,
    sync_state,
    pg_size_pretty(pg_wal_lsn_diff(pg_current_wal_lsn(), replay_lsn)) AS total_lag,
    replay_lag AS time_lag,
    state
FROM pg_stat_replication;
```

### 示例6：编写复制延迟告警脚本

```bash
#!/bin/bash
# replication_monitor.sh — 复制延迟监控脚本

THRESHOLD_BYTES=104857600  # 100MB 阈值
ALERT_EMAIL="dba@example.com"

LAG_BYTES=$(psql -At -c "
SELECT COALESCE(
    MAX(pg_wal_lsn_diff(pg_current_wal_lsn(), replay_lsn)),
    0
)::bigint
FROM pg_stat_replication;
")

if [ "$LAG_BYTES" -gt "$THRESHOLD_BYTES" ]; then
    LAG_PRETTY=$(psql -At -c "SELECT pg_size_pretty(${LAG_BYTES}::bigint);")
    echo "WARNING: 复制延迟超过阈值！当前延迟: $LAG_PRETTY" \
        | mail -s "[PostgreSQL Alert] 复制延迟告警" "$ALERT_EMAIL"
fi

# 检查是否有不活跃的复制槽占用大量空间
INACTIVE_SLOTS=$(psql -At -c "
SELECT string_agg(
    slot_name || ': ' || pg_size_pretty(pg_wal_lsn_diff(pg_current_wal_lsn(), restart_lsn)),
    ', '
)
FROM pg_replication_slots
WHERE NOT active
AND pg_wal_lsn_diff(pg_current_wal_lsn(), restart_lsn) > $THRESHOLD_BYTES;
")

if [ -n "$INACTIVE_SLOTS" ]; then
    echo "WARNING: 不活跃复制槽堆积 WAL: $INACTIVE_SLOTS" \
        | mail -s "[PostgreSQL Alert] 复制槽 WAL 堆积" "$ALERT_EMAIL"
fi
```

## 常见问题

### Q1: 从库启动后显示 "waiting for WAL" 是什么意思？

**A:** 这是正常状态。从库已连接主库但主库暂时没有新的 WAL 数据发送，从库在等待。当主库有写入操作时，WAL 数据会立即流过来。可通过 `pg_stat_wal_receiver` 确认连接状态。

### Q2: 同步复制时从库宕机会导致主库阻塞吗？

**A:** 是的，如果只配置了一个同步从库且该从库宕机，主库的所有写事务都会挂起等待从库确认。解决方案：(1) 配置多个同步从库使用 `ANY` 模式；(2) 使用超时机制自动降级为异步；(3) 手动将 `synchronous_standby_names` 设为空以紧急恢复。

### Q3: 复制槽的 WAL 堆积导致磁盘满了怎么办？

**A:** 这是常见的运维事故。紧急处理步骤：(1) 使用 `pg_drop_replication_slot()` 删除不再使用的复制槽；(2) 如果从库确实无法短期恢复，先删除复制槽释放空间，后续重新做基础备份；(3) 设置 `max_slot_wal_keep_size`（PG 13+）限制单个复制槽保留的 WAL 最大量。
