## 理论讲解

### 1. 什么是视图？

视图（View）是一个**虚拟表**，本身不存储数据，而是保存一条 SQL 查询语句。每次查询视图时，PostgreSQL 会执行底层的 SQL 语句并返回结果。

视图的用途：
- **简化复杂查询**：将多表 JOIN 封装成视图，用户只需 `SELECT * FROM view_name`
- **数据安全**：隐藏敏感列，只暴露必要字段
- **逻辑抽象**：提供稳定的查询接口，底层表结构变化时只需修改视图定义
- **代码复用**：避免重复编写相同的复杂查询

### 2. CREATE VIEW 语法

```sql
CREATE [OR REPLACE] VIEW 视图名 [(列别名列表)] AS
SELECT 查询语句;
```

- `OR REPLACE`：如果视图已存在则替换（列数和类型必须兼容）
- 可以为视图列指定别名

### 3. 可更新视图

在满足以下条件时，视图支持 `INSERT`、`UPDATE`、`DELETE` 操作：

- 查询只引用一个表（FROM 子句中没有 JOIN）
- 没有 `DISTINCT`、`GROUP BY`、`HAVING`、`UNION`
- 没有聚合函数、窗口函数
- 没有 `LIMIT` / `OFFSET`
- SELECT 列表中不包含表达式（只有简单列引用）

### 4. WITH CHECK OPTION

确保通过视图插入或更新的数据**满足视图的 WHERE 条件**：

```sql
CREATE VIEW active_products AS
SELECT * FROM products WHERE status = '在售'
WITH CHECK OPTION;

-- 以下操作会被拒绝（因为不满足 status = '在售'）
INSERT INTO active_products (name, status) VALUES ('测试', '已下架');
-- ERROR: new row violates check option for view "active_products"
```

可选修饰符：
- `WITH LOCAL CHECK OPTION`：只检查当前视图的条件
- `WITH CASCADED CHECK OPTION`（默认）：检查当前视图及所有基础视图的条件

### 5. 物化视图（Materialized View）

物化视图**实际存储查询结果**，是一个"快照"。查询速度快，但数据可能不是最新的。

```sql
CREATE MATERIALIZED VIEW 视图名 AS
SELECT 查询语句
[WITH [NO] DATA];
```

- `WITH DATA`（默认）：创建时立即执行查询并存储结果
- `WITH NO DATA`：只创建定义，不填充数据，查询前必须先 REFRESH

### 6. REFRESH MATERIALIZED VIEW

手动刷新物化视图以更新数据：

```sql
-- 普通刷新（会锁表，刷新期间不可查询）
REFRESH MATERIALIZED VIEW 视图名;

-- 并发刷新（需要唯一索引，刷新期间可以查询旧数据）
REFRESH MATERIALIZED VIEW CONCURRENTLY 视图名;
```

### 7. 普通视图 vs 物化视图

| 特性 | 普通视图 | 物化视图 |
|------|---------|---------|
| 数据存储 | 不存储，每次实时查询 | 存储查询结果快照 |
| 数据实时性 | 始终最新 | 需要手动 REFRESH |
| 查询性能 | 等同于执行底层 SQL | 直接读取已存储的数据，很快 |
| 可创建索引 | 不可以 | 可以 |
| 适用场景 | 简化查询、权限控制 | 复杂聚合、报表、数据仓库 |
| 存储空间 | 不占用 | 占用磁盘空间 |

CREATE TABLE products (
    id        SERIAL PRIMARY KEY,
    name      VARCHAR(100) NOT NULL,

    stock     INT DEFAULT 0,

### 示例 1：创建基本视图

```sql
-- 创建一个在售商品视图，隐藏 stock 字段
CREATE VIEW v_active_products AS
SELECT id, name, category, price
FROM products
WHERE status = '在售';

-- 使用视图查询
SELECT * FROM v_active_products ORDER BY price DESC;

-- 视图可以像表一样使用 WHERE
SELECT * FROM v_active_products WHERE category = '书籍';
```

### 示例 2：创建复杂查询视图

```sql
-- 创建订单详情视图（封装多表 JOIN）
CREATE VIEW v_order_details AS
SELECT
    o.id AS order_id,
    o.customer_id,
    o.order_date,
    p.name AS product_name,
    p.category,
    oi.quantity,
    oi.unit_price,
    oi.quantity * oi.unit_price AS subtotal
FROM orders o
JOIN order_items oi ON o.id = oi.order_id
JOIN products p ON oi.product_id = p.id;

-- 使用视图就像查询普通表
SELECT * FROM v_order_details WHERE customer_id = 1;
```

### 示例 3：创建统计视图

```sql
-- 创建销售统计视图
CREATE VIEW v_sales_summary AS
SELECT
    p.category,
    COUNT(DISTINCT o.id) AS order_count,
    SUM(oi.quantity) AS total_quantity,
    SUM(oi.quantity * oi.unit_price) AS total_revenue,
    ROUND(AVG(oi.unit_price), 2) AS avg_unit_price
FROM orders o
JOIN order_items oi ON o.id = oi.order_id
JOIN products p ON oi.product_id = p.id
GROUP BY p.category;

-- 直接查询统计结果
SELECT * FROM v_sales_summary ORDER BY total_revenue DESC;
```

### 示例 4：可更新视图

```sql
-- 创建可更新视图
CREATE VIEW v_books AS
SELECT id, name, price, stock, status
FROM products
WHERE category = '书籍';

-- 通过视图更新数据
UPDATE v_books SET price = 79.00 WHERE name = 'Python 入门';

-- 通过视图插入数据（注意：category 列不在视图中，会使用默认值 NULL）
-- 这可能不是我们想要的行为
INSERT INTO v_books (name, price, stock, status) VALUES ('新书', 99.00, 50, '在售');
-- 该行的 category 是 NULL，不是 '书籍'
```

### 示例 5：WITH CHECK OPTION

```sql
-- 创建带检查选项的视图
CREATE OR REPLACE VIEW v_active_items AS
SELECT id, name, category, price, stock, status
FROM products
WHERE status = '在售'
WITH CHECK OPTION;

-- 正常更新（满足条件）
UPDATE v_active_items SET price = 79.00 WHERE name = 'PostgreSQL 实战';

-- 以下更新会被拒绝（试图将 status 改为非 '在售'）
-- UPDATE v_active_items SET status = '已下架' WHERE name = 'PostgreSQL 实战';
-- ERROR: new row violates check option for view "v_active_items"
```

### 示例 6：创建物化视图

```sql
-- 创建物化视图
CREATE MATERIALIZED VIEW mv_sales_summary AS
SELECT
    p.category,
    COUNT(*) AS order_count,
    SUM(oi.quantity * oi.unit_price) AS total_revenue
FROM orders o
JOIN order_items oi ON o.id = oi.order_id
JOIN products p ON oi.product_id = p.id
GROUP BY p.category;

-- 查询物化视图（速度很快，直接读取存储的数据）
SELECT * FROM mv_sales_summary ORDER BY total_revenue DESC;

-- 在物化视图上创建索引
CREATE INDEX idx_mv_sales_category ON mv_sales_summary (category);
```

### 示例 7：刷新物化视图

```sql
-- 插入新订单数据
INSERT INTO orders (customer_id, order_date, total_amount) VALUES (2, '2025-04-01', 89.00);
INSERT INTO order_items (order_id, product_id, quantity, unit_price) VALUES (5, 1, 1, 89.00);

-- 此时物化视图的数据还是旧的
SELECT * FROM mv_sales_summary;

-- 刷新物化视图
REFRESH MATERIALIZED VIEW mv_sales_summary;

-- 现在数据更新了
SELECT * FROM mv_sales_summary;

-- 并发刷新（需要先创建唯一索引）
CREATE UNIQUE INDEX idx_mv_sales_cat_unique ON mv_sales_summary (category);
REFRESH MATERIALIZED VIEW CONCURRENTLY mv_sales_summary;
```

### 示例 8：管理视图

```sql
-- 查看视图定义
SELECT pg_get_viewdef('v_active_products', true);

-- 修改视图（OR REPLACE）
CREATE OR REPLACE VIEW v_active_products AS
SELECT id, name, category, price, stock
FROM products
WHERE status = '在售';

-- 删除视图
DROP VIEW IF EXISTS v_active_products;

-- 删除物化视图
DROP MATERIALIZED VIEW IF EXISTS mv_sales_summary;

-- 查看数据库中所有视图
SELECT table_name, table_type
FROM information_schema.tables
WHERE table_schema = 'public' AND table_type = 'VIEW';
```

## 常见问题

### Q1: 视图会影响查询性能吗？

**A**：普通视图本身不影响性能——它只是存储的一条 SQL 语句。查询视图时，PostgreSQL 会将视图定义"展开"并与外层查询合并后统一优化。性能等同于直接执行底层 SQL。但如果视图定义非常复杂（多层嵌套），可能会增加优化器的负担。

### Q2: 物化视图什么时候该用？

**A**：适合以下场景：
- 底层查询复杂且执行时间长（如多表 JOIN + 聚合）
- 数据变化不频繁，允许一定延迟
- 查询频率高，需要快速响应
- 报表、仪表盘等场景

不适合实时性要求高的场景，因为需要手动 REFRESH。

### Q3: 如何自动刷新物化视图？

**A**：PostgreSQL 不支持自动刷新物化视图。常见方案：
- 使用 `pg_cron` 扩展定时执行 `REFRESH MATERIALIZED VIEW CONCURRENTLY`
- 在 ETL 流程中添加刷新步骤
- 使用触发器在底层表数据变更时触发刷新（需谨慎，可能影响写入性能）
