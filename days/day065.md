# Day 065: 索引深入（三）索引策略

## 学习目标

- 掌握覆盖索引（INCLUDE）的概念和使用方法
- 理解表达式索引的创建与应用场景
- 了解 REINDEX 命令的作用与使用时机
- 掌握 CREATE INDEX CONCURRENTLY 的在线创建索引方法
- 学会检测和清理无用索引

## 理论讲解

### 一、覆盖索引（Covering Index / INCLUDE）

#### 1.1 什么是覆盖索引

覆盖索引是指索引中包含了查询所需的所有列，使得查询可以**仅通过索引就获取全部数据，无需回表**访问堆表。这种优化称为 Index-Only Scan。

在 PostgreSQL 11 中引入了 `INCLUDE` 子句，允许在索引中包含额外的非键列。

#### 1.2 INCLUDE 与普通多列索引的区别

```sql
-- 普通多列索引：所有列都参与索引排序和查找
CREATE INDEX idx_a_b ON t(a, b);

-- 覆盖索引：只有 a 是键列，b 只是附带存储
CREATE INDEX idx_a_include_b ON t(a) INCLUDE (b);
```

关键区别：
- INCLUDE 列**不参与索引的排序**
- INCLUDE 列**不能用于 WHERE 条件**的索引扫描
- INCLUDE 列**仅用于避免回表**
- INCLUDE 列不影响索引的唯一性约束判断

#### 1.3 典型场景

```sql
-- 场景：经常查询订单状态和金额
-- 查询：SELECT order_id, amount FROM orders WHERE status = 'pending'

-- 使用 INCLUDE 避免回表
CREATE INDEX idx_orders_status_cover ON orders(status) INCLUDE (order_id, amount);
-- 现在查询可以走 Index-Only Scan，不需要访问堆表
```

#### 1.4 Index-Only Scan 的条件

即使有覆盖索引，Index-Only Scan 也需要满足一个条件：**可见性检查**。PostgreSQL 使用 Visibility Map 来判断页面中的所有行是否对当前事务可见。如果 Visibility Map 标记某页面为"全部可见"，则可以跳过回表检查。

因此，**定期 VACUUM 对 Index-Only Scan 的性能至关重要**，因为 VACUUM 会更新 Visibility Map。

### 二、表达式索引（Expression Index）

#### 2.1 概念

表达式索引是对列的表达式或函数结果建立索引，而不是对列的原始值。

```sql
CREATE INDEX idx_name ON table_name (expression);
```

#### 2.2 常见用途

```sql
-- 1. 不区分大小写的查询
CREATE INDEX idx_users_lower_email ON users (LOWER(email));
-- 配合使用：WHERE LOWER(email) = 'john@example.com'

-- 2. 从 JSONB 中提取字段
CREATE INDEX idx_data_type ON events ((data->>'type'));
-- 配合使用：WHERE data->>'type' = 'purchase'

-- 3. 日期部分查询
CREATE INDEX idx_orders_year ON orders (EXTRACT(YEAR FROM order_date));
-- 配合使用：WHERE EXTRACT(YEAR FROM order_date) = 2024

-- 4. 计算字段
CREATE INDEX idx_items_total ON order_items ((price * quantity));
-- 配合使用：WHERE price * quantity > 1000
```

#### 2.3 注意事项

- 查询中的表达式必须与索引定义**完全匹配**
- 表达式必须是 IMMUTABLE 的（结果不随环境变化）
- 索引维护成本增加（每次写入需要计算表达式）

### 三、REINDEX

#### 3.1 为什么需要 REINDEX

索引在长期的增删改操作中可能会出现：
- **膨胀（Bloat）**：删除的索引条目留下空洞，索引体积增大
- **碎片化**：页面填充不均匀，降低扫描效率
- **损坏**：极端情况下索引可能损坏

#### 3.2 REINDEX 语法

```sql
-- 重建单个索引
REINDEX INDEX index_name;

-- 重建表的所有索引
REINDEX TABLE table_name;

-- 重建整个模式的所有索引
REINDEX SCHEMA schema_name;

-- 重建整个数据库的所有索引
REINDEX DATABASE database_name;

-- PostgreSQL 12+ 支持 CONCURRENTLY
REINDEX (CONCURRENTLY) INDEX index_name;
```

#### 3.3 REINDEX 的影响

- 普通 REINDEX 会**锁表**（对写操作加排他锁）
- `REINDEX CONCURRENTLY`（PostgreSQL 12+）不会锁表，但执行时间更长
- REINDEX 会创建新索引替换旧索引，需要临时的额外磁盘空间

### 四、CREATE INDEX CONCURRENTLY

#### 4.1 为什么需要

普通的 `CREATE INDEX` 会对表加 `ShareLock`，阻止所有写操作（INSERT/UPDATE/DELETE）。对于大表，创建索引可能需要数小时，这期间表完全不可写。

`CREATE INDEX CONCURRENTLY` 允许在创建索引的同时不阻塞写操作。

#### 4.2 工作原理

分为两个阶段：
1. **第一次扫描**：扫描表，构建初始索引，同时记录此期间的修改
2. **第二次扫描**：将第一次扫描期间的修改合并到索引中

#### 4.3 语法

```sql
-- 在线创建索引
CREATE INDEX CONCURRENTLY idx_name ON table_name(column_name);
```

#### 4.4 注意事项

- 执行时间通常是普通 CREATE INDEX 的 2-3 倍
- 如果中途失败，会留下一个 INVALID 索引，需要手动删除
- 不能在事务块中使用
- 不能与 `CREATE INDEX IF NOT EXISTS` 一起使用

```sql
-- 检查是否有无效索引
SELECT indexrelid::regclass AS index_name, indisvalid
FROM pg_index
WHERE NOT indisvalid;

-- 删除无效索引
DROP INDEX CONCURRENTLY idx_name;
```

### 五、无用索引检测

索引不是越多越好。每个索引：
- 占用磁盘空间
- 减慢写入速度（INSERT/UPDATE/DELETE 都需要维护索引）
- 增加 VACUUM 的工作量

#### 5.1 检测方法

使用 `pg_stat_user_indexes` 视图：

```sql
-- 查找很少使用的索引
SELECT
    schemaname,
    tablename,
    indexrelname AS index_name,
    idx_scan AS times_used,
    pg_size_pretty(pg_relation_size(indexrelid)) AS index_size
FROM pg_stat_user_indexes
WHERE idx_scan < 10  -- 使用次数少于 10
ORDER BY pg_relation_size(indexrelid) DESC;
```

#### 5.2 注意事项

- 统计信息从上次 `pg_stat_reset()` 或数据库启动后开始累计
- 确保监控周期足够长（至少覆盖一个完整业务周期）
- 唯一索引即使查询不用也不要删（它保证数据唯一性）
- 有些索引可能只在特定时段使用（如月末报表）

## 代码示例

### 示例1：覆盖索引与 Index-Only Scan

```sql
-- 创建测试表
CREATE TABLE products (
    id SERIAL PRIMARY KEY,
    name TEXT,
    category TEXT,
    price NUMERIC,
    stock INT
);

INSERT INTO products (name, category, price, stock)
SELECT
    'Product_' || i,
    (ARRAY['电子','服装','食品','图书','家居'])[1 + (i % 5)],
    (random() * 1000)::numeric(10,2),
    (random() * 500)::int
FROM generate_series(1, 500000) AS i;

ANALYZE products;
VACUUM products;  -- 更新 Visibility Map

-- 普通索引（需要回表获取 price）
CREATE INDEX idx_products_cat ON products(category);

EXPLAIN ANALYZE
SELECT category, price FROM products WHERE category = '电子';
-- 通常是 Index Scan 或 Bitmap Scan

-- 覆盖索引（包含 price，无需回表）
CREATE INDEX idx_products_cat_cover ON products(category) INCLUDE (price);

EXPLAIN ANALYZE
SELECT category, price FROM products WHERE category = '电子';
-- 应该看到 Index Only Scan
```

### 示例2：表达式索引

```sql
-- 创建用户表
CREATE TABLE users (
    id SERIAL PRIMARY KEY,
    email TEXT NOT NULL,
    full_name TEXT,
    created_at TIMESTAMP DEFAULT now(),
    profile JSONB
);

INSERT INTO users (email, full_name, created_at, profile)
SELECT
    'user_' || i || '@Example.COM',
    'User ' || i,
    '2020-01-01'::timestamp + (random() * 1500 || ' days')::interval,
    jsonb_build_object('level', (random()*10)::int, 'city', '城市' || (i%20))
FROM generate_series(1, 300000) AS i;

ANALYZE users;

-- 不区分大小写的 email 查询
-- 没有索引时：
EXPLAIN ANALYZE
SELECT * FROM users WHERE LOWER(email) = 'user_42@example.com';

-- 创建表达式索引
CREATE INDEX idx_users_lower_email ON users (LOWER(email));

-- 有索引后：
EXPLAIN ANALYZE
SELECT * FROM users WHERE LOWER(email) = 'user_42@example.com';

-- JSONB 字段的表达式索引
CREATE INDEX idx_users_city ON users ((profile->>'city'));

EXPLAIN ANALYZE
SELECT * FROM users WHERE profile->>'city' = '城市5';

-- 日期部分索引
CREATE INDEX idx_users_year ON users (EXTRACT(YEAR FROM created_at));

EXPLAIN ANALYZE
SELECT COUNT(*) FROM users WHERE EXTRACT(YEAR FROM created_at) = 2023;
```

### 示例3：REINDEX 操作

```sql
-- 查看索引的当前大小
SELECT indexname, pg_size_pretty(pg_relation_size(indexrelid)) AS size
FROM pg_stat_user_indexes
WHERE tablename = 'products';

-- 大量删除数据造成索引膨胀
DELETE FROM products WHERE id % 3 = 0;

-- 查看删除后的索引大小（可能没变化，因为空间只是标记可重用）
SELECT indexname, pg_size_pretty(pg_relation_size(indexrelid)) AS size
FROM pg_stat_user_indexes
WHERE tablename = 'products';

-- 重建索引
REINDEX TABLE products;

-- 再次检查索引大小（应该变小了）
SELECT indexname, pg_size_pretty(pg_relation_size(indexrelid)) AS size
FROM pg_stat_user_indexes
WHERE tablename = 'products';
```

### 示例4：CREATE INDEX CONCURRENTLY

```sql
-- 在线创建索引（不阻塞写操作）
CREATE INDEX CONCURRENTLY idx_products_price ON products(price);

-- 验证索引是否有效
SELECT indexrelid::regclass, indisvalid
FROM pg_index
WHERE indexrelid = 'idx_products_price'::regclass;

-- 在线删除索引
DROP INDEX CONCURRENTLY idx_products_price;

-- 模拟创建失败后的清理
-- 如果 CONCURRENTLY 创建失败，索引状态为 INVALID
SELECT indexrelid::regclass AS index_name, indisvalid
FROM pg_index
WHERE NOT indisvalid;
```

### 示例5：无用索引检测与清理

```sql
-- 查看所有索引的使用情况
SELECT
    schemaname || '.' || tablename AS table_full_name,
    indexrelname AS index_name,
    idx_scan AS scan_count,
    idx_tup_read AS tuples_read,
    idx_tup_fetch AS tuples_fetched,
    pg_size_pretty(pg_relation_size(indexrelid)) AS index_size
FROM pg_stat_user_indexes
ORDER BY idx_scan ASC, pg_relation_size(indexrelid) DESC;

-- 查找重复索引（一个索引是另一个的前缀）
SELECT
    a.indexrelid::regclass AS index_a,
    b.indexrelid::regclass AS index_b,
    a.indkey AS keys_a,
    b.indkey AS keys_b
FROM pg_index a
JOIN pg_index b ON a.indrelid = b.indrelid
    AND a.indexrelid != b.indexrelid
    AND a.indkey::text LIKE b.indkey::text || ' %'
WHERE a.indrelid = 'products'::regclass;

-- 索引大小占比分析
SELECT
    tablename,
    pg_size_pretty(pg_relation_size(tablename::regclass)) AS table_size,
    pg_size_pretty(pg_indexes_size(tablename::regclass)) AS total_index_size,
    ROUND(100.0 * pg_indexes_size(tablename::regclass) /
          NULLIF(pg_relation_size(tablename::regclass), 0), 1) AS index_ratio_pct
FROM pg_stat_user_tables
ORDER BY pg_indexes_size(tablename::regclass) DESC;
```

## 常见问题

### Q1：覆盖索引（INCLUDE）和普通多列索引有什么实际区别？

**A**：主要区别在于：
1. **索引体积**：INCLUDE 列不参与 B-tree 的排序结构，通常比多列索引更紧凑
2. **唯一约束**：`UNIQUE INDEX (a) INCLUDE (b)` 只保证 a 的唯一性，而 `UNIQUE INDEX (a, b)` 保证 (a,b) 组合唯一
3. **查询优化**：INCLUDE 列不能用于 WHERE 条件的索引扫描，只能用于避免回表。如果 b 列也需要出现在 WHERE 中，应该用多列索引

### Q2：CREATE INDEX CONCURRENTLY 失败了怎么办？

**A**：如果 `CREATE INDEX CONCURRENTLY` 中途失败（如磁盘空间不足、被取消等），会留下一个状态为 INVALID 的索引。这个索引：
- 不会被查询使用
- 但仍会在写操作时被维护（浪费资源）
- 必须手动删除：`DROP INDEX CONCURRENTLY idx_name;`
- 然后重新创建

### Q3：如何判断一个索引是否真的"无用"？

**A**：判断索引是否无用需要综合考虑：
1. 查看 `pg_stat_user_indexes.idx_scan` 确认使用次数为 0 或极少
2. 确保统计周期覆盖了所有业务场景（月报、季报等）
3. 检查索引是否用于唯一约束或外键约束
4. 考虑索引是否用于其他数据库（如从库上的报表查询）
5. 可以先通过 `UPDATE pg_index SET indisvalid = false` 将索引标记为无效观察一段时间，确认无影响后再删除
