## 理论讲解

### 1. WAL 参数详解

WAL（Write-Ahead Logging）是 PostgreSQL 保证数据持久性和崩溃恢复的核心机制。

**wal_level —— WAL 记录级别：**

| 级别 | 说明 | 适用场景 |
|------|------|----------|
| `minimal` | 最少信息，仅崩溃恢复 | 独立服务器，无需复制 |
| `replica` | 包含复制所需信息（默认） | 流复制、归档 |
| `logical` | 包含逻辑解码所需信息 | 逻辑复制、CDC |

**checkpoint 参数：**
- `max_wal_size`：两个检查点之间允许的最大 WAL 大小（默认 1GB）
- `min_wal_size`：WAL 文件回收的最小保留大小（默认 80MB）
- `checkpoint_timeout`：两个检查点之间的最大时间间隔（默认 5min）
- `checkpoint_completion_target`：检查点写入的目标完成时间占间隔的比例（默认 0.9）

检查点的触发条件（任一满足即触发）：
1. WAL 大小达到 max_wal_size
2. 距上次检查点超过 checkpoint_timeout
3. 手动执行 `CHECKPOINT` 命令

### 2. 并发参数

| 参数 | 说明 | 默认值 |
|------|------|--------|
| `max_connections` | 最大并发连接数 | 100 |
| `superuser_reserved_connections` | 为超级用户保留的连接数 | 3 |
| `max_worker_processes` | 最大后台工作进程数 | 8 |
| `max_parallel_workers` | 最大并行查询工作者数 | 8 |
| `max_parallel_workers_per_gather` | 每个 Gather 节点的最大并行工作者 | 2 |
| `max_parallel_maintenance_workers` | 并行维护操作的工作者数 | 2 |

**max_connections 的注意点：**
- 每个连接消耗约 5~10MB 内存
- 连接过多会导致上下文切换开销增大
- 建议配合连接池（PgBouncer）使用
- 公式参考：实际需要的并发数 × 1.5

### 3. 日志参数

PostgreSQL 的日志系统非常灵活，关键参数：

| 参数 | 说明 | 推荐值 |
|------|------|--------|
| `logging_collector` | 启用日志收集器 | on |
| `log_directory` | 日志存储目录 | 'log' |
| `log_filename` | 日志文件命名模式 | 'postgresql-%Y-%m-%d.log' |
| `log_rotation_age` | 日志轮转时间间隔 | 1d |
| `log_rotation_size` | 日志轮转大小阈值 | 100MB |
| `log_min_duration_statement` | 慢查询阈值 | 1000（毫秒） |
| `log_statement` | 记录哪些语句 | 'ddl' |
| `log_checkpoints` | 记录检查点信息 | on |
| `log_connections` | 记录连接事件 | on |
| `log_disconnections` | 记录断开事件 | on |
| `log_lock_waits` | 记录锁等待 | on |
| `log_temp_files` | 记录临时文件使用 | 0 |
| `log_autovacuum_min_duration` | 记录 autovacuum 活动 | 0 |
| `log_line_prefix` | 日志行前缀格式 | '%t [%p] %u@%d ' |

### 4. Autovacuum 全局参数

| 参数 | 说明 | 默认值 |
|------|------|--------|
| `autovacuum` | 启用自动清理 | on |
| `autovacuum_max_workers` | 最大 autovacuum 工作进程数 | 3 |
| `autovacuum_naptime` | 检查间隔 | 1min |
| `autovacuum_vacuum_threshold` | 触发 VACUUM 的最少变更行数 | 50 |
| `autovacuum_vacuum_scale_factor` | 触发 VACUUM 的变更比例 | 0.2 |
| `autovacuum_analyze_threshold` | 触发 ANALYZE 的最少变更行数 | 50 |
| `autovacuum_analyze_scale_factor` | 触发 ANALYZE 的变更比例 | 0.1 |
| `autovacuum_vacuum_cost_delay` | VACUUM 操作的延迟 | 2ms |
| `autovacuum_vacuum_cost_limit` | VACUUM 操作的成本限制 | -1（使用全局值） |

触发 VACUUM 的公式：
```
变更行数 > autovacuum_vacuum_threshold + autovacuum_vacuum_scale_factor × 表行数
```

## 代码示例

### 示例 1：WAL 参数查看与配置

```sql
-- 查看当前 WAL 配置
SELECT name, setting, unit, context
FROM pg_settings
WHERE name LIKE 'wal%'
   OR name LIKE 'checkpoint%'
   OR name LIKE 'max_wal%'
   OR name LIKE 'min_wal%'
ORDER BY name;

-- 查看 WAL 统计信息
SELECT * FROM pg_stat_wal;

-- 查看检查点统计
SELECT checkpoints_timed,      -- 定时触发的检查点数
       checkpoints_req,         -- 请求触发的检查点数
       checkpoint_write_time,   -- 检查点写入时间（毫秒）
       checkpoint_sync_time,    -- 检查点同步时间（毫秒）
       buffers_checkpoint,      -- 检查点写入的缓冲区数
       buffers_backend,         -- 后端直接写入的缓冲区数
       maxwritten_clean         -- bgwriter 被中断的次数
FROM pg_stat_bgwriter;

-- 如果 checkpoints_req 远大于 checkpoints_timed
-- 说明 max_wal_size 或 checkpoint_timeout 设置偏小

-- WAL 相关推荐配置（写入密集场景）
-- wal_level = 'replica'
-- max_wal_size = '4GB'       -- 增大以减少检查点频率
-- min_wal_size = '1GB'
-- checkpoint_timeout = '15min'  -- 延长检查点间隔
-- checkpoint_completion_target = 0.9
```

### 示例 2：WAL 大小与检查点监控

```sql
-- 查看当前 WAL 位置和大小
```

### 示例 3：并发参数配置

```sql
-- 查看当前连接使用情况
SELECT count(*) AS total_connections,
       sum(CASE WHEN state = 'active' THEN 1 ELSE 0 END) AS active,
       sum(CASE WHEN state = 'idle' THEN 1 ELSE 0 END) AS idle,
       sum(CASE WHEN state = 'idle in transaction' THEN 1 ELSE 0 END) AS idle_in_tx
FROM pg_stat_activity
WHERE backend_type = 'client backend';

-- 查看连接上限
SHOW max_connections;
SHOW superuser_reserved_connections;

-- 查看并行查询参数
SHOW max_parallel_workers;
SHOW max_parallel_workers_per_gather;

-- 并行查询配置推荐（16 核 CPU）
-- max_worker_processes = 16
-- max_parallel_workers = 12
-- max_parallel_workers_per_gather = 4
-- max_parallel_maintenance_workers = 4

-- 测试并行查询
SET max_parallel_workers_per_gather = 4;
EXPLAIN (ANALYZE)
SELECT count(*) FROM large_table WHERE status = 'active';
-- 执行计划中应出现 Parallel Seq Scan 或 Parallel Index Scan

-- 禁用并行查询（对比性能）
SET max_parallel_workers_per_gather = 0;
EXPLAIN (ANALYZE)
SELECT count(*) FROM large_table WHERE status = 'active';
```

### 示例 5：Autovacuum 全局参数配置

```sql
-- 查看当前 autovacuum 参数
SELECT name, setting, unit, short_desc
FROM pg_settings
WHERE name LIKE 'autovacuum%'
ORDER BY name;

-- 生产环境推荐配置
-- autovacuum = on                       -- 必须开启
-- autovacuum_max_workers = 5            -- 增加工作进程
-- autovacuum_naptime = '30s'            -- 缩短检查间隔
-- autovacuum_vacuum_threshold = 50      -- 最小变更行数
-- autovacuum_vacuum_scale_factor = 0.1  -- 降低比例（默认 0.2）
-- autovacuum_analyze_threshold = 50
-- autovacuum_analyze_scale_factor = 0.05 -- 降低分析比例
-- autovacuum_vacuum_cost_delay = '2ms'  -- 降低对正常操作的影响
-- autovacuum_vacuum_cost_limit = 1000   -- 增加以加速 vacuum

-- 查看 autovacuum 当前活动
SELECT pid, datname, relid::regclass AS table_name,
       phase, heap_blks_total, heap_blks_scanned,
       heap_blks_vacuumed, index_vacuum_count,
       max_dead_tuples, num_dead_tuples
FROM pg_stat_progress_vacuum;

-- 查看每张表的 autovacuum 触发情况
SELECT schemaname, relname,
       n_dead_tup,
       n_live_tup,
       CASE WHEN n_live_tup > 0
            THEN round(n_dead_tup::numeric / n_live_tup * 100, 2)
            ELSE 0
       END AS dead_ratio_pct,
       last_vacuum,
       last_autovacuum,
       last_analyze,
       last_autoanalyze
FROM pg_stat_user_tables
ORDER BY n_dead_tup DESC
LIMIT 20;
```

## 常见问题

### Q1：max_wal_size 设大了会有什么影响？

**A：** `max_wal_size` 增大的影响：

**优点：**
- 减少检查点频率，降低 I/O 峰值
- 减少 WAL 段文件的回收和创建开销
- 适合写入密集型工作负载

**缺点：**
- 崩溃恢复时间变长（需要重放更多 WAL）
- 占用更多磁盘空间
- 如果磁盘空间有限，可能导致 WAL 目录撑满

建议：写入密集场景设置 2GB~8GB，确保磁盘有足够空间存储 WAL 文件。

### Q2：log_statement 设为 'all' 会对性能有影响吗？

**A：** 是的，`log_statement = 'all'` 会记录每一条 SQL 语句，对性能有明显影响：

- 写日志本身消耗 I/O
- 高并发下日志文件增长非常快
- 可能成为 I/O 瓶颈

生产环境建议：
- 使用 `log_statement = 'ddl'` 只记录 DDL 语句
- 使用 `log_min_duration_statement` 记录慢查询
- 需要全量审计时，考虑使用 `pgaudit` 扩展并将日志写入独立磁盘

### Q3：如何确定 max_connections 的合理值？

**A：** 确定步骤：
1. **分析实际并发需求**：监控高峰期的活跃连接数
2. **考虑内存开销**：每个连接约 5~10MB，`max_connections × 10MB` 不应超过可用内存
3. **使用连接池**：PgBouncer 的 transaction 模式下，数据库只需少量连接即可支持大量客户端
4. **推荐公式**：`max_connections = CPU核心数 × 2 + 有效磁盘数`（OLTP 场景）

例如 16 核服务器：`16 × 2 + 1 = 33`，加上连接池中间件，数据库侧设置 50~100 即可。
