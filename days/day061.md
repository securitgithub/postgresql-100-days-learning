# Day 061: EXPLAIN 与查询计划（一）

## 学习目标

- 掌握 EXPLAIN 命令的基本用法及其常用选项
- 理解 EXPLAIN ANALYZE 的执行原理与输出含义
- 区分 Seq Scan、Index Scan、Bitmap Scan 三种扫描节点
- 理解 PostgreSQL 的 Cost 估算模型及其参数含义
- 学会从查询计划中判断性能瓶颈

## 理论讲解

### 一、为什么需要 EXPLAIN？

在数据库性能优化中，最关键的第一步就是**了解数据库如何执行你的查询**。PostgreSQL 内部有一个查询规划器（Query Planner），它会根据表的统计信息、索引情况、数据分布等因素，选择一条它认为代价最小的执行路径。`EXPLAIN` 命令可以让我们"窥探"这个执行计划，而不需要实际执行查询。

### 二、EXPLAIN 基本语法

```sql
EXPLAIN [选项] SQL语句;
```

最简单的用法：

```sql
EXPLAIN SELECT * FROM users WHERE id = 100;
```

这会返回查询规划器为该查询选择的执行计划，但**不会真正执行查询**。

### 三、EXPLAIN ANALYZE

`EXPLAIN ANALYZE` 不仅显示计划，还会**实际执行查询**并收集真实的运行时统计信息：

```sql
EXPLAIN ANALYZE SELECT * FROM users WHERE id = 100;
```

输出会额外包含：
- **actual time**：每个节点的实际执行时间（毫秒）
- **rows**：每个节点实际返回的行数
- **loops**：该节点被执行的次数
- **Planning Time**：规划耗时
- **Execution Time**：执行总耗时

> **注意**：`EXPLAIN ANALYZE` 会真正执行语句，对于 INSERT/UPDATE/DELETE，建议包裹在事务中并 ROLLBACK。

```sql
BEGIN;
EXPLAIN ANALYZE DELETE FROM logs WHERE created_at < '2020-01-01';
ROLLBACK;
```

### 四、扫描节点类型

#### 4.1 Seq Scan（顺序扫描）

顺序扫描就是从头到尾读取整张表的每一行，然后检查是否满足 WHERE 条件。

特点：
- 当表较小或需要返回大部分数据时，Seq Scan 可能比索引扫描更快
- 不需要任何索引
- 对于大表且只取少量数据时，性能较差

```sql
-- 典型的顺序扫描场景
EXPLAIN SELECT * FROM orders;
-- 结果示例：
-- Seq Scan on orders  (cost=0.00..1520.00 rows=50000 width=120)
```

#### 4.2 Index Scan（索引扫描）

通过索引定位到满足条件的行，然后回表（到堆表）获取完整数据。

特点：
- 适用于高选择性查询（返回少量行）
- 每找到一条索引记录，就回表读取一次
- 如果返回行数较多，随机 I/O 开销可能很大

```sql
-- 确保 id 列有索引
EXPLAIN SELECT * FROM orders WHERE id = 42;
-- 结果示例：
-- Index Scan using orders_pkey on orders  (cost=0.29..8.31 rows=1 width=120)
--   Index Cond: (id = 42)
```

#### 4.3 Bitmap Scan（位图扫描）

Bitmap Scan 是介于 Seq Scan 和 Index Scan 之间的折中方案。它分两步执行：

1. **Bitmap Index Scan**：扫描索引，构建一个位图（bitmap），记录哪些页面包含匹配行
2. **Bitmap Heap Scan**：按页面顺序读取堆表中标记的页面

特点：
- 适用于中等选择性的查询
- 减少随机 I/O，将其转换为顺序 I/O
- 可以合并多个索引条件（BitmapAnd / BitmapOr）

```sql
EXPLAIN SELECT * FROM orders WHERE status = 'pending' AND amount > 1000;
-- 可能的结果：
-- Bitmap Heap Scan on orders  (cost=12.45..320.50 rows=150 width=120)
--   Recheck Cond: (status = 'pending')
--   Filter: (amount > 1000)
--   ->  Bitmap Index Scan on idx_orders_status  (cost=0.00..12.41 rows=500 width=0)
--         Index Cond: (status = 'pending')
```

### 五、Cost 估算模型

EXPLAIN 输出中的 `cost` 是查询规划器估算的"代价"，单位是抽象的磁盘页读取成本单位。

```
cost=启动成本..总成本
```

- **启动成本（Startup Cost）**：返回第一行之前的成本
- **总成本（Total Cost）**：返回所有行的总成本

影响 Cost 估算的关键参数（可在 `postgresql.conf` 中配置）：

| 参数 | 默认值 | 含义 |
|------|--------|------|
| `seq_page_cost` | 1.0 | 顺序读取一个磁盘页的成本 |
| `random_page_cost` | 4.0 | 随机读取一个磁盘页的成本 |
| `cpu_tuple_cost` | 0.01 | 处理每行数据的 CPU 成本 |
| `cpu_index_tuple_cost` | 0.005 | 处理每条索引记录的 CPU 成本 |
| `cpu_operator_cost` | 0.0025 | 执行每个运算符的 CPU 成本 |
| `effective_cache_size` | 4GB | 可用于缓存的内存估算 |

Seq Scan 的成本估算公式：

```
总成本 = seq_page_cost × 页面数 + cpu_tuple_cost × 行数
```

Index Scan 的成本估算公式（简化）：

```
总成本 = random_page_cost × 预估需要读取的页面数 + cpu_index_tuple_cost × 索引行数 + cpu_tuple_cost × 返回行数
```

### 六、如何查看表的统计信息

```sql
-- 查看表的页面数和行数估算
SELECT relname, relpages, reltuples
FROM pg_class
WHERE relname = 'orders';

-- 查看更详细的表大小
SELECT pg_size_pretty(pg_relation_size('orders')) AS table_size,
       pg_size_pretty(pg_indexes_size('orders')) AS index_size;
```

## 代码示例

### 示例1：对比不同扫描方式

```sql
-- 创建测试表
CREATE TABLE test_scan (
    id SERIAL PRIMARY KEY,
    category INT,
    value TEXT
);

-- 插入10万行测试数据
INSERT INTO test_scan (category, value)
SELECT
    (random() * 100)::int,
    md5(random()::text)
FROM generate_series(1, 100000);

-- 创建索引
CREATE INDEX idx_test_category ON test_scan(category);

-- 更新统计信息
ANALYZE test_scan;

-- 查询1：精确匹配单行 -> 预期 Index Scan
EXPLAIN ANALYZE SELECT * FROM test_scan WHERE id = 500;

-- 查询2：返回约1%的数据 -> 可能 Bitmap Scan
EXPLAIN ANALYZE SELECT * FROM test_scan WHERE category = 42;

-- 查询3：返回大部分数据 -> 预期 Seq Scan
EXPLAIN ANALYZE SELECT * FROM test_scan WHERE category < 90;
```

### 示例2：EXPLAIN 的不同输出格式

```sql
-- 默认文本格式
EXPLAIN SELECT * FROM test_scan WHERE category = 10;

-- 显示详细执行统计
EXPLAIN (ANALYZE, BUFFERS) SELECT * FROM test_scan WHERE category = 10;

-- VERBOSE 显示输出列信息
EXPLAIN (VERBOSE) SELECT * FROM test_scan WHERE category = 10;
```

### 示例3：理解 Cost 数值

```sql
-- 查看当前的成本参数
SHOW seq_page_cost;
SHOW random_page_cost;
SHOW cpu_tuple_cost;

-- 手动验证 Seq Scan 成本
-- 首先获取表的页面数和行数
SELECT relpages, reltuples FROM pg_class WHERE relname = 'test_scan';

-- 假设 relpages = 834, reltuples = 100000
-- Seq Scan 成本 ≈ 1.0 * 834 + 0.01 * 100000 = 1834.00
EXPLAIN SELECT * FROM test_scan;
```

### 示例4：强制使用特定扫描方式（仅用于测试）

```sql
-- 禁用 Seq Scan，观察规划器选择
SET enable_seqscan = off;
EXPLAIN ANALYZE SELECT * FROM test_scan WHERE category < 90;
-- 即使返回大量数据，也会被迫用索引扫描

-- 恢复默认设置
SET enable_seqscan = on;

-- 禁用索引扫描，强制 Seq Scan
SET enable_indexscan = off;
SET enable_bitmapscan = off;
EXPLAIN ANALYZE SELECT * FROM test_scan WHERE category = 42;

-- 恢复
SET enable_indexscan = on;
SET enable_bitmapscan = on;
```

### 示例5：Bitmap Scan 合并多个索引

```sql
-- 创建额外索引
CREATE INDEX idx_test_value ON test_scan(value);

-- 使用多条件查询，观察 BitmapAnd/BitmapOr
EXPLAIN ANALYZE
SELECT * FROM test_scan
WHERE category = 5 OR category = 10;

EXPLAIN ANALYZE
SELECT * FROM test_scan
WHERE category = 5 AND value LIKE 'abc%';
```

## 常见问题

### Q1：EXPLAIN 显示的 Cost 数值大就一定慢吗？

**A**：不一定。Cost 是一个相对值，不是绝对时间。它的作用是让规划器在多个候选计划之间做选择。两个查询的 Cost 不能直接比较，因为涉及的数据量不同。要判断真实性能，需要用 `EXPLAIN ANALYZE` 看 actual time。此外，Cost 估算依赖统计信息的准确度，如果统计过时，估算可能严重偏离实际。

### Q2：为什么有索引但规划器选择了 Seq Scan？

**A**：规划器会综合考虑多种因素：
1. 查询需要返回的行占总行数比例过高（如超过 10-20%），此时顺序扫描的顺序 I/O 比索引的随机 I/O 更快
2. 表非常小（只有几个页面），走索引反而增加额外步骤
3. 统计信息过时，需要 `ANALYZE` 更新
4. `random_page_cost` 设置不合理（如 SSD 环境下应调低）

### Q3：EXPLAIN ANALYZE 会修改数据吗？

**A**：`EXPLAIN ANALYZE` 会真正执行 SQL 语句。如果是 SELECT 则没有副作用。但如果是 INSERT、UPDATE、DELETE，**数据会被实际修改**。所以对写操作使用时，务必放在事务中并 ROLLBACK：

```sql
BEGIN;
EXPLAIN ANALYZE UPDATE orders SET status = 'shipped' WHERE id = 1;
ROLLBACK;
```
