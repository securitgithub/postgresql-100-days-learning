# Day 051: 恢复实战 — 从逻辑备份恢复、物理备份恢复、PITR恢复到指定时间点

## 学习目标

- 掌握使用 `pg_restore` 和 `psql` 从逻辑备份恢复数据库
- 理解物理备份恢复的完整流程与注意事项
- 学会配置 PITR（Point-In-Time Recovery）恢复到指定时间点
- 熟悉 `recovery_target_time`、`recovery_target_lsn`、`recovery_target_xid` 等恢复目标参数
- 能够在生产级场景中制定并执行恢复策略

## 理论讲解

### 1. 恢复的核心概念

数据库恢复是备份的逆过程。PostgreSQL 提供两大类恢复方式：

| 恢复方式 | 适用备份类型 | 恢复粒度 | 停机要求 |
|----------|-------------|---------|---------|
| 逻辑恢复 | pg_dump/pg_dumpall 输出 | 表/库级别 | 可在线恢复到新库 |
| 物理恢复 | pg_basebackup + WAL 归档 | 整个集群 | 需要停机替换数据目录 |
| PITR 恢复 | 物理备份 + WAL | 整个集群到指定时间点 | 需要停机 |

### 2. 逻辑备份恢复

逻辑恢复依据备份格式不同，使用不同的工具：

- **纯文本格式（.sql）**：使用 `psql` 导入
- **自定义格式（.dump / -Fc）**：使用 `pg_restore` 导入
- **目录格式（-Fd）**：使用 `pg_restore`，支持并行恢复
- **tar 格式（-Ft）**：使用 `pg_restore`

恢复时的关键考量：
- 目标数据库是否已存在
- 是否需要先清理已有对象（`--clean`）
- 是否恢复权限和所有者（`--no-owner`、`--no-privileges`）
- 并行恢复的线程数（`-j`）

### 3. 物理备份恢复

物理恢复的完整流程：

1. 停止 PostgreSQL 服务
2. 清空或备份当前数据目录
3. 从备份中恢复数据目录
4. 配置恢复参数（`postgresql.conf` 或 `postgresql.auto.conf`）
5. 创建 `recovery.signal`（PostgreSQL 12+）或 `recovery.conf`（旧版本）
6. 启动 PostgreSQL，等待恢复完成

### 4. PITR（Point-In-Time Recovery）

PITR 允许将数据库恢复到过去的任意时间点，前提条件：
- 拥有该时间点之前的物理备份
- 拥有从备份时刻到目标时间点之间的所有 WAL 文件

恢复目标参数（PostgreSQL 12+，在 `postgresql.conf` 中配置）：

| 参数 | 说明 |
|------|------|
| `recovery_target_time` | 恢复到指定时间戳 |
| `recovery_target_lsn` | 恢复到指定 WAL LSN 位置 |
| `recovery_target_xid` | 恢复到指定事务 ID |
| `recovery_target_name` | 恢复到 `pg_create_restore_point()` 创建的命名点 |
| `recovery_target` | 设为 `immediate` 表示恢复到一致性点即停止 |
| `recovery_target_inclusive` | 是否包含目标点本身，默认 true |
| `recovery_target_action` | 到达目标后的动作：`pause`/`promote`/`shutdown` |

### 5. WAL 归档恢复路径

`restore_command` 是 PITR 的核心配置，它告诉 PostgreSQL 如何获取归档的 WAL 文件：

```
restore_command = 'cp /archive/%f %p'
```

其中 `%f` 是 WAL 文件名，`%p` 是恢复时的目标路径。

## 代码示例

### 示例1：从纯文本备份恢复

```sql
-- 首先创建目标数据库
CREATE DATABASE mydb_restored;
```

```bash
# 使用 psql 从纯文本备份恢复
psql -U postgres -d mydb_restored -f /backup/mydb_backup.sql

# 恢复时显示详细信息
psql -U postgres -d mydb_restored -f /backup/mydb_backup.sql -v ON_ERROR_STOP=1 --echo-errors
```

### 示例2：从自定义格式备份恢复

```bash
# 恢复整个数据库（-Fc 格式）
pg_restore -U postgres -d mydb_restored /backup/mydb_backup.dump

# 并行恢复，使用 4 个线程
pg_restore -U postgres -d mydb_restored -j 4 /backup/mydb_backup.dump

# 恢复前先清理已有对象
pg_restore -U postgres -d mydb_restored --clean --if-exists /backup/mydb_backup.dump

# 仅恢复数据，不恢复表结构
pg_restore -U postgres -d mydb_restored --data-only /backup/mydb_backup.dump

# 仅恢复表结构，不恢复数据
pg_restore -U postgres -d mydb_restored --schema-only /backup/mydb_backup.dump
```

### 示例3：选择性恢复特定表

```bash
# 仅恢复指定的表
pg_restore -U postgres -d mydb_restored -t orders /backup/mydb_backup.dump
pg_restore -U postgres -d mydb_restored -t customers -t products /backup/mydb_backup.dump

# 仅恢复指定 schema 的对象
pg_restore -U postgres -d mydb_restored -n public /backup/mydb_backup.dump

# 查看备份内容（不实际恢复）
pg_restore -l /backup/mydb_backup.dump
```

### 示例4：物理备份恢复

```bash
# 步骤1：停止 PostgreSQL
sudo systemctl stop postgresql

# 步骤2：备份当前数据目录（以防万一）
sudo mv /var/lib/postgresql/16/main /var/lib/postgresql/16/main_old

# 步骤3：从基础备份恢复
sudo tar -xzf /backup/base_backup.tar.gz -C /var/lib/postgresql/16/

# 步骤4：确保目录权限正确
sudo chown -R postgres:postgres /var/lib/postgresql/16/main

# 步骤5：启动 PostgreSQL
sudo systemctl start postgresql
```

### 示例5：PITR 恢复到指定时间点

```bash
# 步骤1：停止数据库
sudo systemctl stop postgresql

# 步骤2：清空数据目录并恢复基础备份
sudo rm -rf /var/lib/postgresql/16/main
sudo tar -xzf /backup/base_backup_20260401.tar.gz -C /var/lib/postgresql/16/

# 步骤3：编辑 postgresql.conf，添加恢复配置
cat >> /var/lib/postgresql/16/main/postgresql.conf << 'EOF'
# PITR 恢复配置
restore_command = 'cp /archive/wal/%f %p'
recovery_target_time = '2026-04-01 14:30:00+08'
recovery_target_action = 'pause'
EOF

# 步骤4：创建恢复信号文件（PostgreSQL 12+）
touch /var/lib/postgresql/16/main/recovery.signal

# 步骤5：修复权限并启动
sudo chown -R postgres:postgres /var/lib/postgresql/16/main
sudo systemctl start postgresql
```

### 示例6：使用 LSN 恢复

```sql
-- 在执行关键操作前记录当前 LSN
SELECT pg_current_wal_lsn();
-- 结果示例: 0/1A000060

-- 创建命名恢复点
SELECT pg_create_restore_point('before_big_delete');
```

```bash
# 使用 LSN 恢复
cat >> /var/lib/postgresql/16/main/postgresql.conf << 'EOF'
restore_command = 'cp /archive/wal/%f %p'
recovery_target_lsn = '0/1A000060'
recovery_target_inclusive = false
recovery_target_action = 'promote'
EOF
```

### 示例7：恢复完成后提升为主库

```sql
-- 查看恢复状态
SELECT pg_is_in_recovery();

-- 如果 recovery_target_action = 'pause'，手动提升
SELECT pg_wal_replay_resume();

-- 或使用 pg_promote()（PostgreSQL 12+）
SELECT pg_promote(wait := true, wait_seconds := 60);
```

### 示例8：验证恢复结果

```sql
-- 检查数据库是否已退出恢复模式
SELECT pg_is_in_recovery();
-- 应返回 false

-- 检查最后恢复的 WAL 位置
SELECT pg_last_wal_replay_lsn();

-- 检查最后恢复的时间戳
SELECT pg_last_xact_replay_timestamp();

-- 验证关键表的数据量
SELECT schemaname, relname, n_live_tup
FROM pg_stat_user_tables
ORDER BY n_live_tup DESC
LIMIT 20;

-- 检查数据库大小
SELECT pg_database.datname,
       pg_size_pretty(pg_database_size(pg_database.datname))
FROM pg_database
ORDER BY pg_database_size(pg_database.datname) DESC;
```

## 常见问题

### Q1：pg_restore 报错 "database already exists" 怎么办？

使用 `--clean --if-exists` 选项可以在恢复前清理已有对象。如果需要恢复到全新数据库，先使用 `createdb` 创建空库再恢复。注意 `--create` 选项会尝试先创建数据库，与手动创建冲突。

### Q2：PITR 恢复后发现恢复时间点不对，可以重新恢复吗？

可以。只要保留了原始的物理备份和 WAL 归档文件，可以重复执行恢复流程。再次停库、重新恢复基础备份、调整 `recovery_target_time`、重新创建 `recovery.signal` 并启动即可。

### Q3：恢复后 `pg_is_in_recovery()` 一直返回 true？

如果 `recovery_target_action` 设为 `pause`，数据库会在到达目标点后暂停并保持恢复模式。需要执行 `SELECT pg_wal_replay_resume();` 或 `SELECT pg_promote();` 来完成恢复并提升为正常读写模式。
