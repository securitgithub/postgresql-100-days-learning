## 理论讲解

### 1. 什么是事务？

事务（Transaction）是一组 SQL 操作的逻辑单元，要么**全部成功**，要么**全部失败**。事务是保证数据一致性的核心机制。

经典例子——银行转账：
```
A 向 B 转账 1000 元：
1. A 的余额减少 1000
2. B 的余额增加 1000
```
这两步必须作为一个整体执行。如果第 1 步成功、第 2 步失败，A 的钱就凭空消失了。

### 2. ACID 特性详解

#### 原子性（Atomicity）
事务中的所有操作要么全部完成，要么全部不执行。如果事务中任何一步失败，之前的所有操作都会被撤销（回滚），数据库恢复到事务开始前的状态。

**PostgreSQL 实现**：通过 WAL（预写日志）和 MVCC（多版本并发控制）实现。失败时通过日志回滚。

#### 一致性（Consistency）
事务执行前后，数据库必须从一个一致状态转换到另一个一致状态。所有约束（主键、外键、CHECK、NOT NULL）在事务完成后都必须满足。

**PostgreSQL 实现**：在 COMMIT 时检查所有约束（部分约束可以设为延迟检查 DEFERRABLE）。

#### 隔离性（Isolation）
并发事务之间互不干扰。一个事务的中间状态对其他事务不可见（取决于隔离级别）。

**PostgreSQL 实现**：通过 MVCC，每个事务看到的是数据的一个"快照"，不需要像传统数据库那样大量使用锁。

#### 持久性（Durability）
事务一旦提交（COMMIT），其结果就是永久的，即使系统崩溃也不会丢失。

**PostgreSQL 实现**：COMMIT 时确保 WAL 日志写入磁盘。崩溃恢复时通过重放 WAL 日志恢复数据。

### 3. BEGIN / COMMIT / ROLLBACK

```sql
-- 开始事务
BEGIN;

-- 执行 SQL 操作
UPDATE accounts SET balance = balance - 1000 WHERE user_id = 1;
UPDATE accounts SET balance = balance + 1000 WHERE user_id = 2;

-- 提交事务（所有操作生效）
COMMIT;

-- 或者回滚事务（撤销所有操作）
ROLLBACK;
```

`BEGIN` 也可以写成 `START TRANSACTION`，两者等价。

### 4. SAVEPOINT（保存点）

`SAVEPOINT` 在事务内部设置一个标记点，可以部分回滚到该点，而不需要回滚整个事务：

```sql
BEGIN;
INSERT INTO orders (...) VALUES (...);

SAVEPOINT sp1;
INSERT INTO order_items (...) VALUES (...);
-- 如果这步出错
ROLLBACK TO SAVEPOINT sp1;  -- 只回滚到 sp1，orders 的插入保留

-- 继续执行其他操作
INSERT INTO order_items (...) VALUES (...);  -- 重试

COMMIT;  -- 提交整个事务
```

相关命令：
- `SAVEPOINT 名称`：创建保存点
- `ROLLBACK TO SAVEPOINT 名称`：回滚到保存点
- `RELEASE SAVEPOINT 名称`：释放保存点（不再需要时）

### 5. PostgreSQL 的自动提交

PostgreSQL 默认工作在**自动提交模式**下：每条 SQL 语句都是一个独立的事务，执行完立即提交。

```sql
-- 以下两条语句是两个独立的事务
UPDATE products SET price = 100 WHERE id = 1;  -- 自动提交
UPDATE products SET price = 200 WHERE id = 2;  -- 自动提交
```

只有显式使用 `BEGIN` 后，才会进入手动事务模式，直到 `COMMIT` 或 `ROLLBACK`。

### 6. 事务中的错误处理

在 PostgreSQL 中，事务内一旦发生错误，事务进入**中止状态（aborted）**，后续的所有命令（除了 ROLLBACK）都会被拒绝：

```sql
BEGIN;
INSERT INTO products (id, name) VALUES (1, 'test');  -- 假设 id=1 已存在，报错
-- 事务进入 aborted 状态
SELECT * FROM products;  -- ERROR: current transaction is aborted
-- 只能 ROLLBACK
ROLLBACK;
```

这就是 SAVEPOINT 的重要用途——可以在保存点处恢复，继续执行。

## 代码示例

### 准备测试数据

```sql
-- 创建银行账户表
CREATE TABLE accounts (
    id       SERIAL PRIMARY KEY,
    name     VARCHAR(50) NOT NULL,
    balance  NUMERIC(12,2) NOT NULL CHECK (balance >= 0)
);

-- 创建转账记录表
CREATE TABLE transfers (
    id          SERIAL PRIMARY KEY,
    from_id     INT REFERENCES accounts(id),
    to_id       INT REFERENCES accounts(id),
    amount      NUMERIC(12,2) NOT NULL CHECK (amount > 0),
    created_at  TIMESTAMP DEFAULT now()
);

-- 插入测试账户
INSERT INTO accounts (name, balance) VALUES
('张三', 10000.00),
('李四', 5000.00),
('王五', 3000.00);
```

### 示例 1：基本事务——银行转账

```sql
-- 张三向李四转账 2000 元
BEGIN;

-- 扣减张三的余额
UPDATE accounts SET balance = balance - 2000 WHERE id = 1;

-- 增加李四的余额
UPDATE accounts SET balance = balance + 2000 WHERE id = 2;

-- 记录转账
INSERT INTO transfers (from_id, to_id, amount) VALUES (1, 2, 2000);

-- 提交事务
COMMIT;

-- 验证结果
SELECT * FROM accounts;
-- 张三: 8000, 李四: 7000
```

### 示例 2：事务回滚

```sql
-- 尝试转账 50000 元（张三余额不足）
BEGIN;

UPDATE accounts SET balance = balance - 50000 WHERE id = 1;
-- ERROR: new row for relation "accounts" violates check constraint "accounts_balance_check"
-- 事务自动进入 aborted 状态

ROLLBACK;
-- 所有操作撤销，余额不变

SELECT * FROM accounts WHERE id = 1;
-- 张三余额仍然是 8000
```

### 示例 3：使用 SAVEPOINT

```sql
BEGIN;

-- 第一步：创建新账户
INSERT INTO accounts (name, balance) VALUES ('赵六', 1000.00);

SAVEPOINT sp_after_account;

-- 第二步：尝试一个可能失败的操作
INSERT INTO accounts (id, name, balance) VALUES (1, '重复', 100.00);
-- ERROR: duplicate key，事务进入 aborted 状态

-- 回滚到保存点（只撤销第二步）
ROLLBACK TO SAVEPOINT sp_after_account;

-- 第一步的操作仍然有效，继续执行
UPDATE accounts SET balance = 2000.00 WHERE name = '赵六';

COMMIT;

-- 验证：赵六账户已创建且余额为 2000
SELECT * FROM accounts WHERE name = '赵六';
```

### 示例 4：多个 SAVEPOINT

```sql
BEGIN;

SAVEPOINT sp1;
INSERT INTO transfers (from_id, to_id, amount) VALUES (1, 2, 100);

SAVEPOINT sp2;
INSERT INTO transfers (from_id, to_id, amount) VALUES (2, 3, 200);

SAVEPOINT sp3;
INSERT INTO transfers (from_id, to_id, amount) VALUES (3, 1, 300);

-- 只回滚 sp3 之后的操作
ROLLBACK TO SAVEPOINT sp3;

-- sp1 和 sp2 的插入仍然有效
COMMIT;

-- 验证：只有前两条转账记录
SELECT * FROM transfers ORDER BY id DESC LIMIT 3;
```

### 示例 5：完整的转账函数（实际应用）

```sql
-- 在事务中实现完整的转账逻辑
BEGIN;

-- 1. 检查转出账户是否存在且余额充足
DO $$
DECLARE
    v_balance NUMERIC;
BEGIN
    SELECT balance INTO v_balance FROM accounts WHERE id = 1;
    IF v_balance IS NULL THEN
        RAISE EXCEPTION '账户不存在';
    END IF;
    IF v_balance < 500 THEN
        RAISE EXCEPTION '余额不足，当前余额: %', v_balance;
    END IF;
END $$;

-- 2. 执行转账
UPDATE accounts SET balance = balance - 500 WHERE id = 1;
UPDATE accounts SET balance = balance + 500 WHERE id = 3;
INSERT INTO transfers (from_id, to_id, amount) VALUES (1, 3, 500);

COMMIT;
```

### 示例 6：查看当前事务状态

```sql
-- 查看当前事务 ID
SELECT txid_current();

-- 查看活跃事务
SELECT pid, state, query, xact_start
FROM pg_stat_activity
WHERE state = 'active' OR state = 'idle in transaction';
```

### 示例 7：自动提交演示

```sql
-- 在自动提交模式下（默认），每条语句立即生效
UPDATE accounts SET balance = balance + 1 WHERE id = 1;
-- 立即提交，无法撤销

-- 如果需要把多条语句作为整体，必须显式使用 BEGIN
BEGIN;
UPDATE accounts SET balance = balance + 1 WHERE id = 1;
UPDATE accounts SET balance = balance + 1 WHERE id = 2;
COMMIT;  -- 两条更新作为一个事务提交
```

## 常见问题

### Q1: 长事务有什么危害？

**A**：长时间未提交的事务会导致多个问题：
- **阻塞其他事务**：持有锁导致其他事务等待
- **阻止 VACUUM 清理**：MVCC 下，事务开始后的所有旧版本行都无法被清理，导致表膨胀
- **连接资源占用**：数据库连接被长期占用
- 建议：事务应尽量短，避免在事务中进行耗时操作（如等待用户输入）。

### Q2: BEGIN 之后如果客户端断开连接会怎样？

**A**：PostgreSQL 会自动 ROLLBACK 该事务。未提交的所有更改都会被撤销。这是安全的设计——确保不会有"半完成"的事务残留。

### Q3: PostgreSQL 支持哪些事务隔离级别？

**A**：PostgreSQL 支持四种隔离级别：
- `READ UNCOMMITTED`（实际等同于 READ COMMITTED）
- `READ COMMITTED`（默认）
- `REPEATABLE READ`
- `SERIALIZABLE`

可以在 BEGIN 时指定：`BEGIN ISOLATION LEVEL REPEATABLE READ;`。默认的 READ COMMITTED 对大多数应用已足够。
