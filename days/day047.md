# Day 047: VACUUM 与 MVCC

## 理论讲解

### 1. MVCC 原理

PostgreSQL 使用 MVCC 来实现并发访问控制，核心思想是**不同事务看到不同版本的数据**。

每一行数据（Tuple）都包含隐藏的系统列：

| 列名 | 说明 |
|------|------|
| `xmin` | 创建该行的事务 ID |
| `xmax` | 删除/更新该行的事务 ID（0 表示未删除） |
| `cmin` | 事务内的命令序号（插入） |
| `cmax` | 事务内的命令序号（删除） |
| `ctid` | 行的物理位置（页号, 偏移） |

**可见性规则简化版：**
- 如果 `xmin` 事务已提交且 `xmax` 为 0：该行可见
- 如果 `xmin` 事务已提交且 `xmax` 事务也已提交：该行不可见（已删除）
- 如果 `xmin` 事务未提交：该行对其他事务不可见
- 实际规则更复杂，涉及快照（Snapshot）机制

**UPDATE 的特殊处理：**
在 PostgreSQL 中，UPDATE 实际上是 **DELETE + INSERT**：
1. 将旧行标记为已删除（设置 xmax）
2. 插入一个新行（新版本）
3. 旧行成为"死元组"

### 2. 死元组（Dead Tuples）

死元组是指已经对所有活跃事务不可见的旧版本行。

**产生原因：**
- UPDATE 操作产生旧版本
- DELETE 操作标记的行
- 事务回滚（ROLLBACK）产生的无效行

**影响：**
- 占用磁盘空间
- 表膨胀（Table Bloat），降低顺序扫描性能
- 索引膨胀，增加索引查找开销
- 如果不清理，最终导致事务 ID 回卷

### 3. VACUUM vs VACUUM FULL

| 特性 | VACUUM | VACUUM FULL |
|------|--------|-------------|
| 锁级别 | 不阻塞读写（共享锁） | 排他锁（阻塞所有操作） |
| 空间回收 | 标记空间可重用，不归还OS | 重建表，归还空间给OS |
| 执行速度 | 较快 | 较慢 |
| 空间需求 | 不需要额外空间 | 需要表大小的额外空间 |
| 适用场景 | 日常维护 | 严重膨胀时的紧急处理 |
| 索引处理 | 清理索引中的死条目 | 重建索引 |

**VACUUM 的工作流程：**
1. 扫描表，收集死元组的位置
2. 清理索引中指向死元组的条目
3. 标记死元组占用的空间为可重用
4. 更新可见性映射（Visibility Map）和空闲空间映射（Free Space Map）
5. 推进 `relfrozenxid`（冻结事务 ID）

### 4. ANALYZE

`ANALYZE` 收集表的统计信息，存储在 `pg_statistic` 系统表中，供查询规划器使用。

统计信息包括：
- 行数估计
- 列的 NULL 值比例
- 列的不同值数量（n_distinct）
- 最常见值及其频率
- 直方图分布

没有准确的统计信息，规划器可能选择糟糕的执行计划。

### 5. 事务 ID 回卷（Transaction ID Wraparound）

PostgreSQL 使用 32 位事务 ID，最多约 42 亿个。由于使用环形比较：
- 在某个事务 ID 前面的约 21 亿个事务被认为是"过去的"（可见）
- 后面的约 21 亿个事务被认为是"未来的"（不可见）

当事务 ID 耗尽时会发生"回卷"，导致新旧数据可见性混乱。

**冻结机制：**
- VACUUM 将足够老的事务 ID 替换为特殊的"冻结"标记（FrozenTransactionId）
- 冻结的行对所有事务都可见
- `vacuum_freeze_min_age`：行必须存活多少个事务后才能被冻结（默认 5000万）
- `vacuum_freeze_table_age`：表的 `relfrozenxid` 达到多少时触发全表冻结扫描
- `autovacuum_freeze_max_age`：超过此值强制触发 autovacuum（默认 2 亿）

## 代码示例

### 示例 1：观察 MVCC 版本机制

```sql
-- 创建测试表
CREATE TABLE mvcc_demo (
    id INT PRIMARY KEY,
    name VARCHAR(50),
    value INT
);

-- 插入数据并查看系统列
INSERT INTO mvcc_demo VALUES (1, '初始数据', 100);

-- 查看隐藏的系统列
SELECT xmin, xmax, cmin, cmax, ctid, *
FROM mvcc_demo;
-- xmin = 创建此行的事务ID
-- xmax = 0（未被删除或更新）
-- ctid = (0,1)（第0页第1个位置）

-- 更新数据
UPDATE mvcc_demo SET value = 200 WHERE id = 1;

-- 再次查看
SELECT xmin, xmax, ctid, *
FROM mvcc_demo;
-- xmin = 新的事务ID（因为 UPDATE = DELETE + INSERT）
-- ctid 可能变化（新行可能在不同位置）
```

### 示例 2：观察死元组

```sql
-- 创建测试表并插入大量数据
CREATE TABLE bloat_test (
    id SERIAL PRIMARY KEY,
    data TEXT,
    status VARCHAR(20) DEFAULT 'active'
);

INSERT INTO bloat_test (data)
SELECT md5(random()::text)
FROM generate_series(1, 100000);

-- 查看初始表大小和行数
SELECT pg_size_pretty(pg_total_relation_size('bloat_test')) AS total_size,
       pg_size_pretty(pg_relation_size('bloat_test')) AS table_size;

-- 查看初始统计
SELECT relname, n_live_tup, n_dead_tup,
       last_vacuum, last_autovacuum
FROM pg_stat_user_tables
WHERE relname = 'bloat_test';

-- 大量更新操作，产生死元组
UPDATE bloat_test SET data = md5(random()::text)
WHERE id <= 50000;

-- 查看更新后的统计
SELECT relname, n_live_tup, n_dead_tup,
       CASE WHEN n_live_tup > 0
            THEN round(n_dead_tup::numeric / n_live_tup * 100, 2)
            ELSE 0
       END AS dead_ratio_pct
FROM pg_stat_user_tables
WHERE relname = 'bloat_test';
-- n_dead_tup ≈ 50000

-- 查看表大小（已膨胀）
SELECT pg_size_pretty(pg_relation_size('bloat_test')) AS table_size;
```

### 示例 3：VACUUM 操作

```sql
-- 执行普通 VACUUM
VACUUM bloat_test;

-- 查看 VACUUM 后的统计
SELECT relname, n_live_tup, n_dead_tup,
       last_vacuum
FROM pg_stat_user_tables
WHERE relname = 'bloat_test';
-- n_dead_tup 应接近 0

-- 注意：表大小可能没有减小！
-- 因为 VACUUM 只是标记空间为可重用，不归还给操作系统
SELECT pg_size_pretty(pg_relation_size('bloat_test')) AS table_size;

-- VACUUM VERBOSE 显示详细信息
VACUUM VERBOSE bloat_test;
-- 输出示例：
-- INFO: vacuuming "public.bloat_test"
-- INFO: table "bloat_test": found XXX removable, XXX nonremovable row versions
--       in XXX pages; XXX pages are entirely empty
-- DETAIL: XXX dead row versions cannot be removed yet...

-- 带 ANALYZE 的 VACUUM（同时更新统计信息）
VACUUM ANALYZE bloat_test;
```

### 示例 4：VACUUM FULL 回收空间

```sql
-- 再次制造膨胀
UPDATE bloat_test SET data = md5(random()::text);

-- 记录当前大小
SELECT pg_size_pretty(pg_relation_size('bloat_test')) AS before_size;

-- 普通 VACUUM
VACUUM bloat_test;
SELECT pg_size_pretty(pg_relation_size('bloat_test')) AS after_vacuum;
-- 大小基本不变

-- VACUUM FULL（注意：会锁表！）
VACUUM FULL bloat_test;
SELECT pg_size_pretty(pg_relation_size('bloat_test')) AS after_vacuum_full;
-- 大小明显减小

-- 替代方案：使用 pg_repack 扩展（不锁表的空间回收）
-- CREATE EXTENSION pg_repack;
-- 命令行执行：pg_repack -t bloat_test -d mydb
```

### 示例 5：ANALYZE 与查询计划

```sql
-- 删除所有统计信息（模拟全新表）
-- 注意：不建议在生产环境执行
DELETE FROM pg_statistic
WHERE starelid = 'bloat_test'::regclass;

-- 不准确统计下的查询计划
EXPLAIN SELECT * FROM bloat_test WHERE id < 100;
-- 规划器可能估算行数不准

-- 执行 ANALYZE
ANALYZE bloat_test;

-- 准确统计下的查询计划
EXPLAIN SELECT * FROM bloat_test WHERE id < 100;
-- 规划器的行数估算更准确

-- ANALYZE VERBOSE 显示详细信息
ANALYZE VERBOSE bloat_test;
-- 输出：
-- INFO: analyzing "public.bloat_test"
-- INFO: "bloat_test": scanned XXXX of XXXX pages, containing XXXXX live rows
--       and XXX dead rows; XXXX rows in sample, XXXX estimated total rows

-- 查看统计信息
SELECT attname, null_frac, n_distinct, most_common_vals
FROM pg_stats
WHERE tablename = 'bloat_test'
  AND schemaname = 'public';
```

### 示例 6：事务 ID 回卷监控

```sql
-- 查看数据库的冻结年龄
SELECT datname,
       age(datfrozenxid) AS frozen_xid_age,
       current_setting('autovacuum_freeze_max_age')::bigint AS max_age,
       round(age(datfrozenxid)::numeric /
             current_setting('autovacuum_freeze_max_age')::numeric * 100, 2)
             AS pct_towards_wraparound
FROM pg_database
ORDER BY age(datfrozenxid) DESC;

-- 查看表级别的冻结年龄
SELECT schemaname, relname,
       age(relfrozenxid) AS frozen_xid_age,
       pg_size_pretty(pg_total_relation_size(relid)) AS total_size
FROM pg_stat_user_tables
ORDER BY age(relfrozenxid) DESC
LIMIT 20;

-- 如果 age 接近 autovacuum_freeze_max_age（默认 2 亿）
-- 需要立即手动 VACUUM FREEZE

-- 手动冻结
VACUUM FREEZE bloat_test;

-- 再次检查
SELECT relname, age(relfrozenxid) AS frozen_xid_age
FROM pg_stat_user_tables
WHERE relname = 'bloat_test';
-- age 应该非常小了

-- 查看当前事务 ID
SELECT txid_current();

-- 相关参数查看
SHOW vacuum_freeze_min_age;        -- 默认 50000000
SHOW vacuum_freeze_table_age;      -- 默认 150000000
SHOW autovacuum_freeze_max_age;    -- 默认 200000000
```

### 示例 7：表膨胀评估

```sql
-- 使用 pgstattuple 扩展评估表膨胀
CREATE EXTENSION IF NOT EXISTS pgstattuple;

-- 获取详细的表空间使用统计
SELECT * FROM pgstattuple('bloat_test');
-- 返回：
-- table_len: 表总大小
-- tuple_count: 活跃元组数
-- tuple_len: 活跃元组总大小
-- tuple_percent: 活跃数据占比
-- dead_tuple_count: 死元组数
-- dead_tuple_len: 死元组总大小
-- dead_tuple_percent: 死元组占比
-- free_space: 空闲空间
-- free_percent: 空闲空间占比

-- 如果 dead_tuple_percent + free_percent 很高，说明表严重膨胀

-- 简化版查询评估膨胀
SELECT schemaname, tablename,
       pg_size_pretty(pg_total_relation_size(schemaname || '.' || tablename)) AS total_size,
       n_dead_tup,
       n_live_tup,
       CASE WHEN n_live_tup > 0
            THEN round(n_dead_tup::numeric / n_live_tup * 100, 2)
            ELSE 0
       END AS bloat_ratio
FROM pg_stat_user_tables
WHERE n_dead_tup > 1000
ORDER BY n_dead_tup DESC;
```

## 常见问题

### Q1：为什么 VACUUM 后表大小没有变小？

**A：** 普通 VACUUM 只是将死元组占用的空间标记为"可重用"，并不将空间归还给操作系统。这意味着：
- 后续的 INSERT 操作会优先使用这些可重用空间
- 表的物理文件大小不会缩小

如果需要实际缩小文件大小：
1. `VACUUM FULL`：重建表，但会锁表
2. `CLUSTER`：按索引重组表，也会锁表
3. `pg_repack` 扩展：不锁表的空间回收（推荐）

一般情况下，普通 VACUUM 就足够了，因为空间会被后续操作重用。

### Q2：如何判断一张表是否需要 VACUUM FULL？

**A：** 评估标准：
1. **膨胀比例**：使用 `pgstattuple` 检查 `dead_tuple_percent + free_percent`。如果超过 50%，考虑 VACUUM FULL
2. **空间浪费**：如果表比预期大很多（例如 100 万行却占用 10GB）
3. **性能影响**：顺序扫描变慢，因为需要读取大量无用页

决策建议：
- 膨胀率 < 20%：不需要处理
- 膨胀率 20%~50%：观察，确保 autovacuum 正常运行
- 膨胀率 > 50%：考虑 VACUUM FULL 或 pg_repack
- 生产环境优先使用 pg_repack，避免长时间锁表

### Q3：事务 ID 回卷发生时会怎样？

**A：** PostgreSQL 有多重保护机制防止回卷：

1. **autovacuum 冻结**：自动在后台冻结旧事务 ID
2. **防护性警告**：当距离回卷还有 1000 万个事务 ID 时，日志中会出现警告
3. **强制关闭**：当距离回卷还有 100 万个事务 ID 时，PostgreSQL 会拒绝新事务，只允许执行 VACUUM

如果遇到紧急情况：
```sql
-- 检查最紧急的表
SELECT relname, age(relfrozenxid)
FROM pg_class
WHERE relkind = 'r'
ORDER BY age(relfrozenxid) DESC
LIMIT 5;

-- 对最老的表执行冻结
VACUUM FREEZE table_name;
```

预防措施是确保 autovacuum 正常运行，不要关闭它。
