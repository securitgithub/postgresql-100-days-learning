## 理论讲解

### 1. PostgreSQL 锁的类型

PostgreSQL 使用多种粒度的锁来保证并发访问的数据一致性：

#### 表级锁（Table-level Locks）

| 锁模式 | 典型触发操作 |
|--------|-------------|
| `AccessShareLock` | SELECT |
| `RowShareLock` | SELECT FOR UPDATE / FOR SHARE |
| `RowExclusiveLock` | INSERT, UPDATE, DELETE |
| `ShareUpdateExclusiveLock` | VACUUM, ANALYZE, CREATE INDEX CONCURRENTLY |
| `ShareLock` | CREATE INDEX（非 CONCURRENTLY） |
| `ShareRowExclusiveLock` | CREATE TRIGGER, 某些 ALTER TABLE |
| `ExclusiveLock` | REFRESH MATERIALIZED VIEW CONCURRENTLY |
| `AccessExclusiveLock` | DROP TABLE, ALTER TABLE, TRUNCATE, REINDEX, VACUUM FULL |

#### 锁兼容性矩阵

`AccessExclusiveLock` 与所有锁冲突，而 `AccessShareLock` 仅与 `AccessExclusiveLock` 冲突。这意味着普通 SELECT 几乎不会被阻塞，但 DDL 操作（如 ALTER TABLE）会阻塞所有查询。

#### 行级锁（Row-level Locks）

| 锁模式 | 触发方式 |
|--------|---------|
| FOR KEY SHARE | 外键检查时自动获取 |
| FOR SHARE | SELECT ... FOR SHARE |
| FOR NO KEY UPDATE | 不修改唯一键列的 UPDATE |
| FOR UPDATE | SELECT ... FOR UPDATE |

行级锁不会在 `pg_locks` 中显示（除非有等待），它们存储在元组头部。

#### 其他锁类型

- **Advisory Locks（咨询锁）**：应用层显式管理的锁
- **Predicate Locks（谓词锁）**：用于 SERIALIZABLE 隔离级别

### 2. pg_locks 视图

`pg_locks` 视图显示当前所有锁的信息：

| 字段 | 说明 |
|------|------|
| `locktype` | 锁类型：relation, tuple, transactionid, virtualxid, advisory 等 |
| `database` | 数据库 OID |
| `relation` | 被锁对象的 OID |
| `pid` | 持有/等待锁的进程 ID |
| `mode` | 锁模式 |
| `granted` | true=已获得，false=正在等待 |
| `waitstart` | 开始等待的时间（PG 14+） |
| `fastpath` | 是否通过快速路径获取 |

### 3. 锁等待链分析

当一个事务等待另一个事务释放锁时，就形成了锁等待链。如果等待链形成环路，就产生了死锁。

PostgreSQL 的死锁检测机制：
- `deadlock_timeout`（默认 1 秒）：等待锁超过此时间后触发死锁检测
- 检测到死锁后，PostgreSQL 会回滚其中一个事务来打破循环
- 死锁事件会被记录到日志中

### 4. 锁监控的重要性

生产环境中常见的锁问题：
- 长事务持有锁导致其他操作等待
- DDL 操作（ALTER TABLE）排队导致连锁阻塞
- `idle in transaction` 状态持有锁不释放
- 热点行争用导致大量行锁等待

## 代码示例

### 示例1：查看当前所有锁

```sql
-- 查看当前所有锁（按锁类型分组统计）
SELECT locktype, mode, count(*)
FROM pg_locks
GROUP BY locktype, mode
ORDER BY count(*) DESC;

-- 查看正在等待的锁
SELECT l.pid, l.locktype, l.mode, l.granted,
       l.relation::regclass AS table_name,
       a.usename, a.datname,
       a.state, left(a.query, 80) AS query
FROM pg_locks l
JOIN pg_stat_activity a ON l.pid = a.pid
WHERE NOT l.granted
ORDER BY l.waitstart;
```

### 示例2：检测锁冲突（阻塞关系）

```sql
-- 查看谁阻塞了谁（核心查询）
SELECT
    blocked_locks.pid AS blocked_pid,
    blocked_activity.usename AS blocked_user,
    left(blocked_activity.query, 60) AS blocked_query,
    blocking_locks.pid AS blocking_pid,
    blocking_activity.usename AS blocking_user,
    left(blocking_activity.query, 60) AS blocking_query,
    blocked_locks.mode AS blocked_mode,
    blocking_locks.mode AS blocking_mode
FROM pg_catalog.pg_locks blocked_locks
JOIN pg_catalog.pg_stat_activity blocked_activity
    ON blocked_activity.pid = blocked_locks.pid
JOIN pg_catalog.pg_locks blocking_locks
    ON blocking_locks.locktype = blocked_locks.locktype
    AND blocking_locks.database IS NOT DISTINCT FROM blocked_locks.database
    AND blocking_locks.relation IS NOT DISTINCT FROM blocked_locks.relation
    AND blocking_locks.page IS NOT DISTINCT FROM blocked_locks.page
    AND blocking_locks.tuple IS NOT DISTINCT FROM blocked_locks.tuple
    AND blocking_locks.virtualxid IS NOT DISTINCT FROM blocked_locks.virtualxid
    AND blocking_locks.transactionid IS NOT DISTINCT FROM blocked_locks.transactionid
    AND blocking_locks.classid IS NOT DISTINCT FROM blocked_locks.classid
    AND blocking_locks.objid IS NOT DISTINCT FROM blocked_locks.objid
    AND blocking_locks.objsubid IS NOT DISTINCT FROM blocked_locks.objsubid
    AND blocking_locks.pid != blocked_locks.pid
JOIN pg_catalog.pg_stat_activity blocking_activity
    ON blocking_activity.pid = blocking_locks.pid
WHERE NOT blocked_locks.granted;
```

### 示例3：使用 pg_blocking_pids 简化查询（PG 9.6+）

```sql
-- pg_blocking_pids() 直接返回阻塞进程列表
SELECT pid,
       usename,
       pg_blocking_pids(pid) AS blocked_by,
       left(query, 80) AS query,
       state,
       wait_event_type,
       wait_event
FROM pg_stat_activity
WHERE cardinality(pg_blocking_pids(pid)) > 0;
```

### 示例4：模拟锁冲突场景

```sql
-- 会话1：开始事务并锁定行
BEGIN;
UPDATE orders SET status = 'processing' WHERE order_id = 1001;
-- 不要 COMMIT，保持事务打开

-- 会话2：尝试更新同一行（将被阻塞）
UPDATE orders SET status = 'shipped' WHERE order_id = 1001;
-- 此查询会等待会话1释放锁

-- 会话3：查看锁等待情况
SELECT pid,
       pg_blocking_pids(pid) AS blocked_by,
       state,
       left(query, 60) AS query,
       now() - query_start AS wait_duration
FROM pg_stat_activity
WHERE cardinality(pg_blocking_pids(pid)) > 0;

-- 解决：在会话1中提交或回滚
COMMIT;  -- 或 ROLLBACK;
```

### 示例5：DDL 操作导致的连锁阻塞

```sql
-- 演示 ALTER TABLE 导致的连锁阻塞

-- 会话1：长事务持有 AccessShareLock
BEGIN;
SELECT count(*) FROM users;
-- 不提交

-- 会话2：ALTER TABLE 需要 AccessExclusiveLock（被会话1阻塞）
ALTER TABLE users ADD COLUMN phone varchar(20);
-- 等待中...

-- 会话3：新的 SELECT 也被阻塞（因为 ALTER TABLE 在队列中）
SELECT * FROM users WHERE id = 1;
-- 也在等待！

-- 这就是著名的 "DDL 排队" 问题
-- 解决方案1：给 DDL 设置锁超时
SET lock_timeout = '3s';
ALTER TABLE users ADD COLUMN phone varchar(20);

-- 解决方案2：终止阻塞的长事务
SELECT pg_terminate_backend(会话1的pid);
```

### 示例6：死锁检测与配置

```sql
-- 查看死锁超时设置
SHOW deadlock_timeout;

-- 设置死锁检测超时
ALTER SYSTEM SET deadlock_timeout = '1s';  -- 默认值
SELECT pg_reload_conf();

-- 模拟死锁：
-- 会话1：
BEGIN;
UPDATE accounts SET balance = balance - 100 WHERE id = 1;

-- 会话2：
BEGIN;
UPDATE accounts SET balance = balance - 50 WHERE id = 2;

-- 会话1：（尝试锁定会话2已锁定的行）
UPDATE accounts SET balance = balance + 100 WHERE id = 2;
-- 等待...

-- 会话2：（尝试锁定会话1已锁定的行 -> 死锁！）
UPDATE accounts SET balance = balance + 50 WHERE id = 1;
-- PostgreSQL 检测到死锁，回滚其中一个事务
-- ERROR: deadlock detected
```

### 示例7：Advisory Lock（咨询锁）

```sql
-- 获取会话级咨询锁
SELECT pg_advisory_lock(12345);

-- 尝试获取（不等待），返回 true/false
SELECT pg_try_advisory_lock(12345);

-- 释放
SELECT pg_advisory_unlock(12345);

-- 获取事务级咨询锁（事务结束自动释放）
SELECT pg_advisory_xact_lock(12345);

-- 查看当前咨询锁
SELECT * FROM pg_locks WHERE locktype = 'advisory';
```

### 示例8：锁超时配置

```sql
-- 设置锁等待超时（防止无限等待）
SET lock_timeout = '5s';

-- 设置语句超时
SET statement_timeout = '30s';

-- 设置 idle_in_transaction 超时（PG 9.6+）
ALTER SYSTEM SET idle_in_transaction_session_timeout = '10min';
SELECT pg_reload_conf();

-- 查看当前超时设置
SHOW lock_timeout;
SHOW statement_timeout;
SHOW idle_in_transaction_session_timeout;
```

### 示例9：锁监控视图封装

```sql
-- 创建锁监控视图
CREATE OR REPLACE VIEW v_lock_monitor AS
SELECT
    COALESCE(bl.relation::regclass::text, bl.locktype) AS locked_item,
    now() - ba.query_start AS waiting_duration,
    ba.pid AS blocked_pid,
    ba.usename AS blocked_user,
    left(ba.query, 60) AS blocked_query,
    la.pid AS blocking_pid,
    la.usename AS blocking_user,
    left(la.query, 60) AS blocking_query,
    la.state AS blocking_state
FROM pg_catalog.pg_locks bl
JOIN pg_catalog.pg_stat_activity ba ON ba.pid = bl.pid
JOIN pg_catalog.pg_locks ll ON
    ll.locktype = bl.locktype
    AND ll.database IS NOT DISTINCT FROM bl.database
    AND ll.relation IS NOT DISTINCT FROM bl.relation
    AND ll.page IS NOT DISTINCT FROM bl.page
    AND ll.tuple IS NOT DISTINCT FROM bl.tuple
    AND ll.pid != bl.pid
JOIN pg_catalog.pg_stat_activity la ON la.pid = ll.pid
WHERE NOT bl.granted
ORDER BY waiting_duration DESC;

-- 使用
SELECT * FROM v_lock_monitor;
```

## 常见问题

### Q1：为什么 SELECT 也可能被锁阻塞？

普通 SELECT 持有 `AccessShareLock`，它只与 `AccessExclusiveLock` 冲突。当有 DDL 操作（如 ALTER TABLE、DROP TABLE）在等待 `AccessExclusiveLock` 时，后续的 SELECT 也会排队等待。这是因为 PostgreSQL 的锁队列是公平的——DDL 排在前面，新的 SELECT 必须在它后面等待。解决方案是给 DDL 设置 `lock_timeout`。

### Q2：deadlock_timeout 设得太小会有什么问题？

`deadlock_timeout` 控制的是"等待多久后检查是否死锁"，而不是"等多久就认为死锁"。设得太小会增加死锁检测的频率，浪费 CPU 资源。设得太大会导致真正的死锁被发现得太晚。默认的 1 秒是合理的平衡值。

### Q3：如何安全地执行 DDL 避免长时间锁等待？

推荐做法：(1) 设置 `lock_timeout` 为较短的值（如 3-5 秒）；(2) 在低峰期执行；(3) 先检查是否有长事务正在运行；(4) 使用 `CONCURRENTLY` 选项（如 `CREATE INDEX CONCURRENTLY`）；(5) 多次重试而非无限等待。
