## 理论讲解

### 1. 性能优化方法论

```
收集指标 → 识别瓶颈 → 制定方案 → 实施优化 → 验证效果 → 持续监控
```

性能优化的黄金法则：

- **不要过早优化**：先跑通业务，再针对实际瓶颈优化
- **用数据说话**：每次优化前后都要有量化对比
- **优先级排序**：先优化影响最大的问题
- **一次只改一个变量**：便于判断效果

### 2. pg_stat_statements 简介

`pg_stat_statements` 是 PostgreSQL 最重要的性能分析扩展，它记录所有执行过的 SQL 语句的统计信息，包括：

- 执行次数（calls）
- 总耗时（total_exec_time）
- 平均耗时（mean_exec_time）
- 读取的行数（rows）
- 缓冲区命中/读取（shared_blks_hit / shared_blks_read）

### 3. 索引优化维度

| 优化方向 | 说明 |
|---------|------|
| 缺失索引 | 高频查询没有合适的索引 |
| 冗余索引 | 多个索引功能重叠，浪费空间和写入性能 |
| 未使用索引 | 创建了但从未被查询使用 |
| 索引类型选择 | B-tree/GIN/GiST/BRIN 选择不当 |
| 复合索引列顺序 | 选择性高的列应放前面 |
| 部分索引 | 只索引满足条件的行 |
| 覆盖索引 | INCLUDE 额外列避免回表 |

### 4. 参数调优核心参数

| 参数 | 默认值 | 推荐值 | 说明 |
|------|-------|--------|------|
| shared_buffers | 128MB | 总内存的25% | 共享缓冲区 |
| effective_cache_size | 4GB | 总内存的75% | 规划器预估可用缓存 |
| work_mem | 4MB | 64-256MB | 排序/哈希操作内存 |
| maintenance_work_mem | 64MB | 512MB-2GB | 维护操作内存 |
| random_page_cost | 4.0 | 1.1 (SSD) | 随机IO成本 |
| effective_io_concurrency | 1 | 200 (SSD) | IO并发度 |
| wal_buffers | -1 | 64MB | WAL缓冲区 |
| checkpoint_completion_target | 0.9 | 0.9 | 检查点完成比例 |

## 代码示例

### 示例1：启用和使用 pg_stat_statements

```sql
-- 安装扩展（需先在 postgresql.conf 中配置）
-- shared_preload_libraries = 'pg_stat_statements'
CREATE EXTENSION IF NOT EXISTS pg_stat_statements;

-- 重置统计数据（优化前先重置，跑一段时间再分析）
SELECT pg_stat_statements_reset();

-- ============================================================
-- 查询 Top 10 慢查询（按总耗时排序）
-- ============================================================
SELECT
    queryid,
    calls,
    round(total_exec_time::numeric, 2) AS total_ms,
    round(mean_exec_time::numeric, 2) AS avg_ms,
    round(max_exec_time::numeric, 2) AS max_ms,
    rows,
    round((shared_blks_hit * 100.0 / NULLIF(shared_blks_hit + shared_blks_read, 0))::numeric, 1)
        AS cache_hit_pct,
    LEFT(query, 100) AS query_preview
FROM pg_stat_statements
WHERE dbid = (SELECT oid FROM pg_database WHERE datname = current_database())
ORDER BY total_exec_time DESC
LIMIT 10;

-- Top 10 最频繁的查询
SELECT
    queryid,
    calls,
    round(mean_exec_time::numeric, 2) AS avg_ms,
    rows / NULLIF(calls, 0) AS avg_rows,
    LEFT(query, 100) AS query_preview
FROM pg_stat_statements
WHERE dbid = (SELECT oid FROM pg_database WHERE datname = current_database())
ORDER BY calls DESC
LIMIT 10;

-- Top 10 缓冲区读取最多的查询（IO密集型）
SELECT
    queryid,
    calls,
    shared_blks_read,
    shared_blks_hit,
    round((shared_blks_hit * 100.0 / NULLIF(shared_blks_hit + shared_blks_read, 0))::numeric, 1)
        AS cache_hit_pct,
    LEFT(query, 100) AS query_preview
FROM pg_stat_statements
WHERE dbid = (SELECT oid FROM pg_database WHERE datname = current_database())
ORDER BY shared_blks_read DESC
LIMIT 10;

-- 查看临时文件使用（排序/哈希溢出到磁盘）
SELECT
    queryid,
    calls,
    temp_blks_read + temp_blks_written AS temp_blks_total,
    round(mean_exec_time::numeric, 2) AS avg_ms,
    LEFT(query, 100) AS query_preview
FROM pg_stat_statements
WHERE temp_blks_read + temp_blks_written > 0
ORDER BY temp_blks_read + temp_blks_written DESC
LIMIT 10;
```

### 示例2：慢查询分析与优化

```sql
-- ============================================================
-- 场景1：商品搜索查询慢
-- ============================================================

-- 原始查询（可能慢）
EXPLAIN (ANALYZE, BUFFERS, FORMAT TEXT)
SELECT id, name, base_price, total_sales
FROM products
WHERE status = 1
  AND base_price BETWEEN 100 AND 500
  AND '热销' = ANY(tags)
ORDER BY total_sales DESC
LIMIT 20;

-- 分析：可能缺少合适的复合索引
-- 优化：创建覆盖索引
CREATE INDEX idx_products_status_price_sales
ON products (status, base_price, total_sales DESC)
INCLUDE (name)                           -- 覆盖索引，避免回表
WHERE status = 1;                        -- 部分索引，只索引上架商品

-- 优化后验证
EXPLAIN (ANALYZE, BUFFERS, FORMAT TEXT)
SELECT id, name, base_price, total_sales
FROM products
WHERE status = 1
  AND base_price BETWEEN 100 AND 500
ORDER BY total_sales DESC
LIMIT 20;

-- ============================================================
-- 场景2：用户订单列表查询慢
-- ============================================================

-- 原始查询
EXPLAIN (ANALYZE, BUFFERS, FORMAT TEXT)
SELECT o.*, COUNT(oi.id) AS item_count
FROM orders o
LEFT JOIN order_items oi ON oi.order_id = o.id
WHERE o.user_id = 12345
GROUP BY o.id
ORDER BY o.created_at DESC
LIMIT 20;

-- 分析：orders 表数据量大，需要索引支持
-- 优化：确认复合索引存在
CREATE INDEX IF NOT EXISTS idx_orders_user_time
ON orders (user_id, created_at DESC);

-- 如果 order_items 的 order_id 没有索引
CREATE INDEX IF NOT EXISTS idx_order_items_order
ON order_items (order_id);

-- ============================================================
-- 场景3：统计报表查询慢（全表扫描）
-- ============================================================

-- 原始查询
EXPLAIN (ANALYZE, BUFFERS)
SELECT
    date_trunc('month', created_at) AS month,
    status,
    COUNT(*) AS order_count,
    SUM(pay_amount) AS total_revenue
FROM orders
WHERE created_at >= '2025-01-01' AND created_at < '2026-01-01'
GROUP BY month, status
ORDER BY month;

-- 优化方案1：确认分区裁剪生效
-- 查看是否扫描了所有分区
SET enable_partition_pruning = on;  -- 确保开启

-- 优化方案2：创建物化视图缓存报表数据
CREATE MATERIALIZED VIEW mv_monthly_order_stats AS
SELECT
    date_trunc('month', created_at) AS month,
    status,
    COUNT(*) AS order_count,
    SUM(pay_amount) AS total_revenue,
    AVG(pay_amount) AS avg_order_value,
    COUNT(DISTINCT user_id) AS unique_buyers
FROM orders
GROUP BY month, status;

CREATE UNIQUE INDEX idx_mv_monthly_stats
ON mv_monthly_order_stats (month, status);

-- 定时刷新（配合 pg_cron）
REFRESH MATERIALIZED VIEW CONCURRENTLY mv_monthly_order_stats;
```

### 示例3：索引分析与清理

```sql
-- ============================================================
-- 查找未使用的索引
-- ============================================================
SELECT
    schemaname,
    relname AS table_name,
    indexrelname AS index_name,
    pg_size_pretty(pg_relation_size(i.indexrelid)) AS index_size,
    idx_scan AS times_used,
    idx_tup_read,
    idx_tup_fetch
FROM pg_stat_user_indexes i
JOIN pg_index pi ON i.indexrelid = pi.indexrelid
WHERE idx_scan = 0                    -- 从未被使用
  AND NOT pi.indisunique              -- 排除唯一索引
  AND NOT pi.indisprimary             -- 排除主键
  AND schemaname = 'public'
ORDER BY pg_relation_size(i.indexrelid) DESC;

-- ============================================================
-- 查找冗余索引（前缀重复）
-- ============================================================
SELECT
    a.indexrelid::regclass AS index_a,
    b.indexrelid::regclass AS index_b,
    pg_size_pretty(pg_relation_size(a.indexrelid)) AS size_a,
    pg_size_pretty(pg_relation_size(b.indexrelid)) AS size_b
FROM pg_index a
JOIN pg_index b ON a.indrelid = b.indrelid
    AND a.indexrelid != b.indexrelid
    AND a.indkey::text LIKE b.indkey::text || '%'
WHERE a.indrelid::regclass::text NOT LIKE 'pg_%';

-- ============================================================
-- 查看索引膨胀情况
-- ============================================================
SELECT
    schemaname || '.' || relname AS table,
    indexrelname AS index,
    pg_size_pretty(pg_relation_size(indexrelid)) AS index_size,
    idx_scan,
    round(100.0 * idx_tup_read / NULLIF(idx_tup_fetch, 0), 1) AS read_fetch_ratio
FROM pg_stat_user_indexes
WHERE schemaname = 'public'
ORDER BY pg_relation_size(indexrelid) DESC
LIMIT 20;

-- 重建膨胀的索引（在线操作，不阻塞写入）
REINDEX INDEX CONCURRENTLY idx_orders_user_time;

-- 或者用新索引替换
CREATE INDEX CONCURRENTLY idx_orders_user_time_new
ON orders (user_id, created_at DESC);

-- 验证新索引可用后删除旧索引
DROP INDEX CONCURRENTLY idx_orders_user_time;
ALTER INDEX idx_orders_user_time_new RENAME TO idx_orders_user_time;

-- ============================================================
-- 删除确认无用的索引
-- ============================================================
-- 先标记为 invalid 观察一段时间
-- UPDATE pg_index SET indisvalid = false WHERE indexrelid = 'idx_xxx'::regclass;
-- 确认无影响后再删除
DROP INDEX CONCURRENTLY IF EXISTS idx_xxx;
```

### 示例4：分区优化

```sql
-- ============================================================
-- 检查分区裁剪是否生效
-- ============================================================
EXPLAIN (ANALYZE)
SELECT COUNT(*)
FROM orders
WHERE created_at >= '2025-06-01' AND created_at < '2025-07-01';
-- 应该只扫描 orders_2025_06 分区

-- 查看各分区的大小和行数
SELECT
    child.relname AS partition_name,
    pg_size_pretty(pg_total_relation_size(child.oid)) AS total_size,
    pg_stat_get_live_tuples(child.oid) AS live_tuples,
    pg_stat_get_dead_tuples(child.oid) AS dead_tuples
FROM pg_inherits
JOIN pg_class parent ON pg_inherits.inhparent = parent.oid
JOIN pg_class child ON pg_inherits.inhrelid = child.oid
WHERE parent.relname = 'orders'
ORDER BY child.relname;

-- ============================================================
-- 自动创建未来分区
-- ============================================================
CREATE OR REPLACE FUNCTION create_future_partitions(
    p_table TEXT,
    p_months_ahead INT DEFAULT 3
)
RETURNS VOID
LANGUAGE plpgsql
AS $$
DECLARE
    v_start DATE;
    v_end DATE;
    v_name TEXT;
BEGIN
    FOR i IN 0..p_months_ahead - 1 LOOP
        v_start := date_trunc('month', NOW() + (i || ' months')::interval)::date;
        v_end := v_start + INTERVAL '1 month';
        v_name := p_table || '_' || to_char(v_start, 'YYYY_MM');

        -- 检查分区是否已存在
        IF NOT EXISTS (
            SELECT 1 FROM pg_class WHERE relname = v_name
        ) THEN
            EXECUTE format(
                'CREATE TABLE %I PARTITION OF %I FOR VALUES FROM (%L) TO (%L)',
                v_name, p_table, v_start, v_end
            );
            RAISE NOTICE '创建分区: %', v_name;
        END IF;
    END LOOP;
END;
$$;

-- 执行
SELECT create_future_partitions('orders', 6);

-- ============================================================
-- 分离旧分区归档
-- ============================================================
-- 将旧分区分离（数据仍在，但不再属于分区表）
ALTER TABLE orders DETACH PARTITION orders_2025_01;

-- 可以对分离的分区进行压缩或移到慢速存储
-- 或者直接删除
-- DROP TABLE orders_2025_01;
```

### 示例5：参数调优

```sql
-- ============================================================
-- 查看当前关键参数
-- ============================================================
SELECT name, setting, unit, context, short_desc
FROM pg_settings
WHERE name IN (
    'shared_buffers',
    'effective_cache_size',
    'work_mem',
    'maintenance_work_mem',
    'random_page_cost',
    'effective_io_concurrency',
    'max_parallel_workers_per_gather',
    'max_worker_processes',
    'wal_buffers',
    'checkpoint_completion_target',
    'max_connections',
    'default_statistics_target'
);

-- ============================================================
-- 缓冲区命中率分析
-- ============================================================
SELECT
    sum(heap_blks_read) AS heap_read,
    sum(heap_blks_hit) AS heap_hit,
    round(sum(heap_blks_hit) * 100.0 / NULLIF(sum(heap_blks_hit) + sum(heap_blks_read), 0), 2)
        AS hit_ratio
FROM pg_statio_user_tables;
-- 目标：> 99%

-- 索引缓冲区命中率
SELECT
    sum(idx_blks_read) AS idx_read,
    sum(idx_blks_hit) AS idx_hit,
    round(sum(idx_blks_hit) * 100.0 / NULLIF(sum(idx_blks_hit) + sum(idx_blks_read), 0), 2)
        AS hit_ratio
FROM pg_statio_user_indexes;

-- ============================================================
-- 基于 16GB 内存服务器的推荐配置
-- ============================================================
/*
-- postgresql.conf 调优

# 内存
shared_buffers = '4GB'              # 总内存的25%
effective_cache_size = '12GB'       # 总内存的75%
work_mem = '128MB'                  # 复杂查询排序内存
maintenance_work_mem = '1GB'        # 维护操作（VACUUM/CREATE INDEX）
huge_pages = try                    # 大页内存

# 磁盘（SSD）
random_page_cost = 1.1              # SSD 随机读接近顺序读
effective_io_concurrency = 200      # SSD 支持高并发IO
seq_page_cost = 1.0

# 并行查询
max_worker_processes = 8
max_parallel_workers_per_gather = 4
max_parallel_workers = 8
parallel_tuple_cost = 0.01
parallel_setup_cost = 100

# WAL
wal_buffers = '64MB'
max_wal_size = '4GB'
min_wal_size = '1GB'
checkpoint_completion_target = 0.9

# 连接
max_connections = 200

# 统计
default_statistics_target = 200     # 提高统计精度
track_io_timing = on                # 跟踪IO耗时
*/

-- 运行时调整（不需要重启）
ALTER SYSTEM SET work_mem = '128MB';
ALTER SYSTEM SET random_page_cost = 1.1;
ALTER SYSTEM SET effective_io_concurrency = 200;
ALTER SYSTEM SET default_statistics_target = 200;
SELECT pg_reload_conf();

-- 需要重启的参数
ALTER SYSTEM SET shared_buffers = '4GB';
ALTER SYSTEM SET max_connections = 200;
-- 需要执行 systemctl restart postgresql

-- ============================================================
-- 收集更精确的统计信息
-- ============================================================
-- 对高频查询涉及的列提高统计精度
ALTER TABLE products ALTER COLUMN base_price SET STATISTICS 500;
ALTER TABLE orders ALTER COLUMN status SET STATISTICS 500;
ALTER TABLE orders ALTER COLUMN user_id SET STATISTICS 500;

-- 重新收集统计信息
ANALYZE products;
ANALYZE orders;
```

## 常见问题

### Q1: 如何判断一个索引是否应该删除？

**A:** 综合考虑以下因素：(1) `pg_stat_user_indexes` 中 `idx_scan = 0` 且数据库运行时间已足够长（至少覆盖一个完整业务周期）；(2) 索引不是唯一约束或主键的一部分；(3) 索引不是其他索引的前缀（即不存在冗余关系中被依赖的情况）。删除前建议先用 `UPDATE pg_index SET indisvalid = false` 将索引标记为无效，观察一段时间确认无影响后再实际删除。

### Q2: work_mem 设大会不会有问题？

**A:** `work_mem` 是每个排序/哈希操作分配的内存，一个复杂查询可能有多个这样的操作，每个操作都会分配 `work_mem`。如果设得太大（如1GB）且并发连接很多，可能导致内存耗尽。安全公式：`work_mem = 可用内存 / (max_connections * 2)`。对于个别大查询，可以在会话级别临时调大：`SET work_mem = '512MB'`。

### Q3: 分区表一定比普通表快吗？

**A:** 不一定。分区表只有在查询条件包含分区键（实现分区裁剪）时才有明显优势。如果查询不包含分区键条件，反而需要扫描所有分区，可能比普通表更慢。此外分区数量过多（数千个）也会导致规划器开销增大。建议单个分区表的分区数量控制在几十到几百个。
