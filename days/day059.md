## 理论讲解

### 1. PostgreSQL 版本号规则

PostgreSQL 10 之后的版本号格式：**主版本.小版本**（如 16.2）

| 升级类型 | 示例 | 停机要求 | 风险等级 |
|----------|------|---------|---------|
| 小版本升级 | 16.1 → 16.2 | 需要短暂重启 | 低 |
| 大版本升级 | 15.x → 16.x | 需要停机或特殊方案 | 中-高 |

**小版本升级**：
- 只包含 Bug 修复和安全补丁
- 不改变内部存储格式
- 不需要数据迁移
- 强烈建议保持最新小版本

**大版本升级**：
- 包含新功能和改进
- 可能改变内部存储格式
- 需要数据迁移
- 每年发布一个大版本（通常 9-10 月）
- 每个大版本支持 5 年

### 2. 大版本升级方案对比

| 方案 | 停机时间 | 复杂度 | 数据量限制 | 回滚方案 |
|------|---------|--------|-----------|---------|
| pg_dump/restore | 长（与数据量正比） | 低 | 大库不适用 | 保留旧库 |
| pg_upgrade | 短（分钟级） | 中 | 无 | 保留旧目录 |
| 逻辑复制 | 近零 | 高 | 无 | 切回旧库 |

### 3. pg_upgrade 工作原理

`pg_upgrade` 是 PostgreSQL 官方提供的大版本升级工具，它的核心思想是：

- **复用数据文件**：不重写数据文件，只升级系统目录
- **两种模式**：
  - **Copy 模式**：复制数据文件（安全，占用双倍空间）
  - **Link 模式**（`--link`）：使用硬链接（快速，不占额外空间，但旧集群不可再用）

pg_upgrade 的工作步骤：
1. 检查新旧集群的兼容性
2. 转储旧集群的 schema
3. 创建新集群的系统目录
4. 导入 schema 到新集群
5. 复制或链接数据文件
6. 应用后续处理（统计信息、扩展等）

### 4. 逻辑复制升级方案

利用逻辑复制实现近零停机升级：

1. 在新版本上创建空数据库
2. 在旧库创建 publication
3. 在新库创建 subscription，开始数据同步
4. 等待数据同步完成
5. 停止应用写入旧库
6. 确认同步完成
7. 将应用指向新库

优点：几乎零停机，可以充分测试新库
缺点：需要额外资源，配置复杂，DDL 不会自动同步

### 5. 数据迁移工具

| 工具 | 用途 | 特点 |
|------|------|------|
| pg_dump/pg_restore | PostgreSQL 内部迁移 | 标准工具，灵活 |
| pg_upgrade | 大版本升级 | 快速，数据文件复用 |
| pgloader | 从其他数据库迁移到 PG | 支持 MySQL、SQLite 等 |
| ora2pg | Oracle 迁移到 PG | 转换 Oracle 特有语法 |
| DMS（AWS/Aliyun） | 云迁移 | 在线迁移，多源支持 |
| pgcopydb | 快速复制整个数据库 | 并行化，比 pg_dump 快 |

## 代码示例

### 示例1：小版本升级流程

```bash
# 检查当前版本
psql -c "SELECT version();"

# 步骤1：查看可用更新（Debian/Ubuntu）
sudo apt update
apt list --upgradable | grep postgresql

# 步骤2：备份数据库（安全第一）
pg_dumpall -U postgres > /backup/pre_upgrade_dump.sql

# 步骤3：停止服务
sudo systemctl stop postgresql

# 步骤4：安装新版本
sudo apt install postgresql-16  # 小版本会自动替换二进制文件

# 步骤5：启动服务
sudo systemctl start postgresql

# 步骤6：验证版本
psql -c "SELECT version();"

# RHEL/CentOS 流程
sudo systemctl stop postgresql-16
sudo yum update postgresql16-server postgresql16
sudo systemctl start postgresql-16
```

### 示例2：pg_upgrade 前置检查

```bash
# 安装新版本（不初始化数据库）
sudo apt install postgresql-17

# 查看旧集群和新集群的路径
# 旧集群：/var/lib/postgresql/16/main
# 新集群：/var/lib/postgresql/17/main

# 初始化新集群（如果未自动初始化）
sudo pg_ctlcluster 17 main start -- --data-checksums
sudo pg_ctlcluster 17 main stop

# 运行兼容性检查（不实际升级）
sudo -u postgres /usr/lib/postgresql/17/bin/pg_upgrade \
    --old-datadir=/var/lib/postgresql/16/main \
    --new-datadir=/var/lib/postgresql/17/main \
    --old-bindir=/usr/lib/postgresql/16/bin \
    --new-bindir=/usr/lib/postgresql/17/bin \
    --check

# 检查结果会报告任何不兼容的问题
```

### 示例3：使用 pg_upgrade 执行升级

```bash
# 步骤1：停止旧集群
sudo systemctl stop postgresql@16-main

# 步骤2：确保新集群也已停止
sudo systemctl stop postgresql@17-main

# 步骤3：执行升级（Copy 模式 - 安全）
sudo -u postgres /usr/lib/postgresql/17/bin/pg_upgrade \
    --old-datadir=/var/lib/postgresql/16/main \
    --new-datadir=/var/lib/postgresql/17/main \
    --old-bindir=/usr/lib/postgresql/16/bin \
    --new-bindir=/usr/lib/postgresql/17/bin \
    --jobs=4 \
    --verbose

# 或使用 Link 模式（更快，但旧集群不可再用）
sudo -u postgres /usr/lib/postgresql/17/bin/pg_upgrade \
    --old-datadir=/var/lib/postgresql/16/main \
    --new-datadir=/var/lib/postgresql/17/main \
    --old-bindir=/usr/lib/postgresql/16/bin \
    --new-bindir=/usr/lib/postgresql/17/bin \
    --link \
    --jobs=4

# 步骤4：启动新集群
sudo systemctl start postgresql@17-main

# 步骤5：恢复统计信息
# pg_upgrade 会生成一个脚本
sudo -u postgres /var/lib/postgresql/17/main/analyze_new_cluster.sh

# 步骤6：验证升级
psql -c "SELECT version();"
psql -c "SELECT datname FROM pg_database;"

# 步骤7：清理旧集群（确认无误后）
# pg_upgrade 生成了删除旧集群的脚本
sudo -u postgres /var/lib/postgresql/17/main/delete_old_cluster.sh
```

### 示例4：pg_upgrade 处理扩展

```sql
-- 升级前检查扩展兼容性
SELECT extname, extversion FROM pg_extension;

-- 确保新版本中也安装了相同的 contrib 包
-- sudo apt install postgresql-17-contrib

-- 升级后检查扩展
SELECT extname, extversion FROM pg_extension;

-- 更新扩展到最新版本
ALTER EXTENSION pg_stat_statements UPDATE;
ALTER EXTENSION pgcrypto UPDATE;
```

### 示例5：逻辑复制方式升级

```sql
-- === 旧集群（PG 16）操作 ===

-- 确认 wal_level = logical
SHOW wal_level;
-- 如果不是 logical，需要修改并重启
ALTER SYSTEM SET wal_level = 'logical';
-- 重启后继续

-- 创建复制用户
CREATE USER repl_user WITH REPLICATION LOGIN PASSWORD 'repl_pass';
GRANT SELECT ON ALL TABLES IN SCHEMA public TO repl_user;

-- 创建发布（所有表）
CREATE PUBLICATION upgrade_pub FOR ALL TABLES;

-- === 新集群（PG 17）操作 ===

-- 先用 pg_dump --schema-only 导入表结构
-- pg_dump -U postgres -h old_host --schema-only mydb | psql -U postgres -h new_host mydb

-- 创建订阅
CREATE SUBSCRIPTION upgrade_sub
    CONNECTION 'host=old_host port=5432 dbname=mydb user=repl_user password=repl_pass'
    PUBLICATION upgrade_pub;

-- 监控同步进度
SELECT subname, received_lsn, latest_end_lsn, latest_end_time
FROM pg_stat_subscription;

-- 查看表的初始同步状态
SELECT srsubid, srrelid::regclass, srsubstate
FROM pg_subscription_rel;
-- srsubstate: i=初始化中, d=数据复制中, s=已同步, r=就绪
```

```bash
# === 切换步骤 ===

# 1. 停止应用写入旧库（或将旧库设为只读）
# psql -c "ALTER DATABASE mydb SET default_transaction_read_only = true;"

# 2. 等待复制追上
# 确认 pg_stat_subscription 中 received_lsn 等于旧库的 pg_current_wal_lsn()

# 3. 在新库删除订阅
# psql -h new_host -c "DROP SUBSCRIPTION upgrade_sub;"

# 4. 在新库创建序列值同步（逻辑复制不同步序列）
# pg_dump -h old_host --data-only -t '*_id_seq' mydb | psql -h new_host mydb

# 5. 将应用指向新库

# 6. 在旧库删除发布
# psql -h old_host -c "DROP PUBLICATION upgrade_pub;"
```

### 示例6：序列值迁移

```sql
-- 逻辑复制不会同步序列值，需要手动处理

-- 在旧库导出序列当前值
SELECT 'SELECT setval(' || quote_literal(schemaname || '.' || sequencename)
       || ', ' || last_value || ');'
FROM pg_sequences
WHERE schemaname = 'public';

-- 在新库执行生成的 setval 语句
SELECT setval('public.users_id_seq', 15432);
SELECT setval('public.orders_id_seq', 89123);
-- ...
```

### 示例7：pgloader 从 MySQL 迁移

```bash
# 安装 pgloader
sudo apt install pgloader

# 创建迁移配置文件
cat > /tmp/mysql_to_pg.load << 'EOF'
LOAD DATABASE
    FROM mysql://root:password@localhost/myapp
    INTO postgresql://postgres:password@localhost/myapp_pg
WITH include no drop,
     create tables,
     create indexes,
     reset sequences,
     workers = 4,
     concurrency = 2
SET work_mem to '128MB',
    maintenance_work_mem to '512MB'
CAST type datetime to timestamptz
                    drop default drop not null using zero-dates-to-null,
     type int with extra auto_increment to serial
ALTER SCHEMA 'myapp' RENAME TO 'public';
EOF

# 执行迁移
pgloader /tmp/mysql_to_pg.load
```

### 示例8：pgcopydb 快速复制数据库

```bash
# 安装 pgcopydb
# https://github.com/dimitri/pgcopydb

# 使用 pgcopydb 复制整个数据库（并行化）
pgcopydb clone \
    --source "postgres://user:pass@source:5432/mydb" \
    --target "postgres://user:pass@target:5432/mydb" \
    --table-jobs 8 \
    --index-jobs 4

# 仅复制 schema
pgcopydb copy schema \
    --source "postgres://user:pass@source:5432/mydb" \
    --target "postgres://user:pass@target:5432/mydb"

# 仅复制数据
pgcopydb copy data \
    --source "postgres://user:pass@source:5432/mydb" \
    --target "postgres://user:pass@target:5432/mydb" \
    --table-jobs 8
```

## 常见问题

### Q1：pg_upgrade 的 --link 模式安全吗？

`--link` 模式使用硬链接共享数据文件，升级速度非常快。但一旦新集群启动，旧集群的数据文件就不能再使用了（因为新集群会修改共享的数据文件）。如果升级后发现问题需要回滚，必须从备份恢复。建议在使用 `--link` 前做一次完整备份。

### Q2：大版本升级后性能变差怎么办？

升级后统计信息需要重新收集。`pg_upgrade` 会生成 `analyze_new_cluster.sh` 脚本，务必运行。此外，新版本可能改变了查询规划器的行为，某些查询计划可能不同。使用 `pg_stat_statements` 对比升级前后的查询性能，对性能退化的查询单独分析和调优。

### Q3：逻辑复制升级方式的局限是什么？

逻辑复制不同步：DDL 变更、序列值、大对象（Large Objects）、分区表的分区信息（PG 13 之前）。此外，逻辑复制需要表有主键或 REPLICA IDENTITY。在切换前必须手动处理这些差异。复杂的应用可能有更多需要注意的点。
