## 理论讲解

### 1. pg_stat_statements 扩展

`pg_stat_statements` 是 PostgreSQL 最重要的性能分析扩展，它跟踪所有 SQL 语句的执行统计信息：

- 调用次数（calls）
- 总执行时间 / 平均时间 / 最大最小时间
- 返回行数
- 共享缓冲区命中/读取/写入
- 临时文件使用
- WAL 生成量（PG 13+）
- JIT 编译信息（PG 15+）

SQL 语句会被"规范化"——常量被替换为参数占位符 `$1, $2`，这样相同模式的查询会归为一条统计记录。

### 2. Prometheus + Grafana 方案

这是目前最流行的开源监控方案：

**架构**：
```
PostgreSQL --> postgres_exporter --> Prometheus --> Grafana
```

- **postgres_exporter**：连接 PostgreSQL，采集指标并暴露为 Prometheus 格式
- **Prometheus**：时序数据库，定期抓取指标并存储
- **Grafana**：可视化平台，从 Prometheus 查询数据并展示仪表盘

关键指标分类：
- 连接数与连接池使用率
- 事务提交/回滚率
- 缓存命中率
- 复制延迟
- 锁等待
- WAL 生成速率
- 表膨胀率

### 3. pgwatch2

pgwatch2 是专为 PostgreSQL 设计的监控工具：
- 内置丰富的采集指标
- 支持 InfluxDB / Prometheus / PostgreSQL 作为存储后端
- 提供预置的 Grafana 仪表盘
- 支持自定义 SQL 指标
- Web UI 管理界面

### 4. 监控告警设计原则

良好的告警应遵循：
- **分级**：Critical / Warning / Info
- **可操作**：每个告警都应有明确的处理方法
- **避免告警疲劳**：不要告警太多无关紧要的事件
- **包含上下文**：告警信息中包含当前值和阈值

## 代码示例

### 示例1：安装和配置 pg_stat_statements

```sql
-- 检查扩展是否可用
SELECT * FROM pg_available_extensions WHERE name = 'pg_stat_statements';

-- 安装扩展
CREATE EXTENSION IF NOT EXISTS pg_stat_statements;

-- 需要在 postgresql.conf 中添加到 shared_preload_libraries（需重启）
-- shared_preload_libraries = 'pg_stat_statements'
```

```sql
-- 关键配置参数
ALTER SYSTEM SET shared_preload_libraries = 'pg_stat_statements';
ALTER SYSTEM SET pg_stat_statements.max = 10000;       -- 最大跟踪语句数
ALTER SYSTEM SET pg_stat_statements.track = 'all';      -- top/all/none
ALTER SYSTEM SET pg_stat_statements.track_utility = on;  -- 跟踪工具命令
ALTER SYSTEM SET pg_stat_statements.track_planning = on; -- 跟踪计划时间（PG 13+）
-- 重启后生效
```

### 示例2：pg_stat_statements 常用查询

```sql
-- TOP 10 最耗时的查询
SELECT queryid,
       calls,
       round(total_exec_time::numeric, 2) AS total_time_ms,
       round(mean_exec_time::numeric, 2) AS avg_time_ms,
       round(max_exec_time::numeric, 2) AS max_time_ms,
       rows,
       left(query, 80) AS query_preview
FROM pg_stat_statements
ORDER BY total_exec_time DESC
LIMIT 10;

-- TOP 10 最频繁的查询
SELECT queryid,
       calls,
       round(total_exec_time::numeric, 2) AS total_time_ms,
       round(mean_exec_time::numeric, 2) AS avg_time_ms,
       rows,
       left(query, 80) AS query_preview
FROM pg_stat_statements
ORDER BY calls DESC
LIMIT 10;

-- TOP 10 平均执行时间最长的查询
SELECT queryid,
       calls,
       round(mean_exec_time::numeric, 2) AS avg_time_ms,
       round(stddev_exec_time::numeric, 2) AS stddev_ms,
       rows / NULLIF(calls, 0) AS avg_rows,
       left(query, 80) AS query_preview
FROM pg_stat_statements
WHERE calls > 10  -- 过滤偶发查询
ORDER BY mean_exec_time DESC
LIMIT 10;

-- 缓存命中率最低的查询
SELECT queryid,
       calls,
       shared_blks_hit,
       shared_blks_read,
       CASE WHEN shared_blks_hit + shared_blks_read > 0
            THEN round(100.0 * shared_blks_hit / (shared_blks_hit + shared_blks_read), 2)
            ELSE 100 END AS hit_ratio,
       left(query, 80) AS query_preview
FROM pg_stat_statements
WHERE shared_blks_hit + shared_blks_read > 100
ORDER BY hit_ratio ASC
LIMIT 10;

-- 生成最多临时文件的查询
SELECT queryid,
       calls,
       temp_blks_read + temp_blks_written AS temp_blks_total,
       left(query, 80) AS query_preview
FROM pg_stat_statements
WHERE temp_blks_read + temp_blks_written > 0
ORDER BY temp_blks_total DESC
LIMIT 10;

-- 重置统计（定期重置以保持数据新鲜）
SELECT pg_stat_statements_reset();
```

### 示例6：Prometheus 告警规则

```yaml
# postgresql_alerts.yml
groups:
  - name: postgresql
    rules:
      # 连接数过高
      - alert: PostgreSQLTooManyConnections
        expr: pg_stat_activity_count > (pg_settings_max_connections * 0.8)
        for: 5m
        labels:
          severity: warning
        annotations:
          summary: "PostgreSQL 连接数超过 80%"
          description: "实例 {{ $labels.instance }} 当前连接 {{ $value }}"

      # 缓存命中率过低
      - alert: PostgreSQLLowCacheHitRatio
        expr: |
          pg_stat_database_blks_hit{datname!~"template.*"}
          / (pg_stat_database_blks_hit{datname!~"template.*"}
             + pg_stat_database_blks_read{datname!~"template.*"}) < 0.98
        for: 15m
        labels:
          severity: warning
        annotations:
          summary: "缓存命中率低于 98%"

      # 复制延迟
      - alert: PostgreSQLReplicationLag
        expr: pg_replication_lag > 30
        for: 5m
        labels:
          severity: critical
        annotations:
          summary: "复制延迟超过 30 秒"

      # 死锁发生
      - alert: PostgreSQLDeadlocks
        expr: rate(pg_stat_database_deadlocks[5m]) > 0
        for: 1m
        labels:
          severity: warning
        annotations:
          summary: "检测到死锁"

      # 数据库大小增长过快
      - alert: PostgreSQLDatabaseSizeGrowth
        expr: |
          (pg_database_size_bytes - pg_database_size_bytes offset 1d)
          / pg_database_size_bytes offset 1d > 0.1
        for: 1h
        labels:
          severity: info
        annotations:
          summary: "数据库大小 24 小时增长超过 10%"
```

### 示例7：自定义监控 SQL 脚本

```sql
-- 综合健康检查脚本
-- 1. 连接状态
SELECT '连接状态' AS check_item,
       json_build_object(
           'total', count(*),
           'active', count(*) FILTER (WHERE state = 'active'),
           'idle', count(*) FILTER (WHERE state = 'idle'),
           'idle_in_transaction', count(*) FILTER (WHERE state = 'idle in transaction'),
           'max_connections', current_setting('max_connections')::int
       ) AS result
FROM pg_stat_activity;

-- 2. 缓存命中率
SELECT '缓存命中率' AS check_item,
       json_build_object(
           'hit_ratio', round(
               sum(blks_hit) * 100.0 / NULLIF(sum(blks_hit) + sum(blks_read), 0), 2
           )
       ) AS result
FROM pg_stat_database;

-- 3. 事务回滚率
SELECT '事务回滚率' AS check_item,
       json_build_object(
           'rollback_ratio', round(
               sum(xact_rollback) * 100.0 / NULLIF(sum(xact_commit) + sum(xact_rollback), 0), 2
           )
       ) AS result
FROM pg_stat_database;

-- 4. 长事务检测
SELECT '长事务' AS check_item,
       json_agg(json_build_object(
           'pid', pid,
           'duration', now() - xact_start,
           'query', left(query, 60)
       )) AS result
FROM pg_stat_activity
WHERE state != 'idle'
  AND xact_start < now() - interval '5 minutes'
  AND pid != pg_backend_pid();

-- 5. 表膨胀 TOP5
SELECT '表膨胀' AS check_item,
       json_agg(json_build_object(
           'table', schemaname || '.' || relname,
           'dead_tuples', n_dead_tup,
           'live_tuples', n_live_tup,
           'dead_ratio', round(100.0 * n_dead_tup / NULLIF(n_live_tup, 0), 2)
       ) ORDER BY n_dead_tup DESC) AS result
FROM (
    SELECT schemaname, relname, n_dead_tup, n_live_tup
    FROM pg_stat_user_tables
    ORDER BY n_dead_tup DESC
    LIMIT 5
) t;
```

## 常见问题

### Q1：pg_stat_statements 会影响性能吗？

影响非常小，通常在 1-3% 以内。它使用固定大小的共享内存（由 `pg_stat_statements.max` 控制）。当跟踪的语句数达到上限时，会根据使用频率进行淘汰。建议在所有生产环境中启用。

### Q2：Prometheus 的数据保留策略如何设置？

Prometheus 默认保留 15 天的数据。通过 `--storage.tsdb.retention.time` 参数调整。对于长期存储，建议使用 Thanos 或 VictoriaMetrics 作为远程存储后端。对 PostgreSQL 监控来说，保留 30-90 天的数据通常足够。

### Q3：如何在不重启的情况下更新 pg_stat_statements？

扩展本身需要通过 `shared_preload_libraries` 加载（需重启），但大部分运行时参数可以通过 `SET` 修改。可以使用 `pg_stat_statements_reset()` 重置统计。版本升级时使用 `ALTER EXTENSION pg_stat_statements UPDATE;`。
