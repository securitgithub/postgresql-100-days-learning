## 理论讲解

### 1. 用户与角色的本质

在 PostgreSQL 中，**角色（Role）** 是权限管理的基本单位。从 PostgreSQL 8.1 开始，用户和组的概念被统一为"角色"。

- `CREATE USER` 等价于 `CREATE ROLE ... LOGIN;`，即创建一个可以登录的角色
- `CREATE GROUP` 等价于 `CREATE ROLE ... NOLOGIN;`，即创建一个不能登录的角色
- 角色既可以代表一个真实用户，也可以代表一组权限的集合（组角色）

角色信息存储在系统目录 `pg_authid` 中，是集群级别的对象，在所有数据库间共享。

### 2. 角色属性详解

创建角色时可以指定多种属性：

| 属性 | 说明 | 默认值 |
|------|------|--------|
| `LOGIN` / `NOLOGIN` | 是否允许登录 | NOLOGIN |
| `SUPERUSER` / `NOSUPERUSER` | 是否为超级用户 | NOSUPERUSER |
| `CREATEDB` / `NOCREATEDB` | 是否允许创建数据库 | NOCREATEDB |
| `CREATEROLE` / `NOCREATEROLE` | 是否允许创建/管理角色 | NOCREATEROLE |
| `INHERIT` / `NOINHERIT` | 是否自动继承组角色权限 | INHERIT |
| `REPLICATION` / `NOREPLICATION` | 是否允许流复制 | NOREPLICATION |
| `BYPASSRLS` / `NOBYPASSRLS` | 是否绕过行级安全 | NOBYPASSRLS |
| `CONNECTION LIMIT` | 最大并发连接数 | -1（无限制） |
| `PASSWORD` | 设置密码 | NULL |
| `VALID UNTIL` | 密码过期时间 | 无过期 |

### 3. 角色继承与组

PostgreSQL 通过 **GRANT 角色 TO 角色** 实现组的概念：

- 当角色 A 被授予角色 B 时，A 成为 B 的成员
- 如果 A 具有 `INHERIT` 属性，则 A 自动继承 B 的所有权限
- 如果 A 是 `NOINHERIT`，则需要通过 `SET ROLE B;` 手动激活 B 的权限
- PostgreSQL 16+ 引入了 `GRANT ... WITH ADMIN OPTION`，允许成员将组角色授予其他角色

### 4. GRANT 与 REVOKE

`GRANT` 和 `REVOKE` 是 PostgreSQL 权限管理的核心命令：

- **GRANT** 用于授予权限，可作用于数据库、Schema、表、函数等多种对象
- **REVOKE** 用于回收权限
- 权限类型包括：SELECT、INSERT、UPDATE、DELETE、TRUNCATE、REFERENCES、TRIGGER、CREATE、CONNECT、EXECUTE、USAGE 等

## 代码示例

### 示例 1：创建基本角色与用户

```sql
-- 创建一个不能登录的角色（组角色）
CREATE ROLE dev_team NOLOGIN;

-- 创建一个可以登录的用户，设置密码和过期时间
CREATE USER alice
    PASSWORD 'Alice@2026!'
    VALID UNTIL '2027-01-01'
    CONNECTION LIMIT 5;

-- 等价写法：使用 CREATE ROLE + LOGIN
CREATE ROLE bob
    LOGIN
    PASSWORD 'Bob@2026!'
    CREATEDB;

-- 创建超级用户（需要由现有超级用户执行）
CREATE ROLE admin_user
    LOGIN
    SUPERUSER
    PASSWORD 'Admin@Secure!';
```

### 示例 2：角色属性修改

```sql
-- 授予 alice 创建数据库的权限
ALTER ROLE alice CREATEDB;

-- 修改密码
ALTER ROLE alice PASSWORD 'NewPass@2026!';

-- 设置密码过期时间
ALTER ROLE bob VALID UNTIL '2026-12-31';

-- 撤销 bob 的创建数据库权限
ALTER ROLE bob NOCREATEDB;

-- 设置角色的默认配置参数
ALTER ROLE alice SET search_path TO myschema, public;
ALTER ROLE alice SET statement_timeout TO '30s';

-- 重命名角色
ALTER ROLE alice RENAME TO alice_dev;
```

### 示例 3：组角色与继承

```sql
-- 创建组角色
CREATE ROLE readonly_group NOLOGIN;
CREATE ROLE readwrite_group NOLOGIN;

-- 授予权限给组角色
GRANT SELECT ON ALL TABLES IN SCHEMA public TO readonly_group;
GRANT SELECT, INSERT, UPDATE, DELETE ON ALL TABLES IN SCHEMA public TO readwrite_group;

-- 将用户加入组
GRANT readonly_group TO alice;
GRANT readwrite_group TO bob;

-- alice 拥有 INHERIT 属性（默认），自动继承 readonly_group 的 SELECT 权限

-- 创建 NOINHERIT 用户
CREATE ROLE carol LOGIN NOINHERIT PASSWORD 'Carol@2026!';
GRANT readwrite_group TO carol;

-- carol 需要手动切换角色才能使用 readwrite_group 的权限
-- 连接后执行：
SET ROLE readwrite_group;
-- 恢复原始角色：
RESET ROLE;
```

### 示例 4：GRANT 与 REVOKE 实战

```sql
-- 创建测试数据库和表
CREATE DATABASE shop_db;
\c shop_db

CREATE TABLE products (
    id SERIAL PRIMARY KEY,
    name VARCHAR(100) NOT NULL,
    price NUMERIC(10,2),
    stock INT DEFAULT 0
);

-- 授予 alice 对 products 表的查询权限
GRANT SELECT ON products TO alice;

-- 授予 bob 对 products 表的增删改查权限
GRANT SELECT, INSERT, UPDATE, DELETE ON products TO bob;

-- 授予对序列的使用权限（INSERT 时需要）
GRANT USAGE, SELECT ON SEQUENCE products_id_seq TO bob;

-- 授予 Schema 使用权限
GRANT USAGE ON SCHEMA public TO alice;
GRANT USAGE ON SCHEMA public TO bob;

-- 授予数据库连接权限
GRANT CONNECT ON DATABASE shop_db TO alice;
GRANT CONNECT ON DATABASE shop_db TO bob;

-- 回收 alice 的 SELECT 权限
REVOKE SELECT ON products FROM alice;

-- 回收所有权限
REVOKE ALL PRIVILEGES ON products FROM bob;
```

### 示例 5：查看角色与权限信息

```sql
-- 查看所有角色
SELECT rolname, rolsuper, rolcreatedb, rolcreaterole, rolcanlogin
FROM pg_roles
ORDER BY rolname;

-- 查看角色成员关系
SELECT r.rolname AS role_name,
       m.rolname AS member_name,
       g.admin_option
FROM pg_auth_members g
JOIN pg_roles r ON r.oid = g.roleid
JOIN pg_roles m ON m.oid = g.member
ORDER BY r.rolname;

-- 查看表权限
SELECT grantee, privilege_type, is_grantable
FROM information_schema.table_privileges
WHERE table_name = 'products';

-- 使用 psql 元命令查看
-- \du           -- 列出所有角色
-- \du+          -- 详细角色信息
-- \dp products  -- 查看 products 表的权限
```

### 示例 6：删除角色

```sql
-- 删除角色前需要先回收其拥有的对象和权限
-- 方法一：重新分配所有对象给另一个角色
REASSIGN OWNED BY alice TO postgres;
-- 然后删除角色拥有的权限
DROP OWNED BY alice;
-- 最后删除角色
DROP ROLE alice;

-- 方法二：使用 IF EXISTS 避免错误
DROP ROLE IF EXISTS bob;
```

## 常见问题

### Q1：CREATE USER 和 CREATE ROLE 到底有什么区别？

**A：** 唯一的区别是默认的 LOGIN 属性：
- `CREATE USER xxx` = `CREATE ROLE xxx LOGIN`
- `CREATE ROLE xxx` 默认是 `NOLOGIN`

在功能上两者完全等价。建议统一使用 `CREATE ROLE`，在需要登录时显式添加 `LOGIN` 属性，这样语义更清晰。

### Q2：为什么 GRANT SELECT 之后用户仍然无法查询表？

**A：** 最常见的原因有三个：
1. **缺少 Schema 权限**：用户需要 `USAGE` 权限才能访问 Schema 中的对象。执行 `GRANT USAGE ON SCHEMA public TO user_name;`
2. **缺少数据库连接权限**：用户需要 `CONNECT` 权限。执行 `GRANT CONNECT ON DATABASE db_name TO user_name;`
3. **Search Path 问题**：用户的 `search_path` 中没有包含目标 Schema，需要使用全限定名如 `schema_name.table_name`

### Q3：如何防止角色的密码被暴露？

**A：** 建议采取以下措施：
1. 使用 `scram-sha-256` 认证方式（而非 `md5`），在 `pg_hba.conf` 中配置
2. 设置 `password_encryption = 'scram-sha-256'` 参数
3. 避免在 SQL 日志中记录包含密码的语句：设置 `log_min_duration_statement` 时注意
4. 使用 `\password` 命令在 psql 中交互式设置密码，避免密码出现在命令历史中
5. 定期通过 `VALID UNTIL` 强制密码轮换
