# Day 066: 查询优化技巧

## 学习目标

- 掌握避免全表扫描的常用方法

- 比较子查询、JOIN、CTE 三种写法的性能差异
- 掌握 EXISTS vs IN 的选择策略
- 了解预编译语句（Prepared Statement）对性能的影响

## 理论讲解

### 一、避免不必要的全表扫描

全表扫描（Seq Scan）在以下情况下是合理的：
- 小表（几百到几千行）
- 需要返回表中大部分数据
- 没有合适的索引

但在大表上进行不必要的全表扫描是最常见的性能杀手。以下是常见原因和解决方案：

#### 1.1 索引列上使用了函数

```sql
-- ❌ 不走索引
SELECT * FROM users WHERE UPPER(name) = 'JOHN';

-- ✅ 创建表达式索引
CREATE INDEX idx_users_upper_name ON users (UPPER(name));
SELECT * FROM users WHERE UPPER(name) = 'JOHN';

-- ✅ 或者在应用层处理
SELECT * FROM users WHERE name = 'John';
```

#### 1.2 隐式类型转换

```sql
-- ❌ phone 是 VARCHAR 类型，但传入数字，触发隐式转换
SELECT * FROM users WHERE phone = 13800138000;

-- ✅ 使用正确的类型
SELECT * FROM users WHERE phone = '13800138000';
```

#### 1.3 LIKE 以通配符开头

```sql
-- ❌ 无法使用 B-tree 索引
SELECT * FROM users WHERE name LIKE '%john%';

-- ✅ 使用 pg_trgm 扩展 + GIN 索引
CREATE EXTENSION pg_trgm;
CREATE INDEX idx_users_name_trgm ON users USING gin (name gin_trgm_ops);
SELECT * FROM users WHERE name LIKE '%john%';
```

#### 1.4 否定条件

```sql
-- ❌ NOT IN / <> 通常不走索引
SELECT * FROM orders WHERE status <> 'completed';

-- ✅ 使用部分索引
CREATE INDEX idx_orders_not_completed ON orders(id)
WHERE status <> 'completed';
```

### 二、JOIN 顺序优化

#### 2.1 规划器的 JOIN 顺序选择

PostgreSQL 的规划器会自动评估不同 JOIN 顺序的代价并选择最优顺序。对于 N 个表的 JOIN，理论上有 N! 种排列，但规划器使用动态规划或遗传算法来减少搜索空间。

关键参数：
- `join_collapse_limit`（默认 8）：小于此值时用动态规划穷举，大于时用遗传算法
- `from_collapse_limit`（默认 8）：控制子查询展开为 JOIN 的阈值
- `geqo_threshold`（默认 12）：大于此值时启用遗传算法（GEQO）

#### 2.2 手动控制 JOIN 顺序

当规划器选择不当时，可以用 `JOIN` 的书写顺序加上参数控制：

```sql
-- 关闭 JOIN 重排序，强制按书写顺序执行
SET join_collapse_limit = 1;

-- 小表（过滤性强的表）放在前面驱动大表
SELECT *
FROM small_filtered_table s
JOIN large_table l ON l.id = s.large_id
JOIN another_table a ON a.id = l.another_id;

-- 恢复默认
RESET join_collapse_limit;
```

### 三、子查询 vs JOIN vs CTE 性能对比

#### 3.1 子查询（Subquery）

```sql
-- 标量子查询
SELECT name,
       (SELECT COUNT(*) FROM orders WHERE orders.user_id = users.id)
FROM users;
-- 问题：对每一行用户都执行一次子查询（相关子查询）
```

#### 3.2 JOIN

```sql
-- 等价的 JOIN 写法
SELECT u.name, COUNT(o.id)
FROM users u
LEFT JOIN orders o ON o.user_id = u.id
GROUP BY u.name;
-- 通常更高效：规划器可以选择 Hash Join 一次处理
```

#### 3.3 CTE（Common Table Expression）

PostgreSQL 12 之前，CTE 是**优化屏障**（Optimization Fence），规划器不会将 CTE 展开到主查询中优化。从 PostgreSQL 12 开始，非递归 CTE 如果只被引用一次，会自动内联。

```sql
-- PostgreSQL 12+：这个 CTE 会被自动内联
WITH active_users AS (
    SELECT * FROM users WHERE status = 'active'
)
SELECT * FROM active_users WHERE city = '北京';

-- 强制不内联（当需要物化 CTE 结果时）
WITH active_users AS MATERIALIZED (
    SELECT * FROM users WHERE status = 'active'
)
SELECT * FROM active_users WHERE city = '北京';

-- 强制内联
WITH active_users AS NOT MATERIALIZED (
    SELECT * FROM users WHERE status = 'active'
)
SELECT * FROM active_users WHERE city = '北京';
```

#### 3.4 性能建议

| 场景 | 推荐写法 |
|------|----------|
| 简单关联查询 | JOIN |
| 存在性检查 | EXISTS 子查询 |
| 结果需要复用多次 | CTE (MATERIALIZED) |
| 提高可读性且不影响性能 | CTE（PostgreSQL 12+ 自动内联）|
| 相关子查询（逐行执行） | 改写为 JOIN |

### 四、EXISTS vs IN

#### 4.1 语义区别

```sql
-- IN：检查值是否在列表中
SELECT * FROM users WHERE id IN (SELECT user_id FROM orders);

-- EXISTS：检查子查询是否返回至少一行
SELECT * FROM users u WHERE EXISTS (SELECT 1 FROM orders o WHERE o.user_id = u.id);
```

#### 4.2 性能差异

在现代 PostgreSQL（9.6+）中，规划器通常能将 IN 和 EXISTS 优化为相同的执行计划。但在某些情况下仍有差异：

- **子查询结果集很大时**：EXISTS 更好，因为它找到第一条匹配就停止
- **子查询结果集有 NULL 值时**：`NOT IN` 的行为可能出乎意料（任何值 NOT IN 包含 NULL 的集合都返回 NULL），推荐用 `NOT EXISTS`
- **子查询结果集很小且有索引时**：IN 和 EXISTS 性能相当

```sql
-- ❌ NOT IN 遇到 NULL 的陷阱
SELECT * FROM users
WHERE id NOT IN (SELECT user_id FROM orders);
-- 如果 orders.user_id 有 NULL 值，这个查询返回空结果！

-- ✅ NOT EXISTS 不受 NULL 影响
SELECT * FROM users u
WHERE NOT EXISTS (SELECT 1 FROM orders o WHERE o.user_id = u.id);
```

### 五、预编译语句（Prepared Statement）

#### 5.1 概念

预编译语句将 SQL 的解析、重写、规划与执行分离。第一次 PREPARE 时完成前三步，之后每次 EXECUTE 只需执行。

#### 5.2 优势

- 避免重复解析和规划的开销
- 防止 SQL 注入（参数化查询）
- 对于频繁执行的简单查询，性能提升显著

#### 5.3 Generic Plan vs Custom Plan

PostgreSQL 对预编译语句有两种规划策略：
- **Custom Plan**：每次执行时根据实际参数值生成定制计划（前 5 次执行）
- **Generic Plan**：使用通用计划，忽略参数值（第 6 次开始，如果通用计划不比定制计划差）

```sql
-- 创建预编译语句
PREPARE user_query(int) AS
SELECT * FROM users WHERE id = $1;

-- 执行
EXECUTE user_query(42);
EXECUTE user_query(100);

-- 查看执行计划
EXPLAIN EXECUTE user_query(42);

-- 释放
DEALLOCATE user_query;
```

## 代码示例

### 示例1：避免全表扫描

```sql
-- 创建测试环境
CREATE TABLE orders (
    id SERIAL PRIMARY KEY,
    user_id INT NOT NULL,
    status TEXT NOT NULL,
    amount NUMERIC NOT NULL,
    created_at TIMESTAMP NOT NULL DEFAULT now()
);

INSERT INTO orders (user_id, status, amount, created_at)
SELECT
    (random() * 50000)::int + 1,
    (ARRAY['pending','processing','completed','cancelled'])[1 + (random()*3)::int],
    (random() * 5000)::numeric(10,2),
    '2023-01-01'::timestamp + (random() * 730 || ' days')::interval
FROM generate_series(1, 1000000);

CREATE INDEX idx_orders_user ON orders(user_id);
CREATE INDEX idx_orders_status ON orders(status);
CREATE INDEX idx_orders_created ON orders(created_at);
ANALYZE orders;

-- ❌ 在索引列上使用函数
EXPLAIN ANALYZE
SELECT * FROM orders WHERE DATE(created_at) = '2024-06-15';

-- ✅ 改写为范围查询
EXPLAIN ANALYZE
SELECT * FROM orders
WHERE created_at >= '2024-06-15' AND created_at < '2024-06-16';

-- ❌ 隐式类型转换
EXPLAIN ANALYZE
SELECT * FROM orders WHERE user_id = '12345';
-- 注意：这里 PostgreSQL 会自动转换，但某些类型不行

-- ✅ OR 改写为 IN
EXPLAIN ANALYZE
SELECT * FROM orders WHERE status = 'pending' OR status = 'processing';

EXPLAIN ANALYZE
SELECT * FROM orders WHERE status IN ('pending', 'processing');
```

### 示例2：JOIN 顺序影响

```sql
-- 创建关联表
CREATE TABLE users (
    id SERIAL PRIMARY KEY,
    name TEXT,
    city TEXT
);

CREATE TABLE products (
    id SERIAL PRIMARY KEY,
    name TEXT,
    category TEXT
);

CREATE TABLE order_items (
    id SERIAL PRIMARY KEY,
    order_id INT REFERENCES orders(id),
    product_id INT REFERENCES products(id),
    quantity INT
);

INSERT INTO users (name, city)
SELECT 'User_' || i, (ARRAY['北京','上海','广州'])[1+(i%3)]
FROM generate_series(1, 50000) AS i;

INSERT INTO products (name, category)
SELECT 'Prod_' || i, (ARRAY['A','B','C','D'])[1+(i%4)]
FROM generate_series(1, 1000) AS i;

INSERT INTO order_items (order_id, product_id, quantity)
SELECT (random()*999999+1)::int, (random()*999+1)::int, (random()*10+1)::int
FROM generate_series(1, 2000000);

ANALYZE users; ANALYZE products; ANALYZE order_items;

-- 让规划器自动选择 JOIN 顺序
EXPLAIN ANALYZE
SELECT u.name, p.name, oi.quantity
FROM order_items oi
JOIN orders o ON o.id = oi.order_id
JOIN users u ON u.id = o.user_id
JOIN products p ON p.id = oi.product_id
WHERE u.city = '北京' AND p.category = 'A';

-- 强制按书写顺序 JOIN
SET join_collapse_limit = 1;
EXPLAIN ANALYZE
SELECT u.name, p.name, oi.quantity
FROM users u
JOIN orders o ON o.user_id = u.id
JOIN order_items oi ON oi.order_id = o.id
JOIN products p ON p.id = oi.product_id
WHERE u.city = '北京' AND p.category = 'A';
RESET join_collapse_limit;
```

### 示例3：子查询 vs JOIN vs CTE

```sql
-- 方法1：相关子查询（通常最慢）
EXPLAIN ANALYZE
SELECT u.name,
       (SELECT COUNT(*) FROM orders o WHERE o.user_id = u.id) AS order_count
FROM users u
WHERE u.city = '上海'
LIMIT 100;

-- 方法2：JOIN + GROUP BY
EXPLAIN ANALYZE
SELECT u.name, COUNT(o.id) AS order_count
FROM users u
LEFT JOIN orders o ON o.user_id = u.id
WHERE u.city = '上海'
GROUP BY u.id, u.name
LIMIT 100;

-- 方法3：CTE
EXPLAIN ANALYZE
WITH user_orders AS (
    SELECT user_id, COUNT(*) AS order_count
    FROM orders
    GROUP BY user_id
)
SELECT u.name, COALESCE(uo.order_count, 0)
FROM users u
LEFT JOIN user_orders uo ON uo.user_id = u.id
WHERE u.city = '上海'
LIMIT 100;
```

### 示例4：EXISTS vs IN

```sql
-- 查找有订单的用户

-- 方法1：IN
EXPLAIN ANALYZE
SELECT * FROM users
WHERE id IN (SELECT user_id FROM orders);

-- 方法2：EXISTS
EXPLAIN ANALYZE
SELECT * FROM users u
WHERE EXISTS (SELECT 1 FROM orders o WHERE o.user_id = u.id);

-- 方法3：JOIN (去重)
EXPLAIN ANALYZE
SELECT DISTINCT u.*
FROM users u
JOIN orders o ON o.user_id = u.id;

-- NOT EXISTS vs NOT IN（注意 NULL 安全性）
-- 查找没有订单的用户

-- ✅ NOT EXISTS（推荐）
EXPLAIN ANALYZE
SELECT * FROM users u
WHERE NOT EXISTS (SELECT 1 FROM orders o WHERE o.user_id = u.id);

-- ⚠️ NOT IN（如果子查询结果有 NULL 可能出问题）
EXPLAIN ANALYZE
SELECT * FROM users
WHERE id NOT IN (SELECT user_id FROM orders WHERE user_id IS NOT NULL);

-- ✅ LEFT JOIN + IS NULL
EXPLAIN ANALYZE
SELECT u.*
FROM users u
LEFT JOIN orders o ON o.user_id = u.id
WHERE o.id IS NULL;
```

### 示例5：预编译语句

```sql
-- 创建预编译语句
PREPARE get_user_orders(int) AS
SELECT o.id, o.amount, o.status
FROM orders o
WHERE o.user_id = $1
ORDER BY o.created_at DESC
LIMIT 10;

-- 多次执行
EXECUTE get_user_orders(100);
EXECUTE get_user_orders(200);
EXECUTE get_user_orders(300);

-- 查看执行计划（前5次是 custom plan）
EXPLAIN EXECUTE get_user_orders(100);

-- 执行更多次后查看是否切换为 generic plan
EXECUTE get_user_orders(400);
EXECUTE get_user_orders(500);
EXECUTE get_user_orders(600);
EXPLAIN EXECUTE get_user_orders(700);

-- 查看当前所有预编译语句
SELECT * FROM pg_prepared_statements;

-- 释放
DEALLOCATE get_user_orders;
```

### 示例6：综合优化案例

```sql
-- 优化前：慢查询
EXPLAIN ANALYZE
SELECT u.name, u.city,
       (SELECT SUM(o.amount) FROM orders o WHERE o.user_id = u.id) AS total_amount,
       (SELECT COUNT(*) FROM orders o WHERE o.user_id = u.id) AS order_count
FROM users u
WHERE u.city IN ('北京', '上海')
ORDER BY total_amount DESC NULLS LAST
LIMIT 20;

-- 优化后：用 JOIN 替代相关子查询
EXPLAIN ANALYZE
SELECT u.name, u.city,
       COALESCE(SUM(o.amount), 0) AS total_amount,
       COUNT(o.id) AS order_count
FROM users u
LEFT JOIN orders o ON o.user_id = u.id
WHERE u.city IN ('北京', '上海')
GROUP BY u.id, u.name, u.city
ORDER BY total_amount DESC
LIMIT 20;
```

## 常见问题

### Q1：为什么有时候 JOIN 比子查询慢？

**A**：并不总是 JOIN 更快：
1. 如果子查询结果很小且被缓存，相关子查询可能很快
2. JOIN 可能产生笛卡尔积然后再过滤，中间结果集爆炸
3. 规划器可能选择了不合适的 JOIN 顺序
建议始终用 EXPLAIN ANALYZE 对比实际执行时间，而不是假设哪种写法更好。

### Q2：预编译语句的 Generic Plan 有什么风险？

**A**：Generic Plan 不考虑参数的实际值，当数据分布不均匀时可能选择次优计划。例如，一个 status 列有 99% 的值是 'completed'，1% 是 'pending'。Custom Plan 对 status='pending' 会选择索引扫描，但 Generic Plan 可能选择顺序扫描。可以通过 `plan_cache_mode` 参数控制（`auto`、`force_custom_plan`、`force_generic_plan`）。

### Q3：CTE 什么时候应该强制 MATERIALIZED？

**A**：当 CTE 被多次引用，且 CTE 内部查询开销大时，使用 MATERIALIZED 可以避免重复计算。另外，当你需要"固定"CTE 的结果（比如在 CTE 中修改数据）时，也必须物化。但如果 CTE 只引用一次，让 PostgreSQL 自动内联通常是更好的选择。
