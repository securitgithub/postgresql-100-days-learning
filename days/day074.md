# Day 074: 事务隔离级别 — 理解并发事务的行为

## 学习目标

- 理解 SQL 标准定义的四种事务隔离级别及其差异
- 掌握 PostgreSQL 对各隔离级别的 MVCC 实现方式
- 能够识别和理解脏读、不可重复读、幻读三种并发异常
- 学会处理 Serializable 隔离级别下的序列化失败和重试机制
- 能根据业务场景选择合适的隔离级别

## 理论讲解

### 1. 四种事务隔离级别

SQL 标准定义了四种隔离级别，从低到高：

| 隔离级别 | 脏读 | 不可重复读 | 幻读 |
|---------|------|-----------|------|
| Read Uncommitted | 可能 | 可能 | 可能 |
| Read Committed | 不可能 | 可能 | 可能 |
| Repeatable Read | 不可能 | 不可能 | 可能* |
| Serializable | 不可能 | 不可能 | 不可能 |

**PostgreSQL 的特殊之处：**
- PostgreSQL 的 Read Uncommitted 实际上等同于 Read Committed（不允许脏读）
- PostgreSQL 的 Repeatable Read 实际上也防止了幻读（比标准更严格）
- PostgreSQL 使用 SSI（Serializable Snapshot Isolation）实现 Serializable 级别

### 2. 并发异常解释

**脏读（Dirty Read）**：一个事务读到另一个尚未提交的事务所修改的数据。如果那个事务回滚，读到的就是无效数据。PostgreSQL 在任何隔离级别下都不允许脏读。

**不可重复读（Non-repeatable Read）**：在同一事务中，两次读取同一行数据得到不同的结果，因为在两次读取之间另一个事务修改并提交了该行。

**幻读（Phantom Read）**：在同一事务中，两次执行相同的范围查询，第二次出现了第一次没有的行（"幻影行"），因为另一个事务插入了满足条件的新行。

### 3. MVCC（多版本并发控制）

PostgreSQL 使用 MVCC 实现事务隔离。每个事务看到的是数据的一个快照，而不是锁定数据。

核心机制：
- 每行数据有 `xmin`（创建该版本的事务ID）和 `xmax`（删除/更新该版本的事务ID）
- 事务根据隔离级别决定能看到哪些版本
- Read Committed：每条 SQL 语句获取一个新快照
- Repeatable Read / Serializable：整个事务使用同一个快照

### 4. 序列化失败与重试

在 Repeatable Read 和 Serializable 级别下，如果检测到冲突，PostgreSQL 会抛出错误要求事务重试：
- `ERROR: could not serialize access due to concurrent update`（Repeatable Read）
- `ERROR: could not serialize access due to read/write dependencies`（Serializable）

应用程序必须捕获这些错误（SQLSTATE `40001`）并重试整个事务。

## 代码示例

### 示例1：查看和设置隔离级别

```sql
-- 查看当前默认隔离级别
SHOW default_transaction_isolation;
-- 结果通常是：read committed

-- 设置全局默认隔离级别（需要 superuser）
ALTER SYSTEM SET default_transaction_isolation = 'repeatable read';
SELECT pg_reload_conf();

-- 为当前会话设置隔离级别
SET SESSION CHARACTERISTICS AS TRANSACTION ISOLATION LEVEL SERIALIZABLE;

-- 为单个事务设置隔离级别
BEGIN ISOLATION LEVEL REPEATABLE READ;
-- ... 事务操作 ...
COMMIT;

-- 也可以在 BEGIN 之后设置
BEGIN;
SET TRANSACTION ISOLATION LEVEL SERIALIZABLE;
-- ... 事务操作 ...
COMMIT;

-- 查看当前事务隔离级别
SELECT current_setting('transaction_isolation');
```

### 示例2：Read Committed 下的不可重复读

```sql
-- 准备数据
CREATE TABLE accounts (
    id INTEGER PRIMARY KEY,
    name TEXT,
    balance NUMERIC(10,2)
);

INSERT INTO accounts VALUES
(1, '张三', 10000.00),
(2, '李四', 5000.00);

-- === 会话 A (Read Committed) ===
BEGIN ISOLATION LEVEL READ COMMITTED;

-- 第一次读取
SELECT balance FROM accounts WHERE id = 1;
-- 结果：10000.00

-- 此时会话 B 修改并提交...

-- 第二次读取（在同一事务中）
SELECT balance FROM accounts WHERE id = 1;
-- 结果：8000.00  ← 不可重复读！同一事务中两次读取结果不同
COMMIT;

-- === 会话 B ===
BEGIN;
UPDATE accounts SET balance = 8000.00 WHERE id = 1;
COMMIT;
```

### 示例3：Repeatable Read 防止不可重复读

```sql
-- === 会话 A (Repeatable Read) ===
BEGIN ISOLATION LEVEL REPEATABLE READ;

-- 第一次读取
SELECT balance FROM accounts WHERE id = 1;
-- 结果：10000.00

-- 此时会话 B 修改并提交...

-- 第二次读取（在同一事务中）
SELECT balance FROM accounts WHERE id = 1;
-- 结果：10000.00  ← 保持一致！事务看到的是开始时的快照
COMMIT;

-- === 会话 B ===
BEGIN;
UPDATE accounts SET balance = 8000.00 WHERE id = 1;
COMMIT;
```

### 示例4：Repeatable Read 下的更新冲突

```sql
-- 重置数据
UPDATE accounts SET balance = 10000.00 WHERE id = 1;

-- === 会话 A ===
BEGIN ISOLATION LEVEL REPEATABLE READ;
SELECT balance FROM accounts WHERE id = 1;
-- 结果：10000.00

-- === 会话 B ===
BEGIN;
UPDATE accounts SET balance = balance - 2000 WHERE id = 1;
COMMIT;
-- 会话 B 成功，balance = 8000

-- === 回到会话 A ===
UPDATE accounts SET balance = balance - 3000 WHERE id = 1;
-- ERROR: could not serialize access due to concurrent update
-- 会话 A 必须回滚并重试！

ROLLBACK;

-- 重试整个事务
BEGIN ISOLATION LEVEL REPEATABLE READ;
SELECT balance FROM accounts WHERE id = 1;
-- 结果：8000.00（现在看到 B 的提交了）
UPDATE accounts SET balance = balance - 3000 WHERE id = 1;
-- 成功！balance = 5000
COMMIT;
```

### 示例5：Serializable 级别防止写偏斜

```sql
-- 场景：值班系统要求至少有一人在岗
CREATE TABLE on_call (
    id SERIAL PRIMARY KEY,
    doctor_name TEXT NOT NULL,
    is_on_duty BOOLEAN NOT NULL DEFAULT true
);

INSERT INTO on_call (doctor_name, is_on_duty) VALUES
('医生A', true),
('医生B', true);

-- === 会话 A (Serializable) ===
BEGIN ISOLATION LEVEL SERIALIZABLE;
-- 检查是否有其他人在岗
SELECT count(*) FROM on_call WHERE is_on_duty = true;
-- 结果：2  → 还有另一人，可以请假
UPDATE on_call SET is_on_duty = false WHERE doctor_name = '医生A';

-- === 会话 B (Serializable) ===
BEGIN ISOLATION LEVEL SERIALIZABLE;
-- 也检查是否有其他人在岗
SELECT count(*) FROM on_call WHERE is_on_duty = true;
-- 结果：2  → 还有另一人，可以请假（看到的是快照）
UPDATE on_call SET is_on_duty = false WHERE doctor_name = '医生B';

-- === 两个会话都尝试提交 ===
-- 会话 A: COMMIT;  → 成功
-- 会话 B: COMMIT;  → ERROR: could not serialize access due to
--                     read/write dependencies among transactions

-- 如果使用 Read Committed，两个都会成功，导致无人值班！
-- Serializable 检测到这种"写偏斜"(write skew)并阻止
```

### 示例6：序列化失败的重试逻辑（PL/pgSQL）

```sql
-- 创建带自动重试的转账函数
CREATE OR REPLACE FUNCTION transfer_with_retry(
    p_from_id INTEGER,
    p_to_id INTEGER,
    p_amount NUMERIC,
    p_max_retries INTEGER DEFAULT 5
) RETURNS TEXT AS $$
DECLARE
    v_retries INTEGER := 0;
    v_balance NUMERIC;
BEGIN
    LOOP
        BEGIN
            -- 开始 Serializable 事务（在函数中通过 SET TRANSACTION）
            -- 注意：函数内部不能用 BEGIN/COMMIT，使用 SAVEPOINT 模拟
            PERFORM set_config('transaction_isolation', 'serializable', true);

            -- 检查余额
            SELECT balance INTO v_balance
            FROM accounts WHERE id = p_from_id;

            IF v_balance < p_amount THEN
                RETURN '余额不足';
            END IF;

            -- 执行转账
            UPDATE accounts SET balance = balance - p_amount WHERE id = p_from_id;
            UPDATE accounts SET balance = balance + p_amount WHERE id = p_to_id;

            RETURN '转账成功';

        EXCEPTION
            WHEN serialization_failure THEN
                v_retries := v_retries + 1;
                IF v_retries >= p_max_retries THEN
                    RAISE EXCEPTION '重试次数超限 (% 次)', p_max_retries;
                END IF;
                -- 等待随机时间后重试
                PERFORM pg_sleep(random() * 0.1 * v_retries);
                -- 继续循环重试
        END;
    END LOOP;
END;
$$ LANGUAGE plpgsql;

-- 使用
SELECT transfer_with_retry(1, 2, 1000.00);
```

### 示例7：查看事务快照信息

```sql
-- 查看当前事务的快照信息
SELECT
    txid_current() AS current_txid,
    txid_current_snapshot() AS snapshot;

-- 快照格式说明：xmin:xmax:xip_list
-- xmin: 最小活跃事务ID（所有小于此ID的事务已完成）
-- xmax: 最大已分配事务ID+1
-- xip_list: 活跃事务ID列表

-- 查看系统中的活跃事务
SELECT
    pid,
    xact_start,
    state,
    query,
    backend_xid,
    backend_xmin
FROM pg_stat_activity
WHERE state != 'idle'
ORDER BY xact_start;

-- 查看行的版本信息
SELECT
    xmin,  -- 创建该行版本的事务ID
    xmax,  -- 删除/锁定该行的事务ID（0表示未删除）
    ctid,  -- 物理位置 (page, offset)
    *
FROM accounts;
```

## 常见问题

### Q1: 为什么 PostgreSQL 的 Read Uncommitted 等同于 Read Committed？

**A:** 这是 PostgreSQL MVCC 架构的自然结果。在 MVCC 中，每个事务看到的是已提交数据的快照，未提交的数据在其他事务中本就不可见。实现脏读反而需要额外的工作，且没有实际价值（脏读有严重的数据一致性风险）。SQL 标准允许实现比要求更严格的隔离级别，所以 PostgreSQL 选择在 Read Uncommitted 级别下也提供 Read Committed 的保证。

### Q2: 什么时候应该使用 Serializable 隔离级别？

**A:** 推荐在以下场景使用 Serializable：
1. **对数据一致性要求极高**：如金融系统、库存管理，不能容忍任何并发异常。
2. **存在写偏斜风险**：多个事务基于读取结果做出互相冲突的写操作。
3. **开发效率优先**：使用 Serializable 可以让开发者不用手动添加 SELECT FOR UPDATE 等锁机制，让数据库自动检测冲突。
但必须注意：使用 Serializable 必须实现重试逻辑，否则序列化失败会导致业务失败。

### Q3: 如何在高并发场景下减少序列化失败？

**A:** 几个策略：
1. **缩短事务时间**：事务越短，冲突概率越低。避免在事务中做网络调用或用户交互。
2. **固定访问顺序**：多个事务访问同一组资源时，使用固定的顺序（如按 ID 升序），减少死锁和冲突。
3. **减少读取范围**：精确查询（WHERE id = ?）比范围查询冲突更少。
4. **使用 Read Committed + 显式锁**：对于热点数据，用 `SELECT FOR UPDATE` 代替高隔离级别，可能性能更好。
5. **指数退避重试**：重试时加入随机延迟，避免多个失败事务同时重试造成再次冲突。
