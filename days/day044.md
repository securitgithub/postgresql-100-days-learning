## 理论讲解

### 1. PostgreSQL 连接流程

客户端连接 PostgreSQL 的完整流程：

```
客户端发起连接请求
    ↓
PostgreSQL 检查 listen_addresses 和 port 配置
    ↓
依次匹配 pg_hba.conf 中的规则（从上到下，首次匹配生效）
    ↓
按匹配到的认证方式进行身份验证
    ↓
验证通过 → 建立连接；验证失败 → 拒绝连接
```

### 2. pg_hba.conf 详解

`pg_hba.conf`（Host-Based Authentication）是 PostgreSQL 的客户端认证配置文件，位于数据目录中。

每行规则的格式：

```
TYPE    DATABASE    USER    ADDRESS    METHOD    [OPTIONS]
```

**连接类型（TYPE）：**

| 类型 | 说明 |
|------|------|
| `local` | Unix 域套接字连接（仅限本地） |
| `host` | TCP/IP 连接（包含 SSL 和非 SSL） |
| `hostssl` | 仅 SSL 加密的 TCP/IP 连接 |
| `hostnossl` | 仅非 SSL 的 TCP/IP 连接 |
| `hostgssenc` | 仅 GSSAPI 加密的连接 |

**DATABASE 字段：**
- 数据库名称、`all`、`sameuser`、`samerole`、`replication`，或逗号分隔的列表

**USER 字段：**
- 用户名、`all`，以 `+` 开头表示组成员匹配，或逗号分隔的列表

**ADDRESS 字段：**
- CIDR 格式（如 `192.168.1.0/24`）、主机名、`all`、`samehost`、`samenet`

### 3. 认证方式对比

| 方式 | 安全级别 | 说明 | 适用场景 |
|------|----------|------|----------|
| `trust` | 极低 | 无条件信任，不需要密码 | 仅限本地开发 |
| `reject` | — | 无条件拒绝 | 显式禁止特定连接 |
| `md5` | 中等 | MD5 哈希密码认证 | 兼容旧客户端 |
| `scram-sha-256` | 高 | SCRAM 挑战-响应认证 | 生产环境推荐 |
| `password` | 低 | 明文密码传输 | 不推荐使用 |
| `cert` | 很高 | SSL 客户端证书认证 | 高安全需求场景 |
| `peer` | 高 | 操作系统用户匹配 | 本地信任连接 |
| `ident` | 中等 | ident 协议验证 | Unix 系统 |
| `gss` | 高 | Kerberos/GSSAPI | 企业环境 |
| `ldap` | 高 | LDAP 目录认证 | 企业集中认证 |

### 4. SSL/TLS 配置

PostgreSQL 支持 SSL 加密来保护客户端与服务器之间的通信：

- 需要 OpenSSL 库支持
- 服务器需要证书（server.crt）和私钥（server.key）
- 可选配置客户端证书认证（双向 TLS）
- 通过 `ssl = on` 启用，在 `postgresql.conf` 中配置

### 5. PgBouncer 连接池简介

PostgreSQL 为每个连接创建一个进程，连接创建和销毁成本较高。PgBouncer 是轻量级连接池中间件：

- **Session 模式**：客户端断开后释放连接（默认）
- **Transaction 模式**：事务结束后释放连接（最常用）
- **Statement 模式**：每条语句后释放连接（限制最多）

PgBouncer 可以显著减少 PostgreSQL 的连接数，提升并发处理能力。

## 代码示例

### 示例 1：pg_hba.conf 基本配置

```bash
# pg_hba.conf 典型配置示例
# TYPE    DATABASE    USER        ADDRESS          METHOD

# 本地连接：操作系统用户匹配
local   all         all                          peer

# 本地 TCP/IP 连接：需要 SCRAM 密码认证
host    all         all         127.0.0.1/32     scram-sha-256
host    all         all         ::1/128          scram-sha-256

# 内网特定网段：需要密码认证
host    all         all         192.168.1.0/24   scram-sha-256

# 特定数据库只允许特定用户从特定网段访问
host    production  app_user    10.0.1.0/24      scram-sha-256
host    production  admin       10.0.0.1/32      scram-sha-256

# 拒绝其他所有对 production 数据库的访问
host    production  all         all              reject

# 复制连接只允许从备库服务器
host    replication repl_user   10.0.2.10/32     scram-sha-256

# SSL 连接要求
hostssl all         all         0.0.0.0/0        scram-sha-256

# 拒绝所有非 SSL 的外部连接
hostnossl all       all         0.0.0.0/0        reject
```

### 示例 2：认证方式配置

```bash
# === trust 认证（仅开发环境） ===
# 本地连接不需要密码
local   all    all                    trust
host    all    all    127.0.0.1/32    trust

# === md5 认证（旧版兼容） ===
host    all    all    192.168.1.0/24  md5

# === scram-sha-256 认证（推荐） ===
# 先在 postgresql.conf 中设置：
# password_encryption = 'scram-sha-256'
host    all    all    10.0.0.0/8      scram-sha-256

# === peer 认证（本地系统用户匹配） ===
# 操作系统用户 postgres 可以直接连接为数据库用户 postgres
local   all    postgres               peer
# 操作系统用户 deploy 映射为数据库用户 app_user
local   all    all                    peer map=mymap

# pg_ident.conf 中的映射配置：
# MAPNAME    SYSTEM-USERNAME    PG-USERNAME
# mymap      deploy             app_user
```

```sql
-- 验证 SSL 状态
SHOW ssl;

-- 查看当前连接的 SSL 信息
SELECT * FROM pg_stat_ssl WHERE pid = pg_backend_pid();

-- 查看所有 SSL 连接
SELECT s.pid, s.usename, s.client_addr,
       l.ssl, l.version AS ssl_version, l.cipher
FROM pg_stat_activity s
JOIN pg_stat_ssl l ON s.pid = l.pid
WHERE s.state = 'active';
```

```bash
# 客户端 SSL 连接参数
# psql "host=db.example.com dbname=mydb user=app_user sslmode=verify-full sslrootcert=root.crt"

# sslmode 选项：
# disable     - 不使用 SSL
# allow       - 先尝试非 SSL，失败后尝试 SSL
# prefer      - 先尝试 SSL，失败后尝试非 SSL（默认）
# require     - 必须使用 SSL，不验证证书
# verify-ca   - 必须使用 SSL，验证 CA
# verify-full - 必须使用 SSL，验证 CA 和主机名（最安全）
```

### 示例 4：客户端证书认证


### 示例 5：PgBouncer 基础配置

```ini
; === pgbouncer.ini ===

[databases]
; 数据库连接映射
mydb = host=127.0.0.1 port=5432 dbname=mydb
; 所有未列出的数据库使用通配符
* = host=127.0.0.1 port=5432

[pgbouncer]
; 监听地址和端口
listen_addr = 0.0.0.0
listen_port = 6432

; 认证类型
auth_type = scram-sha-256
auth_file = /etc/pgbouncer/userlist.txt

; 连接池模式
pool_mode = transaction

; 连接池大小
default_pool_size = 20
max_client_conn = 1000
min_pool_size = 5

; 超时设置
server_idle_timeout = 600
client_idle_timeout = 0
query_timeout = 0

; 日志
logfile = /var/log/pgbouncer/pgbouncer.log
pidfile = /var/run/pgbouncer/pgbouncer.pid

; 管理员
admin_users = pgbouncer_admin
stats_users = pgbouncer_stats
```

```bash
# userlist.txt 格式
# "用户名" "密码"（SCRAM 哈希或 MD5 哈希）
# 可以从 PostgreSQL 中获取密码哈希：
# SELECT usename, passwd FROM pg_shadow;
```

## 常见问题

### Q1：修改 pg_hba.conf 后一定需要重启 PostgreSQL 吗？

**A：** 不需要重启，只需重载配置即可：
```sql
SELECT pg_reload_conf();
```
或命令行 `pg_ctl reload`。但注意 `postgresql.conf` 中的某些参数（如 `ssl`、`listen_addresses`、`port`）修改后需要重启才能生效。

pg_hba.conf 的修改在 reload 后对**新连接**生效，已建立的连接不受影响。

### Q2：pg_hba.conf 规则的匹配顺序是怎样的？

**A：** 从上到下，**首次匹配即生效**。这意味着：
- 更具体的规则应放在前面
- 更宽泛的规则放在后面
- `reject` 规则可以用来显式拒绝特定连接

常见错误：将 `host all all 0.0.0.0/0 trust` 放在文件开头，导致后面的所有安全规则都不会生效。

```bash
# 错误示例：宽泛规则在前，具体规则永远不会被匹配
host  all        all       0.0.0.0/0      trust      # 这条匹配一切
host  production admin     10.0.0.1/32    scram-sha-256  # 永远不会被匹配

# 正确示例：具体规则在前
host  production admin     10.0.0.1/32    scram-sha-256
host  all        all       192.168.0.0/16 scram-sha-256
host  all        all       0.0.0.0/0      reject
```

### Q3：scram-sha-256 和 md5 相比有什么优势？

**A：** scram-sha-256 相比 md5 的主要优势：

1. **更强的哈希算法**：SHA-256 替代 MD5，抗碰撞能力更强
2. **挑战-响应机制**：密码不在网络上传输（即使是哈希形式）
3. **防重放攻击**：每次认证使用随机盐值
4. **服务器端安全**：即使数据库密码哈希泄露，也难以逆向
5. **通道绑定**：可与 SSL 绑定，防中间人攻击

从 PostgreSQL 14 开始，`password_encryption` 默认值为 `scram-sha-256`。建议所有新部署都使用 SCRAM。
