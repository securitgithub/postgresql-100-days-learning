# Day 067: 统计信息与查询规划器

## 学习目标

- 理解 pg_statistic 系统表中统计信息的内容与作用
- 掌握 ANALYZE 命令的使用方法和时机
- 了解 default_statistics_target 参数对统计精度的影响
- 学会使用 CREATE STATISTICS 创建多列统计信息
- 理解统计信息如何影响查询规划器的决策

## 理论讲解

### 一、为什么统计信息如此重要？

PostgreSQL 的查询规划器是**基于代价的优化器（Cost-Based Optimizer）**。它需要估算每种执行计划的代价，然后选择代价最低的。而代价估算的核心输入就是**统计信息**——表有多少行、列的值分布如何、哪些值最常出现等等。

如果统计信息不准确，规划器的估算就会偏离实际，可能导致：
- 选择 Seq Scan 而不是 Index Scan（或反过来）
- 选择错误的 JOIN 算法
- 错误的 JOIN 顺序
- 不合理的内存分配

### 二、pg_statistic 与 pg_stats

#### 2.1 pg_statistic

这是 PostgreSQL 存储统计信息的系统表，结构较为底层。实际使用中更常查看 `pg_stats` 视图，它是 `pg_statistic` 的人类可读版本。

#### 2.2 pg_stats 视图的关键字段

| 字段 | 含义 |
|------|------|
| `schemaname` | 模式名 |
| `tablename` | 表名 |
| `attname` | 列名 |
| `null_frac` | NULL 值的比例（0.0 ~ 1.0） |
| `avg_width` | 列值的平均宽度（字节） |
| `n_distinct` | 不同值的数量。正数为估算值；负数表示比例（如 -0.5 表示不同值占总行数 50%） |
| `most_common_vals` | 最常出现的值列表 |
| `most_common_freqs` | 对应的频率列表 |
| `histogram_bounds` | 等频直方图的边界值 |
| `correlation` | 物理存储顺序与逻辑排序的相关性（-1.0 ~ 1.0） |

#### 2.3 统计信息如何被使用

**等值查询估算**：

```
预估行数 = 总行数 × selectivity
```

对于 `WHERE col = 'value'`：
- 如果 value 在 `most_common_vals` 中，直接使用对应的频率
- 否则，假设其余值均匀分布

**范围查询估算**：
利用 `histogram_bounds` 确定满足条件的范围比例。

**NULL 检查**：
`WHERE col IS NULL` → 预估行数 = 总行数 × `null_frac`

**相关性（correlation）**：
如果 correlation 接近 ±1，说明数据按该列物理有序，BRIN 索引和 Index Scan 更有效。

### 三、ANALYZE 命令

#### 3.1 基本用法

```sql
-- 分析整个数据库
ANALYZE;

-- 分析特定表
ANALYZE table_name;

-- 分析表的特定列
ANALYZE table_name(column1, column2);
```

#### 3.2 何时需要手动 ANALYZE

- 大量数据导入（COPY、批量 INSERT）后
- 大量 DELETE 或 UPDATE 后
- 创建新索引前
- 查询计划突然变差时

#### 3.3 自动 ANALYZE

PostgreSQL 的 autovacuum 进程会自动执行 ANALYZE。触发条件：

```
变更行数 > autovacuum_analyze_threshold + autovacuum_analyze_scale_factor × 表行数
```

默认值：
- `autovacuum_analyze_threshold` = 50
- `autovacuum_analyze_scale_factor` = 0.1（10%）

即表变更超过 50 + 10% × 总行数 时触发自动 ANALYZE。

对于大表，可以调低 scale_factor：

```sql
-- 对特定表调整自动分析的灵敏度
ALTER TABLE large_table SET (autovacuum_analyze_scale_factor = 0.02);
```

### 四、default_statistics_target

#### 4.1 概念

`default_statistics_target` 控制 ANALYZE 收集统计信息时的采样量。默认值为 100，表示：
- `most_common_vals` 最多保存 100 个值
- `histogram_bounds` 最多 100 个桶

值越大，统计越精确，但 ANALYZE 和规划耗时也越大。

#### 4.2 调整方式

```sql
-- 全局调整（影响所有表）
SET default_statistics_target = 200;

-- 对特定列调整（推荐）
ALTER TABLE orders ALTER COLUMN status SET STATISTICS 500;
ALTER TABLE orders ALTER COLUMN user_id SET STATISTICS 1000;

-- 调整后需要重新 ANALYZE
ANALYZE orders;
```

#### 4.3 何时需要增大

- 列的值分布非常不均匀（某些值远多于其他值）
- 规划器的行数估算频繁偏差很大
- 列的不同值非常多（如 user_id）

### 五、CREATE STATISTICS —— 多列统计

#### 5.1 问题背景

默认的统计信息是**逐列独立收集**的，规划器假设列之间相互独立。但现实中列之间常常有关联：

```sql
-- 城市和邮编强相关
-- 规划器独立估算：P(city='北京') × P(zip='100000') 可能严重偏离实际
SELECT * FROM addresses
WHERE city = '北京' AND zip_code = '100000';
```

#### 5.2 扩展统计类型

PostgreSQL 10 引入了 `CREATE STATISTICS`，支持以下类型：

| 类型 | 说明 | 版本 |
|------|------|------|
| `ndistinct` | 多列组合的不同值数量 | 10+ |
| `dependencies` | 列之间的函数依赖关系 | 10+ |
| `mcv` | 多列最常见值组合 | 12+ |

#### 5.3 语法

```sql
-- 创建多列统计
CREATE STATISTICS stat_name (ndistinct, dependencies, mcv)
ON column1, column2 FROM table_name;

-- 创建后需要 ANALYZE
ANALYZE table_name;

-- 查看统计信息
SELECT * FROM pg_statistic_ext WHERE stxname = 'stat_name';
SELECT * FROM pg_statistic_ext_data WHERE stxoid = (
    SELECT oid FROM pg_statistic_ext WHERE stxname = 'stat_name'
);
```

## 代码示例

### 示例1：查看 pg_stats 统计信息

```sql
-- 创建测试表
CREATE TABLE sales (
    id SERIAL PRIMARY KEY,
    product_id INT,
    region TEXT,
    amount NUMERIC,
    sale_date DATE
);

-- 插入数据（region 分布不均匀）
INSERT INTO sales (product_id, region, amount, sale_date)
SELECT
    (random() * 100)::int + 1,
    CASE
        WHEN random() < 0.5 THEN '华东'
        WHEN random() < 0.8 THEN '华北'
        WHEN random() < 0.95 THEN '华南'
        ELSE '西部'
    END,
    (random() * 10000)::numeric(10,2),
    '2023-01-01'::date + (random() * 730)::int
FROM generate_series(1, 500000);

ANALYZE sales;

-- 查看 region 列的统计信息
SELECT
    attname,
    null_frac,
    n_distinct,
    most_common_vals,
    most_common_freqs,
    histogram_bounds,
    correlation
FROM pg_stats
WHERE tablename = 'sales' AND attname = 'region';

-- 查看 amount 列的统计信息
SELECT
    attname,
    null_frac,
    avg_width,
    n_distinct,
    array_length(most_common_vals::text[], 1) AS mcv_count,
    histogram_bounds
FROM pg_stats
WHERE tablename = 'sales' AND attname = 'amount';
```

### 示例2：统计信息对查询计划的影响

```sql
-- 查看规划器的行数估算
EXPLAIN SELECT * FROM sales WHERE region = '华东';
-- 预估行数应该接近 50%

EXPLAIN SELECT * FROM sales WHERE region = '西部';
-- 预估行数应该接近 5%

-- 验证估算精度
EXPLAIN ANALYZE SELECT * FROM sales WHERE region = '西部';
-- 比较 rows（估算）和 actual rows（实际）

-- 如果统计信息过时会怎样？
-- 大量修改数据但不 ANALYZE
UPDATE sales SET region = '华东' WHERE region = '西部';

-- 此时统计信息还是旧的
EXPLAIN SELECT * FROM sales WHERE region = '西部';
-- 估算仍认为有约 5% 的数据

-- 更新统计信息
ANALYZE sales;

-- 现在估算准确了
EXPLAIN SELECT * FROM sales WHERE region = '西部';
-- 估算应接近 0 行
```

### 示例3：调整 statistics_target

```sql
-- 创建一个值分布极不均匀的表
CREATE TABLE user_actions (
    id SERIAL PRIMARY KEY,
    user_id INT,
    action_type TEXT
);

-- 插入数据：少数用户产生大量行为
INSERT INTO user_actions (user_id, action_type)
SELECT
    CASE
        WHEN random() < 0.01 THEN (random() * 10)::int         -- 1% 的数据来自 10 个活跃用户
        ELSE (random() * 100000)::int + 11                      -- 99% 分散在 10 万用户中
    END,
    (ARRAY['view','click','purchase','share'])[1+(random()*3)::int]
FROM generate_series(1, 2000000);

-- 默认 statistics_target
ANALYZE user_actions;

-- 查看 user_id 的统计
SELECT n_distinct,
       array_length(most_common_vals::text[], 1) AS mcv_count
FROM pg_stats
WHERE tablename = 'user_actions' AND attname = 'user_id';
-- mcv_count 应该是 100（默认 target）

-- 检查估算精度
EXPLAIN ANALYZE SELECT * FROM user_actions WHERE user_id = 1;

-- 增大 statistics_target
ALTER TABLE user_actions ALTER COLUMN user_id SET STATISTICS 1000;
ANALYZE user_actions;

-- 再次查看
SELECT n_distinct,
       array_length(most_common_vals::text[], 1) AS mcv_count
FROM pg_stats
WHERE tablename = 'user_actions' AND attname = 'user_id';
-- mcv_count 应该是 1000

-- 估算可能更准确
EXPLAIN ANALYZE SELECT * FROM user_actions WHERE user_id = 1;
```

### 示例4：CREATE STATISTICS 多列统计

```sql
-- 创建有强相关性的表
CREATE TABLE addresses (
    id SERIAL PRIMARY KEY,
    city TEXT,
    district TEXT,
    zip_code TEXT
);

-- 城市、区和邮编强相关
INSERT INTO addresses (city, district, zip_code)
SELECT
    city,
    district,
    zip
FROM (VALUES
    ('北京', '朝阳', '100020'),
    ('北京', '海淀', '100080'),
    ('北京', '东城', '100010'),
    ('上海', '浦东', '200120'),
    ('上海', '徐汇', '200030'),
    ('上海', '静安', '200040'),
    ('广州', '天河', '510620'),
    ('广州', '越秀', '510030'),
    ('深圳', '南山', '518000'),
    ('深圳', '福田', '518033')
) AS v(city, district, zip),
generate_series(1, 30000);

ANALYZE addresses;

-- 没有多列统计时的估算
EXPLAIN ANALYZE
SELECT * FROM addresses WHERE city = '北京' AND zip_code = '100020';
-- 规划器独立估算：P(city='北京') × P(zip='100020') ≈ 0.4 × 0.1 = 0.04
-- 实际 P(city='北京' AND zip='100020') ≈ 0.1
-- 估算可能偏差很大

-- 创建多列统计
CREATE STATISTICS stat_city_zip (dependencies, ndistinct, mcv)
ON city, zip_code FROM addresses;

ANALYZE addresses;

-- 有多列统计后的估算
EXPLAIN ANALYZE
SELECT * FROM addresses WHERE city = '北京' AND zip_code = '100020';
-- 估算应该更准确

-- 查看扩展统计信息
SELECT stxname, stxkeys, stxkind
FROM pg_statistic_ext
WHERE stxname = 'stat_city_zip';
```

### 示例5：correlation 与扫描方式

```sql
-- 查看各列的 correlation
SELECT attname, correlation
FROM pg_stats
WHERE tablename = 'sales'
ORDER BY attname;

-- id 列的 correlation 应该接近 1.0（自增序列，物理有序）
-- sale_date 可能也较高（随机生成但大致有序）
-- region 可能接近 0（随机分布）

-- 高 correlation 的列适合 BRIN 索引
CREATE INDEX idx_sales_id_brin ON sales USING brin(id);
EXPLAIN ANALYZE SELECT * FROM sales WHERE id BETWEEN 10000 AND 20000;
```

### 示例6：自动 ANALYZE 配置

```sql
-- 查看当前自动分析配置
SHOW autovacuum_analyze_threshold;
SHOW autovacuum_analyze_scale_factor;

-- 对大表调整更敏感的自动分析
ALTER TABLE sales SET (
    autovacuum_analyze_threshold = 1000,
    autovacuum_analyze_scale_factor = 0.01
);

-- 查看表级配置
SELECT relname, reloptions
FROM pg_class
WHERE relname = 'sales';

-- 查看自动分析的执行情况
SELECT
    relname,
    last_analyze,
    last_autoanalyze,
    analyze_count,
    autoanalyze_count
FROM pg_stat_user_tables
WHERE relname = 'sales';
```

## 常见问题

### Q1：ANALYZE 会锁表吗？

**A**：ANALYZE 只需要 ShareUpdateExclusiveLock，这是一种非常轻量的锁，**不会阻塞任何读写操作**（SELECT、INSERT、UPDATE、DELETE 都不受影响）。只会阻塞 DDL 操作（如 ALTER TABLE）和 VACUUM FULL。所以可以放心在生产环境执行 ANALYZE。

### Q2：default_statistics_target 设得越大越好吗？

**A**：不是。增大 statistics_target 有以下代价：
1. ANALYZE 执行时间增加（需要采样更多数据）
2. pg_statistic 占用更多空间
3. 查询规划时间增加（需要处理更多统计数据）

建议：全局保持默认 100，只对确实需要更精确统计的列单独调大（如倾斜严重的列、经常出现估算偏差的列）。最大值为 10000。

### Q3：CREATE STATISTICS 会自动更新吗？

**A**：是的。一旦创建了扩展统计对象，后续的 ANALYZE（包括 auto-analyze）会自动更新这些统计信息。不需要额外维护。但创建后需要执行一次 ANALYZE 才能生效。
