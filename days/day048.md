## 理论讲解

### 1. Autovacuum 工作原理

Autovacuum 是 PostgreSQL 的自动维护子系统，由以下组件构成：

**Autovacuum Launcher（启动器）：**
- 常驻后台守护进程
- 每隔 `autovacuum_naptime` 检查一次是否有表需要清理
- 负责启动 Autovacuum Worker 进程

**Autovacuum Worker（工作进程）：**
- 实际执行 VACUUM 和 ANALYZE 的进程
- 最多同时运行 `autovacuum_max_workers` 个
- 每个 Worker 处理一个数据库中的一张表

**执行流程：**
```
Launcher 唤醒
    ↓
检查 pg_stat_user_tables 中每张表的统计信息
    ↓
计算每张表是否满足 VACUUM 或 ANALYZE 的触发条件
    ↓
对需要处理的表按优先级排序
    ↓
启动 Worker 处理最紧急的表
    ↓
Worker 完成后被回收，Launcher 等待下次唤醒
```

### 2. 触发条件详解

**VACUUM 触发条件：**
```
n_dead_tup >= autovacuum_vacuum_threshold + autovacuum_vacuum_scale_factor × reltuples
```

例如默认值（threshold=50, scale_factor=0.2）：
- 1000 行的表：需要 50 + 0.2 × 1000 = 250 个死元组
- 100万行的表：需要 50 + 0.2 × 1000000 = 200050 个死元组
- 1亿行的表：需要 50 + 0.2 × 100000000 = 20000050 个死元组

**问题：** 对于大表，默认的 `scale_factor = 0.2` 意味着需要 20% 的行变成死元组才触发 VACUUM，这可能导致大量膨胀。

**ANALYZE 触发条件：**
```
n_mod_since_analyze >= autovacuum_analyze_threshold + autovacuum_analyze_scale_factor × reltuples
```

**冻结触发条件：**
```
age(relfrozenxid) >= autovacuum_freeze_max_age
```
当表的冻结年龄达到此阈值时，强制执行 anti-wraparound VACUUM，此类 VACUUM 无法被取消。

### 3. 成本控制机制

为了避免 Autovacuum 对正常查询造成过大影响，PostgreSQL 使用成本控制机制：

| 参数 | 说明 | 默认值 |
|------|------|--------|
| `vacuum_cost_page_hit` | 在缓冲区中找到页的成本 | 1 |
| `vacuum_cost_page_miss` | 从磁盘读取页的成本 | 2 |
| `vacuum_cost_page_dirty` | 弄脏一个干净页的成本 | 20 |
| `autovacuum_vacuum_cost_limit` | 成本累计上限 | -1（使用 vacuum_cost_limit=200） |
| `autovacuum_vacuum_cost_delay` | 达到上限后的休眠时间 | 2ms |

计算：每秒可处理的页数 ≈ `cost_limit / cost_delay × 1000 / avg_cost_per_page`

### 4. 单表参数覆盖

PostgreSQL 允许为每张表设置独立的 Autovacuum 参数，这对以下场景特别有用：
- 高更新频率的热表需要更频繁的 VACUUM
- 大表需要更低的 scale_factor
- 日志表可能需要禁用 Autovacuum

## 代码示例

### 示例 1：查看 Autovacuum 配置

```sql
-- 查看所有 autovacuum 相关参数
SELECT name, setting, unit, short_desc
FROM pg_settings
WHERE name LIKE '%autovacuum%'
   OR name LIKE '%vacuum_cost%'
ORDER BY name;

-- 查看 autovacuum 是否启用
SHOW autovacuum;

-- 查看关键参数
SELECT
    current_setting('autovacuum_vacuum_threshold') AS vacuum_threshold,
    current_setting('autovacuum_vacuum_scale_factor') AS vacuum_scale_factor,
    current_setting('autovacuum_analyze_threshold') AS analyze_threshold,
    current_setting('autovacuum_analyze_scale_factor') AS analyze_scale_factor,
    current_setting('autovacuum_max_workers') AS max_workers,
    current_setting('autovacuum_naptime') AS naptime,
    current_setting('autovacuum_freeze_max_age') AS freeze_max_age;
```

### 示例 2：计算每张表的 VACUUM 触发阈值

```sql
-- 查看每张表的实际触发阈值和当前状态
SELECT
    schemaname,
    relname,
    n_live_tup,
    n_dead_tup,
    -- 计算 VACUUM 触发阈值
    (current_setting('autovacuum_vacuum_threshold')::int +
     current_setting('autovacuum_vacuum_scale_factor')::float * n_live_tup)::bigint
        AS vacuum_trigger_threshold,
    -- 是否需要 VACUUM
    CASE WHEN n_dead_tup >= (
        current_setting('autovacuum_vacuum_threshold')::int +
        current_setting('autovacuum_vacuum_scale_factor')::float * n_live_tup
    ) THEN '需要 VACUUM'
    ELSE '正常'
    END AS vacuum_status,
    -- 计算 ANALYZE 触发阈值
    (current_setting('autovacuum_analyze_threshold')::int +
     current_setting('autovacuum_analyze_scale_factor')::float * n_live_tup)::bigint
        AS analyze_trigger_threshold,
    n_mod_since_analyze,
    last_autovacuum,
    last_autoanalyze
FROM pg_stat_user_tables
ORDER BY n_dead_tup DESC;
```

### 示例 3：单表 Autovacuum 参数设置

```sql
-- 场景：orders 表更新频繁，需要更积极的 VACUUM
-- 降低 scale_factor 到 0.01（1%变更即触发）
ALTER TABLE orders SET (
    autovacuum_vacuum_threshold = 100,
    autovacuum_vacuum_scale_factor = 0.01,
    autovacuum_analyze_threshold = 50,
    autovacuum_analyze_scale_factor = 0.005
);

-- 场景：大表（1亿行），默认 scale_factor 太大
-- 使用绝对阈值而非比例
ALTER TABLE huge_logs SET (
    autovacuum_vacuum_threshold = 10000,
    autovacuum_vacuum_scale_factor = 0.0,  -- 禁用比例因子
    autovacuum_analyze_threshold = 5000,
    autovacuum_analyze_scale_factor = 0.0
);

-- 场景：加速 VACUUM 执行速度
ALTER TABLE hot_table SET (
    autovacuum_vacuum_cost_delay = '0ms',   -- 不休眠
    autovacuum_vacuum_cost_limit = 10000    -- 提高成本限制
);

-- 场景：对只追加的日志表，降低 VACUUM 频率
ALTER TABLE audit_log SET (
    autovacuum_vacuum_threshold = 100000,
    autovacuum_vacuum_scale_factor = 0.5,
    autovacuum_enabled = false   -- 或直接禁用（慎用！仍需手动 VACUUM）
);

-- 查看表的自定义 autovacuum 参数
SELECT relname, reloptions
FROM pg_class
WHERE reloptions IS NOT NULL
  AND relkind = 'r';

-- 重置为默认值
ALTER TABLE orders RESET (
    autovacuum_vacuum_threshold,
    autovacuum_vacuum_scale_factor,
    autovacuum_analyze_threshold,
    autovacuum_analyze_scale_factor
);
```

### 示例 4：监控 Autovacuum 活动

```sql
-- 查看正在运行的 autovacuum 进程
SELECT pid, datname, usename,
       query,
       state,
       now() - xact_start AS duration,
       wait_event_type, wait_event
FROM pg_stat_activity
WHERE query LIKE 'autovacuum:%'
ORDER BY xact_start;

-- 查看 VACUUM 进度（PostgreSQL 9.6+）
SELECT p.pid,
       a.datname,
       p.relid::regclass AS table_name,
       p.phase,
       p.heap_blks_total,
       p.heap_blks_scanned,
       p.heap_blks_vacuumed,
       CASE WHEN p.heap_blks_total > 0
            THEN round(p.heap_blks_vacuumed::numeric /
                       p.heap_blks_total * 100, 2)
            ELSE 0
       END AS pct_complete,
       p.index_vacuum_count,
       p.max_dead_tuples,
       p.num_dead_tuples
FROM pg_stat_progress_vacuum p
JOIN pg_stat_activity a ON a.pid = p.pid;

-- 查看每张表的上次 VACUUM/ANALYZE 时间
SELECT schemaname, relname,
       n_live_tup,
       n_dead_tup,
       last_vacuum,
       last_autovacuum,
       last_analyze,
       last_autoanalyze,
       vacuum_count,
       autovacuum_count,
       analyze_count,
       autoanalyze_count
FROM pg_stat_user_tables
ORDER BY last_autovacuum DESC NULLS LAST;

-- 查看长时间未 VACUUM 的表
SELECT schemaname, relname,
       n_dead_tup,
       last_autovacuum,
       CASE WHEN last_autovacuum IS NOT NULL
            THEN now() - last_autovacuum
            ELSE NULL
       END AS time_since_last_autovacuum,
       age(relfrozenxid) AS frozen_age
FROM pg_stat_user_tables
WHERE last_autovacuum IS NULL
   OR now() - last_autovacuum > interval '7 days'
ORDER BY n_dead_tup DESC;
```

### 示例 6：Autovacuum 调优实战

```sql
-- === 高并发 OLTP 系统调优 ===

-- 全局设置
-- autovacuum = on
-- autovacuum_max_workers = 6        -- 增加工作进程
-- autovacuum_naptime = '15s'        -- 更频繁检查
-- autovacuum_vacuum_threshold = 25
-- autovacuum_vacuum_scale_factor = 0.02  -- 2%即触发
-- autovacuum_analyze_scale_factor = 0.01

-- 提高 VACUUM 速度
-- autovacuum_vacuum_cost_delay = '0ms'  -- PostgreSQL 12+ 推荐
-- autovacuum_vacuum_cost_limit = 2000

-- 对核心业务表单独设置
ALTER TABLE orders SET (
    autovacuum_vacuum_scale_factor = 0.01,
    autovacuum_vacuum_cost_delay = '0ms',
    autovacuum_vacuum_cost_limit = 5000
);

ALTER TABLE order_items SET (
    autovacuum_vacuum_scale_factor = 0.01,
    autovacuum_vacuum_cost_delay = '0ms',
    autovacuum_vacuum_cost_limit = 5000
);

-- === 数据仓库系统调优 ===
-- 大表不需要频繁 VACUUM，但需要及时 ANALYZE

-- 降低 VACUUM 频率但确保 ANALYZE 及时
-- autovacuum_vacuum_scale_factor = 0.1
-- autovacuum_analyze_scale_factor = 0.02

-- 大批量加载后手动执行
-- VACUUM ANALYZE large_fact_table;

-- === 综合监控查询：找出需要关注的表 ===
SELECT
    schemaname || '.' || relname AS table_name,
    pg_size_pretty(pg_total_relation_size(relid)) AS total_size,
    n_live_tup,
    n_dead_tup,
    CASE WHEN n_live_tup > 0
         THEN round(n_dead_tup::numeric / n_live_tup * 100, 2)
         ELSE 0
    END AS dead_pct,
    last_autovacuum,
    last_autoanalyze,
    age(relfrozenxid) AS frozen_age,
    CASE
        WHEN age(relfrozenxid) > 150000000 THEN '紧急：接近回卷'
        WHEN n_dead_tup > 100000 AND
             n_dead_tup::float / GREATEST(n_live_tup, 1) > 0.1 THEN '警告：膨胀严重'
        WHEN last_autovacuum IS NULL THEN '注意：从未 autovacuum'
        WHEN now() - last_autovacuum > interval '7 days' THEN '注意：长时间未清理'
        ELSE '正常'
    END AS health_status
FROM pg_stat_user_tables
ORDER BY
    CASE
        WHEN age(relfrozenxid) > 150000000 THEN 1
        WHEN n_dead_tup > 100000 THEN 2
        ELSE 3
    END,
    n_dead_tup DESC;
```

## 常见问题

### Q1：Autovacuum 运行中可以被取消吗？

**A：** 分两种情况：

1. **普通 Autovacuum**：可以被取消。如果 autovacuum worker 与用户查询产生锁冲突，autovacuum 会自动退让（在等待锁超时后放弃）。管理员也可以通过 `pg_cancel_backend(pid)` 取消。

2. **Anti-wraparound Autovacuum**：**不能被取消**。这种是为防止事务 ID 回卷而强制执行的 VACUUM，即使使用 `pg_cancel_backend()` 也会立即重新启动。唯一的处理方式是等待它完成。

```sql
-- 查看是否有 anti-wraparound vacuum
SELECT pid, query, now() - xact_start AS duration
FROM pg_stat_activity
WHERE query LIKE '%wraparound%'
   OR (query LIKE 'autovacuum:%' AND query LIKE '%to prevent wraparound%');
```

### Q2：为什么 Autovacuum 一直在运行但死元组还是很多？

**A：** 常见原因：

1. **长事务阻止清理**：如果存在长时间运行的事务，其快照之后产生的死元组无法被清理
```sql
-- 查找长事务
SELECT pid, usename, state, xact_start,
       now() - xact_start AS duration, query
FROM pg_stat_activity
WHERE state IN ('idle in transaction', 'active')
  AND xact_start < now() - interval '10 minutes'
ORDER BY xact_start;
```

2. **成本限制太低**：autovacuum 被限速太严重
3. **Worker 数量不足**：表太多而 worker 不够
4. **更新速度超过清理速度**：需要提高成本限制或增加 worker

### Q3：禁用某张表的 Autovacuum 安全吗？

**A：** **非常不安全**，除非你有完善的手动维护计划。禁用 autovacuum 的风险：

1. **表膨胀**：死元组持续累积，表越来越大
2. **索引膨胀**：索引中的死条目不被清理
3. **事务 ID 回卷**：最危险的后果，可能导致数据库拒绝服务
4. **统计信息过期**：查询计划退化

如果确实需要禁用（如批量加载期间），务必：
```sql
-- 禁用
ALTER TABLE my_table SET (autovacuum_enabled = false);

-- 批量操作完成后立即重新启用
ALTER TABLE my_table SET (autovacuum_enabled = true);

-- 并手动执行一次
VACUUM ANALYZE my_table;
```

注意：即使设置 `autovacuum_enabled = false`，anti-wraparound VACUUM 仍然会执行（这是安全机制）。
