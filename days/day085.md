## 理论讲解

### 1. 高可用方案全景对比

| 方案 | 类型 | Failover | 读写分离 | 连接池 | 复杂度 |
|------|------|----------|----------|--------|--------|
| Patroni | 集群管理 | 自动 | 需配合 HAProxy | 无 | 中 |
| PgPool-II | 中间件 | 自动 | 内建 | 内建 | 高 |
| repmgr | 复制管理 | 自动/手动 | 需配合其他 | 无 | 中 |
| HAProxy+Keepalived | 负载均衡 | VIP 漂移 | 可实现 | 无 | 中 |
| 云 RDS | 托管服务 | 自动 | 内建 | 可选 | 低 |

### 2. PgPool-II

PgPool-II 是 PostgreSQL 生态中功能最丰富的中间件，提供：

- **连接池**：减少 PostgreSQL 的连接创建开销
- **读写分离**：自动将读查询路由到从库
- **负载均衡**：在多个从库之间分配读请求
- **自动 Failover**：检测主库故障并提升从库
- **在线恢复**：自动重建故障节点
- **查询缓存**：缓存 SELECT 查询结果

```
           ┌──────────────┐
客户端 ──▶ │  PgPool-II   │
           │  (中间件)     │
           │  - 连接池     │
           │  - 读写分离   │
           │  - Failover  │
           └──┬────────┬──┘
              │        │
        ┌─────▼──┐ ┌──▼─────┐
        │ PG 主库 │ │ PG 从库 │
        └────────┘ └────────┘
```

**优点**：功能全面、一个组件解决多个问题
**缺点**：配置复杂、本身是单点（需要 Watchdog 实现 PgPool 高可用）、SQL 解析可能有兼容性问题

### 3. repmgr

repmgr（Replication Manager）是由 EDB（EnterpriseDB）维护的 PostgreSQL 复制管理工具：

- **简化复制配置**：自动配置流复制
- **集群管理**：注册和管理集群节点
- **手动/自动 Switchover**：计划内主从切换
- **自动 Failover**：通过 repmgrd 守护进程实现
- **事件通知**：支持自定义通知脚本

repmgr 比 Patroni 更轻量，但缺少分布式一致性存储（依赖 PostgreSQL 自身存储元数据），在网络分区场景下可能不如 Patroni 安全。

### 4. HAProxy + Keepalived

这是一种通用的高可用负载均衡方案：

- **HAProxy**：TCP/HTTP 负载均衡器，通过健康检查路由流量
- **Keepalived**：基于 VRRP 协议提供虚拟 IP（VIP）漂移

```
            VIP: 192.168.1.100
           ┌──────────┐ ┌──────────┐
           │ HAProxy   │ │ HAProxy   │
           │ (Active)  │ │ (Standby) │
           │ Keepalived│ │ Keepalived│
           └────┬──────┘ └──────────┘
                │
        ┌───────┼───────┐
        │       │       │
   ┌────▼─┐ ┌──▼───┐ ┌─▼────┐
   │PG 主库│ │PG 从1│ │PG 从2│
   └──────┘ └──────┘ └──────┘
```

这种方案本身不负责 PostgreSQL 的 Failover，通常与 Patroni 或 repmgr 配合使用。

### 5. 云服务商 RDS 方案对比

| 特性 | AWS RDS/Aurora | 阿里云 RDS PG | Google Cloud SQL | Azure Database |
|------|---------------|--------------|-----------------|----------------|
| 自动 Failover | 支持（~60s） | 支持（~30s） | 支持 | 支持 |
| 只读副本 | 最多15个(Aurora) | 最多32个 | 多个 | 多个 |
| 跨区域复制 | 支持 | 支持 | 支持 | 支持 |
| 自动备份 | 支持 | 支持 | 支持 | 支持 |
| 版本升级 | 半自动 | 半自动 | 半自动 | 半自动 |
| 扩展支持 | 有限 | 有限 | 有限 | 有限 |
| 成本 | 较高 | 中等 | 较高 | 较高 |

云 RDS 的优势在于**运维简单**，适合没有专业 DBA 团队的中小企业。缺点是灵活性较低、成本较高、扩展插件受限。

## 代码示例

### 示例1：PgPool-II 基础配置

```ini
# /etc/pgpool2/pgpool.conf

# ---- 连接设置 ----
listen_addresses = '*'
port = 9999                    # PgPool 监听端口
socket_dir = '/var/run/pgpool'

# ---- 后端 PostgreSQL 节点 ----
backend_hostname0 = '192.168.1.101'
backend_port0 = 5432
backend_weight0 = 1            # 负载均衡权重
backend_data_directory0 = '/var/lib/postgresql/16/main'
backend_flag0 = 'ALLOW_TO_FAILOVER'

backend_hostname1 = '192.168.1.102'
backend_port1 = 5432
backend_weight1 = 1
backend_data_directory1 = '/var/lib/postgresql/16/main'
backend_flag1 = 'ALLOW_TO_FAILOVER'

backend_hostname2 = '192.168.1.103'
backend_port2 = 5432
backend_weight2 = 1
backend_data_directory2 = '/var/lib/postgresql/16/main'
backend_flag2 = 'ALLOW_TO_FAILOVER'

# ---- 连接池 ----
num_init_children = 32         # 连接池子进程数
max_pool = 4                   # 每个子进程的连接池大小
connection_cache = on

# ---- 读写分离与负载均衡 ----
load_balance_mode = on         # 启用读负载均衡
master_slave_mode = on         # 主从模式
master_slave_sub_mode = 'stream'  # 流复制子模式

# 读写分离：SELECT 路由到从库，写操作路由到主库
# 以下函数列表中的函数会被发送到主库
black_function_list = 'currval,lastval,nextval,setval'

# ---- 健康检查 ----
health_check_period = 10       # 健康检查间隔（秒）
health_check_timeout = 20
health_check_user = 'pgpool'
health_check_password = 'pgpool_pass'
health_check_max_retries = 3

# ---- 自动 Failover ----
failover_command = '/etc/pgpool2/failover.sh %d %h %p %D %m %H %M %P %r %R'
follow_primary_command = '/etc/pgpool2/follow_primary.sh %d %h %p %D %m %H %M %P %r %R'

# ---- Watchdog（PgPool 自身高可用）----
use_watchdog = on
wd_hostname = '192.168.1.201'
wd_port = 9000
delegate_IP = '192.168.1.200'  # 虚拟 IP
```

### 示例2：PgPool-II Failover 脚本

```bash
#!/bin/bash
# /etc/pgpool2/failover.sh
# PgPool-II failover 回调脚本

FAILED_NODE_ID="$1"
FAILED_HOST="$2"
FAILED_PORT="$3"
FAILED_DB_DIR="$4"
NEW_MASTER_ID="$5"
NEW_MASTER_HOST="$6"
OLD_MASTER_ID="$7"
OLD_PRIMARY_NODE_ID="$8"

PGHOME=/usr/lib/postgresql/16/bin
REPL_USER=replicator
REPL_PASSWORD=ReplP@ss

echo "$(date) Failover triggered: failed_node=$FAILED_NODE_ID new_master=$NEW_MASTER_HOST" \
    >> /var/log/pgpool/failover.log

# 如果失败的是主库，则提升新主库
if [ "$FAILED_NODE_ID" = "$OLD_PRIMARY_NODE_ID" ]; then
    echo "Primary node failed. Promoting $NEW_MASTER_HOST..." \
        >> /var/log/pgpool/failover.log

    # 在新主库上执行 promote
    ssh -T postgres@$NEW_MASTER_HOST \
        "$PGHOME/pg_ctl promote -D $FAILED_DB_DIR"

    if [ $? -eq 0 ]; then
        echo "Successfully promoted $NEW_MASTER_HOST as new primary" \
            >> /var/log/pgpool/failover.log
    else
        echo "ERROR: Failed to promote $NEW_MASTER_HOST" \
            >> /var/log/pgpool/failover.log
        exit 1
    fi
fi

exit 0
```

### 示例3：repmgr 安装与配置

```bash

sudo apt-get install postgresql-16-repmgr
```

```ini
# /etc/repmgr/16/repmgr.conf — 主库节点配置

# 日志
log_level=INFO
log_file='/var/log/repmgr/repmgr.log'
```

### 示例4：repmgr 集群搭建步骤

```bash
# ===== 主库操作 =====

# 1. 创建 repmgr 用户和数据库
sudo -u postgres psql -c "CREATE ROLE repmgr WITH SUPERUSER LOGIN PASSWORD 'RepmgrP@ss';"
sudo -u postgres psql -c "CREATE DATABASE repmgr OWNER repmgr;"

# 2. 注册主库
repmgr -f /etc/repmgr/16/repmgr.conf primary register

# 3. 验证
repmgr -f /etc/repmgr/16/repmgr.conf cluster show

# ===== 从库操作 =====

# 4. 克隆从库（自动执行 pg_basebackup）
repmgr -h 192.168.1.101 -U repmgr -d repmgr \
    -f /etc/repmgr/16/repmgr.conf standby clone --dry-run

# 确认无误后执行
repmgr -h 192.168.1.101 -U repmgr -d repmgr \
    -f /etc/repmgr/16/repmgr.conf standby clone

# 5. 启动从库
sudo systemctl start postgresql

# 6. 注册从库
repmgr -f /etc/repmgr/16/repmgr.conf standby register

# 7. 查看集群状态
repmgr -f /etc/repmgr/16/repmgr.conf cluster show

# 输出示例：
#  ID | Name     | Role    | Status    | Upstream | Location | Prio | TL
# ----+----------+---------+-----------+----------+----------+------+----
#  1  | pg-node1 | primary | * running |          | default  | 100  | 1
#  2  | pg-node2 | standby |   running | pg-node1 | default  | 100  | 1
```

### 示例6：启动 repmgrd 实现自动 Failover

```bash
# 在所有节点启动 repmgrd 守护进程
repmgrd -f /etc/repmgr/16/repmgr.conf --daemonize

# 或通过 systemd 管理
sudo systemctl start repmgrd
sudo systemctl enable repmgrd

# repmgrd 会持续监控集群状态
# 当检测到主库不可达时，自动触发 Failover
```

### 示例7：Keepalived 配置示例

```ini
# /etc/keepalived/keepalived.conf — HAProxy 节点1

global_defs {
    router_id HAProxy_PG_1
}

vrrp_script check_haproxy {
    script "/usr/bin/killall -0 haproxy"
    interval 2
    weight 2
}

vrrp_instance VI_PG {
    state MASTER              # 另一个节点设为 BACKUP
    interface eth0
    virtual_router_id 51
    priority 101              # 另一个节点设为 100
    advert_int 1

    authentication {
        auth_type PASS
        auth_pass PgHA2024
    }

    virtual_ipaddress {
        192.168.1.200/24      # 虚拟 IP 地址
    }

    track_script {
        check_haproxy
    }
}
```

## 常见问题

### Q1: Patroni 和 repmgr 该选哪个？

**A:** 如果你需要最高的可靠性和社区支持，推荐 Patroni。Patroni 依赖外部 DCS（如 etcd）进行 Leader 选举，在网络分区场景下更安全，不容易出现"脑裂"。repmgr 更轻量，没有外部依赖，适合规模较小或对复杂度敏感的团队。目前业界主流趋势是选择 Patroni。

### Q2: PgPool-II 的 SQL 兼容性有什么问题？

**A:** PgPool-II 需要解析 SQL 来判断读写操作，但其 SQL 解析器不如 PostgreSQL 完善。某些复杂 SQL、CTE、函数调用可能被错误分类。例如，包含写操作的函数调用如果被当作 SELECT 路由到从库就会失败。建议在使用前做充分的 SQL 兼容性测试，并利用 `black_function_list` 和 `white_function_list` 来手动调整路由规则。

### Q3: 云 RDS 和自建 PostgreSQL 如何选择？

**A:** 关键考量因素：(1) 团队能力 — 没有专业 DBA 选云 RDS；(2) 成本 — 长期大规模使用自建更便宜；(3) 定制需求 — 需要特殊扩展（如 PostGIS、TimescaleDB）自建更灵活；(4) 合规要求 — 某些行业要求数据不出境或私有部署。一般建议初期用云 RDS 快速起步，规模增长后评估是否迁移到自建方案。
