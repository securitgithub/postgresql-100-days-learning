## 理论讲解

### 1. 表空间（Tablespace）概述

表空间是 PostgreSQL 中管理磁盘布局的逻辑概念，允许管理员将数据文件分布到不同的物理磁盘上。

使用场景：
- 将频繁访问的表放在 SSD 上，归档数据放在 HDD 上
- 将索引与表数据分离到不同磁盘
- 管理磁盘空间不足时扩展存储
- 为不同数据库或用户分配独立的存储位置

默认表空间：
- `pg_default`：数据目录下的 `base` 子目录，默认存储位置
- `pg_global`：数据目录下的 `global` 子目录，存储集群级共享表

### 2. 数据目录结构

PostgreSQL 的数据目录（`PGDATA`）结构如下：

```
PGDATA/
├── base/                  # 默认表空间，每个数据库一个子目录
│   ├── 1/                 # template1 数据库
│   ├── 13067/             # template0 数据库（OID）
│   └── 13068/             # postgres 数据库（OID）
│       ├── 16384          # 表文件（OID）
│       ├── 16384_fsm      # Free Space Map
│       └── 16384_vm       # Visibility Map
├── global/                # 集群级共享表（pg_database等）
├── pg_commit_ts/          # 事务提交时间戳数据
├── pg_dynshmem/           # 动态共享内存文件
├── pg_logical/            # 逻辑复制数据
├── pg_multixact/          # 多事务状态数据
├── pg_notify/             # LISTEN/NOTIFY 数据
├── pg_replslot/           # 复制槽数据
├── pg_serial/             # 已提交的序列化事务信息
├── pg_snapshots/          # 导出的快照
├── pg_stat/               # 永久统计子系统数据
├── pg_stat_tmp/           # 临时统计数据
├── pg_subtrans/           # 子事务状态数据
├── pg_tblspc/             # 表空间的符号链接
├── pg_twophase/           # 两阶段事务文件
├── pg_wal/                # WAL 文件
│   └── archive_status/    # WAL 归档状态
├── pg_xact/               # 事务提交状态数据（原 pg_clog）
├── postgresql.conf        # 主配置文件
├── postgresql.auto.conf   # ALTER SYSTEM 生成的配置
├── pg_hba.conf            # 客户端认证配置
├── pg_ident.conf          # 用户名映射配置
├── PG_VERSION             # 主版本号
├── postmaster.opts        # 最近启动参数
└── postmaster.pid         # PID 文件
```

### 3. 文件存储细节

- 每个表和索引对应一个或多个文件，文件名为对象的 OID（或 relfilenode）
- 单个文件最大 1GB，超过后自动分段（如 `16384`, `16384.1`, `16384.2`）
- 每个表附带辅助文件：
  - `_fsm`：Free Space Map，记录每页的可用空间
  - `_vm`：Visibility Map，记录哪些页全部可见（用于 Index-Only Scan 和 VACUUM 优化）

### 4. TOAST 机制

TOAST（The Oversized-Attribute Storage Technique）是 PostgreSQL 处理大字段值的机制。

触发条件：当一行数据超过约 2KB 时（页面大小 8KB 的约 1/4），TOAST 会介入。

TOAST 策略：

| 策略 | 说明 |
|------|------|
| `PLAIN` | 不压缩，不外存。仅用于定长类型 |
| `EXTENDED` | 先尝试压缩，若仍过大则外存（默认策略） |
| `EXTERNAL` | 不压缩，直接外存。适合已压缩的数据 |
| `MAIN` | 先尝试压缩，尽量不外存 |

TOAST 表：
- 每个含有可 TOAST 列的表都有一个关联的 TOAST 表
- TOAST 表存储在 `pg_toast` schema 下
- 命名格式：`pg_toast.pg_toast_<oid>`
- TOAST 表自带索引用于快速查找

### 5. 磁盘空间管理

磁盘空间不足是数据库最危险的故障之一。需要监控：
- 数据目录空间
- WAL 目录空间
- 表空间空间
- 临时文件空间
- 表膨胀导致的空间浪费

## 代码示例

### 示例1：表空间创建与管理

```sql
-- 查看现有表空间
SELECT spcname, pg_tablespace_location(oid),
       pg_size_pretty(pg_tablespace_size(spcname)) AS size
FROM pg_tablespace;

-- 创建表空间（需要超级用户，目录必须已存在且属于 postgres 用户）
-- 先在系统中创建目录：mkdir -p /data/fast_ssd && chown postgres:postgres /data/fast_ssd
CREATE TABLESPACE fast_ssd LOCATION '/data/fast_ssd';
CREATE TABLESPACE slow_hdd LOCATION '/data/slow_hdd';

-- 设置表空间所有者
ALTER TABLESPACE fast_ssd OWNER TO dba_admin;

-- 设置表空间的默认参数
ALTER TABLESPACE fast_ssd SET (seq_page_cost = 1.0, random_page_cost = 1.1);
ALTER TABLESPACE slow_hdd SET (seq_page_cost = 1.0, random_page_cost = 4.0);
```

### 示例2：在指定表空间创建对象

```sql
-- 在指定表空间创建表
CREATE TABLE hot_data (
    id serial PRIMARY KEY,
    data jsonb NOT NULL,
    created_at timestamptz DEFAULT now()
) TABLESPACE fast_ssd;

-- 在指定表空间创建索引
CREATE INDEX idx_hot_data_created ON hot_data (created_at) TABLESPACE fast_ssd;

-- 将已有表移到新表空间
ALTER TABLE archive_data SET TABLESPACE slow_hdd;

-- 将索引移到新表空间
ALTER INDEX idx_archive_data_date SET TABLESPACE slow_hdd;

-- 将数据库默认表空间设为 fast_ssd
ALTER DATABASE myapp SET TABLESPACE fast_ssd;

-- 为用户设置默认表空间
ALTER USER app_user SET default_tablespace = 'fast_ssd';

-- 批量迁移所有表到新表空间
ALTER TABLE ALL IN TABLESPACE pg_default SET TABLESPACE fast_ssd;

-- 删除表空间（必须先清空其中所有对象）
DROP TABLESPACE slow_hdd;
```

### 示例3：数据目录与文件映射

```sql
-- 查看表对应的文件路径
SELECT pg_relation_filepath('hot_data');
-- 返回类似: base/16384/24576

-- 查看表的 OID 和 relfilenode
SELECT oid, relname, relfilenode, reltablespace
FROM pg_class
WHERE relname = 'hot_data';

-- 查看数据库 OID
SELECT oid, datname FROM pg_database;

-- 查看完整的文件路径
SELECT pg_relation_filepath(c.oid) AS filepath,
       c.relname,
       CASE c.relkind
           WHEN 'r' THEN 'table'
           WHEN 'i' THEN 'index'
           WHEN 't' THEN 'toast'
           WHEN 'S' THEN 'sequence'
       END AS type,
       pg_size_pretty(pg_relation_size(c.oid)) AS size
FROM pg_class c
WHERE c.relnamespace = 'public'::regnamespace
ORDER BY pg_relation_size(c.oid) DESC
LIMIT 20;
```

### 示例4：磁盘空间监控查询

```sql
-- 数据库大小排名
SELECT datname,
       pg_size_pretty(pg_database_size(datname)) AS db_size
FROM pg_database
ORDER BY pg_database_size(datname) DESC;

-- 当前数据库中最大的表
SELECT schemaname,
       tablename,
       pg_size_pretty(pg_total_relation_size(schemaname || '.' || tablename)) AS total_size,
       pg_size_pretty(pg_table_size(schemaname || '.' || tablename)) AS table_size,
       pg_size_pretty(pg_indexes_size(schemaname || '.' || tablename)) AS index_size,
       pg_size_pretty(
           pg_total_relation_size(schemaname || '.' || tablename)
           - pg_table_size(schemaname || '.' || tablename)
           - pg_indexes_size(schemaname || '.' || tablename)
       ) AS toast_size
FROM pg_tables
WHERE schemaname NOT IN ('pg_catalog', 'information_schema')
ORDER BY pg_total_relation_size(schemaname || '.' || tablename) DESC
LIMIT 20;

-- 表空间使用情况
SELECT spcname,
       pg_size_pretty(pg_tablespace_size(spcname)) AS size,
       pg_tablespace_location(oid) AS location
FROM pg_tablespace
ORDER BY pg_tablespace_size(spcname) DESC;

-- 表膨胀估算（简化版）
SELECT schemaname, relname,
       n_live_tup,
       n_dead_tup,
       pg_size_pretty(pg_table_size(relid)) AS table_size,
       CASE WHEN n_live_tup > 0
            THEN round(100.0 * n_dead_tup / (n_live_tup + n_dead_tup), 1)
            ELSE 0 END AS bloat_pct,
       last_autovacuum
FROM pg_stat_user_tables
WHERE n_dead_tup > 10000
ORDER BY n_dead_tup DESC
LIMIT 20;
```

### 示例5：TOAST 相关查询

```sql
-- 查看表的 TOAST 表
SELECT c.relname AS table_name,
       t.relname AS toast_table,
       pg_size_pretty(pg_relation_size(t.oid)) AS toast_size
FROM pg_class c
JOIN pg_class t ON c.reltoastrelid = t.oid
WHERE c.relnamespace = 'public'::regnamespace
ORDER BY pg_relation_size(t.oid) DESC;

-- 查看列的 TOAST 策略
SELECT attname, atttypid::regtype,
       CASE attstorage
           WHEN 'p' THEN 'PLAIN'
           WHEN 'e' THEN 'EXTERNAL'
           WHEN 'x' THEN 'EXTENDED'
           WHEN 'm' THEN 'MAIN'
       END AS storage_strategy
FROM pg_attribute
WHERE attrelid = 'hot_data'::regclass
  AND attnum > 0
  AND NOT attisdropped;

-- 修改列的 TOAST 策略
ALTER TABLE hot_data ALTER COLUMN data SET STORAGE EXTERNAL;
-- EXTERNAL：不压缩直接外存，适合已压缩的数据（如图片）
-- EXTENDED：先压缩再外存（默认，适合 JSON/TEXT）
-- MAIN：尽量压缩不外存

-- 查看 TOAST 块大小
SHOW block_size;   -- 通常 8192 字节
-- TOAST 块大小约为 block_size 的 1/4 ≈ 2000 字节
```

### 示例6：pg_size_pretty 系列函数

```sql
-- 各种大小查询函数
SELECT
    pg_size_pretty(pg_database_size(current_database())) AS database_size,
    pg_size_pretty(pg_total_relation_size('hot_data')) AS total_size,
    pg_size_pretty(pg_table_size('hot_data')) AS table_only,
    pg_size_pretty(pg_indexes_size('hot_data')) AS indexes_size,
    pg_size_pretty(pg_relation_size('hot_data')) AS main_fork;

-- 查看表的各个 fork 大小
SELECT pg_size_pretty(pg_relation_size('hot_data', 'main')) AS main,
       pg_size_pretty(pg_relation_size('hot_data', 'fsm')) AS fsm,
       pg_size_pretty(pg_relation_size('hot_data', 'vm')) AS vm,
       pg_size_pretty(pg_relation_size('hot_data', 'init')) AS init;
```

### 示例7：临时文件监控

```sql
-- 查看临时文件使用
SELECT datname,
       temp_files,
       pg_size_pretty(temp_bytes) AS temp_bytes
FROM pg_stat_database
WHERE temp_files > 0
ORDER BY temp_bytes DESC;

-- 配置临时文件告警
SHOW temp_file_limit;  -- 默认 -1（不限制）
-- 设置临时文件大小限制（防止单个查询占满磁盘）
ALTER SYSTEM SET temp_file_limit = '5GB';
SELECT pg_reload_conf();
```

## 常见问题

### Q1：删除大量数据后，磁盘空间为什么没有释放？

PostgreSQL 的 MVCC 机制意味着 DELETE 只是将行标记为"死元组"，不会立即释放磁盘空间。普通 `VACUUM` 会将空间标记为可重用（但不还给操作系统），`VACUUM FULL` 会重写整张表，真正释放空间但需要独占锁。在大多数场景下，普通 VACUUM 配合后续数据写入重用空间就足够了。

### Q2：表空间可以跨文件系统吗？

表空间的 LOCATION 可以指向任何文件系统上的目录，只要 PostgreSQL 进程（postgres 用户）有读写权限。使用 NFS 或网络存储作为表空间虽然技术上可行，但可能导致性能问题和数据安全隐患。建议使用本地磁盘或可靠的块存储设备。

### Q3：TOAST 表需要手动维护吗？

通常不需要。TOAST 表会随主表一起被 VACUUM 处理。但如果主表大量更新大字段，TOAST 表可能膨胀严重。可以通过 `pg_toast.pg_toast_<oid>` 直接查看 TOAST 表大小。如果膨胀严重，可对主表执行 `VACUUM FULL` 或 `pg_repack` 来同时回收 TOAST 表空间。
