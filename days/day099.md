## 理论讲解

### 1. 运维体系全景

```
┌─────────────────────────────────────────────┐
│               数据库运维体系                   │
├───────────┬──────────┬──────────┬────────────┤

└───────────┴──────────┴──────────┴────────────┘
```

### 2. 备份策略设计

备份策略应遵循 **3-2-1 原则**：
- **3** 份数据副本
- **2** 种不同存储介质
- **1** 份异地备份

PostgreSQL 备份方式：

| 方式 | 工具 | 特点 | 适用场景 |
|------|------|------|---------|
| 逻辑备份 | pg_dump/pg_dumpall | SQL 级别，可选表 | 小库、迁移 |
| 物理备份 | pg_basebackup | 文件级别，完整拷贝 | 大库、PITR |
| WAL 归档 | archive_command | 增量备份 | 配合物理备份做 PITR |
| 第三方工具 | pgBackRest, Barman | 增量/差异/并行 | 生产环境推荐 |

### 3. 监控指标体系

核心监控指标分为四大类：

- **性能指标**：QPS、TPS、查询延迟、缓冲区命中率
- **资源指标**：连接数、CPU、内存、磁盘IO、磁盘空间
- **复制指标**：复制延迟、WAL 堆积
- **健康指标**：死锁、长事务、膨胀表、失败的 autovacuum

### 4. 高可用架构

```
┌─────────┐         ┌─────────┐
│  主库    │ ──WAL──→│  备库    │
│ Primary │         │ Standby │
└────┬────┘         └────┬────┘
     │                    │
     ▼                    ▼
┌─────────┐         ┌─────────┐
│ 读写请求 │         │ 只读请求 │
└─────────┘         └─────────┘
```

流复制模式：
- **异步复制**（默认）：性能最好，可能丢失少量数据
- **同步复制**：零数据丢失，但写入延迟增加

### 5. 安全加固层次

- **网络层**：防火墙、VPN、SSL/TLS
- **认证层**：pg_hba.conf、SCRAM-SHA-256
- **授权层**：最小权限原则、行级安全策略
- **数据层**：列级加密、传输加密
- **审计层**：pgAudit 扩展

## 代码示例

### 示例1：逻辑备份策略

```bash
#!/bin/bash
# ============================================================
# backup_logical.sh - PostgreSQL 逻辑备份脚本
# ============================================================

# 配置
DB_HOST="localhost"
DB_PORT="5432"
DB_NAME="ecommerce"
DB_USER="postgres"
BACKUP_DIR="/data/backups/logical"
RETENTION_DAYS=30
DATE=$(date +%Y%m%d_%H%M%S)
LOG_FILE="${BACKUP_DIR}/backup_${DATE}.log"

# 创建备份目录
mkdir -p "${BACKUP_DIR}"

echo "$(date): 开始逻辑备份" | tee -a "${LOG_FILE}"

# 全库备份（自定义格式，支持并行恢复）
pg_dump \
    -h "${DB_HOST}" \
    -p "${DB_PORT}" \
    -U "${DB_USER}" \
    -d "${DB_NAME}" \
    -Fc \
    --verbose \
    --no-owner \
    --no-privileges \
    -f "${BACKUP_DIR}/full_${DATE}.dump" \
    2>> "${LOG_FILE}"

if [ $? -eq 0 ]; then
    echo "$(date): 全库备份成功" | tee -a "${LOG_FILE}"
    # 记录备份大小
    ls -lh "${BACKUP_DIR}/full_${DATE}.dump" >> "${LOG_FILE}"
else
    echo "$(date): 全库备份失败！" | tee -a "${LOG_FILE}"
    # 发送告警（可接入告警系统）
    exit 1
fi

# 仅备份 Schema（用于文档和审计）
pg_dump \
    -h "${DB_HOST}" \
    -p "${DB_PORT}" \
    -U "${DB_USER}" \
    -d "${DB_NAME}" \
    --schema-only \
    -f "${BACKUP_DIR}/schema_${DATE}.sql" \
    2>> "${LOG_FILE}"

# 清理过期备份
find "${BACKUP_DIR}" -name "*.dump" -mtime +${RETENTION_DAYS} -delete
find "${BACKUP_DIR}" -name "*.sql" -mtime +${RETENTION_DAYS} -delete
find "${BACKUP_DIR}" -name "*.log" -mtime +${RETENTION_DAYS} -delete

echo "$(date): 已清理 ${RETENTION_DAYS} 天前的备份" | tee -a "${LOG_FILE}"

# 备份验证（恢复到临时数据库测试）
echo "$(date): 开始备份验证..." | tee -a "${LOG_FILE}"
createdb -h "${DB_HOST}" -U "${DB_USER}" "verify_${DATE}" 2>/dev/null
pg_restore \
    -h "${DB_HOST}" \
    -U "${DB_USER}" \
    -d "verify_${DATE}" \
    --jobs=4 \
    "${BACKUP_DIR}/full_${DATE}.dump" \
    2>> "${LOG_FILE}"

if [ $? -eq 0 ]; then
    # 简单验证数据
    ROW_COUNT=$(psql -h "${DB_HOST}" -U "${DB_USER}" -d "verify_${DATE}" \
        -t -c "SELECT COUNT(*) FROM orders" 2>/dev/null)
    echo "$(date): 备份验证成功，订单数: ${ROW_COUNT}" | tee -a "${LOG_FILE}"
else
    echo "$(date): 备份验证失败！" | tee -a "${LOG_FILE}"
fi

# 清理验证数据库
dropdb -h "${DB_HOST}" -U "${DB_USER}" "verify_${DATE}" 2>/dev/null

echo "$(date): 备份流程完成" | tee -a "${LOG_FILE}"
```

#!/bin/bash
# ============================================================
# 使用 pg_basebackup 做物理基础备份
# ============================================================

BACKUP_DIR="/data/backups/physical"
DATE=$(date +%Y%m%d_%H%M%S)

mkdir -p "${BACKUP_DIR}/${DATE}"

pg_basebackup \
    -h localhost \
    -U replicator \
    -D "${BACKUP_DIR}/${DATE}" \
    -Ft \
    -z \
    --wal-method=stream \
    --checkpoint=fast \
    --progress \
    --verbose

echo "物理备份完成: ${BACKUP_DIR}/${DATE}"
ls -lh "${BACKUP_DIR}/${DATE}/"

### 示例3：监控告警

```sql
-- ============================================================
-- 监控查询合集
-- ============================================================

-- 1. 连接数监控
SELECT
    count(*) AS total_connections,
    count(*) FILTER (WHERE state = 'active') AS active,
    count(*) FILTER (WHERE state = 'idle') AS idle,
    count(*) FILTER (WHERE state = 'idle in transaction') AS idle_in_tx,
    (SELECT setting::int FROM pg_settings WHERE name = 'max_connections') AS max_conn,
    round(count(*) * 100.0 / (SELECT setting::int FROM pg_settings WHERE name = 'max_connections'), 1)
        AS usage_pct
FROM pg_stat_activity
WHERE backend_type = 'client backend';

-- 2. 长事务检测（超过5分钟的事务）
SELECT
    pid,
    usename,
    state,
    age(clock_timestamp(), xact_start) AS tx_duration,
    age(clock_timestamp(), query_start) AS query_duration,
    LEFT(query, 100) AS query_preview,
    wait_event_type,
    wait_event
FROM pg_stat_activity
WHERE state != 'idle'
  AND xact_start < NOW() - INTERVAL '5 minutes'
ORDER BY xact_start;

-- 3. 锁等待检测
SELECT
    blocked.pid AS blocked_pid,
    blocked.usename AS blocked_user,
    blocking.pid AS blocking_pid,
    blocking.usename AS blocking_user,
    blocked.query AS blocked_query,
    blocking.query AS blocking_query,
    age(clock_timestamp(), blocked.query_start) AS waiting_duration
FROM pg_stat_activity blocked
JOIN pg_locks blocked_locks ON blocked.pid = blocked_locks.pid
JOIN pg_locks blocking_locks ON blocked_locks.locktype = blocking_locks.locktype
    AND blocked_locks.relation = blocking_locks.relation
    AND blocked_locks.pid != blocking_locks.pid
JOIN pg_stat_activity blocking ON blocking_locks.pid = blocking.pid
WHERE NOT blocked_locks.granted;

-- 4. 表膨胀估算
SELECT
    schemaname || '.' || relname AS table_name,
    pg_size_pretty(pg_total_relation_size(relid)) AS total_size,
    n_live_tup AS live_rows,
    n_dead_tup AS dead_rows,
    CASE WHEN n_live_tup > 0
        THEN round(n_dead_tup * 100.0 / n_live_tup, 1)
        ELSE 0
    END AS dead_pct,
    last_vacuum,
    last_autovacuum,
    last_analyze,
    last_autoanalyze
FROM pg_stat_user_tables
ORDER BY n_dead_tup DESC
LIMIT 20;

-- 5. 磁盘空间监控
SELECT
    tablename,
    pg_size_pretty(pg_total_relation_size(schemaname || '.' || tablename)) AS total_size,
    pg_size_pretty(pg_relation_size(schemaname || '.' || tablename)) AS table_size,
    pg_size_pretty(pg_indexes_size(schemaname || '.' || tablename)) AS index_size
FROM pg_tables
WHERE schemaname = 'public'
ORDER BY pg_total_relation_size(schemaname || '.' || tablename) DESC
LIMIT 20;

-- 6. 复制延迟监控
SELECT
    client_addr,
    application_name,
    state,
    sent_lsn,
    write_lsn,
    flush_lsn,
    replay_lsn,
    pg_wal_lsn_diff(sent_lsn, replay_lsn) AS replay_lag_bytes,
    pg_size_pretty(pg_wal_lsn_diff(sent_lsn, replay_lsn)) AS replay_lag
FROM pg_stat_replication;
```

```bash
#!/bin/bash
# ============================================================
# monitoring_check.sh - 定期健康检查脚本
# ============================================================

DB_HOST="localhost"
DB_USER="postgres"
DB_NAME="ecommerce"
ALERT_THRESHOLD_CONN_PCT=80
ALERT_THRESHOLD_LONG_TX_MIN=10
ALERT_THRESHOLD_REPL_LAG_MB=100

# 检查连接使用率
CONN_PCT=$(psql -h ${DB_HOST} -U ${DB_USER} -d ${DB_NAME} -t -c "
    SELECT round(count(*) * 100.0 /
        (SELECT setting::int FROM pg_settings WHERE name = 'max_connections'), 0)
    FROM pg_stat_activity WHERE backend_type = 'client backend'
")

if [ "${CONN_PCT}" -gt "${ALERT_THRESHOLD_CONN_PCT}" ]; then
    echo "[ALERT] 连接使用率过高: ${CONN_PCT}%"
fi

# 检查长事务
LONG_TX=$(psql -h ${DB_HOST} -U ${DB_USER} -d ${DB_NAME} -t -c "
    SELECT count(*) FROM pg_stat_activity
    WHERE state != 'idle'
      AND xact_start < NOW() - INTERVAL '${ALERT_THRESHOLD_LONG_TX_MIN} minutes'
")

if [ "${LONG_TX}" -gt 0 ]; then
    echo "[ALERT] 发现 ${LONG_TX} 个长事务（超过 ${ALERT_THRESHOLD_LONG_TX_MIN} 分钟）"
fi

# 检查复制延迟
REPL_LAG=$(psql -h ${DB_HOST} -U ${DB_USER} -d ${DB_NAME} -t -c "
    SELECT COALESCE(MAX(pg_wal_lsn_diff(sent_lsn, replay_lsn) / 1024 / 1024)::int, 0)
    FROM pg_stat_replication
")

if [ "${REPL_LAG}" -gt "${ALERT_THRESHOLD_REPL_LAG_MB}" ]; then
    echo "[ALERT] 复制延迟过大: ${REPL_LAG}MB"
fi

echo "$(date): 健康检查完成 - 连接:${CONN_PCT}% 长事务:${LONG_TX} 复制延迟:${REPL_LAG}MB"
```

-- ============================================================
-- 认证与权限安全加固
-- ============================================================

-- 1. 使用 SCRAM-SHA-256 认证（替代旧的 MD5）
-- pg_hba.conf:
-- host all all 0.0.0.0/0 scram-sha-256
-- postgresql.conf:
-- password_encryption = scram-sha-256

-- 2. 创建最小权限的应用账号
CREATE ROLE app_readonly WITH LOGIN PASSWORD 'strong_password_here';
GRANT CONNECT ON DATABASE ecommerce TO app_readonly;
GRANT USAGE ON SCHEMA public TO app_readonly;
GRANT SELECT ON ALL TABLES IN SCHEMA public TO app_readonly;
ALTER DEFAULT PRIVILEGES IN SCHEMA public GRANT SELECT ON TABLES TO app_readonly;

CREATE ROLE app_readwrite WITH LOGIN PASSWORD 'another_strong_password';
GRANT CONNECT ON DATABASE ecommerce TO app_readwrite;
GRANT USAGE ON SCHEMA public TO app_readwrite;
GRANT SELECT, INSERT, UPDATE, DELETE ON ALL TABLES IN SCHEMA public TO app_readwrite;
GRANT USAGE ON ALL SEQUENCES IN SCHEMA public TO app_readwrite;
ALTER DEFAULT PRIVILEGES IN SCHEMA public GRANT SELECT, INSERT, UPDATE, DELETE ON TABLES TO app_readwrite;
ALTER DEFAULT PRIVILEGES IN SCHEMA public GRANT USAGE ON SEQUENCES TO app_readwrite;

-- 3. 行级安全策略（多租户场景）
ALTER TABLE orders ENABLE ROW LEVEL SECURITY;

CREATE POLICY orders_user_policy ON orders
    FOR ALL
    TO app_readwrite
    USING (user_id = current_setting('app.current_user_id')::bigint);

-- 使用时设置当前用户
SET app.current_user_id = '12345';
SELECT * FROM orders;  -- 只能看到 user_id=12345 的订单

### 练习1：完整备份策略实施

在测试环境中实施完整的备份策略：配置每日逻辑备份、开启 WAL 归档、做一次物理基础备份。模拟数据误删场景，使用 PITR 恢复到误操作前1分钟。

### 练习2：监控仪表盘搭建

编写一组监控查询脚本，输出格式化的数据库健康报告，包含：连接状态、缓冲命中率、Top5 慢查询、表膨胀排名、复制延迟。

### 练习3：流复制搭建

在两台服务器（或两个 Docker 容器）上搭建一主一备的流复制架构。在主库写入数据，验证备库同步。模拟主库故障，执行故障切换。

### 练习4：安全加固实施

对项目数据库实施安全加固：创建最小权限应用账号、配置 SSL、启用行级安全策略、安装 pgAudit 审计扩展。验证各安全措施是否生效。

### 练习5：运维手册编写

## 常见问题

### Q1: 逻辑备份和物理备份应该如何搭配使用？

**A:** 推荐两者结合使用。物理备份（pg_basebackup + WAL 归档）作为主备份方案，支持 PITR 恢复到任意时间点，恢复速度快。逻辑备份（pg_dump）作为补充，用于：(1) 跨版本迁移；(2) 恢复单个表；(3) 备份验证（可以直接查看 SQL 内容）。典型策略：每天一次 pg_basebackup，持续 WAL 归档，每周一次 pg_dump 作为额外保障。

### Q2: 流复制的同步模式和异步模式怎么选？

**A:** 取决于业务对数据丢失的容忍度。异步复制（默认）性能更好，主库不需要等待备库确认，但主库故障时可能丢失最后几秒的数据。同步复制保证零数据丢失，但每次 COMMIT 都要等待备库确认写入，写入延迟会增加。折中方案是 `synchronous_commit = remote_write`，等待备库写入操作系统缓存（不等待落盘），兼顾安全性和性能。

### Q3: 如何监控 autovacuum 是否正常工作？

**A:** 关注以下指标：(1) `pg_stat_user_tables` 中的 `n_dead_tup` 持续增长说明 autovacuum 跟不上；(2) `last_autovacuum` 和 `last_autoanalyze` 如果长时间没有更新说明 autovacuum 没有执行；(3) 查看 `pg_stat_activity` 中是否有 autovacuum worker 在运行。如果大表的 autovacuum 效率不足，可以针对特定表调整参数：`ALTER TABLE orders SET (autovacuum_vacuum_scale_factor = 0.01)`。
