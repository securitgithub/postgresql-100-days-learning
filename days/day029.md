## 理论讲解

### 什么是 CTE

CTE（公用表表达式）通过 `WITH` 子句定义一个临时的命名结果集，可以在后续的查询中像表一样引用它。CTE 使复杂查询更加清晰、易于维护。

```sql
WITH cte_name AS (
    SELECT ...
)
SELECT * FROM cte_name;
```

### CTE 的优势

1. **提高可读性**：将复杂查询分解为逻辑清晰的步骤
2. **避免重复**：同一个 CTE 可以在查询中多次引用
3. **支持递归**：`WITH RECURSIVE` 可以实现递归查询
4. **可用于 DML**：CTE 可以在 INSERT、UPDATE、DELETE 中使用

### CTE 的物化行为

PostgreSQL 12+ 引入了 CTE 物化控制：

- `WITH cte AS MATERIALIZED (...)` — 强制物化（执行一次，结果缓存）
- `WITH cte AS NOT MATERIALIZED (...)` — 不物化（内联到主查询中优化）
- 默认行为：非递归 CTE 且只被引用一次时，PostgreSQL 可能自动内联

### 多 CTE 组合

可以在一个 WITH 子句中定义多个 CTE，后面的 CTE 可以引用前面的：

```sql
WITH
    cte1 AS (SELECT ...),
    cte2 AS (SELECT ... FROM cte1),
    cte3 AS (SELECT ... FROM cte1 JOIN cte2 ...)
SELECT * FROM cte3;
```

### WITH RECURSIVE 递归查询

递归 CTE 由两部分组成：

```sql
WITH RECURSIVE cte_name AS (
    -- 基础查询（锚点）：递归的起点
    SELECT ...
    UNION [ALL]
    -- 递归查询：引用自身
    SELECT ... FROM cte_name WHERE ...
)
SELECT * FROM cte_name;
```

执行过程：
1. 执行基础查询，得到初始结果集
2. 用上一轮的结果执行递归查询，得到新的行
3. 将新行添加到结果集中
4. 重复步骤 2-3，直到递归查询不再产生新行
5. 返回所有累积的结果

> **重要**：递归查询必须有终止条件（通常在 WHERE 子句中），否则会无限循环。PostgreSQL 有 `max_recursive_iterations` 参数作为安全保护。

## 代码示例

### 示例一：基本 CTE

```sql
-- 使用 CTE 使查询更清晰
WITH high_revenue AS (
    SELECT
        department,
        SUM(revenue) AS total_revenue
    FROM monthly_revenue
    GROUP BY department
    HAVING SUM(revenue) > 250000
)
SELECT
    department,
    total_revenue,
    ROUND(total_revenue / 6, 2) AS monthly_avg
FROM high_revenue
ORDER BY total_revenue DESC;
```

### 示例二：CTE 替代子查询

```sql
-- 不使用 CTE（嵌套子查询，难以阅读）
SELECT * FROM (
    SELECT employee, SUM(amount) AS total
    FROM sales GROUP BY employee
) t
WHERE total > (
    SELECT AVG(total) FROM (
        SELECT SUM(amount) AS total FROM sales GROUP BY employee
    ) t2
);

-- 使用 CTE（清晰的分步逻辑）
WITH employee_totals AS (
    SELECT employee, SUM(amount) AS total
    FROM sales
    GROUP BY employee
),
avg_total AS (
    SELECT AVG(total) AS avg_val FROM employee_totals
)
SELECT
    et.employee,
    et.total,
    at.avg_val AS company_avg
FROM employee_totals et
CROSS JOIN avg_total at
WHERE et.total > at.avg_val
ORDER BY et.total DESC;
```

### 示例三：多 CTE 组合 — 分步分析

```sql
-- 多步分析：部门汇总 → 排名 → 占比
WITH
dept_summary AS (
    -- 第一步：按部门汇总
    SELECT
        department,
        COUNT(*) AS sale_count,
        SUM(amount) AS total_amount,
        ROUND(AVG(amount), 2) AS avg_amount
    FROM sales
    GROUP BY department
),
ranked AS (
    -- 第二步：排名
    SELECT
        *,
        RANK() OVER (ORDER BY total_amount DESC) AS revenue_rank
    FROM dept_summary
),
with_pct AS (
    -- 第三步：计算占比
    SELECT
        *,
        ROUND(total_amount * 100.0 / SUM(total_amount) OVER (), 2) AS pct
    FROM ranked
)
SELECT * FROM with_pct ORDER BY revenue_rank;
```

### 示例四：CTE 用于 DML — INSERT

```sql
-- 使用 CTE 的结果插入另一张表
CREATE TABLE dept_report (
    department VARCHAR(50),
    total_sales NUMERIC(12,2),
    report_date DATE
);

WITH summary AS (
    SELECT
        department,
        SUM(amount) AS total_sales
    FROM sales
    GROUP BY department
)
INSERT INTO dept_report (department, total_sales, report_date)
SELECT department, total_sales, current_date
FROM summary;
```

### 示例五：WITH RECURSIVE — 数字序列

```sql
-- 生成 1 到 10 的序列（递归方式）
WITH RECURSIVE numbers AS (
    -- 锚点：起始值
    SELECT 1 AS n
    UNION ALL
    -- 递归：每次加 1
    SELECT n + 1 FROM numbers WHERE n < 10
)
SELECT * FROM numbers;
```

### 示例六：WITH RECURSIVE — 阶乘计算

```sql
-- 计算 1! 到 10! 的阶乘
WITH RECURSIVE factorial AS (
    SELECT 1 AS n, 1::BIGINT AS fact
    UNION ALL
    SELECT n + 1, fact * (n + 1) FROM factorial WHERE n < 10
)
SELECT n, fact FROM factorial;
```

### 示例七：WITH RECURSIVE — 树形结构

```sql
-- 创建组织架构表
CREATE TABLE org_tree (
    id          INTEGER PRIMARY KEY,
    name        VARCHAR(50),
    parent_id   INTEGER REFERENCES org_tree(id),
    title       VARCHAR(100)
);

INSERT INTO org_tree VALUES
    (1, '张总', NULL, 'CEO'),
    (2, '李VP', 1, '技术副总裁'),
    (3, '王VP', 1, '销售副总裁'),
    (4, '赵经理', 2, '开发经理'),
    (5, '孙经理', 2, 'QA经理'),
    (6, '周经理', 3, '华北销售经理'),
    (7, '吴经理', 3, '华东销售经理'),
    (8, '郑工', 4, '高级开发'),
    (9, '王工', 4, '开发工程师'),
    (10, '刘工', 5, '测试工程师');

-- 从 CEO 开始递归查询整棵组织树
WITH RECURSIVE org_hierarchy AS (
    -- 锚点：从根节点（CEO）开始
    SELECT
        id, name, title, parent_id,
        0 AS level,
        name::TEXT AS path
    FROM org_tree
    WHERE parent_id IS NULL

    UNION ALL

    -- 递归：查找子节点
    SELECT
        ot.id, ot.name, ot.title, ot.parent_id,
        oh.level + 1,
        oh.path || ' > ' || ot.name
    FROM org_tree ot
    JOIN org_hierarchy oh ON ot.parent_id = oh.id
)
SELECT
    -- 缩进显示层级
    repeat('  ', level) || name AS org_chart,
    title,
    level,
    path
FROM org_hierarchy
ORDER BY path;
```

### 示例八：递归 — 查找某个节点的所有上级

```sql
-- 查找"郑工"的所有上级（向上递归）
WITH RECURSIVE ancestors AS (
    -- 锚点：从目标节点开始
    SELECT id, name, title, parent_id, 0 AS distance
    FROM org_tree
    WHERE name = '郑工'

    UNION ALL

    -- 递归：查找父节点
    SELECT ot.id, ot.name, ot.title, ot.parent_id, a.distance + 1
    FROM org_tree ot
    JOIN ancestors a ON ot.id = a.parent_id
)
SELECT name, title, distance FROM ancestors ORDER BY distance;
-- 结果：郑工(0) → 赵经理(1) → 李VP(2) → 张总(3)
```

### 示例九：递归 — 查找某个节点的所有下属

```sql
-- 查找"李VP"的所有下属
WITH RECURSIVE subordinates AS (
    SELECT id, name, title, parent_id
    FROM org_tree
    WHERE name = '李VP'

    UNION ALL

    SELECT ot.id, ot.name, ot.title, ot.parent_id
    FROM org_tree ot
    JOIN subordinates s ON ot.parent_id = s.id
)
SELECT * FROM subordinates;
-- 结果：李VP, 赵经理, 孙经理, 郑工, 王工, 刘工
```

### 示例十：递归 — 日期序列生成

```sql
-- 用递归 CTE 生成日期序列（替代 generate_series）
WITH RECURSIVE date_range AS (
    SELECT '2025-01-01'::DATE AS d
    UNION ALL
    SELECT d + 1 FROM date_range WHERE d < '2025-01-31'
)
SELECT
    d AS date_val,
    to_char(d, 'Day') AS day_name,
    extract(DOW FROM d) AS dow
FROM date_range;
```

## 常见问题

### Q1：CTE 和子查询哪个性能更好？
**A**：在 PostgreSQL 12+ 中，非递归 CTE 在只被引用一次时默认会被内联（等同于子查询），性能相同。如果 CTE 被引用多次，物化（MATERIALIZED）可以避免重复计算。可以使用 `EXPLAIN` 查看实际执行计划。

### Q2：递归 CTE 会不会死循环？
**A**：如果递归查询没有正确的终止条件，确实会死循环。PostgreSQL 默认没有递归深度限制，但可以通过 `SET max_recursive_iterations = 1000;`（PostgreSQL 14+）设置上限。在编写递归 CTE 时，务必确保 WHERE 条件能终止递归。

### Q3：UNION 和 UNION ALL 在递归 CTE 中有什么区别？
**A**：`UNION` 会在每次递归迭代时去重，可以防止环形数据导致的死循环，但性能较低。`UNION ALL` 不去重，性能更好，但需要确保数据不存在环或者手动维护访问路径来避免重复访问。大多数树形结构使用 `UNION ALL` 即可。
