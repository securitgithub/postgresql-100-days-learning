## 理论讲解

### 1. ORDER BY 基础

在 SQL 中，查询返回的行默认是**无序的**。如果需要对结果集进行排序，必须使用 `ORDER BY` 子句。

```
SELECT 列名 FROM 表名 ORDER BY 排序列 [ASC | DESC];
```

- `ASC`：升序排列（默认值，可以省略）
- `DESC`：降序排列

PostgreSQL 中 `ORDER BY` 位于 `SELECT` 语句的末尾，在 `WHERE`、`GROUP BY`、`HAVING` 之后执行。

### 2. 单列排序

对查询结果按照某一列进行排序。排序列可以是表中的列名，也可以是列的别名或列序号（从 1 开始）。

```sql
-- 按列名排序
SELECT * FROM products ORDER BY price ASC;

-- 按列序号排序（第 3 列）
SELECT id, name, price FROM products ORDER BY 3 DESC;

-- 按别名排序
SELECT name, price * 0.9 AS discounted FROM products ORDER BY discounted;
```

### 3. 多列排序

当需要在第一列值相同的情况下再按另一列排序时，可以指定多个排序列：

```sql
SELECT * FROM employees
ORDER BY department ASC, salary DESC;
```

以上语句先按 `department` 升序排列，同一部门内再按 `salary` 降序排列。每个列可以单独指定排序方向。

### 4. NULLS FIRST 与 NULLS LAST

PostgreSQL 中 NULL 值在排序时的默认行为：
- **升序（ASC）**：NULL 排在**最后**（NULLS LAST）
- **降序（DESC）**：NULL 排在**最前**（NULLS FIRST）

可以通过显式声明来覆盖默认行为：

```sql
-- 升序排列，但 NULL 排在最前
SELECT * FROM products ORDER BY price ASC NULLS FIRST;

-- 降序排列，但 NULL 排在最后
SELECT * FROM products ORDER BY price DESC NULLS LAST;
```

### 5. LIMIT 与 OFFSET

`LIMIT` 限制返回的行数，`OFFSET` 跳过前 N 行。这两个子句常用于实现分页。

```sql
SELECT * FROM products ORDER BY id LIMIT 10 OFFSET 20;
```

- `LIMIT 10`：最多返回 10 行
- `OFFSET 20`：跳过前 20 行（即从第 21 行开始）

**分页公式**：第 N 页（每页 page_size 条）:
```
OFFSET = (N - 1) * page_size
LIMIT  = page_size
```

### 6. FETCH FIRST（SQL 标准语法）

`FETCH FIRST` 是 SQL:2008 标准语法，功能等同于 `LIMIT`：

```sql
-- 等价于 LIMIT 10
SELECT * FROM products ORDER BY id
FETCH FIRST 10 ROWS ONLY;

-- 等价于 LIMIT 10 OFFSET 20
SELECT * FROM products ORDER BY id
OFFSET 20 ROWS
FETCH FIRST 10 ROWS ONLY;
```

`FETCH FIRST` 可读性更强，且在 PostgreSQL、Oracle、DB2 等数据库中通用。

### 7. 性能注意事项

- **大 OFFSET 的问题**：`OFFSET 100000` 意味着数据库需要扫描并丢弃前 100000 行，效率极低。
- **游标分页（Keyset Pagination）**：用 `WHERE id > last_seen_id ORDER BY id LIMIT 10` 代替大 OFFSET，性能稳定。
- **索引加速排序**：如果 `ORDER BY` 的列上有索引，PostgreSQL 可以直接利用索引的有序性避免额外排序。
- **排序内存**：大量数据排序可能使用磁盘临时文件，可通过 `work_mem` 参数调整。

```sql
('笔记本支架', '配件', NULL, 90),

```

### 示例 1：单列排序

```sql
-- 按价格升序排列
SELECT name, price FROM products ORDER BY price;

-- 按价格降序排列
SELECT name, price FROM products ORDER BY price DESC;
```

### 示例 2：多列排序

```sql
-- 先按类别升序，同类别内按价格降序
SELECT name, category, price
FROM products
ORDER BY category ASC, price DESC;
```

### 示例 3：NULL 排序控制

```sql
-- 升序排列，NULL 价格的商品排在最前面
SELECT name, price
FROM products
ORDER BY price ASC NULLS FIRST;

-- 降序排列，NULL 价格的商品排在最后面
SELECT name, price
FROM products
ORDER BY price DESC NULLS LAST;
```

### 示例 4：LIMIT / OFFSET 分页

```sql
-- 第 1 页（每页 3 条）
SELECT id, name, price FROM products ORDER BY id LIMIT 3 OFFSET 0;

-- 第 2 页
SELECT id, name, price FROM products ORDER BY id LIMIT 3 OFFSET 3;

-- 第 3 页
SELECT id, name, price FROM products ORDER BY id LIMIT 3 OFFSET 6;
```

### 示例 5：FETCH FIRST 语法

```sql
-- 获取价格最高的前 3 件商品
SELECT name, price FROM products
ORDER BY price DESC NULLS LAST
FETCH FIRST 3 ROWS ONLY;
```

### 示例 6：游标分页（Keyset Pagination）

```sql
-- 第一次查询
SELECT id, name, price FROM products
ORDER BY id
LIMIT 3;

-- 假设上一页最后一条记录的 id = 3，获取下一页
SELECT id, name, price FROM products
WHERE id > 3
ORDER BY id
LIMIT 3;
```

## 常见问题

### Q1: 不使用 ORDER BY 时，结果的顺序是固定的吗？

**A**：不是。如果没有 `ORDER BY`，PostgreSQL 不保证任何特定的返回顺序。即使数据看上去按插入顺序排列，这也只是巧合。在表被更新、vacuum 或并发访问后，顺序可能随时改变。**如果需要确定的顺序，必须使用 ORDER BY。**

### Q2: LIMIT/OFFSET 在数据量大时为什么会慢？

**A**：`OFFSET N` 本质上需要先获取前 N 行再丢弃，数据库仍然做了全量扫描和排序。当 N 很大时（如 `OFFSET 100000`），即使只要 10 行也需要处理 100010 行。推荐使用游标分页（Keyset Pagination），通过 `WHERE id > last_id` 方式实现，配合索引性能远优于大 OFFSET。

### Q3: ORDER BY 可以使用不在 SELECT 列表中的列吗？

**A**：可以。PostgreSQL 允许按不在 `SELECT` 列表中的列排序。但如果查询使用了 `DISTINCT`，则 `ORDER BY` 中的列必须出现在 `SELECT` 列表中。
