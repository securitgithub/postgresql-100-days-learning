## 理论讲解

### 1. 为什么需要分布式 PostgreSQL

当单机 PostgreSQL 达到瓶颈时（存储上限、写入瓶颈、查询性能不足），有两种扩展方向：

- **垂直扩展（Scale Up）**：升级硬件，但有上限且成本指数增长
- **水平扩展（Scale Out）**：将数据分散到多台机器，理论上可无限扩展

Citus 是 PostgreSQL 的分布式扩展，将标准 PostgreSQL 转变为分布式数据库，同时保持 SQL 兼容性。2022 年 Citus 已完全开源（原为商业产品）。

### 2. Citus 架构

```
┌──────────────────────────────────────────────────────┐
│                    客户端应用                          │
└──────────────────────┬───────────────────────────────┘
                       │
              ┌────────▼────────┐
              │   协调节点        │
              │ (Coordinator)    │
              │                  │
              │  - 接收 SQL      │
              │  - 查询规划      │
              │  - 分发子查询    │
              │  - 合并结果      │
              └──┬─────┬─────┬──┘
                 │     │     │
         ┌───────▼┐ ┌──▼───┐ ┌▼───────┐
         │Worker 1│ │Worker2│ │Worker 3│
         │        │ │       │ │        │
         │ Shard  │ │ Shard │ │ Shard  │
         │ 1,4,7  │ │ 2,5,8 │ │ 3,6,9  │
         └────────┘ └───────┘ └────────┘
```

- **协调节点（Coordinator）**：接收客户端查询，制定分布式执行计划，分发子查询到工作节点，汇总结果返回
- **工作节点（Worker）**：存储实际数据分片（Shard），执行协调节点下发的子查询
- **分片（Shard）**：表的水平分区，每个分片是工作节点上的一张普通 PostgreSQL 表

### 3. 三种表类型

#### 分布式表（Distributed Table）
- 数据按分片键（Distribution Column）哈希分散到多个工作节点
- 适用于大表（事实表、日志表、事件表）
- 查询效率取决于是否能按分片键路由

#### 参考表（Reference Table）
- 数据完整复制到所有工作节点
- 适用于小的维度表（如城市表、类别表、配置表）
- 可以与分布式表高效 JOIN

#### 本地表（Local Table）
- 只存在于协调节点上的普通表
- 适用于不需要分布式处理的小表
- 不能直接与分布式表 JOIN（需要特殊处理）

### 4. 分片键的选择

分片键的选择直接影响查询性能和数据分布。选择原则：

| 原则 | 说明 |
|------|------|
| **高基数** | 分片键应有足够多的唯一值，确保数据均匀分布 |
| **查询频率** | 最常出现在 WHERE、JOIN 条件中的列 |
| **避免热点** | 不要选择导致大量数据集中在少数分片的列 |
| **关联查询** | 需要 JOIN 的表应使用相同的分片键（Co-location） |

常见分片键选择：
- **多租户应用**：`tenant_id`（租户ID）— 每个租户的数据在同一分片，查询效率高
- **实时分析**：`user_id` 或 `device_id` — 按用户/设备维度分析
- **时序数据**：`sensor_id` + 时间分区的组合

### 5. Co-location（共置）

当两张分布式表使用相同的分片键和分片数量时，相同分片键值的数据会存储在同一个工作节点上。这使得这些表之间的 JOIN 可以在本地完成，无需跨节点数据传输：

```
Worker 1:
  orders (tenant_id=1)   ←── 本地 JOIN ──▶  order_items (tenant_id=1)

Worker 2:
  orders (tenant_id=2)   ←── 本地 JOIN ──▶  order_items (tenant_id=2)
```

### 示例1：安装和初始化 Citus

```bash
# ===== 在所有节点上安装 Citus =====

# Ubuntu/Debian
curl https://install.citusdata.com/community/deb.sh | sudo bash
sudo apt-get install postgresql-16-citus-12.1

# CentOS/RHEL
curl https://install.citusdata.com/community/rpm.sh | sudo bash
sudo yum install citus121_16

# 配置 PostgreSQL 加载 Citus 扩展
echo "shared_preload_libraries = 'citus'" | sudo tee -a /etc/postgresql/16/main/postgresql.conf

# 重启 PostgreSQL
sudo systemctl restart postgresql
```

```sql
-- 在协调节点上创建 Citus 扩展
CREATE EXTENSION citus;

-- 注册工作节点
SELECT citus_set_coordinator_host('192.168.1.100', 5432);
SELECT citus_add_node('192.168.1.101', 5432);
SELECT citus_add_node('192.168.1.102', 5432);
SELECT citus_add_node('192.168.1.103', 5432);

-- 验证集群状态
SELECT * FROM citus_get_active_worker_nodes();

-- 查看节点信息
SELECT nodeid, nodename, nodeport, isactive
FROM pg_dist_node;
```

### 示例2：创建分布式表

```sql
-- 创建多租户 SaaS 应用的表结构

-- 租户表（参考表 — 复制到所有节点）
CREATE TABLE tenants (
    id SERIAL PRIMARY KEY,
    name VARCHAR(100) NOT NULL,
    plan VARCHAR(20) DEFAULT 'free',
    created_at TIMESTAMP DEFAULT now()
);

-- 将 tenants 设为参考表
SELECT create_reference_table('tenants');

-- 用户表（分布式表 — 按 tenant_id 分片）
CREATE TABLE users (
    id BIGSERIAL,
    tenant_id INT NOT NULL REFERENCES tenants(id),
    username VARCHAR(50) NOT NULL,
    email VARCHAR(100),
    created_at TIMESTAMP DEFAULT now(),
    PRIMARY KEY (tenant_id, id)
);

-- 将 users 设为分布式表，以 tenant_id 为分片键
SELECT create_distributed_table('users', 'tenant_id');

-- 订单表（分布式表 — 与 users 共置）
CREATE TABLE orders (
    id BIGSERIAL,
    tenant_id INT NOT NULL,
    user_id BIGINT NOT NULL,
    amount NUMERIC(10,2),
    status VARCHAR(20) DEFAULT 'pending',
    created_at TIMESTAMP DEFAULT now(),
    PRIMARY KEY (tenant_id, id)
);

SELECT create_distributed_table('orders', 'tenant_id');

-- 订单明细表（分布式表 — 与 orders 共置）
CREATE TABLE order_items (
    id BIGSERIAL,
    tenant_id INT NOT NULL,
    order_id BIGINT NOT NULL,
    product_name VARCHAR(100),
    quantity INT,
    price NUMERIC(10,2),
    PRIMARY KEY (tenant_id, id)
);

SELECT create_distributed_table('order_items', 'tenant_id');
```

### 示例3：查看分片信息

```sql
-- 查看分布式表信息
SELECT
    logicalrelid AS table_name,
    partmethod,           -- h=hash, a=append, n=none
    colocationid,         -- 共置组 ID
    repmodel              -- s=statement, t=streaming
FROM pg_dist_partition;

-- 查看分片分布情况
SELECT
    logicalrelid::text AS table_name,
    nodename,
    count(*) AS shard_count
FROM pg_dist_shard_placement
JOIN pg_dist_shard USING (shardid)
GROUP BY logicalrelid, nodename
ORDER BY table_name, nodename;

-- 查看特定表的分片详情
SELECT
    shardid,
    nodename,
    nodeport,
    shardminvalue,
    shardmaxvalue
FROM pg_dist_shard
JOIN pg_dist_shard_placement USING (shardid)
WHERE logicalrelid = 'orders'::regclass
ORDER BY shardminvalue::int;

-- 查看共置组信息
SELECT * FROM pg_dist_colocation;

-- 查看分片大小
SELECT
    logicalrelid::text AS table_name,
    pg_size_pretty(citus_total_relation_size(logicalrelid)) AS total_size
FROM pg_dist_partition
ORDER BY citus_total_relation_size(logicalrelid) DESC;
```

### 示例4：分布式查询示例

```sql
-- 插入测试数据
INSERT INTO tenants (name, plan) VALUES
    ('公司A', 'enterprise'),
    ('公司B', 'pro'),
    ('公司C', 'free');

INSERT INTO users (tenant_id, username, email) VALUES
    (1, 'alice', 'alice@companya.com'),
    (1, 'bob', 'bob@companya.com'),
    (2, 'charlie', 'charlie@companyb.com'),
    (3, 'dave', 'dave@companyc.com');

INSERT INTO orders (tenant_id, user_id, amount, status) VALUES
    (1, 1, 199.99, 'completed'),
    (1, 1, 89.50, 'completed'),
    (1, 2, 350.00, 'pending'),
    (2, 3, 1200.00, 'completed'),
    (3, 4, 25.00, 'cancelled');

-- 查询1：按租户查询（最高效 — 路由到单个分片）
-- Citus 检测到 WHERE 包含分片键，只查询对应的单个工作节点
SELECT u.username, o.amount, o.status
FROM users u JOIN orders o ON u.tenant_id = o.tenant_id AND u.id = o.user_id
WHERE u.tenant_id = 1;

-- 查询2：跨租户聚合（需要查询所有分片，但在各节点上并行执行）
SELECT
    t.name AS tenant_name,
    COUNT(o.id) AS order_count,
    SUM(o.amount) AS total_amount,
    AVG(o.amount) AS avg_amount
FROM tenants t
LEFT JOIN orders o ON t.id = o.tenant_id
GROUP BY t.id, t.name
ORDER BY total_amount DESC;

-- 查询3：使用 EXPLAIN 查看分布式查询计划
EXPLAIN (VERBOSE, COSTS OFF)
SELECT * FROM orders WHERE tenant_id = 1;

-- 查询4：分布式子查询
SELECT tenant_id, total_orders, total_amount
FROM (
    SELECT
        tenant_id,
        COUNT(*) AS total_orders,
        SUM(amount) AS total_amount
    FROM orders
    GROUP BY tenant_id
) sub
WHERE total_amount > 100
ORDER BY total_amount DESC;
```

### 示例5：分布式 DDL 和维护操作

```sql
-- 分布式 DDL 会自动在所有分片上执行

-- 添加列
ALTER TABLE orders ADD COLUMN notes TEXT;

-- 创建索引（在所有分片上并行创建）
CREATE INDEX idx_orders_status ON orders (tenant_id, status);
CREATE INDEX idx_orders_created ON orders (tenant_id, created_at);

-- 添加约束
ALTER TABLE orders ADD CONSTRAINT orders_amount_positive CHECK (amount >= 0);

-- 分布式 VACUUM
-- Citus 会自动将 VACUUM 分发到所有工作节点
VACUUM ANALYZE orders;

-- 重新平衡分片（添加新节点后）
SELECT citus_rebalance_start();

-- 查看重新平衡进度
SELECT * FROM citus_rebalance_status();
```

### 示例6：使用 Docker Compose 快速搭建 Citus 集群

```yaml
# docker-compose.yml — Citus 集群
version: '3.8'

services:
  coordinator:
    image: citusdata/citus:12.1
    container_name: citus-coordinator
    ports:
      - "5432:5432"
    environment:
      POSTGRES_PASSWORD: CitusP@ss
      POSTGRES_USER: citus
      POSTGRES_DB: myapp
    volumes:
      - coordinator_data:/var/lib/postgresql/data
    labels:
      - "com.citusdata.role=Coordinator"

  worker1:
    image: citusdata/citus:12.1
    container_name: citus-worker1
    environment:
      POSTGRES_PASSWORD: CitusP@ss
      POSTGRES_USER: citus
    volumes:
      - worker1_data:/var/lib/postgresql/data
    labels:
      - "com.citusdata.role=Worker"

  worker2:
    image: citusdata/citus:12.1
    container_name: citus-worker2
    environment:
      POSTGRES_PASSWORD: CitusP@ss
      POSTGRES_USER: citus
    volumes:
      - worker2_data:/var/lib/postgresql/data
    labels:
      - "com.citusdata.role=Worker"

  manager:
    image: citusdata/membership-manager:0.3.0
    container_name: citus-manager
    volumes:
      - /var/run/docker.sock:/var/run/docker.sock
    depends_on:
      - coordinator
      - worker1
      - worker2
    environment:
      CITUS_HOST: coordinator

volumes:
  coordinator_data:
  worker1_data:
  worker2_data:
```

```bash
# 启动集群
docker compose up -d

# 等待所有节点就绪后验证
docker exec citus-coordinator psql -U citus -d myapp -c "SELECT * FROM citus_get_active_worker_nodes();"
```

### 示例7：性能优化与最佳实践

```sql
-- 1. 查看分布式查询的执行计划
EXPLAIN (ANALYZE, VERBOSE)
SELECT tenant_id, COUNT(*), SUM(amount)
FROM orders
WHERE created_at > '2024-01-01'
GROUP BY tenant_id;

-- 2. 查看每个分片的大小分布（检查是否均匀）
SELECT
    shardid,
    nodename,
    pg_size_pretty(shard_size) AS size
FROM (
    SELECT
        shardid,
        nodename,
        citus_shard_sizes.shard_size
    FROM pg_dist_shard
    JOIN pg_dist_shard_placement USING (shardid)
    JOIN (
        SELECT shardid, result::bigint AS shard_size
        FROM run_command_on_shards('orders', 'SELECT pg_total_relation_size(''%s'')')
    ) citus_shard_sizes USING (shardid)
    WHERE logicalrelid = 'orders'::regclass
) sub
ORDER BY shard_size DESC;

-- 3. 设置分片数量（创建表前设置，默认为32）
SET citus.shard_count = 64;

-- 4. 启用分片级别的并行查询
SET citus.max_adaptive_executor_pool_size = 4;

-- 5. 查看当前连接使用的分片路由信息
SET citus.explain_all_tasks = on;
EXPLAIN SELECT * FROM orders WHERE tenant_id = 1;
```

## 常见问题

### Q1: Citus 支持所有 PostgreSQL 功能吗？

**A:** 大部分支持，但有一些限制：(1) 跨分片的外键约束不支持（同一分片键的外键支持）；(2) 跨分片的唯一约束必须包含分片键；(3) 某些窗口函数在跨分片场景下可能受限；(4) CTE（WITH 查询）在某些场景下可能需要 push-down 优化。(5) 全局的序列号需要使用 bigserial 并依赖 PostgreSQL 的序列机制。建议在迁移前做完整的 SQL 兼容性测试。

### Q2: 分片键选错了怎么办？

**A:** 更改分片键需要重新分布数据，这是一个比较重的操作。步骤：(1) 创建新的分布式表使用新的分片键；(2) 将数据从旧表迁移到新表；(3) 重命名表。Citus 提供了 `alter_distributed_table()` 函数（Citus 11+）可以更方便地变更分片键，但对大表仍然需要较长时间。因此，在设计阶段仔细选择分片键至关重要。

### Q3: Citus 和原生 PostgreSQL 分区表有什么区别？

**A:** PostgreSQL 内建分区（Partitioning）是将数据分散在同一台服务器的不同表中，主要解决大表的管理和查询优化问题。Citus 是将数据分散在多台服务器上，解决单机存储和计算能力的限制。两者可以组合使用：先用 Citus 做分布式分片，每个分片内部再使用时间分区。
