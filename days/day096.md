## 理论讲解

### 1. 项目选择

综合项目应覆盖 PostgreSQL 的核心特性。以下两个方向均适合：

**方向A：电商平台**
- 用户管理、商品管理、购物车
- 订单管理、支付、库存
- 搜索、推荐、评价
- 高并发、事务一致性要求高

**方向B：SaaS 多租户平台**
- 租户隔离、权限管理
- 订阅计费、用量统计
- 数据导入导出
- 行级安全策略、Schema 隔离

本项目选择**电商平台**作为综合案例，因为它覆盖了：复杂数据模型、高并发事务、全文搜索、JSON 数据、分区表、各类索引优化等核心知识点。

### 2. 需求分析方法

```
业务需求 → 功能清单 → 数据实体识别 → ER 图 → 物理设计
```

需求分析的关键步骤：

1. **识别核心业务流程**：注册→浏览→加购→下单→支付→发货→收货→评价
2. **列出功能模块**：用户、商品、订单、支付、物流、评价、运营
3. **确定数据实体**：哪些"东西"需要在数据库中存储
4. **定义实体关系**：实体之间的关联和基数（1:1、1:N、M:N）
5. **识别关键查询**：系统最频繁、最重要的查询模式

### 3. ER 图设计原则

- **实体（Entity）**：具有独立意义的数据对象（如用户、商品、订单）
- **属性（Attribute）**：实体的特征（如用户名、价格）
- **关系（Relationship）**：实体间的关联及基数

设计三大范式：
- **1NF**：每列原子性，不可再分
- **2NF**：非键属性完全依赖于主键
- **3NF**：非键属性不传递依赖于主键

实际项目中适度反范式化可以提升查询性能。

### 4. 物理设计考虑因素

| 考虑因素 | 决策内容 |
|---------|---------|
| 数据类型选择 | 合适的类型（如用 NUMERIC 而非 FLOAT 存储金额） |
| 主键策略 | 自增 BIGSERIAL vs UUID |
| 索引设计 | 哪些列需要索引，用什么索引类型 |
| 分区策略 | 大表是否需要分区，按什么维度 |
| 约束 | NOT NULL、UNIQUE、CHECK、外键 |
| 默认值 | 时间戳、状态字段默认值 |
| JSON 字段 | 哪些数据适合用 JSONB 存储 |

### 5. 命名规范

建议采用以下命名规范保持一致性：

- 表名：小写复数形式（`users`, `orders`, `order_items`）
- 列名：小写下划线分隔（`created_at`, `total_amount`）
- 主键：`id`（BIGSERIAL）
- 外键：`关联表单数_id`（`user_id`, `product_id`）
- 索引：`idx_表名_列名`（`idx_orders_user_id`）
- 唯一约束：`uk_表名_列名`
- 检查约束：`chk_表名_描述`

### 示例3：物理表设计 —— 用户与地址

```sql
-- ============================================================
-- 用户表
-- ============================================================
CREATE TABLE users (
    id              BIGSERIAL PRIMARY KEY,
    username        VARCHAR(50) UNIQUE NOT NULL,
    email           VARCHAR(200) UNIQUE NOT NULL,
    phone           VARCHAR(20) UNIQUE,
    password_hash   VARCHAR(255) NOT NULL,     -- bcrypt 哈希
    nickname        VARCHAR(100),
    avatar_url      TEXT,
    status          SMALLINT NOT NULL DEFAULT 1,  -- 1:正常 0:禁用 -1:注销
    email_verified  BOOLEAN NOT NULL DEFAULT FALSE,
    last_login_at   TIMESTAMPTZ,
    created_at      TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    updated_at      TIMESTAMPTZ NOT NULL DEFAULT NOW(),

    CONSTRAINT chk_users_status CHECK (status IN (-1, 0, 1)),
    CONSTRAINT chk_users_email CHECK (email ~* '^[^@]+@[^@]+\.[^@]+$')
);

-- 索引
CREATE INDEX idx_users_email ON users (email);
CREATE INDEX idx_users_phone ON users (phone) WHERE phone IS NOT NULL;
CREATE INDEX idx_users_status ON users (status) WHERE status != 1;  -- 部分索引
CREATE INDEX idx_users_created_at ON users (created_at);

-- 自动更新 updated_at 的触发器
CREATE OR REPLACE FUNCTION update_updated_at()
RETURNS TRIGGER AS $$
BEGIN
    NEW.updated_at = NOW();
    RETURN NEW;
END;
$$ LANGUAGE plpgsql;

CREATE TRIGGER trg_users_updated_at
    BEFORE UPDATE ON users
    FOR EACH ROW EXECUTE FUNCTION update_updated_at();

-- ============================================================
-- 用户收货地址表
-- ============================================================
CREATE TABLE user_addresses (
    id          BIGSERIAL PRIMARY KEY,
    user_id     BIGINT NOT NULL REFERENCES users(id) ON DELETE CASCADE,
    label       VARCHAR(20) DEFAULT '家',    -- 标签：家、公司等
    receiver    VARCHAR(50) NOT NULL,         -- 收件人
    phone       VARCHAR(20) NOT NULL,
    province    VARCHAR(50) NOT NULL,
    city        VARCHAR(50) NOT NULL,
    district    VARCHAR(50) NOT NULL,
    address     TEXT NOT NULL,                -- 详细地址
    postal_code VARCHAR(10),
    is_default  BOOLEAN NOT NULL DEFAULT FALSE,
    created_at  TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    updated_at  TIMESTAMPTZ NOT NULL DEFAULT NOW()
);

CREATE INDEX idx_user_addresses_user_id ON user_addresses (user_id);
CREATE TRIGGER trg_user_addresses_updated_at
    BEFORE UPDATE ON user_addresses
    FOR EACH ROW EXECUTE FUNCTION update_updated_at();
```

### 示例4：物理表设计 —— 商品与 SKU

```sql
-- ============================================================
-- 商品分类表（支持多级分类）
-- ============================================================
CREATE TABLE categories (
    id          SERIAL PRIMARY KEY,
    parent_id   INTEGER REFERENCES categories(id),
    name        VARCHAR(100) NOT NULL,
    slug        VARCHAR(100) UNIQUE NOT NULL,   -- URL 友好名称
    level       SMALLINT NOT NULL DEFAULT 1,     -- 层级：1/2/3
    sort_order  INTEGER NOT NULL DEFAULT 0,
    is_active   BOOLEAN NOT NULL DEFAULT TRUE,
    created_at  TIMESTAMPTZ NOT NULL DEFAULT NOW()
);

CREATE INDEX idx_categories_parent ON categories (parent_id);

-- ============================================================
-- 商品表
-- ============================================================
CREATE TABLE products (
    id              BIGSERIAL PRIMARY KEY,
    category_id     INTEGER NOT NULL REFERENCES categories(id),
    name            VARCHAR(200) NOT NULL,
    slug            VARCHAR(200) UNIQUE NOT NULL,
    description     TEXT,
    detail_html     TEXT,                       -- 商品详情（富文本）
    brand           VARCHAR(100),
    tags            TEXT[],                     -- 标签数组
    attributes      JSONB DEFAULT '{}',         -- 扩展属性（颜色/材质等）
    base_price      NUMERIC(12, 2) NOT NULL,    -- 基础价格
    status          SMALLINT NOT NULL DEFAULT 0, -- 0:草稿 1:上架 2:下架
    total_sales     INTEGER NOT NULL DEFAULT 0,
    review_count    INTEGER NOT NULL DEFAULT 0,
    avg_rating      NUMERIC(3, 2) DEFAULT 0,
    search_vector   TSVECTOR,                   -- 全文搜索向量
    created_at      TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    updated_at      TIMESTAMPTZ NOT NULL DEFAULT NOW(),

    CONSTRAINT chk_products_price CHECK (base_price >= 0),
    CONSTRAINT chk_products_status CHECK (status IN (0, 1, 2))
);

-- 索引策略
CREATE INDEX idx_products_category ON products (category_id);
CREATE INDEX idx_products_status ON products (status);
CREATE INDEX idx_products_tags ON products USING GIN (tags);          -- 数组索引
CREATE INDEX idx_products_attrs ON products USING GIN (attributes);   -- JSONB 索引
CREATE INDEX idx_products_search ON products USING GIN (search_vector); -- 全文搜索索引
CREATE INDEX idx_products_sales ON products (total_sales DESC) WHERE status = 1; -- 热门排序

-- 自动更新全文搜索向量
CREATE OR REPLACE FUNCTION update_product_search_vector()
RETURNS TRIGGER AS $$
BEGIN
    NEW.search_vector :=
        setweight(to_tsvector('simple', COALESCE(NEW.name, '')), 'A') ||
        setweight(to_tsvector('simple', COALESCE(NEW.brand, '')), 'B') ||
        setweight(to_tsvector('simple', COALESCE(NEW.description, '')), 'C');
    RETURN NEW;
END;
$$ LANGUAGE plpgsql;

CREATE TRIGGER trg_products_search_vector
    BEFORE INSERT OR UPDATE OF name, brand, description ON products
    FOR EACH ROW EXECUTE FUNCTION update_product_search_vector();

CREATE TRIGGER trg_products_updated_at
    BEFORE UPDATE ON products
    FOR EACH ROW EXECUTE FUNCTION update_updated_at();

-- ============================================================
-- 商品 SKU 表
-- ============================================================
CREATE TABLE product_skus (
    id          BIGSERIAL PRIMARY KEY,
    product_id  BIGINT NOT NULL REFERENCES products(id) ON DELETE CASCADE,
    sku_code    VARCHAR(50) UNIQUE NOT NULL,    -- SKU 编码
    attributes  JSONB NOT NULL DEFAULT '{}',    -- SKU 属性（如 {"颜色":"红","尺码":"L"}）
    price       NUMERIC(12, 2) NOT NULL,
    stock       INTEGER NOT NULL DEFAULT 0,
    reserved    INTEGER NOT NULL DEFAULT 0,     -- 已预留库存（已下单未支付）
    image_url   TEXT,
    is_active   BOOLEAN NOT NULL DEFAULT TRUE,

    CONSTRAINT chk_skus_price CHECK (price >= 0),
    CONSTRAINT chk_skus_stock CHECK (stock >= 0),
    CONSTRAINT chk_skus_reserved CHECK (reserved >= 0 AND reserved <= stock)
);

CREATE INDEX idx_skus_product ON product_skus (product_id);
CREATE INDEX idx_skus_code ON product_skus (sku_code);
```

### 示例5：物理表设计 —— 订单（分区表）

```sql
-- ============================================================
-- 订单表（按月范围分区）
-- ============================================================
CREATE TABLE orders (
    id              BIGSERIAL,
    order_no        VARCHAR(30) NOT NULL,        -- 订单编号
    user_id         BIGINT NOT NULL,
    address_snapshot JSONB NOT NULL,              -- 下单时地址快照
    total_amount    NUMERIC(12, 2) NOT NULL,
    discount_amount NUMERIC(12, 2) DEFAULT 0,
    pay_amount      NUMERIC(12, 2) NOT NULL,     -- 实付金额
    status          SMALLINT NOT NULL DEFAULT 0,
    -- 状态: 0待支付 1已支付 2已发货 3已收货 4已完成 5已取消 6已退款
    payment_method  VARCHAR(20),
    paid_at         TIMESTAMPTZ,
    shipped_at      TIMESTAMPTZ,
    completed_at    TIMESTAMPTZ,
    remark          TEXT,
    created_at      TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    updated_at      TIMESTAMPTZ NOT NULL DEFAULT NOW(),

    PRIMARY KEY (id, created_at),               -- 分区表主键必须包含分区键
    CONSTRAINT chk_orders_amounts CHECK (
        total_amount >= 0 AND pay_amount >= 0 AND discount_amount >= 0
    )
) PARTITION BY RANGE (created_at);

-- 创建分区（2025年每月一个分区）
DO $$
DECLARE
    start_date DATE := '2025-01-01';
    end_date DATE;
    partition_name TEXT;
BEGIN
    FOR i IN 0..11 LOOP
        end_date := start_date + INTERVAL '1 month';
        partition_name := 'orders_' || to_char(start_date, 'YYYY_MM');

        EXECUTE format(
            'CREATE TABLE %I PARTITION OF orders FOR VALUES FROM (%L) TO (%L)',
            partition_name,
            start_date,
            end_date
        );

        start_date := end_date;
    END LOOP;
END $$;

-- 默认分区（兜底）
CREATE TABLE orders_default PARTITION OF orders DEFAULT;

-- 索引（会自动继承到所有分区）
CREATE INDEX idx_orders_user ON orders (user_id, created_at DESC);
CREATE INDEX idx_orders_no ON orders (order_no);
CREATE INDEX idx_orders_status ON orders (status, created_at DESC);

-- ============================================================
-- 订单项表
-- ============================================================
CREATE TABLE order_items (
    id          BIGSERIAL PRIMARY KEY,
    order_id    BIGINT NOT NULL,
    product_id  BIGINT NOT NULL,
    sku_id      BIGINT NOT NULL,
    sku_snapshot JSONB NOT NULL,                -- 下单时 SKU 快照
    quantity    INTEGER NOT NULL,
    unit_price  NUMERIC(12, 2) NOT NULL,
    subtotal    NUMERIC(12, 2) NOT NULL,

    CONSTRAINT chk_order_items_qty CHECK (quantity > 0),
    CONSTRAINT chk_order_items_price CHECK (unit_price >= 0)
);

CREATE INDEX idx_order_items_order ON order_items (order_id);
CREATE INDEX idx_order_items_product ON order_items (product_id);

-- ============================================================
-- 订单状态变更日志
-- ============================================================
CREATE TABLE order_status_logs (
    id          BIGSERIAL PRIMARY KEY,
    order_id    BIGINT NOT NULL,
    from_status SMALLINT,
    to_status   SMALLINT NOT NULL,
    operator    VARCHAR(50),                   -- 操作人
    remark      TEXT,
    created_at  TIMESTAMPTZ NOT NULL DEFAULT NOW()
);

CREATE INDEX idx_status_logs_order ON order_status_logs (order_id, created_at);
```

## 常见问题

### Q1: 主键用自增 ID 还是 UUID？

**A:** 各有优劣。自增 BIGSERIAL 的优点是紧凑（8字节）、有序（对 B-tree 索引友好）、可读性好。UUID 的优点是全局唯一（适合分布式）、不可猜测（更安全）。电商系统推荐：内部 ID 用 BIGSERIAL，对外暴露的编号用业务规则生成（如订单号 `ORD20250101000001`）。如果未来可能需要分库分表，也可以考虑 Snowflake ID。

### Q2: 为什么订单中要存地址快照和 SKU 快照？

**A:** 因为用户地址和商品信息是可变的。如果订单只存地址 ID 和 SKU ID 的引用，当用户修改地址或商品调价后，历史订单的信息也会跟着变，这是不正确的。存储快照（JSONB 格式）可以保证订单数据的历史一致性——下单时什么样就永远是什么样。

### Q3: 分区表的分区应该多久创建一次？

**A:** 推荐提前自动创建。可以在 pg_cron 中设置定时任务，每月1号自动创建下一个月的分区。也可以提前创建全年的分区。重要的是不要忘记创建，否则数据会写入默认分区（如果有的话），或者直接报错。生产环境务必有监控告警机制检测分区是否已创建。
