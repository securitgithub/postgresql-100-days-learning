# Day 062: EXPLAIN 与查询计划（二）

## 学习目标

- 理解三种 JOIN 节点：Nested Loop、Hash Join、Merge Join 的原理与适用场景
- 掌握 Sort 和 Aggregate 节点的执行方式
- 学会使用 EXPLAIN 的 BUFFERS、TIMING、FORMAT JSON 等高级选项
- 了解查询计划的可视化工具及使用方法
- 能够分析复杂多表 JOIN 查询的执行计划

## 理论讲解

### 一、JOIN 节点类型

当查询涉及多表关联时，规划器必须选择一种 JOIN 算法。PostgreSQL 提供了三种核心的 JOIN 执行策略。

#### 1.1 Nested Loop（嵌套循环连接）

原理类似于两层 for 循环：对外表的每一行，扫描内表中所有匹配的行。

```
for each row in outer_table:
    for each row in inner_table:
        if join_condition matches:
            output combined row
```

特点：
- 外表行数少时非常高效
- 内表有索引时，内层循环变成 Index Scan，性能极好
- 如果两个表都很大且无索引，性能会急剧下降（O(N×M)）
- 是唯一支持所有 JOIN 条件（包括不等连接）的方式

#### 1.2 Hash Join（哈希连接）

分为两个阶段：
1. **Build 阶段**：扫描内表（较小的表），按 JOIN 键建立哈希表
2. **Probe 阶段**：扫描外表（较大的表），用 JOIN 键探测哈希表

特点：
- 适合两个大表的等值连接
- 需要额外内存存放哈希表（受 `work_mem` 控制）
- 如果哈希表超出 `work_mem`，会分批写入磁盘（代价增大）
- 只支持等值连接（`=`）
- 启动成本较高（需先建完哈希表才能输出第一行）

#### 1.3 Merge Join（归并连接）

要求两个输入都按 JOIN 键排序，然后同步推进两个有序流进行匹配。

特点：
- 两个输入必须有序（可以利用索引顺序，也可以先 Sort）
- 对大数据集的等值连接非常高效
- 天然支持结果排序
- 也支持不等连接（如 `<`、`>`、`BETWEEN`）
- 如果数据已排序（如通过索引），无需额外排序开销

### 二、Sort 节点

Sort 节点出现在 ORDER BY、Merge Join 前的预排序、DISTINCT、GROUP BY 等场景。

PostgreSQL 的排序方式：
- **内存排序（quicksort）**：数据量小于 `work_mem` 时
- **外部排序（external merge sort）**：数据量超过 `work_mem` 时，使用磁盘临时文件
- **Top-N 排序**：当有 `LIMIT` 时，使用堆排序只保留前 N 行

```sql
EXPLAIN ANALYZE SELECT * FROM orders ORDER BY created_at LIMIT 10;
-- Sort Method: top-N heapsort  Memory: 25kB
```

### 三、Aggregate 节点

常见的聚合节点：

| 节点类型 | 说明 |
|----------|------|
| Aggregate | 简单聚合（无 GROUP BY） |
| HashAggregate | 使用哈希表进行 GROUP BY 聚合 |
| GroupAggregate | 对已排序数据进行 GROUP BY 聚合 |
| Partial/Finalize Aggregate | 并行聚合中的部分和最终聚合 |

HashAggregate 在内存中为每个 GROUP 键创建一个哈希桶，适合分组数较少的情况。GroupAggregate 要求输入已排序，适合数据量大但已有排序索引的场景。

### 四、EXPLAIN 的高级选项

#### 4.1 BUFFERS

显示每个节点的缓冲区使用情况：

```sql
EXPLAIN (ANALYZE, BUFFERS) SELECT * FROM orders WHERE id = 1;
```

输出中会看到：
- `shared hit=N`：从共享缓冲区（内存）读取的页面数
- `shared read=N`：从磁盘读取的页面数
- `shared dirtied=N`：本次查询弄脏的页面数
- `shared written=N`：本次查询写回磁盘的页面数
- `temp read/written`：临时文件读写（排序或哈希溢出时出现）

#### 4.2 TIMING

控制是否收集每个节点的时间信息（默认开启）：

```sql
-- 关闭时间收集（减少开销，某些系统 gettimeofday 很慢）
EXPLAIN (ANALYZE, TIMING OFF) SELECT * FROM orders;
```

#### 4.3 FORMAT

支持四种输出格式：

```sql
-- 默认文本格式
EXPLAIN (FORMAT TEXT) SELECT * FROM orders;

-- JSON 格式（适合程序解析）
EXPLAIN (FORMAT JSON) SELECT * FROM orders;

-- XML 格式
EXPLAIN (FORMAT XML) SELECT * FROM orders;

-- YAML 格式
EXPLAIN (FORMAT YAML) SELECT * FROM orders;
```

JSON 格式特别适合与可视化工具配合使用。

#### 4.4 SETTINGS（PostgreSQL 12+）

显示影响查询计划且不是默认值的配置参数：

```sql
EXPLAIN (SETTINGS) SELECT * FROM orders;
```

#### 4.5 WAL（PostgreSQL 13+）

显示 WAL 日志的生成情况：

```sql
EXPLAIN (ANALYZE, WAL) INSERT INTO orders (amount) VALUES (100);
```

### 五、可视化工具

#### 5.1 explain.dalibo.com

一个免费的在线工具，粘贴 EXPLAIN ANALYZE 的输出即可获得可视化图表，展示每个节点的耗时占比、行数偏差等。

#### 5.2 pgAdmin 内置可视化

pgAdmin 4 在查询工具中内置了 EXPLAIN 图形化展示，点击"Explain"或"Explain Analyze"按钮即可。

#### 5.3 depesz.com

另一个流行的在线工具，由 depesz 开发，可以用颜色高亮标注慢节点。

商业工具，提供更智能的优化建议。

## 代码示例

### 示例1：观察不同 JOIN 算法

```sql
-- 创建测试表
CREATE TABLE customers (
    id SERIAL PRIMARY KEY,
    name TEXT,
    city TEXT
);

CREATE TABLE orders (
    id SERIAL PRIMARY KEY,
    customer_id INT REFERENCES customers(id),
    amount NUMERIC,
    order_date DATE
);

-- 插入数据
INSERT INTO customers (name, city)
SELECT
    'Customer_' || i,
    (ARRAY['北京','上海','广州','深圳','杭州'])[1 + (i % 5)]
FROM generate_series(1, 10000) AS i;

INSERT INTO orders (customer_id, amount, order_date)
SELECT
    (random() * 9999 + 1)::int,
    (random() * 10000)::numeric(10,2),
    '2024-01-01'::date + (random() * 365)::int
FROM generate_series(1, 200000);

ANALYZE customers;
ANALYZE orders;

-- Nested Loop：小表驱动，内表有索引
EXPLAIN ANALYZE
SELECT c.name, o.amount
FROM customers c
JOIN orders o ON o.customer_id = c.id
WHERE c.id = 42;

-- Hash Join：两个较大结果集的等值连接
EXPLAIN ANALYZE
SELECT c.name, o.amount
FROM customers c
JOIN orders o ON o.customer_id = c.id
WHERE c.city = '北京';

-- Merge Join：观察排序后的归并
SET enable_hashjoin = off;
EXPLAIN ANALYZE
SELECT c.name, o.amount
FROM customers c
JOIN orders o ON o.customer_id = c.id;
SET enable_hashjoin = on;
```

### 示例2：Sort 节点分析

```sql
-- 内存排序
EXPLAIN (ANALYZE, BUFFERS)
SELECT * FROM orders ORDER BY amount LIMIT 20;

-- 外部排序（减小 work_mem 触发）
SET work_mem = '64kB';
EXPLAIN (ANALYZE, BUFFERS)
SELECT * FROM orders ORDER BY amount;
-- 观察 Sort Method: external merge  Disk: xxxkB
RESET work_mem;
```

### 示例3：Aggregate 节点对比

```sql
-- HashAggregate
EXPLAIN ANALYZE
SELECT city, COUNT(*) FROM customers GROUP BY city;

-- GroupAggregate（先排序再聚合）
CREATE INDEX idx_customers_city ON customers(city);
SET enable_hashagg = off;
EXPLAIN ANALYZE
SELECT city, COUNT(*) FROM customers GROUP BY city;
SET enable_hashagg = on;

-- 简单聚合（无 GROUP BY）
EXPLAIN ANALYZE
SELECT COUNT(*), AVG(amount) FROM orders;
```

### 示例4：BUFFERS 选项详解

```sql
-- 第一次执行（冷缓存可能有 shared read）
EXPLAIN (ANALYZE, BUFFERS)
SELECT * FROM orders WHERE customer_id = 100;

-- 第二次执行（数据已在缓存中，应该都是 shared hit）
EXPLAIN (ANALYZE, BUFFERS)
SELECT * FROM orders WHERE customer_id = 100;
```

### 示例5：JSON 格式输出

```sql
-- JSON 格式，适合程序解析或贴入可视化工具
EXPLAIN (ANALYZE, BUFFERS, FORMAT JSON)
SELECT c.name, SUM(o.amount) AS total
FROM customers c
JOIN orders o ON o.customer_id = c.id
WHERE c.city = '上海'
GROUP BY c.name
ORDER BY total DESC
LIMIT 10;
```

### 示例6：多表 JOIN 的完整计划分析

```sql
-- 创建产品表
CREATE TABLE products (
    id SERIAL PRIMARY KEY,
    name TEXT,
    price NUMERIC
);

CREATE TABLE order_items (
    id SERIAL PRIMARY KEY,
    order_id INT REFERENCES orders(id),
    product_id INT REFERENCES products(id),
    quantity INT
);

INSERT INTO products (name, price)
SELECT 'Product_' || i, (random() * 1000)::numeric(10,2)
FROM generate_series(1, 500) AS i;

INSERT INTO order_items (order_id, product_id, quantity)
SELECT
    (random() * 199999 + 1)::int,
    (random() * 499 + 1)::int,
    (random() * 10 + 1)::int
FROM generate_series(1, 500000);

ANALYZE products;
ANALYZE order_items;

-- 三表 JOIN 查询计划
EXPLAIN (ANALYZE, BUFFERS, TIMING)
SELECT c.name, p.name AS product, oi.quantity, o.order_date
FROM customers c
JOIN orders o ON o.customer_id = c.id
JOIN order_items oi ON oi.order_id = o.id
JOIN products p ON p.id = oi.product_id
WHERE c.city = '深圳'
  AND o.order_date >= '2024-06-01'
LIMIT 100;
```

## 动手练习

### 练习1：JOIN 算法识别

创建两个表（一大一小），执行 JOIN 查询，识别规划器选择了哪种 JOIN 算法。通过调整 `enable_hashjoin`、`enable_mergejoin`、`enable_nestloop` 参数强制使用不同算法，比较执行时间。

### 练习2：work_mem 对排序的影响

将 `work_mem` 设为不同值（64kB、1MB、64MB），对大表执行带 ORDER BY 的查询，观察 Sort 节点的 Sort Method 变化（quicksort vs external merge），记录执行时间差异。

### 练习3：BUFFERS 分析

对同一查询连续执行两次 `EXPLAIN (ANALYZE, BUFFERS)`，对比两次的 `shared hit` 和 `shared read` 数值，理解缓冲区缓存的作用。

### 练习4：可视化工具实践

执行一个复杂的多表 JOIN 查询，使用 `EXPLAIN (ANALYZE, BUFFERS, FORMAT JSON)` 获取 JSON 输出，粘贴到 explain.dalibo.com 进行可视化分析。

### 练习5：聚合节点优化

对一个包含 GROUP BY 的查询，分别在有索引和无索引的情况下执行，观察 HashAggregate 和 GroupAggregate 的切换以及性能差异。

## 常见问题

### Q1：Hash Join 和 Merge Join 哪个更快？

**A**：没有绝对的快慢，取决于具体场景：
- **Hash Join** 在两表大小差异大时通常更快，因为小表建哈希表，大表探测。但如果 `work_mem` 不够导致哈希表溢出到磁盘，性能会急剧下降。
- **Merge Join** 在数据已排序时很高效（如利用索引的自然顺序），不需要额外内存。但如果需要额外排序步骤，可能不如 Hash Join。
- 通常让规划器自己选择即可，它会根据统计信息做出合理判断。

### Q2：EXPLAIN ANALYZE 中 actual rows 和 rows 差异很大怎么办？

**A**：这说明规划器的估算不准确，可能导致选择了次优的执行计划。解决方法：
1. 执行 `ANALYZE` 更新表的统计信息
2. 增大 `default_statistics_target`（如从 100 改为 500），让统计采样更精确
3. 如果是多列相关性问题，考虑使用 `CREATE STATISTICS` 创建扩展统计
4. 检查是否有跨表的数据倾斜

### Q3：查询计划中出现了 temp read/written 意味着什么？

**A**：这表示查询执行过程中数据量超过了 `work_mem`，不得不使用磁盘临时文件。常见于：
- 大数据量的 Sort 操作
- Hash Join 的哈希表溢出
- HashAggregate 的分组过多

解决方法是适当增大 `work_mem`（注意是每个操作独立使用的，并发高时需谨慎）。
