## 理论讲解

### 1. 什么是索引？

索引是数据库中用于**加速数据检索**的数据结构，类似于书籍的目录。没有索引时，查询需要扫描整个表（全表扫描/Sequential Scan）；有索引时，可以快速定位到目标行。

**类比**：
- 全表扫描 = 逐页翻书找某个词
- 索引查找 = 先查目录，直接翻到对应页码

### 2. PostgreSQL 的索引类型

PostgreSQL 支持多种索引类型：

| 类型 | 适用场景 |
|------|---------|
| **B-tree**（默认） | 等值查询、范围查询、排序 |
| Hash | 等值查询 |
| GiST | 几何数据、全文搜索 |
| SP-GiST | 非平衡数据结构 |
| GIN | 数组、JSONB、全文搜索 |
| BRIN | 物理有序的大表 |

本节重点学习最常用的 **B-tree 索引**。

### 3. B-tree 索引原理

B-tree（平衡树）是 PostgreSQL 默认的索引类型。它将数据组织成一棵**平衡多叉树**：

- **根节点**：树的顶部，指向子节点
- **内部节点**：存储键值和子节点指针
- **叶子节点**：存储键值和对应行的物理位置（TID）

查找过程：从根节点开始，根据比较结果向下遍历，直到叶子节点。时间复杂度 O(log N)。

B-tree 支持的操作：`<`、`<=`、`=`、`>=`、`>`、`BETWEEN`、`IN`、`IS NULL`、`LIKE 'abc%'`（前缀匹配）。

### 4. CREATE INDEX 语法

```sql
-- 基本语法
CREATE INDEX 索引名 ON 表名 (列名);

-- 指定索引类型
CREATE INDEX 索引名 ON 表名 USING btree (列名);

-- 多列索引（复合索引）
CREATE INDEX 索引名 ON 表名 (列1, 列2);

-- 降序索引
CREATE INDEX 索引名 ON 表名 (列名 DESC);

-- 条件索引（部分索引）
CREATE INDEX 索引名 ON 表名 (列名) WHERE 条件;

-- 并发创建索引（不阻塞写操作，生产环境推荐）
CREATE INDEX CONCURRENTLY 索引名 ON 表名 (列名);
```

### 5. 唯一索引

唯一索引确保索引列的值不重复：

```sql
CREATE UNIQUE INDEX 索引名 ON 表名 (列名);
```

`PRIMARY KEY` 和 `UNIQUE` 约束会自动创建唯一索引。

### 6. 索引对性能的影响

**好处**：
- 大幅加速 `SELECT` 查询（尤其是 WHERE、JOIN、ORDER BY）
- 加速 `UPDATE`/`DELETE` 的行定位

**代价**：
- 占用磁盘空间
- 减慢 `INSERT`/`UPDATE`/`DELETE`（每次写入都需要维护索引）
- 增加 VACUUM 的工作量

### 7. 何时创建索引？

**应该创建索引的情况**：
- 经常出现在 `WHERE` 条件中的列
- 经常用于 `JOIN` 的列（外键列）
- 经常用于 `ORDER BY` 的列
- 高选择性的列（不同值多的列，如用户ID）

**不应该创建索引的情况**：
- 表很小（几百行），全表扫描更快
- 低选择性的列（如性别、布尔值，只有几个不同值）
- 很少查询、经常批量写入的表
- 频繁更新的列

## 代码示例

### 准备测试数据

```sql
-- 创建一个较大的测试表
CREATE TABLE test_users (
    id         SERIAL PRIMARY KEY,
    username   VARCHAR(50) NOT NULL,
    email      VARCHAR(100),
    age        INT,
    city       VARCHAR(50),
    status     VARCHAR(20) DEFAULT 'active',
    created_at TIMESTAMP DEFAULT now()
);

-- 插入 10 万条测试数据
INSERT INTO test_users (username, email, age, city, status)
SELECT
    'user_' || i,
    'user_' || i || '@example.com',
    (random() * 60 + 18)::INT,                           -- 18-78 岁
    (ARRAY['北京','上海','广州','深圳','杭州'])[floor(random()*5+1)::INT],
    (ARRAY['active','inactive','banned'])[floor(random()*3+1)::INT]
FROM generate_series(1, 100000) AS s(i);

-- 确认数据量
SELECT COUNT(*) FROM test_users;
```

### 示例 1：无索引查询（全表扫描）

```sql
-- 开启执行时间显示
\timing on

-- 查看查询计划——无索引时使用 Seq Scan
EXPLAIN ANALYZE
SELECT * FROM test_users WHERE email = 'user_50000@example.com';

-- 结果类似：
-- Seq Scan on test_users  (cost=... rows=1 width=...)
--   Filter: ((email)::text = 'user_50000@example.com'::text)
--   Rows Removed by Filter: 99999
```

### 示例 2：创建索引后查询

```sql
-- 在 email 列上创建索引
CREATE INDEX idx_users_email ON test_users (email);

-- 再次查看执行计划
EXPLAIN ANALYZE
SELECT * FROM test_users WHERE email = 'user_50000@example.com';

-- 结果类似：
-- Index Scan using idx_users_email on test_users  (cost=... rows=1 width=...)
--   Index Cond: ((email)::text = 'user_50000@example.com'::text)
-- 执行时间从几十毫秒降到 0.x 毫秒
```

### 示例 3：唯一索引

```sql
-- 创建唯一索引
CREATE UNIQUE INDEX idx_users_username ON test_users (username);

-- 尝试插入重复值会失败
-- INSERT INTO test_users (username, email, age, city)
-- VALUES ('user_1', 'dup@example.com', 25, '北京');
-- ERROR: duplicate key value violates unique constraint "idx_users_username"
```

### 示例 4：多列索引（复合索引）

```sql
-- 创建 city + status 的复合索引
CREATE INDEX idx_users_city_status ON test_users (city, status);

-- 以下查询可以使用该索引
EXPLAIN ANALYZE
SELECT * FROM test_users WHERE city = '北京' AND status = 'active';

-- 只用第一列条件也可以使用索引（最左前缀原则）
EXPLAIN ANALYZE
SELECT * FROM test_users WHERE city = '上海';

-- 只用第二列条件通常不能使用该索引
EXPLAIN ANALYZE
SELECT * FROM test_users WHERE status = 'active';
```

### 示例 5：部分索引（条件索引）

```sql
-- 只为活跃用户创建索引（节省空间）
CREATE INDEX idx_users_active_email ON test_users (email)
WHERE status = 'active';

-- 以下查询可以使用部分索引
EXPLAIN ANALYZE
SELECT * FROM test_users
WHERE status = 'active' AND email = 'user_100@example.com';
```

### 示例 6：索引与排序

```sql
-- 在 created_at 上创建索引
CREATE INDEX idx_users_created ON test_users (created_at DESC);

-- 查询计划中可以看到 Index Scan（避免了排序操作）
EXPLAIN ANALYZE
SELECT * FROM test_users ORDER BY created_at DESC LIMIT 10;
```

### 示例 7：查看表上的索引

```sql
-- 查看表上的所有索引
SELECT
    indexname,
    indexdef
FROM pg_indexes
WHERE tablename = 'test_users'
ORDER BY indexname;

-- 查看索引大小
SELECT
    pg_size_pretty(pg_relation_size('idx_users_email')) AS index_size;

-- 查看表和索引的总大小
SELECT
    pg_size_pretty(pg_total_relation_size('test_users')) AS total_size,
    pg_size_pretty(pg_relation_size('test_users')) AS table_size,
    pg_size_pretty(pg_indexes_size('test_users')) AS indexes_size;
```

### 示例 8：删除索引

```sql
-- 删除索引
DROP INDEX IF EXISTS idx_users_city_status;

-- 并发删除（不阻塞查询）
DROP INDEX CONCURRENTLY IF EXISTS idx_users_active_email;
```

## 常见问题

### Q1: 索引越多越好吗？

**A**：不是。每个索引都会增加写入开销（INSERT/UPDATE/DELETE 都要同步更新索引）、占用磁盘空间，并增加 VACUUM 的负担。应该根据实际查询模式，只在需要的列上创建索引。定期审查未使用的索引并删除它们。可以通过 `pg_stat_user_indexes` 查看索引的使用情况。

### Q2: 主键和 UNIQUE 约束会自动创建索引吗？

**A**：是的。`PRIMARY KEY` 会自动创建唯一的 B-tree 索引；`UNIQUE` 约束也会自动创建唯一索引。但 `FOREIGN KEY`（外键）**不会**自动创建索引。在外键列上手动创建索引是一个常见的最佳实践，因为 JOIN 和级联删除都需要在外键列上查找。

### Q3: CREATE INDEX 会锁表吗？

**A**：普通的 `CREATE INDEX` 会获取 `SHARE` 锁，允许读但阻塞写操作。在大表上创建索引可能需要较长时间，会影响在线业务。使用 `CREATE INDEX CONCURRENTLY` 可以在不阻塞写操作的情况下创建索引，但创建时间更长，且不能在事务中使用。**生产环境中始终使用 CONCURRENTLY。**
