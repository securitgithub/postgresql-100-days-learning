# Day 069: 表分区（二）分区管理

## 学习目标

- 掌握动态创建、附加、分离和删除分区的方法
- 理解 Partition Pruning（分区裁剪）的工作机制
- 学会为分区表设计合适的索引策略
- 掌握默认分区（DEFAULT）的使用
- 了解分区表的日常维护最佳实践

## 理论讲解

### 一、分区的生命周期管理

分区表的核心优势之一就是可以灵活管理分区。与 DELETE 大量数据相比，删除一个分区几乎是瞬间完成的。

#### 1.1 创建新分区

随着时间推移，需要定期创建新分区。建议**提前创建**，避免数据插入时找不到合适的分区。

```sql
-- 创建下月分区
CREATE TABLE orders_2025_01 PARTITION OF orders
    FOR VALUES FROM ('2025-01-01') TO ('2025-02-01');
```

#### 1.2 附加分区（ATTACH PARTITION）

将一个已存在的普通表作为分区挂载到分区表下。

```sql
ALTER TABLE parent_table
ATTACH PARTITION child_table
FOR VALUES FROM (lower) TO (upper);
```

PostgreSQL 会自动验证该表中的所有数据是否满足分区约束。如果表很大，这个验证过程可能很慢。可以先添加 CHECK 约束来加速：

```sql
-- 先在子表上添加约束
ALTER TABLE child_table ADD CONSTRAINT check_range
    CHECK (created_at >= '2025-01-01' AND created_at < '2025-02-01');

-- 附加时 PostgreSQL 会跳过全表验证
ALTER TABLE orders ATTACH PARTITION child_table
    FOR VALUES FROM ('2025-01-01') TO ('2025-02-01');
```

#### 1.3 分离分区（DETACH PARTITION）

将分区从分区表中移除，但保留为独立的普通表。

```sql
-- 普通分离（会短暂加锁）
ALTER TABLE orders DETACH PARTITION orders_2024_01;

-- PostgreSQL 14+ 支持 CONCURRENTLY（不阻塞查询）
ALTER TABLE orders DETACH PARTITION orders_2024_01 CONCURRENTLY;
```

分离后的表是一个独立的普通表，可以：
- 归档到冷存储
- 直接 DROP 删除
- 移动到其他表空间

#### 1.4 删除分区

```sql
-- 直接删除（数据也被删除，比 DELETE 快得多）
DROP TABLE orders_2024_01;
```

### 二、Partition Pruning（分区裁剪）详解

#### 2.1 静态裁剪（Plan-Time Pruning）

在查询规划阶段，根据 WHERE 条件中的常量值确定需要扫描的分区。

```sql
-- 常量条件，规划阶段就能确定
SELECT * FROM orders WHERE created_at >= '2024-06-01' AND created_at < '2024-07-01';
```

#### 2.2 动态裁剪（Execution-Time Pruning）

PostgreSQL 11+ 支持在执行阶段根据运行时参数值进行分区裁剪。

```sql
-- 参数化查询，执行时裁剪
PREPARE q(timestamp) AS SELECT * FROM orders WHERE created_at >= $1 AND created_at < $1 + interval '1 month';
EXECUTE q('2024-06-01');

-- 子查询结果作为条件
SELECT * FROM orders
WHERE created_at >= (SELECT MAX(created_at) - interval '7 days' FROM orders);
```

#### 2.3 裁剪失效的情况

以下情况分区裁剪**不会生效**：

```sql
-- ❌ 分区键上使用函数
SELECT * FROM orders WHERE DATE(created_at) = '2024-06-01';

-- ❌ 分区键参与表达式
SELECT * FROM orders WHERE created_at + interval '1 day' > '2024-06-01';

-- ❌ 没有分区键条件
SELECT * FROM orders WHERE user_id = 100;

-- ✅ 应该直接使用分区键
SELECT * FROM orders WHERE created_at >= '2024-06-01' AND created_at < '2024-06-02';
```

### 三、分区索引策略

#### 3.1 全局索引 vs 本地索引

PostgreSQL 的分区表只支持**本地索引**（每个分区独立维护自己的索引）。在父表上创建的索引会自动在所有分区上创建对应的本地索引。

```sql
-- 在父表上创建索引
CREATE INDEX idx_orders_user ON orders(user_id);
-- 每个分区自动获得一个 idx_orders_user 的本地索引
```

#### 3.2 分区特定索引

也可以为某些分区单独创建索引：

```sql
-- 只对最近的分区创建额外索引
CREATE INDEX idx_orders_2024_12_status ON orders_2024_12(status);
```

#### 3.3 索引策略建议

1. **分区键列通常不需要额外索引**：分区裁剪已经大幅减少了扫描范围
2. **非分区键的常用查询列需要索引**：如 user_id、status 等
3. **考虑历史分区**：冷数据分区可以删除不必要的索引以节省空间
4. **唯一索引必须包含分区键**

### 四、默认分区（DEFAULT）

默认分区用于存储不匹配任何已定义分区的数据。

```sql
CREATE TABLE orders_default PARTITION OF orders DEFAULT;
```

#### 4.1 默认分区的作用

- 防止因缺少目标分区而导致 INSERT 失败
- 作为"兜底"接收意外数据

#### 4.2 注意事项

- 创建新分区时，如果默认分区中有数据属于新分区的范围，PostgreSQL 会自动将这些行移动到新分区（可能很慢）
- 默认分区不应该积累大量数据
- 应定期检查默认分区是否有数据（说明缺少了对应分区）

### 五、分区表的日常维护

#### 5.1 自动化分区创建

可以通过以下方式自动化新分区的创建：
- 定时任务（cron + SQL 脚本）
- `pg_partman` 扩展（推荐）
- 应用层逻辑

#### 5.2 分区表的 VACUUM

```sql
-- 可以单独 VACUUM 某个分区
VACUUM ANALYZE orders_2024_12;

-- 也可以 VACUUM 整个分区表（所有分区都会被处理）
VACUUM ANALYZE orders;
```

#### 5.3 分区表的统计信息

```sql
-- ANALYZE 整个分区表
ANALYZE orders;

-- 或单独分析某个分区
ANALYZE orders_2024_12;
```

## 代码示例

### 示例1：完整的分区生命周期管理

```sql
-- 创建分区表
CREATE TABLE logs (
    id BIGSERIAL,
    level TEXT NOT NULL,
    message TEXT,
    created_at TIMESTAMP NOT NULL DEFAULT now(),
    PRIMARY KEY (id, created_at)
) PARTITION BY RANGE (created_at);

-- 创建初始分区
CREATE TABLE logs_2024_q1 PARTITION OF logs
    FOR VALUES FROM ('2024-01-01') TO ('2024-04-01');
CREATE TABLE logs_2024_q2 PARTITION OF logs
    FOR VALUES FROM ('2024-04-01') TO ('2024-07-01');
CREATE TABLE logs_2024_q3 PARTITION OF logs
    FOR VALUES FROM ('2024-07-01') TO ('2024-10-01');
CREATE TABLE logs_2024_q4 PARTITION OF logs
    FOR VALUES FROM ('2024-10-01') TO ('2025-01-01');

-- 创建默认分区
CREATE TABLE logs_default PARTITION OF logs DEFAULT;

-- 插入数据
INSERT INTO logs (level, message, created_at)
SELECT
    (ARRAY['INFO','WARN','ERROR','DEBUG'])[1+(random()*3)::int],
    'Log message #' || i,
    '2024-01-01'::timestamp + (random() * 365 || ' days')::interval
FROM generate_series(1, 500000) AS i;

-- 提前创建下一季度分区
CREATE TABLE logs_2025_q1 PARTITION OF logs
    FOR VALUES FROM ('2025-01-01') TO ('2025-04-01');

-- 查看各分区数据量
SELECT
    child.relname AS partition_name,
    pg_size_pretty(pg_relation_size(child.oid)) AS size
FROM pg_inherits
JOIN pg_class parent ON pg_inherits.inhparent = parent.oid
JOIN pg_class child ON pg_inherits.inhrelid = child.oid
WHERE parent.relname = 'logs'
ORDER BY child.relname;
```

### 示例2：ATTACH 和 DETACH 操作

```sql
-- 场景：将历史数据迁移到分区表

-- 1. 准备一个普通表
CREATE TABLE logs_2023_q4 (
    id BIGINT,
    level TEXT NOT NULL,
    message TEXT,
    created_at TIMESTAMP NOT NULL
);

INSERT INTO logs_2023_q4 (id, level, message, created_at)
SELECT
    i,
    'INFO',
    'Historical log #' || i,
    '2023-10-01'::timestamp + (random() * 91 || ' days')::interval
FROM generate_series(1, 100000) AS i;

-- 2. 添加约束加速 ATTACH
ALTER TABLE logs_2023_q4 ADD CONSTRAINT check_range
    CHECK (created_at >= '2023-10-01' AND created_at < '2024-01-01');

-- 3. 附加为分区
ALTER TABLE logs ATTACH PARTITION logs_2023_q4
    FOR VALUES FROM ('2023-10-01') TO ('2024-01-01');

-- 验证
EXPLAIN ANALYZE
SELECT * FROM logs WHERE created_at >= '2023-11-01' AND created_at < '2023-12-01';

-- 4. 分离旧分区（归档）
ALTER TABLE logs DETACH PARTITION logs_2023_q4;

-- 分离后的表是独立的，可以归档或删除
-- DROP TABLE logs_2023_q4;  -- 如果要删除
```

### 示例3：默认分区的使用

```sql
-- 尝试插入一条不在任何分区范围内的数据
-- 如果没有默认分区，这会失败
INSERT INTO logs (level, message, created_at)
VALUES ('ERROR', 'Future log', '2026-01-01');

-- 数据应该在默认分区中
SELECT * FROM logs_default WHERE created_at >= '2026-01-01';

-- 现在创建 2026 年分区
-- PostgreSQL 会自动将默认分区中属于该范围的行移过去
CREATE TABLE logs_2026_q1 PARTITION OF logs
    FOR VALUES FROM ('2026-01-01') TO ('2026-04-01');

-- 验证数据已移动
SELECT * FROM logs_default WHERE created_at >= '2026-01-01';
-- 应该为空

SELECT * FROM logs_2026_q1;
-- 应该能看到之前插入的数据
```

### 示例4：分区裁剪验证

```sql
-- 启用分区裁剪（默认就是启用的）
SET enable_partition_pruning = on;

-- 静态裁剪
EXPLAIN (ANALYZE, COSTS OFF)
SELECT COUNT(*) FROM logs
WHERE created_at >= '2024-07-01' AND created_at < '2024-10-01';
-- 只扫描 logs_2024_q3

-- 动态裁剪（使用参数）
PREPARE log_query(timestamp, timestamp) AS
SELECT * FROM logs WHERE created_at >= $1 AND created_at < $2;

EXPLAIN ANALYZE EXECUTE log_query('2024-04-01', '2024-07-01');

-- 关闭分区裁剪观察差异
SET enable_partition_pruning = off;
EXPLAIN (ANALYZE, COSTS OFF)
SELECT COUNT(*) FROM logs
WHERE created_at >= '2024-07-01' AND created_at < '2024-10-01';
-- 所有分区都会被扫描

SET enable_partition_pruning = on;  -- 恢复
```

### 示例5：分区索引策略

```sql
-- 在父表上创建索引（自动传播到所有分区）
CREATE INDEX idx_logs_level ON logs(level);

-- 验证每个分区都有了索引
SELECT tablename, indexname
FROM pg_indexes
WHERE tablename LIKE 'logs_2024%'
  AND indexname LIKE '%level%'
ORDER BY tablename;

-- 在分区表上创建唯一索引（必须包含分区键）
CREATE UNIQUE INDEX idx_logs_unique ON logs(id, created_at);

-- 为特定分区创建额外索引
CREATE INDEX idx_logs_q4_message ON logs_2024_q4(message)
WHERE level = 'ERROR';

-- 查看所有分区的索引情况
SELECT
    tablename,
    indexname,
    pg_size_pretty(pg_relation_size(indexname::regclass)) AS size
FROM pg_indexes
WHERE tablename LIKE 'logs_%'
ORDER BY tablename, indexname;
```

### 示例6：分区维护脚本

```sql
-- 自动创建下个月分区的函数
CREATE OR REPLACE FUNCTION create_monthly_partition(
    parent_table TEXT,
    year INT,
    month INT
) RETURNS VOID AS $$
DECLARE
    partition_name TEXT;
    start_date DATE;
    end_date DATE;
BEGIN
    partition_name := parent_table || '_' || to_char(year, 'FM0000') || '_' || to_char(month, 'FM00');
    start_date := make_date(year, month, 1);
    end_date := start_date + interval '1 month';

    EXECUTE format(
        'CREATE TABLE IF NOT EXISTS %I PARTITION OF %I FOR VALUES FROM (%L) TO (%L)',
        partition_name, parent_table, start_date, end_date
    );

    RAISE NOTICE 'Created partition: %', partition_name;
END;
$$ LANGUAGE plpgsql;

-- 使用示例：创建 2025 年所有月份的分区
DO $$
BEGIN
    FOR m IN 1..12 LOOP
        PERFORM create_monthly_partition('orders', 2025, m);
    END LOOP;
END $$;

-- 清理旧分区的函数
CREATE OR REPLACE FUNCTION drop_old_partitions(
    parent_table TEXT,
    retention_months INT
) RETURNS VOID AS $$
DECLARE
    partition_rec RECORD;
    cutoff_date DATE := CURRENT_DATE - (retention_months || ' months')::interval;
BEGIN
    FOR partition_rec IN
        SELECT child.relname AS partition_name
        FROM pg_inherits
        JOIN pg_class parent ON pg_inherits.inhparent = parent.oid
        JOIN pg_class child ON pg_inherits.inhrelid = child.oid
        WHERE parent.relname = parent_table
          AND child.relname != parent_table || '_default'
        ORDER BY child.relname
    LOOP
        -- 这里需要更复杂的逻辑来判断分区范围
        -- 简化示例：按名称中的日期判断
        RAISE NOTICE 'Checking partition: %', partition_rec.partition_name;
    END LOOP;
END;
$$ LANGUAGE plpgsql;
```

### 示例7：监控默认分区

```sql
-- 定期检查默认分区是否有数据（说明需要创建新分区）
CREATE OR REPLACE FUNCTION check_default_partition(parent_table TEXT)
RETURNS TABLE(default_partition TEXT, row_count BIGINT) AS $$
DECLARE
    default_part TEXT;
    cnt BIGINT;
BEGIN
    SELECT child.relname INTO default_part
    FROM pg_inherits
    JOIN pg_class parent ON pg_inherits.inhparent = parent.oid
    JOIN pg_class child ON pg_inherits.inhrelid = child.oid
    JOIN pg_partitioned_table pt ON pt.partrelid = parent.oid
    WHERE parent.relname = parent_table
      AND child.relname LIKE '%default%';

    IF default_part IS NOT NULL THEN
        EXECUTE format('SELECT COUNT(*) FROM %I', default_part) INTO cnt;
        RETURN QUERY SELECT default_part, cnt;
    END IF;
END;
$$ LANGUAGE plpgsql;

-- 检查
SELECT * FROM check_default_partition('logs');
```

## 常见问题

### Q1：分区数量有限制吗？太多分区会有什么影响？

**A**：PostgreSQL 没有硬性限制分区数量，但过多分区会有影响：
1. **规划时间增加**：规划器需要评估每个分区，分区过多时规划可能变慢
2. **内存消耗**：每个分区都有独立的文件描述符、缓存条目等
3. **DDL 操作变慢**：对父表的操作需要遍历所有分区

一般建议：
- 几百个分区通常没问题
- 超过 1000 个分区需要谨慎
- PostgreSQL 13+ 在分区裁剪方面有显著优化

### Q2：DETACH PARTITION CONCURRENTLY 和普通 DETACH 有什么区别？

**A**：
- **普通 DETACH**：需要对父表加 ACCESS EXCLUSIVE 锁，会阻塞所有对父表的查询
- **CONCURRENTLY（PG14+）**：使用两阶段操作，第一阶段加较弱的锁标记分区为"正在分离"，第二阶段等待所有使用该分区的事务结束后完成分离。不会阻塞 SELECT/INSERT 等操作

在生产环境中，推荐使用 CONCURRENTLY 方式。

### Q3：分区表支持外键吗？

**A**：PostgreSQL 对分区表的外键支持有限：
- **分区表作为外键的被引用表**：PostgreSQL 12+ 支持，其他表可以引用分区表的列
- **分区表作为外键的引用方**：支持，分区表可以有指向其他表的外键
- 外键约束会自动传播到所有分区

但在实际生产中，由于外键会增加写入开销，大表通常不使用外键约束，而是在应用层保证引用完整性。
