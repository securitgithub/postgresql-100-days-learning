# Day 079: 安全加固 — 全方位保护 PostgreSQL 数据库

## 学习目标

- 理解并实践最小权限原则（Principle of Least Privilege）
- 掌握 pgcrypto 扩展进行数据加密
- 学会配置 SSL/TLS 传输加密
- 了解 pgaudit 审计日志的配置和使用
- 掌握 SQL 注入的防护策略

## 理论讲解

### 1. 最小权限原则

最小权限原则要求每个用户和应用程序只拥有完成其任务所需的最少权限。在 PostgreSQL 中：

- 不要使用 superuser 运行应用
- 为不同应用创建不同的数据库角色
- 使用 GRANT/REVOKE 精确控制权限
- 使用 Schema 隔离不同的业务模块
- 使用 Row Level Security（RLS）实现行级权限

### 2. pgcrypto 数据加密

`pgcrypto` 是 PostgreSQL 自带的加密扩展，提供：
- 单向哈希：`digest()`、`crypt()`、`gen_salt()`
- 对称加密：`encrypt()`/`decrypt()`、`pgp_sym_encrypt()`/`pgp_sym_decrypt()`
- 非对称加密：`pgp_pub_encrypt()`/`pgp_pub_decrypt()`
- 随机数据生成：`gen_random_bytes()`、`gen_random_uuid()`

### 3. SSL 传输加密

PostgreSQL 支持 SSL/TLS 加密客户端与服务器之间的通信，防止数据在传输过程中被窃听或篡改。

配置要素：
- 服务器证书和私钥
- CA 证书（用于双向认证）
- pg_hba.conf 中的连接规则
- 客户端的 sslmode 参数

### 4. pgaudit 审计日志

`pgaudit` 扩展提供详细的会话和对象级别的审计日志。它记录：
- 哪个用户在什么时间执行了什么操作
- 对哪些对象进行了读取或修改
- DDL 操作（CREATE、ALTER、DROP）
- 权限变更（GRANT、REVOKE）

### 5. SQL 注入防护

SQL 注入是最常见的数据库攻击方式。PostgreSQL 层面的防护包括：
- 使用参数化查询（Prepared Statements）
- 使用 `quote_literal()`/`quote_ident()` 函数
- 限制用户权限减少攻击面
- 监控异常查询模式

## 代码示例

### 示例1：最小权限配置

```sql
-- 创建应用专用角色（不带登录权限的组角色）
CREATE ROLE app_readonly NOLOGIN;
CREATE ROLE app_readwrite NOLOGIN;
CREATE ROLE app_admin NOLOGIN;

-- 创建登录用户并分配到组
CREATE USER web_app WITH PASSWORD 'strong_password_here' LOGIN;
CREATE USER report_app WITH PASSWORD 'another_strong_pwd' LOGIN;
CREATE USER dba_user WITH PASSWORD 'dba_strong_pwd' LOGIN;

GRANT app_readwrite TO web_app;
GRANT app_readonly TO report_app;
GRANT app_admin TO dba_user;

-- 精确授权
-- 只读角色：只能 SELECT
GRANT USAGE ON SCHEMA public TO app_readonly;
GRANT SELECT ON ALL TABLES IN SCHEMA public TO app_readonly;
ALTER DEFAULT PRIVILEGES IN SCHEMA public
    GRANT SELECT ON TABLES TO app_readonly;

-- 读写角色：SELECT + INSERT + UPDATE + DELETE（无 DDL 权限）
GRANT USAGE ON SCHEMA public TO app_readwrite;
GRANT SELECT, INSERT, UPDATE, DELETE ON ALL TABLES IN SCHEMA public TO app_readwrite;
GRANT USAGE, SELECT ON ALL SEQUENCES IN SCHEMA public TO app_readwrite;
ALTER DEFAULT PRIVILEGES IN SCHEMA public
    GRANT SELECT, INSERT, UPDATE, DELETE ON TABLES TO app_readwrite;
ALTER DEFAULT PRIVILEGES IN SCHEMA public
    GRANT USAGE, SELECT ON SEQUENCES TO app_readwrite;

-- 管理角色：完整 DDL + DML 权限
GRANT ALL ON SCHEMA public TO app_admin;
GRANT ALL ON ALL TABLES IN SCHEMA public TO app_admin;
GRANT ALL ON ALL SEQUENCES IN SCHEMA public TO app_admin;

-- 撤销 public 模式的默认权限（重要！）
REVOKE ALL ON SCHEMA public FROM PUBLIC;
REVOKE CREATE ON SCHEMA public FROM PUBLIC;

-- 列级权限：只允许查看特定列
GRANT SELECT (id, name, email) ON users TO app_readonly;
-- 不授予对 password_hash、ssn 等敏感列的访问

-- 行级安全策略（RLS）
ALTER TABLE orders ENABLE ROW LEVEL SECURITY;

CREATE POLICY orders_isolation ON orders
    FOR ALL
    USING (tenant_id = current_setting('app.tenant_id')::integer);

-- 设置当前租户
SET app.tenant_id = '42';
SELECT * FROM orders;  -- 只能看到 tenant_id = 42 的数据
```

-- TYPE  DATABASE    USER            ADDRESS         METHOD

SELECT
    usename,
    ssl,
    client_addr,
    application_name

### 示例3：pgcrypto 数据加密

```sql
-- 安装 pgcrypto
CREATE EXTENSION pgcrypto;

-- === 密码哈希（单向）===
-- 使用 crypt + gen_salt（推荐 bcrypt）
CREATE TABLE secure_users (
    id SERIAL PRIMARY KEY,
    username TEXT UNIQUE NOT NULL,
    password_hash TEXT NOT NULL,
    created_at TIMESTAMP DEFAULT now()
);

-- 存储密码（使用 bcrypt，自动加盐）
INSERT INTO secure_users (username, password_hash)
VALUES ('alice', crypt('my_secret_password', gen_salt('bf', 10)));
-- bf = bcrypt, 10 = 工作因子（越大越慢越安全）

-- 验证密码
SELECT username FROM secure_users
WHERE username = 'alice'
  AND password_hash = crypt('my_secret_password', password_hash);
-- 密码正确返回行，错误返回空

-- 错误密码验证
SELECT username FROM secure_users
WHERE username = 'alice'
  AND password_hash = crypt('wrong_password', password_hash);
-- 返回空

-- === 数据哈希 ===
-- SHA-256 哈希
SELECT digest('hello world', 'sha256');
SELECT encode(digest('hello world', 'sha256'), 'hex');

-- HMAC（基于密钥的哈希）
SELECT encode(hmac('message', 'secret_key', 'sha256'), 'hex');

-- === 对称加密（PGP）===
-- 加密
INSERT INTO secure_data (encrypted_ssn)
VALUES (pgp_sym_encrypt('123-45-6789', 'encryption_key_here'));

-- 解密
SELECT pgp_sym_decrypt(encrypted_ssn, 'encryption_key_here')
FROM secure_data;

-- 带选项的加密
SELECT pgp_sym_encrypt(
    '敏感数据内容',
    'my_secret_key',
    'compress-algo=2, cipher-algo=aes256'
);

-- === 完整的敏感数据存储示例 ===
CREATE TABLE patient_records (
    id SERIAL PRIMARY KEY,
    patient_name TEXT NOT NULL,
    -- 身份证号加密存储
    id_number_encrypted BYTEA NOT NULL,
    -- 用于快速查找的哈希（不可逆）
    id_number_hash TEXT NOT NULL,
    diagnosis TEXT,
    created_at TIMESTAMP DEFAULT now()
);

-- 插入加密数据
INSERT INTO patient_records (patient_name, id_number_encrypted, id_number_hash, diagnosis)
VALUES (
    '张三',
    pgp_sym_encrypt('110101199001011234', current_setting('app.encryption_key')),
    encode(digest('110101199001011234', 'sha256'), 'hex'),
    '普通感冒'
);

-- 通过哈希快速查找（不需要解密）
SELECT patient_name, diagnosis
FROM patient_records
WHERE id_number_hash = encode(digest('110101199001011234', 'sha256'), 'hex');

-- 需要时解密身份证号
SELECT
    patient_name,
    pgp_sym_decrypt(id_number_encrypted, current_setting('app.encryption_key')) AS id_number
FROM patient_records
WHERE id = 1;

-- === 随机数据生成 ===
-- 生成 UUID
SELECT gen_random_uuid();

-- 生成随机字节
SELECT encode(gen_random_bytes(32), 'hex') AS random_token;

-- 生成随机密码
SELECT encode(gen_random_bytes(18), 'base64') AS random_password;
```

### 示例4：SSL 传输加密配置

```sql
-- === 服务器端配置 ===
-- postgresql.conf
-- ssl = on
-- ssl_cert_file = '/etc/postgresql/server.crt'
-- ssl_key_file = '/etc/postgresql/server.key'
-- ssl_ca_file = '/etc/postgresql/ca.crt'        -- 双向认证时需要
-- ssl_min_protocol_version = 'TLSv1.2'          -- 最低 TLS 版本
-- ssl_ciphers = 'HIGH:MEDIUM:+3DES:!aNULL'      -- 密码套件

-- 生成自签名证书（测试用）
-- openssl req -new -x509 -days 365 -nodes \
--   -out server.crt -keyout server.key \
--   -subj "/CN=pg-server"
-- chmod 600 server.key
-- chown postgres:postgres server.crt server.key

-- 查看当前 SSL 状态
SHOW ssl;

-- 查看 SSL 连接信息
SELECT
    pid,
    ssl,
    version AS ssl_version,
    cipher AS ssl_cipher,
    bits AS ssl_bits,
    client_dn AS client_certificate
FROM pg_stat_ssl
JOIN pg_stat_activity USING (pid)
WHERE datname = current_database();

-- === 客户端连接 ===
-- psql "host=db-server port=5432 dbname=mydb sslmode=require"

-- sslmode 选项：
-- disable:    不使用 SSL
-- allow:      先尝试非 SSL，失败则 SSL
-- prefer:     先尝试 SSL，失败则非 SSL（默认）
-- require:    必须 SSL，不验证证书
-- verify-ca:  必须 SSL，验证 CA
-- verify-full: 必须 SSL，验证 CA + 主机名（最安全）

-- 强制所有连接使用 SSL（在 pg_hba.conf 中）
-- hostssl  all  all  0.0.0.0/0  scram-sha-256
-- 使用 hostssl 而非 host

-- 验证客户端是否使用 SSL
SELECT ssl, client_addr FROM pg_stat_ssl
JOIN pg_stat_activity USING (pid)
WHERE pid = pg_backend_pid();
```

### 示例5：pgaudit 审计日志

```sql
-- 安装 pgaudit（需要先编译安装扩展）
-- shared_preload_libraries = 'pgaudit'  -- 在 postgresql.conf 中
-- 重启 PostgreSQL

CREATE EXTENSION pgaudit;

-- === 会话级审计 ===
-- 记录所有 DDL 和 DML 操作
SET pgaudit.log = 'ddl, write, role';

-- pgaudit.log 可选值：
-- read:     SELECT, COPY TO
-- write:    INSERT, UPDATE, DELETE, TRUNCATE, COPY FROM
-- function: 函数调用
-- role:     GRANT, REVOKE, CREATE/ALTER/DROP ROLE
-- ddl:      所有 DDL
-- misc:     DISCARD, FETCH, CHECKPOINT, VACUUM 等
-- all:      全部

-- 全局设置（postgresql.conf）
-- pgaudit.log = 'ddl, role'
-- pgaudit.log_catalog = off        -- 不记录系统表查询
-- pgaudit.log_client = off         -- 不在客户端显示审计日志
-- pgaudit.log_level = 'log'        -- 日志级别
-- pgaudit.log_parameter = on       -- 记录参数值
-- pgaudit.log_statement_once = off -- 每个子语句都记录

-- === 对象级审计 ===
-- 创建审计角色
CREATE ROLE auditor NOLOGIN;

-- 设置对象审计角色
SET pgaudit.role = 'auditor';
-- 或在 postgresql.conf: pgaudit.role = 'auditor'

-- 对特定表设置审计
GRANT SELECT ON accounts TO auditor;
-- 现在所有对 accounts 的 SELECT 都会被记录

GRANT INSERT, UPDATE, DELETE ON accounts TO auditor;
-- 对 accounts 的所有 DML 都会被记录

-- 查看审计日志（在 PostgreSQL 日志中）
-- 格式：AUDIT: SESSION,1,1,DDL,CREATE TABLE,,,CREATE TABLE test (id int);,<not logged>
-- 格式：AUDIT: SESSION,2,1,WRITE,INSERT,TABLE,public.accounts,INSERT INTO accounts...,<not logged>

-- === 审计日志分析 ===
-- 如果使用 csvlog 格式，可以通过 file_fdw 查询
-- 或使用 pgaudit 的 JSON 格式输出配合 ELK 分析

-- 查看当前审计配置
SHOW pgaudit.log;
SHOW pgaudit.role;
```

### 示例6：SQL 注入防护

```sql
-- === 错误示范（易受 SQL 注入攻击）===
-- 在 PL/pgSQL 中使用字符串拼接（危险！）
CREATE OR REPLACE FUNCTION unsafe_search(p_name TEXT)
RETURNS SETOF users AS $$
BEGIN
    -- 危险！直接拼接用户输入
    RETURN QUERY EXECUTE
        'SELECT * FROM users WHERE name = ''' || p_name || '''';
END;
$$ LANGUAGE plpgsql;

-- 攻击者输入: ' OR '1'='1
-- 生成的 SQL: SELECT * FROM users WHERE name = '' OR '1'='1'
-- 返回所有用户！

-- === 正确做法 1：使用参数化查询 ===
CREATE OR REPLACE FUNCTION safe_search(p_name TEXT)
RETURNS SETOF users AS $$
BEGIN
    -- 使用 $1 参数占位符
    RETURN QUERY EXECUTE
        'SELECT * FROM users WHERE name = $1'
        USING p_name;
END;
$$ LANGUAGE plpgsql;

-- === 正确做法 2：使用 quote_literal / quote_ident ===
CREATE OR REPLACE FUNCTION safe_dynamic_query(
    p_table TEXT,
    p_column TEXT,
    p_value TEXT
) RETURNS SETOF RECORD AS $$
BEGIN
    RETURN QUERY EXECUTE format(
        'SELECT * FROM %I WHERE %I = %L',
        p_table,     -- %I: quote_ident（标识符引用）
        p_column,    -- %I: quote_ident
        p_value      -- %L: quote_literal（值引用）
    );
END;
$$ LANGUAGE plpgsql;

-- === 正确做法 3：使用 format 函数 ===
-- %I = 标识符（表名、列名），自动加双引号
-- %L = 字面量（值），自动加单引号并转义
-- %s = 简单字符串（不推荐用于用户输入）

SELECT format('SELECT * FROM %I WHERE %I = %L', 'users', 'name', 'test''inject');
-- 结果：SELECT * FROM users WHERE name = 'test''inject'
-- 单引号被正确转义

-- === 数据库层面的额外防护 ===
-- 限制查询权限
REVOKE ALL ON SCHEMA pg_catalog FROM app_readwrite;

-- 设置语句超时（防止恶意慢查询）
ALTER ROLE web_app SET statement_timeout = '30s';

-- 限制连接数（防止连接耗尽攻击）
ALTER ROLE web_app CONNECTION LIMIT 50;

-- 监控可疑查询
SELECT
    usename,
    query,
    calls,
    mean_exec_time
FROM pg_stat_statements
WHERE query ILIKE '%union%select%'
   OR query ILIKE '%information_schema%'
   OR query ILIKE '%pg_shadow%'
ORDER BY calls DESC;
```

### 示例7：综合安全检查清单

```sql
-- === 安全审计查询 ===

-- 1. 检查 superuser 账号
SELECT usename, usesuper FROM pg_user WHERE usesuper = true;

-- 2. 检查没有密码的账号
SELECT rolname FROM pg_roles
WHERE rolcanlogin = true AND rolpassword IS NULL;

-- 3. 检查密码加密方式
SHOW password_encryption;
-- 应该是 scram-sha-256

-- 4. 检查 SSL 配置
SHOW ssl;
SELECT * FROM pg_stat_ssl;

-- 5. 检查连接限制
SELECT rolname, rolconnlimit FROM pg_roles
WHERE rolcanlogin = true AND rolconnlimit = -1;

-- 6. 检查 public schema 权限
SELECT grantee, privilege_type
FROM information_schema.schema_privileges
WHERE schema_name = 'public';

-- 7. 检查是否有表对 PUBLIC 开放
SELECT
    schemaname, tablename, tableowner,
    has_table_privilege('PUBLIC', schemaname || '.' || tablename, 'SELECT') AS public_select
FROM pg_tables
WHERE schemaname = 'public';

-- 8. 检查 pg_hba.conf 中是否有 trust 认证
SELECT * FROM pg_hba_file_rules WHERE auth_method = 'trust';

-- 9. 检查监听地址
SHOW listen_addresses;
-- 不应该是 '*' 除非有防火墙保护

-- 10. 检查日志配置
SHOW log_connections;
SHOW log_disconnections;
SHOW log_statement;
SHOW log_min_duration_statement;
```

## 常见问题

### Q1: 数据加密应该在数据库层还是应用层做？

**A:** 两者各有优劣，最佳实践是结合使用：
- **数据库层加密（pgcrypto）**：适合存储加密，如密码哈希。优点是实现简单，所有应用都能使用。缺点是密钥管理困难（密钥在 SQL 中传输），DBA 可以看到解密数据。
- **应用层加密**：适合端到端加密需求。优点是密钥不经过数据库，更安全。缺点是无法在数据库层查询加密字段。
- **推荐策略**：密码哈希用 pgcrypto 的 crypt()，高度敏感的数据（如信用卡号）在应用层加密，传输用 SSL 保护。

### Q2: Row Level Security（RLS）对性能影响大吗？

**A:** RLS 会为每条查询添加额外的过滤条件，性能影响取决于：
1. 策略表达式的复杂度：简单的等值比较（如 `tenant_id = $1`）影响很小。
2. 是否有适当的索引：确保 RLS 条件列有索引。
3. 查询优化器能否有效整合 RLS 条件和原始查询条件。
在正确使用的情况下（有索引、简单策略），性能开销通常不超过 5%。

### Q3: 如何安全地管理数据库密钥和密码？

**A:** 不要在代码或 SQL 中硬编码密钥。推荐做法：
1. 使用环境变量或配置管理工具（如 Vault、AWS Secrets Manager）存储密钥。
2. 通过 `current_setting('app.encryption_key')` 使用会话变量传递密钥。
3. 应用连接时设置密钥：`SET app.encryption_key = '...'`。
4. 使用 `.pgpass` 文件存储数据库密码，权限设为 600。
5. 定期轮换密钥和密码。
