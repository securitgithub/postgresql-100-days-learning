# Day 068: 表分区（一）声明式分区

## 学习目标

- 理解表分区的概念、目的和适用场景
- 掌握 Range 分区的创建方法和使用场景
- 掌握 List 分区的创建方法和使用场景
- 掌握 Hash 分区的创建方法和使用场景
- 了解分区表的查询行为和分区裁剪原理

## 理论讲解

### 一、什么是表分区？

表分区是将一个逻辑上的大表按照一定规则拆分为多个物理上的小表（子表/分区）。对用户来说，操作的仍然是一个表，但在存储层面数据被分散存储。

#### 1.1 为什么需要分区？

当表的数据量增长到数亿甚至数十亿行时，会出现以下问题：
- **查询性能下降**：即使有索引，B-tree 的深度增加，查询变慢
- **维护困难**：VACUUM、REINDEX 等操作耗时极长
- **数据管理不便**：删除旧数据（如保留近一年数据）需要大量 DELETE，非常慢

分区的好处：
- **查询加速**：分区裁剪（Partition Pruning）只扫描相关分区
- **高效维护**：可以单独对某个分区进行 VACUUM、REINDEX
- **快速数据清理**：`DROP` 或 `DETACH` 整个分区比 DELETE 快得多
- **并行处理**：分区可以分布在不同表空间

#### 1.2 分区的适用场景

- 时间序列数据（日志、监控、订单按月/年分区）
- 多租户数据（按租户 ID 分区）
- 地理数据（按地区分区）
- 表超过几千万行且有明确的分区键

#### 1.3 声明式分区 vs 继承分区

PostgreSQL 10 引入了**声明式分区**（Declarative Partitioning），语法更简洁、性能更好。之前版本使用的是基于继承的手动分区方案。本课只讨论声明式分区。

### 二、Range 分区（范围分区）

最常用的分区方式，按键值的连续范围划分数据。

#### 2.1 典型场景

- 按日期/时间范围分区（最常见）
- 按 ID 范围分区
- 按数值范围分区

#### 2.2 语法

```sql
-- 创建分区表（父表）
CREATE TABLE table_name (
    columns...
) PARTITION BY RANGE (partition_key);

-- 创建分区（子表）
CREATE TABLE partition_name PARTITION OF table_name
    FOR VALUES FROM (lower_bound) TO (upper_bound);
```

注意：范围是**左闭右开**的 `[lower, upper)`。

### 三、List 分区（列表分区）

按键值的离散值列表划分数据。

#### 3.1 典型场景

- 按地区/国家分区
- 按状态/类型分区
- 按租户 ID 分区

#### 3.2 语法

```sql
CREATE TABLE table_name (
    columns...
) PARTITION BY LIST (partition_key);

CREATE TABLE partition_name PARTITION OF table_name
    FOR VALUES IN (value1, value2, ...);
```

### 四、Hash 分区（哈希分区）

PostgreSQL 11 引入，按键值的哈希值均匀分配数据。

#### 4.1 典型场景

- 无法按范围或离散值自然分区时
- 需要将数据均匀分散到多个分区
- 按用户 ID 分区但无法预知 ID 范围

#### 4.2 语法

```sql
CREATE TABLE table_name (
    columns...
) PARTITION BY HASH (partition_key);

-- modulus 是总分区数，remainder 是余数（从 0 开始）
CREATE TABLE partition_name PARTITION OF table_name
    FOR VALUES WITH (MODULUS m, REMAINDER r);
```

### 五、分区键的选择

分区键必须满足以下条件：
- 必须是主键或唯一约束的一部分（PostgreSQL 要求分区键包含在所有唯一约束中）
- 查询中应经常出现在 WHERE 条件中（否则无法进行分区裁剪）
- 不应频繁更新（更新分区键值可能导致行跨分区移动）

### 六、分区裁剪（Partition Pruning）

当查询的 WHERE 条件包含分区键时，规划器能够在计划阶段排除不相关的分区。从 PostgreSQL 11 开始还支持**执行时裁剪**（Runtime Pruning）。

```sql
-- 确认分区裁剪已启用（默认启用）
SHOW enable_partition_pruning;
```

## 代码示例

### 示例1：Range 分区 —— 按月分区的订单表

```sql
-- 创建分区父表
CREATE TABLE orders (
    id BIGSERIAL,
    user_id INT NOT NULL,
    amount NUMERIC(12,2) NOT NULL,
    status TEXT NOT NULL DEFAULT 'pending',
    created_at TIMESTAMP NOT NULL DEFAULT now(),
    -- 分区键必须包含在主键中
    PRIMARY KEY (id, created_at)
) PARTITION BY RANGE (created_at);

-- 创建 2024 年各月分区
CREATE TABLE orders_2024_01 PARTITION OF orders
    FOR VALUES FROM ('2024-01-01') TO ('2024-02-01');
CREATE TABLE orders_2024_02 PARTITION OF orders
    FOR VALUES FROM ('2024-02-01') TO ('2024-03-01');
CREATE TABLE orders_2024_03 PARTITION OF orders
    FOR VALUES FROM ('2024-03-01') TO ('2024-04-01');
CREATE TABLE orders_2024_04 PARTITION OF orders
    FOR VALUES FROM ('2024-04-01') TO ('2024-05-01');
CREATE TABLE orders_2024_05 PARTITION OF orders
    FOR VALUES FROM ('2024-05-01') TO ('2024-06-01');
CREATE TABLE orders_2024_06 PARTITION OF orders
    FOR VALUES FROM ('2024-06-01') TO ('2024-07-01');
CREATE TABLE orders_2024_07 PARTITION OF orders
    FOR VALUES FROM ('2024-07-01') TO ('2024-08-01');
CREATE TABLE orders_2024_08 PARTITION OF orders
    FOR VALUES FROM ('2024-08-01') TO ('2024-09-01');
CREATE TABLE orders_2024_09 PARTITION OF orders
    FOR VALUES FROM ('2024-09-01') TO ('2024-10-01');
CREATE TABLE orders_2024_10 PARTITION OF orders
    FOR VALUES FROM ('2024-10-01') TO ('2024-11-01');
CREATE TABLE orders_2024_11 PARTITION OF orders
    FOR VALUES FROM ('2024-11-01') TO ('2024-12-01');
CREATE TABLE orders_2024_12 PARTITION OF orders
    FOR VALUES FROM ('2024-12-01') TO ('2025-01-01');

-- 插入测试数据
INSERT INTO orders (user_id, amount, status, created_at)
SELECT
    (random() * 10000)::int + 1,
    (random() * 5000)::numeric(12,2),
    (ARRAY['pending','completed','cancelled'])[1+(random()*2)::int],
    '2024-01-01'::timestamp + (random() * 365 || ' days')::interval
FROM generate_series(1, 1000000);

ANALYZE orders;

-- 查看分区信息
SELECT
    parent.relname AS parent_table,
    child.relname AS partition,
    pg_size_pretty(pg_relation_size(child.oid)) AS size
FROM pg_inherits
JOIN pg_class parent ON pg_inherits.inhparent = parent.oid
JOIN pg_class child ON pg_inherits.inhrelid = child.oid
WHERE parent.relname = 'orders'
ORDER BY child.relname;

-- 分区裁剪：只查询3月数据，只扫描 orders_2024_03
EXPLAIN ANALYZE
SELECT * FROM orders
WHERE created_at >= '2024-03-01' AND created_at < '2024-04-01';

-- 跨月查询：扫描两个分区
EXPLAIN ANALYZE
SELECT * FROM orders
WHERE created_at >= '2024-03-15' AND created_at < '2024-04-15';

-- 不包含分区键的查询：扫描所有分区
EXPLAIN ANALYZE
SELECT * FROM orders WHERE user_id = 100;
```

### 示例2：List 分区 —— 按地区分区

```sql
-- 创建按地区分区的销售表
CREATE TABLE sales (
    id BIGSERIAL,
    region TEXT NOT NULL,
    product_id INT NOT NULL,
    amount NUMERIC(12,2) NOT NULL,
    sale_date DATE NOT NULL,
    PRIMARY KEY (id, region)
) PARTITION BY LIST (region);

-- 创建各地区分区
CREATE TABLE sales_east PARTITION OF sales
    FOR VALUES IN ('华东');
CREATE TABLE sales_north PARTITION OF sales
    FOR VALUES IN ('华北');
CREATE TABLE sales_south PARTITION OF sales
    FOR VALUES IN ('华南');
CREATE TABLE sales_west PARTITION OF sales
    FOR VALUES IN ('西部', '西北', '西南');

-- 插入测试数据
INSERT INTO sales (region, product_id, amount, sale_date)
SELECT
    (ARRAY['华东','华北','华南','西部','西北','西南'])[1+(random()*5)::int],
    (random() * 100)::int + 1,
    (random() * 10000)::numeric(12,2),
    '2024-01-01'::date + (random() * 365)::int
FROM generate_series(1, 500000);

ANALYZE sales;

-- List 分区裁剪
EXPLAIN ANALYZE
SELECT * FROM sales WHERE region = '华东';

-- 多值匹配
EXPLAIN ANALYZE
SELECT * FROM sales WHERE region IN ('西部', '西北');
-- 只扫描 sales_west 分区
```

### 示例3：Hash 分区 —— 按用户 ID 分区

```sql
-- 创建 Hash 分区表
CREATE TABLE user_events (
    id BIGSERIAL,
    user_id INT NOT NULL,
    event_type TEXT NOT NULL,
    payload JSONB,
    created_at TIMESTAMP DEFAULT now(),
    PRIMARY KEY (id, user_id)
) PARTITION BY HASH (user_id);

-- 创建 8 个哈希分区
CREATE TABLE user_events_p0 PARTITION OF user_events
    FOR VALUES WITH (MODULUS 8, REMAINDER 0);
CREATE TABLE user_events_p1 PARTITION OF user_events
    FOR VALUES WITH (MODULUS 8, REMAINDER 1);
CREATE TABLE user_events_p2 PARTITION OF user_events
    FOR VALUES WITH (MODULUS 8, REMAINDER 2);
CREATE TABLE user_events_p3 PARTITION OF user_events
    FOR VALUES WITH (MODULUS 8, REMAINDER 3);
CREATE TABLE user_events_p4 PARTITION OF user_events
    FOR VALUES WITH (MODULUS 8, REMAINDER 4);
CREATE TABLE user_events_p5 PARTITION OF user_events
    FOR VALUES WITH (MODULUS 8, REMAINDER 5);
CREATE TABLE user_events_p6 PARTITION OF user_events
    FOR VALUES WITH (MODULUS 8, REMAINDER 6);
CREATE TABLE user_events_p7 PARTITION OF user_events
    FOR VALUES WITH (MODULUS 8, REMAINDER 7);

-- 插入数据
INSERT INTO user_events (user_id, event_type, payload, created_at)
SELECT
    (random() * 100000)::int + 1,
    (ARRAY['login','logout','click','purchase'])[1+(random()*3)::int],
    jsonb_build_object('page', '/p/' || (random()*100)::int),
    now() - (random() * 90 || ' days')::interval
FROM generate_series(1, 800000);

ANALYZE user_events;

-- 查看各分区的数据分布（应接近均匀）
SELECT
    child.relname,
    pg_size_pretty(pg_relation_size(child.oid)) AS size,
    (SELECT COUNT(*) FROM ONLY child.relname::regclass) -- 不太准确，仅示意
FROM pg_inherits
JOIN pg_class parent ON pg_inherits.inhparent = parent.oid
JOIN pg_class child ON pg_inherits.inhrelid = child.oid
WHERE parent.relname = 'user_events'
ORDER BY child.relname;

-- Hash 分区裁剪：查询特定用户只扫描一个分区
EXPLAIN ANALYZE
SELECT * FROM user_events WHERE user_id = 42;
```

### 示例4：多级分区（子分区）

```sql
-- 先按地区 List 分区，再按时间 Range 子分区
CREATE TABLE global_sales (
    id BIGSERIAL,
    region TEXT NOT NULL,
    amount NUMERIC(12,2),
    sale_date DATE NOT NULL,
    PRIMARY KEY (id, region, sale_date)
) PARTITION BY LIST (region);

-- 地区级分区（本身也是分区表）
CREATE TABLE global_sales_asia PARTITION OF global_sales
    FOR VALUES IN ('CN', 'JP', 'KR')
    PARTITION BY RANGE (sale_date);

CREATE TABLE global_sales_europe PARTITION OF global_sales
    FOR VALUES IN ('UK', 'DE', 'FR')
    PARTITION BY RANGE (sale_date);

-- 亚洲地区的时间子分区
CREATE TABLE global_sales_asia_2024h1 PARTITION OF global_sales_asia
    FOR VALUES FROM ('2024-01-01') TO ('2024-07-01');
CREATE TABLE global_sales_asia_2024h2 PARTITION OF global_sales_asia
    FOR VALUES FROM ('2024-07-01') TO ('2025-01-01');

-- 欧洲地区的时间子分区
CREATE TABLE global_sales_europe_2024h1 PARTITION OF global_sales_europe
    FOR VALUES FROM ('2024-01-01') TO ('2024-07-01');
CREATE TABLE global_sales_europe_2024h2 PARTITION OF global_sales_europe
    FOR VALUES FROM ('2024-07-01') TO ('2025-01-01');

-- 插入数据
INSERT INTO global_sales (region, amount, sale_date)
SELECT
    (ARRAY['CN','JP','KR','UK','DE','FR'])[1+(random()*5)::int],
    (random() * 5000)::numeric(12,2),
    '2024-01-01'::date + (random() * 365)::int
FROM generate_series(1, 200000);

ANALYZE global_sales;

-- 多级分区裁剪
EXPLAIN ANALYZE
SELECT * FROM global_sales
WHERE region = 'CN' AND sale_date >= '2024-08-01';
-- 只扫描 global_sales_asia_2024h2
```

## 常见问题

### Q1：分区表的主键必须包含分区键，这是否有问题？

**A**：这是声明式分区的限制。因为 PostgreSQL 无法跨分区维护全局唯一约束（每个分区独立存储），所以唯一约束必须包含分区键，确保唯一性在单分区内就能判断。如果业务上需要全局唯一的 ID，可以使用：
1. UUID（天然全局唯一）
2. 序列 + 分区键组合作为业务唯一标识
3. 应用层保证唯一性

### Q2：分区表一定比普通表快吗？

**A**：不一定。分区的性能优势取决于：
1. 查询是否能进行分区裁剪（WHERE 条件包含分区键）
2. 表是否足够大（小表分区反而增加开销）
3. 分区数量是否合理（过多分区会增加规划器开销）

如果查询经常不包含分区键条件，需要扫描所有分区，可能比单表还慢。

### Q3：已有的大表能否转换为分区表？

**A**：PostgreSQL 没有直接的 `ALTER TABLE ... PARTITION BY` 命令。转换需要：
1. 创建新的分区表
2. 将数据从旧表迁移到新表（`INSERT INTO new_table SELECT * FROM old_table`）
3. 在事务中重命名表
4. 或者使用 `pg_partman` 扩展辅助迁移

PostgreSQL 14+ 支持 `ALTER TABLE ... ATTACH PARTITION`，可以将现有表作为分区挂载到分区表下。
